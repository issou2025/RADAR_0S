# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**Spec: migrate to a pure Dart FFI package (3.0.0) and fix macOS defects**

*Project description:*
## Problem Statement

`flutter_ime` ships the same seven-function API three times: once in Dart, once in C++ for Windows, once in Swift for macOS. The three are held together by a hand-synchronised platform-channel contract, and every change has to be applied in lockstep across all of them.

For people **using** the package, this costs them:

- A native toolchain is required to build. Windows consumers need Visual Studio and CMake; macOS consumers need Xcode and CocoaPods. Build failures in the plugin's native layer surface as failures in the consumer's app.
- The package cannot be used outside Flutter at all, even though everything it does is a plain system-library call.
- On macOS, anyone using a non-QWERTY English layout — Dvorak, Colemak, British, Australian, Canadian — is classified as *not* using an English keyboard. Combined with the "Force English" recipe in the README, the package actively fights them: it detects their layout as non-English and forcibly switches them to ABC.
- On macOS, Caps Lock change monitoring depends on a global event monitor for key-related events, which only delivers events when the process is trusted for Accessibility. The package neither requests that permission nor documents the requirement, so for most consumers that monitor silently delivers nothing.

For the **maintainer**, this costs:

- Three test frameworks to keep alive: `dart test`, GoogleTest, XCTest — plus a three-platform CI matrix to run them.
- A channel-name contract duplicated across a Dart constants file, a C++ header, and a Swift enum, where drift is silent and only shows up at runtime.
- Native code that is difficult to reason about and, as it turns out, partly unnecessary.

That last point is now established rather than suspected. A spike on `spike/wndproc-block-necessity` showed that `disableIME()` on Windows is two independent mechanisms, and only one of them does any work:

1. `ImmAssociateContextEx(hwnd, NULL, 0)` — detaches the IME context.
2. WndProc interception of `WM_IME_*` and Hangul-range `WM_CHAR` messages.

With mechanism (2) toggled off at runtime, Korean input was still impossible, while a control field in the same app that never calls `disableIME()` accepted Korean normally. Mechanism (2) was guarding messages that mechanism (1) already prevents from being generated. The entire window-subclassing apparatus — the one part of the plugin that genuinely cannot be expressed in Dart FFI — is dead weight.

## Solution

Rewrite `flutter_ime` as a **pure Dart package that calls system libraries directly through `dart:ffi`**, with no native sources, no CMake, no podspec, and no plugin registration. Ship it as `3.0.0`.

The public API does not change. All seven functions and both streams keep their current names, signatures, and return types, and the opaque input-source tokens keep their current formats, so tokens saved by 2.x can still be restored by 3.x. Consumers upgrade by changing a version constraint.

Alongside the migration, fix the macOS defects the rewrite touches anyway: English-layout classification, Caps Lock monitoring, and dead template code.

Consumers who require the strict WndProc-level message blocking that 2.x nominally offered can stay on 2.x, which is frozen but remains published. The spike indicates that blocking never contributed anything beyond what 3.x still does.

## Implementation Decisions

### Package shape

- The package stops being a Flutter plugin and becomes a plain Dart package. The `flutter.plugin.platforms` section is removed from the manifest so no plugin registration is generated in consumer apps.
- The `windows/` and `macos/` source trees are deleted in full, including the CMake project, the podspec, the privacy manifest, the GoogleTest harness, and the XCTest target. The privacy manifest was never actually bundled — the podspec line that would include it is commented out — so deleting it changes nothing that shipped.
- Supported platforms are declared explicitly in the manifest rather than inferred.
- The dependency on the Flutter SDK is removed if nothing in the runtime path still needs it, so the package is usable from plain Dart. `plugin_platform_interface` is retained because the platform-interface seam is retained.

### Public API

- All seven top-level functions and both streams keep their current names, parameter types, and return types. `Future`-returning signatures are kept even though the underlying FFI calls are synchronous, so consumer code is unaffected.
- Input-source token formats are unchanged on both platforms: the macOS input-source ID string, and the Windows `KLID:conversion:sentence` triple. Tokens remain opaque in the documented contract.
- The method-channel implementation class and the channel-name constants are removed. These were public within the library and their removal is part of what makes this a major version.
- The macOS-only `getPlatformVersion` handler is removed. It was plugin-template residue and was never part of the platform interface or the public API.

### Seam and layering

Confirmed with the maintainer: **one injectable seam**, the existing platform interface.

- The existing platform interface remains the single injectable seam. The top-level functions delegate to it exactly as they do now, and the existing mock and platform-gating tests continue to apply unchanged.
- The default instance becomes an FFI-backed implementation instead of the method-channel one.
- That implementation is a **thin adapter with no branching logic**. Everything with a decision in it is extracted into pure functions that take and return plain values, so they need no seam at all and are tested directly.
- The pure functions are: Windows input-source token parsing and formatting, macOS English-input-source classification, and Caps Lock change detection. These are direct ports of logic that already exists as testable units in the native code.
- Polling is encapsulated in a small poller type that takes a `bool Function()` reader callback and emits only on change. The callback is the injection point, so the poller is tested with a fake reader under `FakeAsync` and never touches an OS.
- The existing platform-support gating policy and its test override are retained as-is.

### Web and unsupported platforms

- `dart:ffi` is unavailable on web, so importing it unconditionally would break web compilation for consumers. The FFI implementation is reached through a conditional import with a stub that is selected on web. The stub throws only if actually invoked; the platform gating means it never is.
- On Linux, Android, and iOS the behaviour is unchanged: every function short-circuits in the gating layer and returns the documented default without touching FFI.

### Windows

- The system libraries are opened directly. The required calls are the IMM32 context, conversion-status, and context-association functions, plus the user32 keyboard-layout and key-state functions. No wrapper library is introduced.
- **Window handle resolution is the main risk in this migration.** The plugin currently gets the Flutter view handle from the plugin registrar, which does not exist in a pure Dart package. The replacement resolves the handle by locating a top-level window belonging to the current process whose class matches the Flutter runner window class, falling back to the foreground window. The existing implementation already falls back to the foreground window, so this is a narrowing of an existing weakness rather than a new one. The resolved handle is cached and re-resolved when it becomes invalid.
- `disableIME()` performs only the IME-context detachment. The WndProc subclassing and the IME message filter are deleted. This is supported by the spike result recorded below.
- `enableIME()` restores the default context association, which the spike also exercised via the control field.
- Both event streams are driven by polling on a fixed internal interval, since a synchronous WndProc callback is not expressible in Dart FFI. The interval is an internal constant and is deliberately not exposed, to keep the API surface stable.
- One behavioural nuance to preserve deliberately: the current implementation emits an input-source event on every layout-change message, which can emit the same value twice in a row. The polling implementation emits only on change. This matches the documented contract ("emits when the input source changes") more closely than the current behaviour does, and is treated as a fix rather than a regression.

### macOS

- Text Input Source Services is reached through the Carbon framework. These are C functions and are callable directly. CoreFoundation string and dictionary construction is needed to build the input-source filter and to read string properties back.
- The property-name constants used to build filters and read properties are exported data symbols, not functions, and must be looked up as symbols and dereferenced rather than called.
- **English classification is fixed.** Instead of substring-matching two hardcoded input-source IDs, the implementation asks the input source for its languages and treats a source whose primary language is English as an English keyboard. Confirmed with the maintainer as an intentional behaviour change for 3.0.0. This makes Dvorak, Colemak, British, Australian, and Canadian layouts classify correctly.
- Note for the implementer: this makes the two platforms' `isEnglishKeyboard` semantics explicitly different — on Windows it means "the IME is not in native-language conversion mode", on macOS it means "the selected layout is an English-language layout". That divergence already exists today; it should be documented rather than papered over.
- **Caps Lock reading moves to the CoreGraphics event-source flags state**, a plain C call, replacing the Objective-C modifier-flags read. Verify during implementation that this call requires no Accessibility permission; if it turns out to, fall back to polling a different source rather than reintroducing an event monitor.
- **The global event monitor is removed.** It required Accessibility permission that the package never requested, so it was largely inert; polling the flags state works regardless of focus and needs no permission, making this a net improvement rather than a downgrade.
- **Input-source change notifications stay push-based.** The distributed notification centre is reachable through its CoreFoundation C interface, and its observer callback returns no value, so it can be bridged with a listener-style native callable. This is the key asymmetry with Windows: the Windows blocker was the requirement to return a value synchronously from the window procedure, not the existence of a callback. macOS therefore keeps instant input-source change delivery and only Caps Lock is polled.

### Documentation and release

- Confirmed with the maintainer: the macOS fixes ship inside 3.0.0 rather than as an interim 2.1.5 release, because the rewrite replaces that code anyway and fixing it twice in two languages is wasted work.
- 2.x is frozen, not unpublished. The README gains a short note explaining that 3.x is pure Dart, that 2.x remains available, and what the difference in `disableIME()` guarantees is.
- The documented contract for `disableIME()` is rewritten to describe what it actually does — detaching the IME context to prevent composition — and to state plainly that it does not and never did prevent pasted text or programmatically injected input.
- The README's macOS platform-support table is updated for the corrected English-layout behaviour.
- CI collapses from a three-platform native matrix to analysis and tests, plus an example-app build smoke test per platform.
- The spike branch is not merged. Its native toggle and example page are scaffolding and are deleted with the rest of the native tree.

## Testing Decisions

A good test here asserts **externally observable behaviour** — what a caller passes in and what comes back — and never reaches into how the result was produced. Tests must not assert that a particular system function was called, nor depend on the internals of the FFI adapter. The migration is a good forcing function for this: the current method-channel tests assert that a specific channel method name was invoked, which is precisely the kind of implementation coupling that makes them worthless the moment the transport changes.

**Modules under test:**

- **Windows input-source token parsing and formatting.** Pure functions. Round-trip a token, reject malformed and empty input, handle a token carrying only a layout identifier with no conversion segments. Directly ports the existing native token tests, including the malformed-token regression that motivated the 2.1.4 fix.
- **macOS English input-source classification.** Pure function. Must cover the layouts the current implementation gets wrong — Dvorak, Colemak, British, Australian, Canadian must classify as English — as well as IMEs and non-English layouts classifying as non-English. These cases are the regression net for the behaviour change.
- **Caps Lock change detection.** Pure function. Emits only on an actual transition.
- **The poller.** Constructed with a fake reader callback and driven under `FakeAsync`. Asserts that it emits only on change, that it starts on first listen and stops on last cancel, and that it does no work with no listeners.
- **Platform gating.** The existing gating tests and mock platform carry over unchanged and remain the tests for "unsupported platforms return documented defaults and never delegate".
- **Web compilation.** A check that the package compiles for web via the conditional-import stub. This is a build-level check rather than a unit test.

**Tests that are deleted:**

- The method-channel tests, which assert channel method names that will no longer exist.
- The channel-name constant tests, which assert a contract that is being removed.
- The GoogleTest and XCTest suites, whose subject matter moves into the pure-function tests above.

**Prior art in this repo:** the existing gating tests and mock platform show the established pattern for testing the top-level functions against the seam. The existing native token and classification tests are the closest prior art for the pure-function tests, and their cases should be carried over rather than reinvented.

**Verification the unit tests deliberately do not cover:** the thin FFI adapter is not unit-tested, by design — that is the point of keeping it logic-free. It is covered by the example app, which must exercise every public function on real Windows and real macOS before release. The `disableIME` / `enableIME` round trip in particular needs manual confirmation on Windows with a real IME, since the spike's control field is the only thing that proves the disable actually did something.

## Out of Scope

- Linux support. The package has never supported it and this work does not add it.
- Restoring message-level input blocking on Windows in any form. The spike showed it contributed nothing, and it is not expressible in Dart FFI.
- Blocking pasted or programmatically injected text. Neither 2.x nor 3.x does this; it is called out here only because the question came up during the spike, and the documentation should say so.
- `disableIME()` / `enableIME()` on macOS. Still unsupported, unchanged.
- Making the polling interval configurable. It stays an internal constant so the public API does not grow.
- Federating the package across separate platform packages. The single-package structure is retained.
- Any change to the shape of the public API. Additions, removals, and signature changes are all outside this work; the only public-surface removals are the method-channel implementation class and the channel constants, which are transport details.
- Unpublishing or altering the 2.x release line beyond freezing it.

## Further Notes

**Spike evidence.** Run on branch `spike/wndproc-block-necessity`, which added a runtime toggle for the WndProc-blocking half of `disableIME()` plus an example page to drive it. Results:

- Mode A (context detachment + message blocking, i.e. current 2.x): Korean input blocked.
- Mode B (context detachment only, i.e. what FFI can still do): Korean input blocked.
- Control field in the same app that never calls `disableIME()`: Korean typed normally, confirming the IME was live and the experiment was not vacuous.
- Paste bypassed the block in both modes, as expected, confirming that the message filter was never a security boundary.

The runtime toggle mattered here: comparing two builds would have left open whether the difference came from the build rather than the mechanism. Both modes were exercised inside a single process against the same IME state.

**Why Windows loses push events but macOS does not.** Dart FFI offers two kinds of native callbacks. An isolate-local callable is bound to the thread that owns its isolate, and a listener callable can be invoked from any thread but is asynchronous and returns no value. A window procedure runs on the platform thread and must return a value synchronously to influence message handling, so neither kind fits. A distributed-notification observer returns nothing and is fire-and-forget, so the listener kind fits it exactly. The constraint is the synchronous return value, not the callback itself — which is why the two platforms end up with different event strategies.

**Risk register.**

- *Window handle resolution on Windows* is the highest-risk item. Multi-window apps and calls made while the app is backgrounded are the failure cases. Worth exercising in the example app with the window unfocused.
- *CoreFoundation bridging on macOS* is the highest-effort item: dictionary and string construction, symbol lookups for property-name constants, and correct release semantics for the copy-versus-get distinction in the Text Input Source API. Leaks here are silent.
- *The macOS English-classification change is user-visible.* It is a fix, but any consumer who built logic around the old ABC/US-only behaviour will see different results. It belongs in the release notes, not just the changelog.
- *Polling latency* on Windows event streams is a real if minor behavioural change. The example app is the place to confirm the chosen interval feels instant.
