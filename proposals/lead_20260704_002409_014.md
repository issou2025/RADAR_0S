## Summary

`deft triage:bulk-*` defaults to a Python subprocess bridge (`createPythonActionsModule`) that resolves and executes `scripts/triage_actions.py` from the project working tree (or from `DEFT_ROOT` when set). A malicious or compromised project checkout can plant both a valid-looking `.deft-cache/github-issue/<owner>/<repo>/<n>/` cache entry and a replacement `scripts/triage_actions.py`. When a developer or headless automation runs a bulk triage action against that repo, the attacker-controlled Python script is executed with the operator's user privileges.

Per-issue triage actions were ported to native TypeScript in #1725, but the bulk path still defaults to the legacy Python bridge — reintroducing an arbitrary-code-execution trust boundary that the TS port was meant to close.

## Threat model

| Dimension | Detail |
|-----------|--------|
| **Attacker** | Author or collaborator of an untrusted project repository |
| **Victim** | Developer or automation that runs `deft triage:bulk-*` from (or against) that checkout |
| **Controlled input** | Project files under `.deft-cache/github-issue/<owner>/<repo>/<n>/` plus `scripts/triage_actions.py` in the same checkout (or any tree reachable via `DEFT_ROOT`) |
| **Prerequisite** | Victim executes a bulk triage command while cwd (or `DEFT_ROOT`) points at the malicious tree, and at least one cached issue record matches the bulk filter |

## Attack path

1. Attacker contributes a project containing:
   - A crafted `.deft-cache/github-issue/<owner>/<repo>/<n>/` entry (with schema-valid `meta.json` / `raw.json`) that matches common bulk filters (e.g. a specific `--label` or `--author`).
   - A malicious `scripts/triage_actions.py` (or relies on `DEFT_ROOT` pointing at a tree that contains one).
2. Victim (or CI agent) runs e.g. `deft triage:bulk-accept --repo owner/repo --label triage-candidate` from the project root.
3. `packages/cli/src/triage-bulk.ts` resolves:
   - `projectRoot = process.cwd()`
   - `frameworkRoot = resolve(process.env.DEFT_ROOT ?? projectRoot)`
   - `scriptsDir = join(frameworkRoot, "scripts")`
4. `bulkActionWithDefaults()` reads attacker-provided cache records via `createFilesystemCacheModule()`, filters to matches, then calls `createPythonActionsModule(scriptsDir)` as the default `actionsModule`.
5. `createPythonActionsModule()` runs:

   ```
   uv run python <scriptsDir>/triage_actions.py <cmd> <issueNumber> --repo <repo> ...
   ```

   with `cwd: dirname(scriptsDir)` — no package-root confinement, no integrity check, no verification that the script belongs to the signed framework install.

6. Attacker Python executes at user privilege level in the victim's environment.

## Impact

- **User-level arbitrary code execution** in local and headless triage automation contexts.
- Attacker gains access to environment variables, filesystem, and any credentials available to the invoking user/process (API tokens, SSH keys, cloud-agent secrets, etc.).
- Does **not** require push access to `deftai/directive` — only write access to the victim's project checkout (or ability to influence `DEFT_ROOT`).

## Affected code

```typescript
// packages/cli/src/triage-bulk.ts — resolves scripts from project/DEFT_ROOT
const frameworkRoot = resolve(process.env.DEFT_ROOT ?? projectRoot);
bulkActionWithDefaults(args.action, args.repo, {
  deftRoot: projectRoot,
  scriptsDir: join(frameworkRoot, "scripts"),
});

// packages/core/src/triage/bulk/index.ts — default actions module is Python bridge
actionsModule: options.actionsModule ?? createPythonActionsModule(scriptsDir),
```

`createPythonActionsModule()` spawns `uv run python <scriptsDir>/triage_actions.py` with no confinement to the framework package root.

## Remediation

### Preferred (closes the trust boundary)

Remove the Python bridge from the bulk path entirely. Route bulk actions through the native TypeScript triage action module (`@deftai/core/triage/actions`), which #1725 already ships for per-issue verbs. `bulkActionWithDefaults()` should default to a TS `actionsModule`, not `createPythonActionsModule()`.

### If a subprocess fallback must remain

- Resolve the script **only** from the signed package / framework install root (e.g. the npm package's bundled `scripts/` or an install-manifest-verified path).
- Reject project-relative `scripts/` paths and any `DEFT_ROOT` override that does not resolve inside the verified framework root.
- Add an integrity check (install manifest hash or package signature) before subprocess execution.

### Defense-in-depth (optional)

- Document that `.deft-cache/` and `scripts/triage_actions.py` in consumer projects are **untrusted input** when running triage against third-party checkouts.
- Consider a `deft verify:triage-trust-boundary` gate that fails when bulk would invoke project-local Python.

## Acceptance criteria

- [ ] `deft triage:bulk-*` never executes `scripts/triage_actions.py` from the consumer project root by default.

- [ ] Regression test: malicious `scripts/triage_actions.py` in a temp project root is **not** invoked during bulk execution.
