## Part 2 — E2E runner architecture (simulators/emulators per target)
- **Android**: existing loop (emulator container, hardware GL) — the model to replicate.
- **iOS**: Simulator requires macOS+Xcode → runs on the macbook. Design the remote loop: macbook enrolled on agent-hub → a runner agent there executes `flutter test integration_test -d <simulator>` + screenshots, reports back via agent-mail; mirror emulator-ctl.sh conventions (sim-ctl: boot/reuse/screenshot/logs). Note: simulator is x86_64/arm64 Apple-Silicon native — FFI needs simulator-slice binaries.
- **macOS**: no emulator needed — the app runs natively on the runner Mac; `flutter test integration_test -d macos` + screenshot capture. Same runner agent.
- **Legality/practicality note**: macOS/iOS VMs on non-Apple Linux hosts are EULA-violating and fragile — do not design around them; the macbook IS the runner.
- **What stays on this Linux box**: headless flutter test (host VM, ffi tag), the Linux DESKTOP target as the cheap cross-desktop proxy (Phase 0, running separately), Android emulator.
- Deliverable: the runner design as bot-sized slices (enrollment prerequisites → sim-ctl scripts → hub protocol → first green cross-target run), with test-sshd reachability from the Mac (Tailscale — test-sshd is Docker-internal; needs a plan: expose via fd-dev's tailnet or run a target on the Mac).
