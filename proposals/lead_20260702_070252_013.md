## Project Understanding
You need assistance with: 
**[security] WebVNC and Code bridge tickets survive share revocation**

*Project description:*
## Summary

WebVNC and Code bridge-agent tickets can still be consumed after the manager who minted the ticket has been removed from the lease share. Crabbox documents lease sharing as a trusted-team model, but it also treats lease ownership, share roles, and bridge management as supported coordinator authorization boundaries. A `manage` share recipient can mint a `wvnc_...` or `code_...` ticket, lose manage access, and still attach the bridge agent within the ticket's 120 second lifetime.

Related: #604 covered already-open WebVNC/Code viewer sockets after revocation, and #730 covered active egress bridges. This report is the remaining unused WebVNC/Code bridge-agent ticket case: the ticket can be minted while authorized, then consumed after the manager share is removed.

The stale-ticket path is asymmetric with the egress bridge implementation in the same file. `consumeEgressTicket` reloads the current lease and rejects the ticket when `leaseManagerAuthorized(...)` no longer authorizes the ticket principal. `consumeWebVNCTicket` and `consumeCodeTicket` reload the lease and check ticket format, expiry, and lease identifier binding, but they do not re-check the ticket principal against the current lease share before accepting and deleting the ticket.

This is in scope under Crabbox's trusted-operator policy because it is not a claim that `use` recipients must be read-only, and it does not depend on hostile arbitrary tenants sharing a broker. The crossed boundary is narrower: after the owner or another manager records a revocation, a previously authorized manager can still perform a manage-only bridge-establishment action using an unused ticket.

## Attack Path

Attacker role: a previously trusted lease share recipient whose role was `manage` when they obtained the bridge-agent ticket, but whose share was later downgraded or removed.

Prerequisites:

- The lease is active and has WebVNC or Code capability enabled.
- The attacker had `manage` access long enough to mint a WebVNC or Code agent ticket.
- The attacker retained the one-shot ticket value and uses it before its 120 second expiry.
- The owner or another manager has already removed the attacker from the lease share, or downgraded them below `manage`.

Steps:

1. Alice owns lease `L` and shares it with Bob as `manage`.
2. Bob calls `POST /v1/leases/L/webvnc/ticket` or `POST /v1/leases/L/code/ticket`.
3. The coordinator authorizes Bob with `leaseManageableByRequest(...)` and stores a `wvnc_...` or `code_...` ticket with Bob's `owner` and `org`.
4. Alice removes Bob from the share, or changes his role so he no longer has manage access.
5. The share update calls `putLeaseShareAndRevokeBridges(...)`. Because access shrank, `revokeUnauthorizedLeaseBridges(...)` closes unauthorized live WebVNC viewers, Code viewers, and egress sessions.
6. Unused `webvnc-ticket:` and `code-ticket:` storage entries are not removed by that revocation path.
7. Bob opens the WebSocket agent route with `Authorization: Bearer wvnc_...` or `X-Crabbox-Bridge-Ticket: code_...`.
8. `consumeWebVNCTicket(...)` or `consumeCodeTicket(...)` accepts the ticket if it is well-formed, unexpired, present in storage, and bound to the requested lease identifier.
9. Unlike `consumeEgressTicket(...)`, the WebVNC and Code paths do not call `leaseManagerAuthorized(lease, ticketPrincipal)` after reloading the current lease.
10. The coordinator accepts the stale ticket and attaches the bridge agent even though Bob's current share no longer permits bridge establishment.

The direct effect is stale manager authority over bridge-agent establishment. Viewer access is still constrained by the separate viewer paths: source review found that portal and viewer connections continue to call the lease visibility/session checks, and live viewers are revoked when their share access is removed. The issue is therefore not framed as a full viewer-authorization bypass.

## Impact

A removed manager can keep a short-lived bridge establishment capability alive for up to 120 seconds after revocation. That weakens the authority of share revocation for WebVNC and Code bridges, and it is especially confusing because the neighboring egress bridge already treats current manager authorization as part of ticket consumption.

Security consequences are bounded but real:

- Revocation is not immediately authoritative for unused WebVNC and Code agent tickets.
- A stale ticket can attach or replace the lease-side bridge agent after the ticket principal no longer has current manage access.
- If an owner or authorized viewer connects while the stale agent is attached, bridge traffic is routed through the agent that was accepted from a revoked principal. The viewer path still performs its own authorization checks, so this should be treated as limited confidentiality and integrity impact, not broad cross-owner lease access.
- The issue is single-use per ticket and time-bounded by the 120 second TTL.

Crabbox's `SECURITY.md` says mutually adversarial users behind one coordinator are not a supported isolation claim. This report does not rely on that unsupported model; it relies on a supported coordinator control: the owner or manager changed the share state, and WebVNC/Code ticket consumption failed to honor that current state.

## Recommended Remediation

Mirror the egress authorization pattern in WebVNC and Code ticket consumption:

- After loading the lease in `consumeWebVNCTicket(...)`, check whether the ticket principal still has manager authorization on the current lease.
- Apply the same check in `consumeCodeTicket(...)`.
- Delete the stale ticket and return `invalid` when authorization has lapsed, matching `consumeEgressTicket(...)`.

The helper can be shared across the three bridge families to avoid future asymmetry. A minimal shape would be:

```ts
if (!this.leaseManagerAuthorized(lease, ticketPrincipal(ticket))) {
  await this.state.storage.delete(key);
  return { status: "invalid" };
}
```

Also consider deleting unused `webvnc-ticket:` and `code-ticket:` entries when `revokeUnauthorizedLeaseBridges(...)` observes that share access shrank. That cleanup would reduce stale capability lifetime even if a future consume path regresses.

## Validation

Validation method: static source review in the isolated verifier workspace against current `main` at `9c041ebb3eb580864c517a481ab135c836bdaa06`.

Evidence:

- WebVNC ticket creation requires manage access and stores the principal: `worker/src/fleet.ts:7033-7055`.
- Code ticket creation requires manage access and stores the principal: `worker/src/fleet.ts:7272-7294`.
- The share update path calls `putLeaseShareAndRevokeBridges(...)` when share state changes: `worker/src/fleet.ts:4800-4824`.
- The revocation function closes unauthorized live WebVNC viewers, Code viewers, and egress sessions, but it only walks in-memory bridge session maps and does not delete unused `webvnc-ticket:` or `code-ticket:` storage entries: `worker/src/fleet.ts:4830-4880`.
- `leaseManagerAuthorized(...)` is the current-share manager predicate used by the egress path: `worker/src/fleet.ts:4882-4890`.
- `consumeWebVNCTicket(...)` and `consumeCodeTicket(...)` accept a valid, unexpired, lease-matching ticket without using that predicate: `worker/src/fleet.ts:8343-8368` and `worker/src/fleet.ts:8382-8407`.
- `consumeEgressTicket(...)` performs the missing check and deletes a stale ticket when the ticket principal no longer has manage access: `worker/src/fleet.ts:8431-8461`.
- Existing tests cover wrong-lease ticket rejection, query-ticket handling, manage-vs-use ticket creation, and live viewer revocation, but not the stale unused-ticket case after a manager is revoked.

Counterevidence and limits:

- The issue does not grant anonymous access. The attacker needs a ticket minted while they had manage access.
- The ticket is single-use and time-bounded to 120 seconds (`worker/src/fleet.ts:153-157`).
- Viewer routes still perform separate authorization/session checks, and share revocation closes unauthorized live viewers. This limits the finding to stale bridge-agent establishment and bridge traffic routed through that agent, not general lease or viewer access.
- The verifier did not run Vitest because `crabbox/worker/node_modules` is absent in this isolated workspace and dependency installation is outside this child verifier's no-edit target scope.

Suggested regression tests:

- Add a `worker/test/fleet.test.ts` case that creates a lease shared to Bob as `manage`, has Bob mint a WebVNC ticket, removes Bob's share, then asserts `consumeWebVNCTicket(...)` returns `invalid` and deletes the stored ticket.
- Add the same case for Code tickets.
- Keep the existing positive case that an unrevoked manager can consume a freshly minted ticket.
