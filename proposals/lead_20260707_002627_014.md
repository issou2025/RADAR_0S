*Project description:*
## Summary

`scripts/dump_tool_descriptions.py` still reads the capability tool descriptions from `src/lingtai/i18n` inside this repo, but this repo has had no Python `src/` tree since all Python moved to the sibling `lingtai-kernel` repo (see "Repository Scope" in `CLAUDE.md`). Because `load_descriptions()` returns `{}` for missing files and `print_section()` silently skips empty sections, the script exits 0 and prints a "Tool Descriptions" document that is missing the entire **Capabilities** section (12 tools: `read`, `write`, `edit`, `glob`, `grep`, ...) with no error or warning.

## Narrative

The script exists to answer a simple, recurring question for prompt and spec work: "what exactly do the agents see as tool descriptions, in all three locales?" It merges the kernel-intrinsic descriptions (`soul`, `psyche`, `system_tool`, `notification_tool`, `email`) with the capability descriptions (file I/O, search, etc.) into one markdown document with English / 中文 / 文言 columns per tool. Anyone auditing prompt wording, checking translation parity, or reviewing tool-surface changes reaches for it.

The path constants tell the story of the repo split. At the top of `scripts/dump_tool_descriptions.py` (lines 18-20):

```python
BASE = Path(__file__).resolve().parent.parent
KERNEL_I18N = BASE.parent / "lingtai-kernel" / "src" / "lingtai_kernel" / "i18n"
LINGTAI_I18N = BASE / "src" / "lingtai" / "i18n"
```

When this was written, the `lingtai` repo still contained the batteries-included Python wrapper under `src/lingtai/`, and only the kernel had been split out — so "kernel i18n lives in the sibling checkout, capability i18n lives here" was correct. Since then, *all* Python moved to `lingtai-kernel` (published as the `lingtai` package on PyPI), and this repo now contains only the Go TUI and portal. The capability i18n now lives at `lingtai-kernel/src/lingtai/i18n/{en,zh,wen}.json` — same sibling checkout as the kernel i18n, different package directory. Nobody updated the script during the migration because nothing forced the issue: it kept running "successfully."

That silence is the real bug. Two design choices compound into it. First, `load_descriptions()` (lines 23-28) treats a missing file as an empty result:

```python
def load_descriptions(path: Path) -> dict[str, str]:
    if not path.is_file():
        return {}
```

Second, `print_section()` (lines 46-48) treats an empty section as "nothing to print":

```python
def print_section(title: str, tools: dict[str, dict[str, str]]) -> None:
    if not tools:
        return
```

Each choice is individually defensible — but together they convert "the source directory vanished in a repo reorganization" into a clean exit-0 run that quietly drops 12 of 17 tools. Someone dumping descriptions today gets a plausible-looking document headed "Kernel Intrinsics" and may not notice the Capabilities section is gone unless they already know it should exist. For a script whose whole purpose is completeness auditing, silently incomplete output is worse than a crash.

There is a second, latent instance of the same failure mode: `KERNEL_I18N` assumes a sibling checkout at `BASE.parent / "lingtai-kernel"`. If someone clones only this repo (it is the public-facing one), the script prints just `# Tool Descriptions` and nothing else — still exit 0, still no hint about what went wrong or where it looked.

Better looks like: the script points both sections at the kernel checkout's two i18n directories (`src/lingtai_kernel/i18n` and `src/lingtai/i18n`), accepts the kernel checkout location as a CLI argument or environment variable for non-sibling layouts, and fails loudly with the resolved paths when a source directory is missing rather than emitting a truncated document.

## Current behavior

- `scripts/dump_tool_descriptions.py:20` sets `LINGTAI_I18N = BASE / "src" / "lingtai" / "i18n"`, a directory that does not exist anywhere in this repo (there is no top-level `src/`).
- `collect(LINGTAI_I18N)` (called from `main()`, line 59) therefore returns `{}` for every locale, via the `return {}` branch of `load_descriptions()` (line 26).
- `print_section("Capabilities", lingtai)` (line 63) hits the `if not tools: return` guard (line 47) and prints nothing.
- Net effect, verified against a current checkout: the script exits 0 and prints only the `## Kernel Intrinsics` section with 5 tools; the 12 capability descriptions present in `lingtai-kernel/src/lingtai/i18n/en.json` (`read.description`, `write.description`, `edit.description`, `glob.description`, `grep.description`, ...) are absent from the output. No warning is emitted.
- Additionally, if `lingtai-kernel` is not checked out as a sibling directory, the kernel section vanishes the same way and the script prints an empty document, still exit 0.

## Detailed implementation plan

1. **Fix the capability path** in `scripts/dump_tool_descriptions.py`. Both i18n trees live in the kernel checkout now; derive both from one root:

   ```python
   DEFAULT_KERNEL_REPO = BASE.parent / "lingtai-kernel"

   def i18n_dirs(kernel_repo: Path) -> tuple[Path, Path]:
       return (
           kernel_repo / "src" / "lingtai_kernel" / "i18n",   # kernel intrinsics
           kernel_repo / "src" / "lingtai" / "i18n",          # capabilities
       )
   ```

   Delete the stale `LINGTAI_I18N = BASE / "src" / "lingtai" / "i18n"` constant.

2. **Accept the kernel checkout path as input.** Add a minimal `argparse` block in `main()`: positional-optional `kernel_repo` (or `--kernel-repo`), falling back to the `LINGTAI_KERNEL_REPO` environment variable, then to `DEFAULT_KERNEL_REPO`. This keeps `python scripts/dump_tool_descriptions.py` working unchanged for the documented sibling-checkout layout while unblocking anyone with a different layout.

3. **Fail loudly on missing sources.** Replace the silent-empty behavior with explicit validation before collecting:

   ```python
   def require_dir(path: Path, what: str) -> None:
       if not path.is_dir():
           sys.exit(f"error: {what} i18n directory not found: {path}\n"
                    f"hint: pass the lingtai-kernel checkout path "
                    f"(arg or LINGTAI_KERNEL_REPO)")
   ```

   Call it for both directories in `main()`. With the directory guaranteed to exist, change `load_descriptions()` to let a missing per-locale file raise (or `sys.exit` with the path) instead of `return {}` — a missing `wen.json` is a real defect the script should surface, not smooth over.

4. **Guard the section printer.** In `print_section()`, replace `if not tools: return` with a hard failure (`sys.exit(f"error: no tool descriptions found for section {title!r}")`). Once the paths are validated, an empty section can only mean the JSON files lost their `.description` keys — also worth failing on for a completeness-audit tool.

5. **Update the docstring** (lines 1-9) to document the argument/env var and state that both i18n trees are read from the `lingtai-kernel` checkout.

6. **Back-compat:** none needed beyond step 2's defaulting — the script has no importers (it is `__main__`-only) and no other file in the repo references it. Output format is unchanged, so any downstream diffing of the markdown only gains the previously missing Capabilities section.

## Risks and alternatives

- **Risk: stricter failure breaks an automation that tolerated the empty output.** A repo-wide search finds no CI job, script, or doc invoking `dump_tool_descriptions.py`, so the blast radius is interactive use — where loud failure is strictly better than a silently truncated report.
- **Risk: hardcoded sibling default still wrong for some layouts.** Mitigated by step 2; the error message in step 3 names the override mechanism, so the failure is self-explanatory.
- **Alternative: vendor/copy the i18n JSONs into this repo.** Rejected — it reintroduces a second source of truth for description text that the repo split deliberately eliminated; the files would drift.
- **Alternative: fetch the JSONs from PyPI/GitHub at runtime.** Rejected as overkill — the script is a maintainer tool that already assumes a dev environment with the kernel checkout, and network fetching adds failure modes and version-skew questions.
- **Alternative: keep `return {}` but print a warning to stderr.** Rejected — warnings on a script whose stdout is typically redirected to a file are easy to miss; the truncated document still gets produced and circulated. Hard failure wins because the script's only job is a complete report.

## Testing plan

The repo currently has no Python test harness (Go-only), so tests can be a small `scripts/test_dump_tool_descriptions.py` run ad hoc or wired into CI later; each test builds a temp directory tree and invokes the script via `subprocess`.

- `test_happy_path_includes_both_sections` — fake kernel checkout with both i18n dirs populated (`en/zh/wen.json` each containing one `*.description` key); assert exit 0 and that stdout contains both `## Kernel Intrinsics` and `## Capabilities`, plus each tool name and all three language labels.
- `test_missing_capability_dir_fails_loudly` — fake checkout with only `src/lingtai_kernel/i18n`; assert nonzero exit and that stderr names the missing `src/lingtai/i18n` path. This is the regression test for this issue.
- `test_missing_kernel_checkout_fails_loudly` — point `--kernel-repo` at a nonexistent directory; assert nonzero exit and an error message mentioning the override mechanism.
- `test_kernel_repo_arg_and_env_override` — place the fake checkout in a non-sibling location; assert the script succeeds when the path is supplied via the CLI argument, and again via `LINGTAI_KERNEL_REPO` with no argument.
- `test_missing_locale_file_fails` — populated dirs but delete `wen.json` from one; assert nonzero exit naming the missing file (locking in the removal of the silent `return {}`).
- `test_non_description_keys_ignored` — JSONs containing both `foo.description` and unrelated keys (`foo.hint`); assert only description keys appear in output (guards the existing filter in `load_descriptions()` through the refactor).
