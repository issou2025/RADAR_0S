# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**clarify_mode synced preference (1-1 / n-m) + Settings toggle**

*Project description:*
## User Story

As a user who sometimes needs one Capture to become several Outcomes (or several Captures to merge into one), I want to choose between the quick one-to-one clarify flow and the split/merge flow, so the simple case stays fast and the complex case is possible.

## Scope

The synced preference and its plumbing only — **not** the n-m UI itself (that's the sibling issue, which depends on this).

- `clarify_mode` key in `user_preferences`, values `oneToOne` (default) / `nToM`, replicated via the existing `SyncedPreferencesNotifier`.
- A Conflict Strategy registry entry (`services/user_preferences_conflict.dart`) — LWW is almost certainly right for a scalar, but it must be registered explicitly rather than falling through, per ADR-0011.
- A Settings toggle exposing it, using canonical vocabulary.
- `ClarifyCard` reads the mode and branches; in `oneToOne` the shipped behaviour is **byte-identical to today** (first Outcome link auto-stamps `clarified_at` and advances).
- Tests: default is `oneToOne`; toggle round-trips; the 1-1 path is unchanged.

## Acceptance Criteria

- [ ] `clarify_mode` persists and replicates across devices
- [ ] A registered Conflict Strategy exists for the key (not an implicit default)
- [ ] Default is `oneToOne`; existing 1-1 clarify behaviour is unchanged and test-pinned
- [ ] Settings exposes the toggle with canonical copy
- [ ] `docs/SYNC.md` ownership matrix updated for the new key

## Out of Scope

- The unified capture-to-Outcome field, swap-label footer, and unlink semantics — sibling issue.
