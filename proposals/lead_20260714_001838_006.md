## Executive Summary

This document defines a unified Continuous Integration (CI) strategy across three application repositories — **backend**, **frontend**, and **mobile** — with a shared workflow:

1. Developers open Pull Requests (PRs) targeting `staging`.
2. Automated test suites run on every PR.
3. Test results are summarized and posted to a dedicated **Slack channel** so product and engineering have immediate visibility.
4. Code merges only when **both** automated tests pass **and** a peer developer approves the PR.
5. **Staging auto-deploy runs only after merge** — not on every green PR — keeping the staging environment stable for QA.

This approach gives the team fast feedback, PM-friendly visibility, and a clean separation between *validation* (PR checks) and *release* (post-merge deploy).

---

## Goals and Non-Goals

### Goals

- Run automated tests on every PR to `staging` across all three repos.
- Standardize branch naming with Jira ticket IDs for traceability.
- Surface pass/fail summaries in Slack with author, ticket, and PR link.
- Block merges until CI is green **and** at least one engineer has approved.
- Deploy to staging only after approved code is merged into `staging`.
- Notify QA when staging is updated and ready for manual exploratory testing.

### Non-Goals (for this phase)

- Replacing human code review with automation.
- Auto-deploying to staging on every passing PR (intentionally deferred).
- Running full end-to-end tests across all three apps in a single monolithic pipeline (each repo owns its own CI; coordination happens at the process level).
- Blocking merges based on code quality metrics beyond the defined test and lint gates.

---

## Core Principle: Tests Are Necessary, Not Sufficient

Passing tests confirm that the code **works as written**. They do **not** guarantee that the code is:

- Well-structured or maintainable
- Secure
- Architecturally sound
- Aligned with product intent

Human peer review remains mandatory. The merge gate requires **both**:

| Gate | What it validates |
|------|-------------------|
| **Green CI** | Automated tests, linting, and configured checks pass |
| **Developer approval** | At least one engineer has reviewed and approved the change |

Merging only when both conditions are met keeps the `staging` branch — and therefore the staging environment — in a trustworthy state.

## Phase 1: Enforce the Git Workflow

Before automation can provide reliable traceability, branch naming and PR targets must be consistent.

### Branch naming rule

All feature branches created off `staging` must follow:

```text
[JIRA-ID]-[short-description]
```

**Examples:**

- `PROJ-123-login-fix`
- `PROJ-456-meal-recommendations`
- `PROJ-789-push-notification-badge`

The Jira ID must appear at the **start** of the branch name so it can be extracted automatically for Slack messages and PM tracking.

### Guardrail: branch name validation

Add a lightweight check that **fails the PR** if the branch name does not match the expected pattern.

**Options:**

1. **GitHub Action** (recommended) — runs on every `pull_request` and validates `github.head_ref`.
2. **Local pre-push hook** — catches issues before push (supplementary, not a substitute for CI).

**Example validation regex:**

```regex
^[A-Z]+-[0-9]+-[a-z0-9-]+$
```

**Example GitHub Action step:**

```yaml
- name: Validate branch name
  run: |
    BRANCH="${{ github.head_ref }}"
    if ! echo "$BRANCH" | grep -qE '^[A-Z]+-[0-9]+-[a-z0-9-]+$'; then
      echo "Branch name must follow: JIRA-ID-short-description (e.g. PROJ-123-login-fix)"
      exit 1
    fi
```

### PR target branch

- PRs that trigger this pipeline must target **`staging`**.
- Promotion from `staging` → `main` follows the existing release process.

## Phase 2: Build the GitHub Actions Triggers

Create or update `.github/workflows/ci.yml` in each of the three repositories.

### Trigger configuration

```yaml
name: CI Pipeline

on:
  pull_request:
    branches: [staging]
```

**Key decisions:**

| Setting | Value | Rationale |
|---------|-------|-----------|
| Event | `pull_request` only | Tests run on proposed changes, not on every push to long-lived branches |
| Target branch | `staging` | Aligns with QA hand-off and staging deploy |
| Deploy trigger | Separate workflow on `push` to `staging` | Keeps staging stable; deploy happens post-merge only |

### Workflow steps (common pattern)

1. Check out code
2. Validate branch name (Phase 1)
3. Install dependencies
4. Run linting (if applicable)
5. Run test suite with **JUnit XML output**
6. Parse results and publish to Slack (Phase 3)
7. Upload test artifacts (optional, for debugging failed runs)

### JUnit XML output

All test runners must emit JUnit XML so results can be parsed consistently:

| Platform | Command |
|----------|---------|
| Backend (Pytest) | `pytest --junitxml=report.xml` |
| Frontend (Jest) | `npm test -- --ci --reporters=default --reporters=jest-junit` |
| Frontend (Vitest) | `npx vitest run --reporter=junit --outputFile=report.xml` |
| Mobile (Flutter) | `flutter test --reporter json` (convert to JUnit) or use a JUnit reporter package |

### Separate deploy workflow

Keep deployment in a **distinct** workflow file (e.g. `.github/workflows/deploy-staging.yml`):

```yaml
on:
  push:
    branches: [staging]
```

This ensures:

- Opening or updating a PR runs tests but does **not** deploy.
- Merging an approved PR to `staging` triggers the staging deploy.

## Phase 3: Parse Results and Notify Slack

Raw JUnit XML is not readable for non-engineers. An intermediary step parses the report and sends a structured Slack message.

### Data extraction

Use one of:

- **[EnricoMi/publish-unit-test-result-action](https://github.com/EnricoMi/publish-unit-test-result-action)** — publishes results to the PR check summary and can feed downstream steps.
- **Custom script** — parse `report.xml` with Python or Node and compute totals.

**Metrics to extract:**

- Total tests run
- Passed
- Failed
- Skipped
- Duration (optional)
- Failed test names (truncated list for Slack)

### Context from GitHub Actions

Inject these into every Slack message:

| Variable | Source | Purpose |
|----------|--------|---------|
| Author | `${{ github.actor }}` | Who opened or updated the PR |
| Branch / Jira ID | `${{ github.head_ref }}` | Ticket traceability |
| PR link | `${{ github.event.pull_request.html_url }}` | One-click navigation |
| Repository | `${{ github.repository }}` | Backend vs frontend vs mobile |
| Commit SHA | `${{ github.sha }}` | Exact revision tested |
| Workflow run | `${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}` | Link to full logs |

### Slack delivery

Use a **Slack Incoming Webhook** scoped to a dedicated channel (e.g. `#engineering-ci` or `#pr-updates`).

Store the webhook URL as a GitHub secret: `SLACK_WEBHOOK_URL`.

**Color coding:**

- **Green** (`#36a64f`) — all tests passed
- **Red** (`#e01e5a`) — one or more tests failed

See [Slack Message Specification](#slack-message-specification) below for the proposed layout.

## Phase 4: Configure Branch Protection Gatekeepers

Branch protection on `staging` enforces the two-gate merge policy.

### Required settings on `staging`

| Rule | Setting |
|------|---------|
| Require pull request before merging | Enabled |
| Required approvals | **1** (engineer peer review) |
| Dismiss stale approvals on new commits | Enabled |
| Require status checks to pass | Enabled |
| Required status checks | `test` (and `lint` if separate job) |
| Require branches to be up to date | Enabled |
| Allow force pushes | Disabled |
| Allow deletions | Disabled |

### Resulting behavior

- Merge button stays **disabled** until CI reports green.
- Merge button stays **disabled** until at least one approving review is present.
- A PM seeing a green Slack message does **not** mean the code is merged — only that tests passed on the PR.

Apply equivalent rules in all three repositories.

## Phase 5: Hand-off to QA and Staging Deploy

Once a PR is **approved** and **tests are green**, the developer merges into `staging`.

### Post-merge sequence

```text
PR merged to staging
        │
        ▼
deploy-staging.yml runs
        │
        ▼
Staging environment updated
        │
        ▼
Slack notification to QA channel:
"Staging has been updated with PROJ-123 — ready for testing"
        │
        ▼
QA begins manual exploratory testing on live staging
```

### QA notification content

| Field | Example |
|-------|---------|
| Jira ticket | `PROJ-123` (parsed from merged branch name or PR title) |
| Author | `@developer` |
| Repositories updated | Backend / Frontend / Mobile (as applicable) |
| Staging URL | `https://staging.example.com` |
| What changed | Brief PR title or release note |

### QA team

Non-technical QAs perform manual exploratory testing on the live staging app after deploy. Automated tests reduce regression risk; manual testing validates user experience, edge cases, and product fit.

## Platform-Specific Test Configuration

### Backend (Django / Pytest)

**Repository:** `acme-api`

```yaml
- name: Run tests
  env:
    DJANGO_SETTINGS_MODULE: app.settings.test
    DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test_db
  run: pytest -v --junitxml=report.xml
```

### Frontend (React)

```yaml
- name: Install dependencies
  run: npm ci   # or: yarn install --frozen-lockfile

- name: Run tests
  run: npm test -- --ci --coverage --reporters=default --reporters=jest-junit
  env:
    JEST_JUNIT_OUTPUT_FILE: report.xml
```

Adjust for Vitest or the project's actual test runner.

### Mobile (Flutter)

```yaml
- name: Set up Flutter
  uses: subosito/flutter-action@v2
  with:
    flutter-version: '3.x.x'   # pin to project version
    channel: stable

- name: Install dependencies
  run: flutter pub get

- name: Run tests
  run: flutter test --reporter expanded
```

For JUnit-compatible output, add a reporter package or a post-step that converts Flutter JSON output to JUnit XML.

## Success Criteria

The plan is successfully implemented when:

1. Every PR to `staging` in all three repos triggers automated tests.
2. Slack receives a formatted pass/fail summary within minutes of each PR update.
3. Merges to `staging` are blocked without both green CI and developer approval.
4. Staging deploys **only** after merge, not on open PRs.
5. QA receives a Slack notification when staging is updated.
6. Branch names are consistently traceable to Jira tickets.
