# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**Feature: Dust i18n extraction-first runtime, ARB, and generated bootstrap**

*Project description:*
## Problem

Dust should provide a first-class i18n workflow for Flutter apps without forcing teams to hand-maintain ARB files, generated bootstrap code, and runtime lookup wiring separately.

The feature should be extraction-first: developers write translation calls in Dart, `dust i18n build` discovers static keys, updates ARB files deterministically, validates translation quality, and generates the runtime bootstrap needed by the app.

## Configuration

```yaml
i18n:
  locales: [en, my]
```

Defaults:

```txt
assets/i18n/{locale}/{namespace}.arb
lib/i18n/app_i18n.g.dart
namespace = first key segment
default/fallback locale = first locale
```

## Target Developer API

```dart
TranslatedText("home.title")

TranslatedText(
  "home.greeting",
  defaultText: "Hello {name}",
  args: {"name": user.name},
)

TranslatedText.dynamic(
  product.titleKey,
  fallback: product.title,
)

context.tr("home.title")
```

Generated bootstrap usage:

```dart
void main() {
  runApp(
    const AppI18n(
      child: App(),
    ),
  );
}
```

## Runtime Scope

- Add `TranslatedText` for statically discoverable keys.
- Add `TranslatedText.dynamic` for API or JSON driven keys that cannot be scanned safely.
- Add `context.tr(...)` as the ergonomic runtime lookup API.
- Add an app-level provider/controller API with final naming decided before stabilization.
- Add runtime configuration API with final naming decided before stabilization.
- Add ARB asset loading by locale and namespace.
- Support `{name}` placeholder interpolation.
- Preserve ARB metadata keys when writing:
  - `@@locale`
  - `@key`
- Use this lookup order:

```txt
remote override
asset namespace
fallback locale namespace
defaultText/fallback
key
```

## Generated Bootstrap Scope

- Generate `lib/i18n/app_i18n.g.dart`.
- Generate `AppI18n` widget/facade.
- Generate locale constants from config.
- Generate asset pattern constants.
- Generate default/fallback locale from the first configured locale.
- Keep generated output deterministic, analyzer-clean, and small enough for real apps.

## Dart Scanner Scope

- Scan `TranslatedText("key")`.
- Scan `defaultText`.
- Scan `args` placeholders.
- Detect namespace from the first key segment.
- Ignore `TranslatedText.dynamic(...)` because dynamic keys are intentionally runtime-only.
- Warn on `TranslatedText(dynamicKey)` and suggest `TranslatedText.dynamic(...)`.
- Warn on hardcoded UI text where it is safe and low-noise to detect.

## ARB Writer Scope

- Create missing locale folders.
- Create missing namespace ARB files.
- Append static keys to the template locale.
- Append empty values to non-template locales.
- Generate `@@locale`.
- Generate `@key` metadata.
- Preserve existing translations.
- Preserve existing metadata.
- Keep deterministic key order.

## Remote Override Scope

## Acceptance Criteria

- Static translation keys can be discovered from Dart source without manually editing ARB files first.
- ARB output is deterministic and preserves existing translations and metadata.
- Placeholder names in Dart calls and ARB metadata are validated.
- Generated `AppI18n` bootstrap works in a Flutter app with configured locales.
- Runtime lookup follows the documented override, asset, fallback, default, and key order.
- Dynamic keys are supported explicitly without being treated as scanner failures.
- `dust i18n check` gives CI-friendly diagnostics for stale, missing, invalid, or low-quality translations.
- The implementation scales to at least 10,000 keys without unacceptable build-time or runtime overhead.

## Validation

- Placeholder interpolation and mismatch tests.
- ARB parser tests.
- ARB writer preservation and ordering tests.
- Dart scanner tests.
- Hardcoded text scanner tests.
- Remote override tests.
- Generated bootstrap analyzer tests.
- 10,000 key performance test.

## Non-goals For v0.1

- Full ICU message-format implementation unless required by the runtime API slice.
- Machine translation integration.
- Translation management SaaS integration.
- Semantic detection of every possible hardcoded UI string.

## Architecture

```txt
User writes TranslatedText
  -> dust i18n build
  -> append assets/i18n/{locale}/{namespace}.arb
  -> generate AppI18n bootstrap
  -> TranslatedText reads from runtime controller/provider
  -> remote overrides can update translations at runtime
```
