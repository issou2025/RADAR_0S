# Project Proposal — Quantity Takeoff

## Project Understanding
You need assistance with: 
**Batch room detection (detect_rooms) — propose every room on a sheet at once**

*Project description:*
## Summary

Flesh out the `detect_rooms` endpoint on the AI socket into a real feature: instead of tracing one room per click (One-Click Area), **propose every room on a sheet in one pass** as editable proposals the estimator accepts/rejects/tweaks.

## Current state

The socket + contract exist but ship **empty** (no model), mirroring `parse_schedule`:

- `server/adapters/base.py` — `detect_rooms(width, height, segments) -> {"rooms": [{"verts": [[x,y],...], "area_px": float}], "note": str}`
- `server/adapters/heuristic.py` — returns `{"rooms": [], "note": "No model loaded…"}`
- `server/app.py` — `POST /ai/detect-rooms`, `DetectRoomsIn { width, height, segments? }`, `Room { verts, area_px }`

The input already accepts **`segments`** (flat boundary linework `[x1,y1,x2,y2,…]`) — and the client already extracts that vector geometry (`extractVectorGeometry`, feeds One-Click's flood-fill). So a real implementation can work from the drawing's actual geometry, not just a raster.

## Two implementation paths (to decide during planning)

1. **Pure geometry, in-browser (likely v1).** Treat the vector segments as a planar graph and find its **enclosed faces** (cycle/face detection on the planar subdivision). Deterministic, no ML, no server, stays local-first — reuses the segments we already have. Turns a 40-click sheet into a review pass.
2. **Vision model (server).** A segmentation model (SAM-style) or doc-VLM reads the raster and returns room polygons — more robust on messy scans, but needs a GPU/server and breaks the local-first default, so it'd be an opt-in, login-gated path (like the schedule scan path).

## UX

Same "AI proposes, human confirms" gate as the schedule importer and One-Click: detected rooms arrive as **editable ghost proposals** reusing the existing One-Click proposal handles (drag corner/edge, Shift-click to add a point, Delete a point). Nothing commits until Create; the estimator assigns a condition.

## Why it's high-leverage

It's the missing half of a full auto-takeoff. Combine:
- `detect_rooms` → all room polygons (area)
- the finish plan's **room tags** (`557 SF / CE-5 / CORRIDOR`, scattered floor codes) → which finish goes where
- the **schedule importer** (#80) → the conditions those codes map to

…and you can **auto-assign a finish to every detected room** — the whole takeoff proposed in one pass.

## Planning questions

- Geometry-only v1 vs. vision path — and where each lives (browser vs. login-gated serverless)?
- Robustness of planar face-detection on real, messy linework (gaps, overlapping hatch, door swings) — how to filter non-room faces (title block, legend, hatched fills)?
- Proposal UX at scale — 30+ rooms proposed at once; review/accept ergonomics.
- Integration with room-tag/finish auto-assignment (separate follow-up, but design the seam now).
