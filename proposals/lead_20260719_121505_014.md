\`\`\`
cmd.exe /c "tray.bat --restart" 2>&1
\`\`\`

returned immediately with no error (exit 0-looking), printing only:

\`\`\`

E:\automation\voice-transcriber>

No "Stopping previous ... tray", no "Starting ... tray", no verification line. The existing tray process (PID unchanged across two separate invocations, ~5 min apart) kept running the pre-merge build, and the webapp on \`:8443\` never came back up — polled for 90s+ each time, never bound.

Calling the same underlying script directly via PowerShell (bypassing \`tray.bat\`'s \`cmd.exe\` wrapper) worked correctly on the first try — forward-slash path, quoted:

\`\`\`
"C:/Windows/System32/WindowsPowerShell/v1.0/powershell.exe" -NoProfile -NonInteractive -File "$env:USERPROFILE/.claude/tray/tray_lifecycle.ps1" launch -AppName "VoiceTranscriber" -ScriptDir "E:/automation/voice-transcriber" -VenvDir "E:/automation/voice-transcriber/.venv" -TrayMatch "launcher\.py\s+tray" -Ports "8443" -TrayLaunch "launcher.py tray" -VersionUrl "https://127.0.0.1:8443/api/version" -Restart

## Why this matters

\`tray.bat\`'s own header comment already documents almost this exact failure mode ("Both cmd shapes have failed under non-interactive nested callers (Git Bash -> \`cmd /c \"tray.bat --restart\"\`, or a finisher skill's Bash tool): detect output came back empty, nothing was killed, and --restart silently degraded to a plain start that adopted the stale webapp and reported success") and claims it was fixed by delegating to the shared \`tray_lifecycle.ps1\`. That fix clearly doesn't cover this specific caller shape — a Claude Code session's Bash tool (Git-Bash-backed) shelling out to \`cmd.exe /c \"tray.bat --restart\"\` — since it reproduced twice in a row just now on voice-transcriber.

This is exactly the "restart reports success but serves the stale build" failure #54 was meant to prevent, and it's silent — no non-zero exit, no visible error — so an agent-driven \`/issue-finish\`/\`/issue-yolo\` run has no signal that the restart didn't actually happen unless it independently polls \`/api/version\` and compares against \`HEAD\` (which is what caught this here).

## Suspected root cause (unconfirmed)

Likely something about how Git-Bash/ConPTY allocates a console for a nested \`cmd.exe /c \"...\"\` invocation from a non-interactive parent — the banner text printed (\`Microsoft Windows [Version ...]\` + copyright + bare prompt) looks like an *interactive* \`cmd.exe\` session got spawned rather than the batch file actually executing under \`/c\`. Needs a repro harness that shells out through Git Bash's actual invocation path (not a human's interactive terminal) to confirm.

## Suggested fix direction

- Reproduce via a Python subprocess call shaped like Claude Code's Bash tool (Git Bash -> \`cmd.exe /c \"...\"\`) rather than a human terminal, to isolate whether it's a quoting/escaping issue or a console-allocation issue.
- Consider having \`tray.bat\` (or agent-facing docs) recommend/require invoking \`tray_lifecycle.ps1\` directly via \`powershell.exe -File\` from automation contexts, bypassing the \`cmd.exe\` layer entirely, if the \`cmd.exe\` wrapper can't be made reliable from this caller shape.
- Add this caller shape to whatever behavioral harness \`#152\` set up, since \`#152\`'s stated purpose was exactly to catch this class of regression.

## Workaround (used to unblock voice-transcriber#144 / PR #145)

Call \`tray_lifecycle.ps1\` directly via \`powershell.exe -File\` with the same arguments \`tray.bat\` would pass, instead of \`cmd.exe /c "tray.bat --restart"\`.
