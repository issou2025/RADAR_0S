## Part 1: Production Infrastructure

### Current state
- `docker-compose.yml` (repo root) already **defines** Postgres, Redis, LocalStack, Elasticsearch, **Prometheus, Grafana**, Mailhog, backend, and frontend services — but this is a local dev-orchestration file, not a deployed environment. Nothing in this compose file is running anywhere but a developer's machine.
- No domain is registered/pointed at anything.
- No CI/CD deployment step exists — `.github/workflows/test-and-build.yml` runs tests/build only, no deploy stage.
- `serviceAccountKey.json` (Firebase) sat unprotected at the repo root before a recent `.gitignore` fix — flagged separately in `CLAUDE.md` as needing credential rotation in the Firebase console regardless of gitignore status.

### Gaps to close

- [ ] **Domain & DNS**: register production domain, configure DNS (A/CNAME records for apex + `www` + `api` subdomain).
- [ ] **SSL/TLS**: provision certificates (Let's Encrypt via Certbot, or managed via hosting provider/CDN) for both frontend and API domains; enforce HTTPS redirect; verify auto-renewal.
- [ ] **Hosting/deployment target**: decide and provision (e.g. a VPS + Docker Compose, or split — frontend on Vercel/Netlify/Cloudflare Pages, backend on Railway/Render/Fly.io/ECS). Current `docker-compose.yml` assumes a single-host deployment; confirm this is still the intended production topology or re-architect for the chosen platform.
- [ ] **Database**: production Postgres instance (managed, e.g. RDS/Supabase/Neon, or self-hosted with proper backup tooling) — distinct from the local dev container currently in use.
- [ ] **Backups**: automated, scheduled backups for Postgres (point-in-time recovery ideally) and any persisted volumes (Elasticsearch indices, S3/LocalStack-equivalent object storage). Define and document a restore procedure — an untested backup is not a backup.
- [ ] **Object storage**: move from LocalStack (dev-only S3 emulation, per `services/s3.ts`) to real AWS S3 (or equivalent) in production.
- [ ] **Environment/secrets management**: production `.env` equivalents (JWT secrets, DB credentials, M-Pesa keys, Firebase service account, SMS/email provider keys, Google Maps API key) need a real secrets store (e.g. AWS Secrets Manager, Doppler, or platform-native env var encryption) — not committed files.
- [ ] **Monitoring**: the `docker-compose.yml` Prometheus + Grafana containers exist but aren't wired to a live `/metrics` endpoint in production, and there's no alerting configured. Needs:
  - [ ] Live-deployed Prometheus scraping the backend
  - [ ] Grafana dashboards (API latency, error rate, DB connection pool, queue depth if applicable)
  - [ ] Alerting (PagerDuty/Slack/email) for critical thresholds (5xx rate, DB down, disk space)
- [ ] **Error tracking**: `@sentry/node` is already a backend dependency (`config/sentry.ts` referenced in `server.ts`) — confirm a production Sentry DSN is configured and events are actually flowing, not just wired in code.
- [ ] **Logging**: `pino`/`pino-pretty` are used for structured logs (`middleware/logging.ts`) — needs a production log aggregation destination (e.g. CloudWatch, Loki, Datadog) rather than stdout-only.
- [ ] **Rate limiting backing store**: `rate-limit-redis` is a dependency — confirm production Redis is provisioned (not just local dev Redis) since rate limiting silently degrades to in-memory (per-instance, not shared) without it.
- [ ] **CI/CD deploy pipeline**: extend `.github/workflows/test-and-build.yml` (or add a new workflow) to actually deploy on merge to `main` — build → test → build Docker images → push to registry → deploy → smoke test.
- [ ] **Health checks & uptime monitoring**: external uptime monitor (e.g. UptimeRobot, Better Uptime) hitting `/health`, with alerting independent of the internal Prometheus/Grafana stack (so you find out even if the monitoring stack itself is down).
- [ ] **Load/capacity planning**: no load testing has been done against realistic concurrent user counts — needed before any real launch traffic.
- [ ] **Credential rotation**: rotate the previously-exposed `serviceAccountKey.json` Firebase credential in the Firebase console (Project Settings → Service Accounts) — flagged in `CLAUDE.md` as not yet confirmed done.

### Suggested sequencing
1. Domain + SSL + basic hosting (frontend + backend reachable over HTTPS)
2. Production DB + backups + secrets management
3. Monitoring + error tracking + alerting
4. CI/CD deploy automation
5. Load testing + go-live checklist
