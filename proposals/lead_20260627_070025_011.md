### Proposal Summary

While experimenting with the AI Gateway, I noticed that gateway providers and models appear to be managed primarily through the UI. This works well for interactive administration, but it makes it more difficult to automate AI Gateway provisioning as part of an Infrastructure-as-Code or CI/CD workflow. It appears that there is source code that could be built upon to support this idea, so not necessarily looking to "reinvent the wheel" here.

I’m interested in understanding whether programmatic management of AI Gateway configuration is something that would align with the project’s long-term direction before investing time in a potential implementation.

### Motivation

> #### What is the use case for this feature?
The use case is enabling administrators to provision and manage AI Gateway configuration programmatically rather than exclusively through the UI. This would allow Gateway providers and models to be created, updated, and managed as part of automated deployment workflows.

> #### Why is this use case valuable to support for MLflow users in general?
Many MLflow deployments are provisioned and maintained through automation rather than manual configuration. Supporting programmatic administration would make AI Gateway easier to integrate into existing deployment workflows, improve repeatability across environments, and reduce manual configuration steps. It would also make it easier to recreate environments consistently after upgrades or disaster recovery.

> #### Why is this use case valuable to support for your project(s) or organization?
I'll be completely honest: I have a YouTube channel where I teach things related to AI/ML, and I'm currently working on content on working with MLflow. I like to teach my learners how to do things programmatically, and I generally lean with a flavoring of how I would approach things in my own job as a Staff ML Engineer at State Farm. As such, I prefer to lean toward doing things in a programmatic, repeatable way, so it feels like a gap today that there's not a clean way to do this outside of the UI. I can imagine that many in a similar position as me would greatly benefit from something like this to support their own respective organizations.

> #### Why is it currently difficult to achieve this use case?
The current AI Gateway leans toward doing things through the UI. As a professional Staff ML Engineer, this is not preferable as a) there is no trace back to how a human user would set this up and b) it is not easily repeatable. In terms of implementing this feature, it seems like it shouldn't be too bad as the MLflow source code already leans this way already.

### Details

I intentionally haven’t proposed a concrete implementation because I’d first like to understand whether this capability aligns with the long-term direction of AI Gateway.

At a high level, I imagine this could take the form of a supported programmatic administration interface that allows AI Gateway configuration to be managed without relying on the UI. Depending on MLflow’s preferred architecture, this could potentially be exposed through REST endpoints, Python client APIs, or another administrative interface.

From an administrator’s perspective, the desired capabilities would include managing Gateway providers and models (e.g., create, update, list, and delete operations), enabling AI Gateway configuration to be incorporated into automated deployment and provisioning workflows.

If this is an area the maintainers are interested in pursuing, I’d be happy to collaborate on a design and contribute an implementation.
