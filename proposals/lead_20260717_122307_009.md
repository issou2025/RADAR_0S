# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**Expand localhost technology detection and fix runtime/service icons**

*Project description:*
### Problem

PortPeek's detection is strong for several web ecosystems, but many common localhost runtimes and infrastructure services still appear as generic process names. The current issue only covers .NET/ASP.NET and Java/Spring Boot, while the same cheap process, command, and project-file signals can identify more useful technologies.

Technology icons also need a deterministic precedence rule. Today `Favicon.svelte` tries a cached project favicon before the detected brand. A Java listener launched from the PortPeek working tree can therefore show the PortPeek logo instead of Java/OpenJDK. SSH/OpenSSH listeners similarly need a recognizable technology icon or a safe built-in fallback.

### Proposed solution

Expand the existing confidence-scored detector with a focused table of common localhost development runtimes, frameworks, servers, and services. Keep detection offline and bounded: use process name, command line, executable path, and nearby project markers already available during enrichment.

Update brand selection so a confidently detected runtime/service can use its technology logo instead of an unrelated project favicon. Continue using the current Simple Icons source where a maintained brand exists; use an existing Lucide icon when it does not.

**Delivery rule:** Phase 1 uses distinctive executable, command, or config signals with low false-positive risk. Phase 2 requires dependency-file inspection or shares an ambiguous runtime, so it may ship in later PRs. Process-name infrastructure matches must still verify enough executable or command context to avoid substring collisions.

### User scenarios

- A developer runs an ASP.NET API, a Spring Boot service, and a Flask worker on nearby ports. Each listener has a useful label instead of only `dotnet`, `java`, or `python`.
- A Java process started from the PortPeek repository displays the Java/OpenJDK or Spring Boot logo—not PortPeek's project favicon.
- An OpenSSH listener displays an OpenSSH logo when supported, otherwise a consistent terminal/key icon rather than the generic PortPeek/app logo.

### Scope

**Included:**

- Deliver all **Phase 1** technologies using cheap offline signals; track Phase 2 rows as incremental follow-up PRs under this issue.
- Add the required exact and bounded wildcard project-root markers.
- Keep generic runtime labels when a specific framework cannot be established confidently.
- Add technology brand mappings and deterministic icon precedence.
- Preserve local project favicons for actual web projects when they are the most relevant identity.
- Add positive, negative, collision, and icon-fallback tests.

**Excluded:**

- Deep dependency graph resolution or executing package managers.
- Exact framework/runtime version detection.
- HTTP probing.
- Scanning outside the bounded project-root search.
- Downloading or bundling a large icon pack.
- Treating arbitrary `java`, `python`, `node`, `ssh`, or `dotnet` processes as a specific framework without supporting evidence.

### Icon precedence

Use a small, explicit rule rather than always preferring the project favicon:

1. A high/medium-confidence detected infrastructure service or runtime brand (for example OpenSSH, Java/OpenJDK, Tomcat, Nginx) wins over an unrelated project favicon.
2. A specifically detected web framework brand (for example Spring Boot, ASP.NET Core, Flask, NestJS) is preferred when it provides the clearest identity.
3. A valid project favicon remains useful for an otherwise generic web project.
4. If a remote brand icon fails, fall back to a built-in Lucide category icon; never substitute PortPeek's own logo unless the listener is actually PortPeek.

### Acceptance criteria

- [ ] Every Phase 1 row in the matrix has positive and negative detection coverage.
- [ ] Each Phase 2 detector is implemented only when it has a bounded, testable signal stronger than a generic runtime process.
- [ ] Specific frameworks require stronger evidence than the generic runtime label.
- [ ] Generic Java and .NET listeners fall back to Java/.NET rather than a false high-confidence framework.
- [ ] Java/OpenJDK, Spring Boot, and OpenSSH listeners render their technology icon or documented fallback.
- [ ] A Java listener in the PortPeek working directory does not render the PortPeek favicon.
- [ ] OpenSSH uses its maintained brand icon when available and a built-in terminal/key fallback otherwise.
- [ ] Failed CDN icons fall back without layout shift, broken-image UI, or repeated error loops.
- [ ] Existing framework and database icon mappings continue to work.
- [ ] Existing Django, FastAPI, and Laravel detection and logos have regression coverage; Symfony is added without weakening their confidence rules.
- [ ] TablePlus.exe is labeled TablePlus and renders the TablePlus brand or database-client fallback, never the PortPeek project favicon.
- [ ] agy.exe is labeled Antigravity and renders the Antigravity logo or orbit/sparkles fallback, never an unrelated project or PortPeek favicon.
- [ ] Detection remains offline and does not materially slow the normal refresh.
- [ ] Rust and TypeScript framework shapes remain synchronized.
- [ ] Tests cover ambiguous commands, project-marker collisions, icon precedence, and fallback behavior.

### Technical considerations

Likely areas:

- `src-tauri/src/domain/detection/framework.rs`: new generic runtime, framework, server, and service matches with confidence.
- `src-tauri/src/domain/detection/project.rs`: additional exact markers plus a bounded way to recognize project extensions such as `*.csproj` without an unbounded recursive scan.
- `src/lib/utils/ports.js`: expand `frameworkBrands` / `appBrands` and extract the icon-choice precedence into testable logic if needed.
- `src/lib/components/ports/Favicon.svelte`: apply the chosen source order and built-in fallback icon.
- `src/lib/utils/ports.test.mjs` and Rust unit tests: detection, collision, precedence, and fallback cases.

Keep this pure/offline so the Tauri app and reusable Rust detection logic stay lightweight. The CLI currently skips framework/favicon enrichment because it has no `AppHandle`; do not expand this issue into redesigning the CLI enrichment path.
