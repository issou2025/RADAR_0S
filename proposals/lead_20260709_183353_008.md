# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**[Feature]: `winapp new` command to scaffold new projects**

*Project description:*
## Is your feature request related to a problem? Please describe.

There is no first-class way to create a new Windows app from the `winapp` CLI. `winapp` is the front door for building Windows apps across frameworks (WinUI, Tauri, Electron, Flutter, Rust) but it can't yet *start* one. Today a developer (or agent) has to leave the CLI, find the right scaffolder for their framework, and wire up Windows packaging/identity by hand.

Each framework already has its own create tool (`npm create tauri-app`, `npm create electron-app`, `flutter create`, `cargo new`), and WinUI now has official `dotnet new` templates (`Microsoft.WindowsAppSDK.WinUI.CSharp.Templates`: `winui`, `winui-navview`, `winui-tabview`, `winui-mvvm`, `winui-lib`, `winui-unittest`) that work without Visual Studio. What's missing is a single `winapp new` that meets developers at any of these, with WinUI as the first-class default path.

## Current state

- **No `new`/scaffold command exists** today
- Each framework is scaffolded by its **own external tool** — `npm create tauri-app`, `npm create electron-app`, `flutter create`, `cargo new`, and for WinUI the `dotnet new` template pack (preview, WASDK-owned: `dotnet new install Microsoft.WindowsAppSDK.WinUI.CSharp.Templates`). `winapp` ties none of these together today.
- Windows packaging/identity (`winapp init`) is a separate manual step after creation, regardless of framework.

## Describe the solution you'd like

Add a top-level **`winapp new`** command with a unified create experience. Framework choice happens inside `winapp` (WinUI at the top of the picker), not via a separate CLI or namespace. For each supported framework, `winapp new` delegates to that framework's **native scaffolder** and produces a Windows app that's ready to run.

- **Interactive picker** — framework, then framework-specific options (for WinUI: which template; for others: whatever their scaffolder asks), then details (name, output, and an optional "set up Windows packaging/identity" step).
- **Non-interactive / agent mode** — every prompt has a flag equivalent, with deterministic, machine-readable output for one-shot scaffolding.
- **Delegate to native scaffolders** — `winapp new` owns the orchestration + UX; the actual project creation is the framework's own tool.
- **Optional `winapp init`** — packaging/identity setup (manifest, SDK packages, projections)
  is an **optional** step in the form and via a flag for agents — never forced. Reuses the
  existing `init`; no new packaging logic.
- **Extensible** — leave a clean seam for additional frameworks/authoring modes later.

This largely **codifies the existing getting-started guides** (which already document
`cargo new`, `npm create tauri-app`, `npm create electron-app`, `flutter create`, and the
WinUI templates) into a single command.

## Proposed architecture (for review)

`winapp new` is a **thin orchestration layer**: it does not author templates or reimplement packaging. It resolves inputs, delegate creation to the framework's native scaffolder, then optionally runs `winapp init`:

```
winapp new
  ├─ 1. resolve inputs (interactive picker  OR  flags / --json for agents)
  ├─ 2. scaffold → delegate to the framework's native tool
  │        winui    → dotnet new <winui template>
  │        tauri    → npm create tauri-app
  │        electron → npm create electron-app
  │        flutter  → flutter create
  │        rust     → cargo new
  └─ 3. (optional) winapp init → manifest, SDK packages, projections
```

- **Ownership split:** `winapp new` owns input resolution + UX + orchestration; each native tool owns its scaffolding; `winapp init` owns packaging/identity. No new scaffolding or packaging logic is introduced.
- **Consistency:** mirrors `winapp run` (which wraps dotnet/msbuild) — `new`/`build`/`run` share the same "thin winapp wrapper over the underlying toolchain" pattern.
- **Prerequisite handling (needs a decision):** delegating means the user must already have the framework's toolchain installed (Node for Tauri/Electron, Flutter SDK, Cargo, .NET for WinUI). `winapp new` should **detect a missing toolchain and fail with a clear, actionable message** (or point to the relevant guide) rather than a raw error. It should not try to install these itself.
- **WinUI specifics:** the WinUI template pack is an external preview dependency (WASDK-owned); `winapp` installs/manages it on demand.

## Work breakdown (sub-issues)

This is an epic; each framework is a sub-issue so they can land independently on top of the shared orchestration/UX layer.

### MVP
- [ ] **P0**: `winapp new` orchestration + UX/picker, flag/`--json` agent mode + WinUI support (delegate to the official `dotnet new` WinUI templates) + Missing-toolchain detection (needed to support additional frameworks)
- [ ] **P1** Optional `winapp init` chaining step

- [ ] **P1** Tauri — delegate to `npm create tauri-app`.
