# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**Standardize Flutter version across dev + CI (FVM) to fix the unreliable dart-format gate**

*Project description:*
## Problem

CI's `code_tests` job runs `dart format lib/ test/ --set-exit-if-changed` under a **pinned** Flutter (`FLUTTER_VERSION=3.38.7` in `.github/workflows/versions.env`). Developers run a newer local Flutter (e.g. 3.38.9), and the two minor versions format some constructs differently. So files committed with a dev's local formatter fail CI's format check even though nothing was "mis-formatted" — `code_tests` goes **red on files unrelated to the PR**, and the format gate becomes noise we routinely admin-merge through (eroding the signal).

Recent fallout:
- #7295 (auth refactor) went red on `choreo_endpoint_test.dart` / `endpoint_test_env.dart` and was admin-merged through it.
- #7293 (#7267 fix) went red on 4 files it never touched (`practice_generation_repo.dart`, `app_version_util.dart`, `choreo_endpoint_test.dart`, `endpoint_test_env.dart`); rebasing onto main did not help because main itself is dirty per 3.38.7.

## Proposed fix — pin ONE Flutter version for everyone

1. Adopt **FVM** (Flutter Version Management): commit an `.fvmrc` pinning the project's Flutter version so every dev's `flutter`/`dart format` matches.
2. Point CI at that same version (set `FLUTTER_VERSION` in `.github/workflows/versions.env` to match, ideally derived from the FVM pin so they can't drift again).
3. One-time: reformat the currently-skewed files under the chosen version so `dart format lib/ test/ --set-exit-if-changed` is clean on main.
4. Document the FVM workflow in the contributor / run-flutter docs.

## TO TEST

No client testing needed — CI / dev-tooling change. Verify by: a no-op PR's `code_tests` passes the format step on a clean tree, and `dart format lib/ test/ --set-exit-if-changed` exits 0 locally under the pinned FVM version.
