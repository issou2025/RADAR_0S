# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**[FamilyManager] Content-hash дедупликация: предотвращение импорта дубликатов через PartAtom-based fingerprint**

*Project description:*
## Контекст

В модуле FamilyManager есть давняя проблема: при попытке импортировать семейство, которое **уже есть в каталоге по содержимому** (но, например, имеет другое имя файла или было получено из другого источника), система не распознаёт это как дубликат.

### История вопроса

**До v2.0.0** использовался SHA-256 от содержимого `.rfa` файла как content hash. Это работало, пока пользователи не начали пересохранять `.rfa` файлы через "Save As" / "Save" в Revit — бинарный формат `.rfa` (OLE Compound Document) меняется при каждом сохранении (timestamps, внутренние GUID, индексы), даже если **содержимое семейства не изменилось**. Результат: ложные "новые" импорты одинаковых семейств → раздувание каталога, путаница в версиях.

В **v2.0.0 (ADR-035)** SHA-256 dedup был полностью удалён:
- Schema v14 дропнул `family_files.sha256`, `catalog_versions.sha256`, `family_files.size_bytes`
- `Sha256FileHasher.cs` удалён
- `FindByHashAsync` удалён из `LocalCatalogProvider`
- `FamilyBatchImportStatus.Duplicate` удалён из enum
- Дедупликация осталась **только по нормализованному имени** (`FamilyNameNormalizer` + `FindByNormalizedNameAsync`)

Это решило проблему ложных "новых" импортов, но **вернуло старую боль**: пользователь не может отличить "это то же самое семейство, просто с другим именем" от "это новая версия того же семейства".

### Бизнес-сценарии, которые сейчас сломаны

**Сценарий A — "Family1" и "Family1 копия":**
- BIM-инженер получил от производителя 2 файла: `PipeFitting_DN50.rfa` и `PipeFitting_DN50_v2.rfa`
- На самом деле это **одно и то же семейство** (одинаковые параметры, геометрия, типоразмеры), просто производитель сделал Save As и переименовал
- Сейчас: оба файла показываются в batch dialog как `Existing` (по нормализованному имени), пользователь должен **вручную** понять, что это дубликат
- Ожидаемое поведение: второй файл показывается как `Duplicate` с auto-Skip

**Сценарий B — Импорт из разных папок:**
- Пользователь скачал ту же библиотеку семейств дважды (из сетевой папки и из облака), оба раза с одинаковыми именами файлов
- Сейчас: при импорте второй раз все показываются как `Existing`, но это новая попытка → новая версия (`v2`)
- Ожидаемое поведение: дубликаты по содержимому не создают новых версий

**Сценарий C — Импорт системного семейства (system families):**
- В активном проекте две категории `OST_PipeFitting` с одинаковым набором типоразмеров (например, шаблон проекта содержит дефолтные типы, а пользователь добавил свои — но они идентичны)
- Сейчас: всегда создаётся новая запись в каталоге
- Ожидаемое поведение: дубликат определяется по содержимому

## Цель

Ввести **стабильный content-hash** для семейств, **независимый от файловых метаданных** (timestamps, OLE internal GUIDs), который:
1. **Меняется** при изменении содержимого семейства (другой параметр, тип, материал)
2. **НЕ меняется** при SaveAs / Save / Upgrade Revit version
3. Используется для определения дубликатов в batch dialog
4. Хранится в SQLite для быстрой проверки без пересчёта

## Техническое решение: PartAtom XML

После исследования ([Jeremy Tammik blog #0248](https://jeremytammik.github.io/tbc/a/0248_extract_part_atoms.htm), [#1640](https://jeremytammik.github.io/tbc/a/1640_part_atom_direct.html), [Autodesk Developer Blog "Comparing Family Part Atoms"](https://blog.autodesk.io/swissbau-basel-and-comparing-family-part-atoms/)) выбран **PartAtom XML fingerprint**.

### Что такое PartAtom

`PartAtom` — это **официальный канонический XML-формат Autodesk** для описания содержимого семейства. Используется:
- Внутри Revit для контент-поиска (Autodesk Seek)
- Внутри самого `.rfa` файла как plain text (XML блок встроен в OLE container)
- Извлекается через `Application.ExtractPartAtomFromFamilyFile(rfaPath, xmlPath)` или через pure C# re-implementation Håvard Dagsvik (Symetri)

**Реальный пример XML из `.rfa` (из блога Jeremy Tammik):**

```xml
<entry xmlns="http://www.w3.org/2005/Atom" xmlns:A="urn:schemas-autodesk-com:partatom">
  <link rel="design-2d" type="application/rfa" href=".">
    <A:design-file>
      <A:title>column.rfa</A:title>
      <A:product>Revit</A:product>
      <A:updated>2009-11-11T08:12:08Z</A:updated>   ← нестабильный timestamp (фильтруем)
    </A:design-file>
  </link>
  <A:family type="user">
    <A:variationCount>4</A:variationCount>           ← кол-во типоразмеров
    <A:instance>...</A:instance>
    <A:type name="600x600">                          ← КОНКРЕТНЫЙ ТИПОРАЗМЕР
      <A:param name="Width">
        <A:type>length</A:type>                      ← параметр с типом и значением
        <A:value>600</A:value>
      </A:param>
      <A:param name="Material">
        <A:type>material</A:type>                    ← материалы
        <A:value>Concrete</A:value>
      </A:param>
    </A:type>
    <A:dim>...</A:dim>                               ← dimensions
  </A:family>
</entry>
```

### Подтверждение стабильности

✅ **Jeremy Tammik** ([TheBuildingCoder #0248](https://jeremytammik.github.io/tbc/a/0248_extract_part_atoms.htm)):
> *"You can see the same four types with the same parameter values listed. The unnamed type is skipped, as well as empty parameter values."*

✅ **Matt Taylor** в комментарии к [Autodesk blog](https://blog.autodesk.io/swissbau-basel-and-comparing-family-part-atoms/):
> *"I would say that this is a good candidate for the ExtractPartAtomFromFamilyFile and ExtractPartAtom methods."*

✅ **Håvard Dagsvik (Symetri)** в [TheBuildingCoder #1640](https://jeremytammik.github.io/tbc/a/1640_part_atom_direct.html):
> *"my own custom method is much faster than ExtractPartAtom. More importantly, it can be used without Revit, and it doesn't save to an external XML file that we don't need anyway."*

Jeremy Tammik добавил метод Håvard в **официальный Building Coder samples** (release 2018.0.138.4):
> *"I added Håvard's method to the existing external command CmdPartAtom in The Building Coder samples... and cleaned up the existing code calling the built-in Revit API method ExtractPartAtomFromFamilyFile at the same time."*

### Почему НЕ другие подходы

| Подход | Почему отвергнут |
|---|---|
| `BasicFileInfo.GetDocumentVersion().VersionGUID` | Меняется при **каждом Save** (Autodesk Community подтверждено) |
| `Family.UniqueId` / `FamilySymbol.UniqueId` | Jeremy Tammik: *"Neither of the identifiers types are stable across individual Revit documents. If one takes an existing file and makes a copy of it, no one would probably expect elements to change their Ids in that copied file"* |
| SHA-256 от файла | Меняется при SaveAs (OLE Compound Document timestamps, GUIDs) — отключено в ADR-035 |
| Ручной хеш параметров через Revit API | Дорого (открытие документа, IFamilyManagerAwaitableEvent маршалинг, freeze dialog workaround). PartAtom уже **включает** все параметры + типы + materials + dimensions |
| `Solid.Volume` сумма | Нестабильно для curved surfaces (approximation). Используем только для system families (`.rvt`) где PartAtom не работает |

Алгоритм для `.rfa` (Håvard Dagsvik подход):
```csharp
private static string ExtractPartAtomXml(string rfaPath)
{
    var bytes = File.ReadAllBytes(rfaPath);
    var text = Encoding.UTF8.GetString(bytes);
    var start = text.IndexOf("<entry", StringComparison.Ordinal);
    if (start < 0) return null;
    var end = text.IndexOf("/entry>", start, StringComparison.Ordinal);
    if (end < 0) return null;
    return text.Substring(start, end + "/entry>".Length - start);
}

private static string Sanitize(string xml)
{
    var doc = XDocument.Parse(xml);
    XNamespace a = "urn:schemas-autodesk-com:partatom";
    // Удаляем нестабильное: timestamp, design-file metadata
    foreach (var updated in doc.Descendants(a + "updated").ToList()) updated.Remove();
    foreach (var meta in doc.Descendants(a + "design-file").ToList()) meta.Remove();
    return doc.ToString(SaveOptions.DisableFormatting);
}

public async Task<FamilyContentHash?> ComputeForRfaAsync(string rfaPath, CancellationToken ct)
{
    var xml = ExtractPartAtomXml(rfaPath);
    if (xml == null) return null;
    var sanitized = Sanitize(xml);
    var hash = SHA256.HashData(Encoding.UTF8.GetBytes(sanitized));
    return new FamilyContentHash(
        HexString: Convert.ToHexString(hash),
        FormatVersion: FamilyContentHashFormat.CurrentVersion,
        ComputedAtUtc: DateTimeOffset.UtcNow,
        ContentKind: "rfa");
}
```

**Для `.rvt`** (system families): используем Revit API внутри existing staging flow:
```csharp
public async Task<FamilyContentHash?> ComputeForRvtAsync(string rvtPath, CancellationToken ct)
{
    // Открываем через Application.OpenDocumentFile (уже делается в StageSystemFamiliesFromMetadataAsync)
    var doc = app.OpenDocumentFile(rvtPath);
    try
    {
        var hashInput = BuildCanonicalInput(doc);
        // hashInput = "FAMILY|" + FamilyCategory.BuiltInCategory +
        //             "|PARAMS|" + sorted(params) + "|TYPES|" + sorted(types) +
        //             "|SOLID_VOLUMES|" + sum + ":" + count + ":" + min + ":" + max +
        //             "|FACES|" + sum_faces + "|EDGES|" + sum_edges
        var hash = SHA256.HashData(Encoding.UTF8.GetBytes(hashInput));
        return new FamilyContentHash(...);
    }
    finally
    {
        doc.Close(false);
    }
}
```

## Risks & Mitigations

| Риск | Митигация |
|---|---|
| `<A:updated>` timestamp меняется → ложные дубликаты | `Sanitize()` удаляет перед хешированием, покрыто unit-тестом "same content different timestamps → same hash" |
| PartAtom XML format change в новой версии Revit | `FormatVersion` в `FamilyContentHash` — при изменении формата lazy re-fill |
| `AccessViolationException` на legacy Revit | Зафиксировано Jeremy Tammik в 2011 для Revit 2011 64-bit — давно исправлено. R19+ работает стабильно. Если проявится → fallback на `null` hasher → batch dialog использует `Existing` вместо `Duplicate` |
| OLE structure изменён в будущем | Если `IndexOf("<entry") == -1` → return null → fallback на legacy path (без дедупа) |
| Performance regression для batch из 50+ семейств | Pure C# подход = 20-100 мс × N. Для 1-5 семейств (99% кейсов) — незаметно. Для 50+ — прерывистое UI feedback через `IProgress<>` |
| System families hash дороже (открытие .rvt) | Делаем **только** внутри existing staging flow → нет дополнительных затрат |

## Out of scope

- ❌ Хранение hash в ExtensibleStorage самого `.rfa` (over-engineered, требует регистрации schema у пользователя)
- ❌ Полный геометрический хеш через `Solid.Volume` каждого solid (используем только для `.rvt`)
- ❌ Recursive hash для вложенных shared families (PartAtom уже включает их references)
- ❌ Кросс-БД дедупликация (только в рамках одной active DB)
- ❌ Migration backfill для существующих записей (lazy-fill при первом использовании)
