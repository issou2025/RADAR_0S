## Summary

Eight findings from integrating poke-me-sdk-dart **v0.3.0/v0.4.0** into a Flutter app (iOS + macOS BYOA inbox) where the device kept ending up in the server's "no push target" state and pushes never reached the app. Cause turned out to be a chain of SDK behaviours that quietly compound a single misconfiguration into a permanent broken state.

The bullets below are mostly independent — feel free to split into separate issues. Listed in priority order (impact first).

---

### 1. `apnsEnvironment` is a footgun in `PokeMe.init`

The current contract requires the caller to pass `ApnsEnvironment.sandbox | .production`, with no auto-detection. The natural choice — gate on Dart's `kReleaseMode` / `kDebugMode` — is **wrong** for any build that isn't a real App Store / TestFlight archive:

- `flutter run --release` to a connected developer device is signed with the **development** APS entitlement → APNs issues a **sandbox** token.
- The Dart side has no idea — `kReleaseMode == true`.
- If the caller naively sends `production`, every subsequent push triggers BadDeviceToken on Apple's side, the poke-me fanout worker cascade-revokes the device's `push_token`, the dashboard shows "no push target", and the SDK happily keeps thinking it's registered. Repeats forever.

Suggested fixes (any one would help, in increasing scope):

- **Doc warning** in the SDK's `PokeMe.init` docstring and README: "Do NOT gate on `kReleaseMode`. The APNs environment is determined by the *entitlement the binary was signed with*, not by Dart compile mode."
- **Auto-detect** on iOS/macOS by reading the `aps-environment` entry from the embedded mobileprovision (parseable from `[[NSBundle mainBundle] pathForResource:@"embedded" ofType:@"mobileprovision"]`). Apple's own examples do this. Falls back to caller-supplied if not detectable.
- **Refuse the default** — make the parameter required, with the docstring explaining the trap.

This was the root cause of our entire debugging session. None of the other issues below would have hurt us in isolation.

### 2. macOS plugin claims `UNUserNotificationCenter.delegate` without chaining

In `macos/Classes/PokemePlugin.swift`:

```swift
public static func register(with registrar: FlutterPluginRegistrar) {
    ...
    UNUserNotificationCenter.current().delegate = instance
}
```

There can only be one `UNUserNotificationCenter.delegate`. If another plugin sets itself later — `flutter_local_notifications` (18.x) is a common offender, and its `willPresent` does an early `return` without calling `completionHandler` for non-flutter-local notifications — the pokeme plugin's `willPresent` never fires and the Dart `pushes` stream stays silent even though the OS displayed the banner. We confirmed this with foreground delivery on macOS: banner visible, no Dart-side `_onPush` callback.

Fix is the standard chained-delegate pattern: capture the previous delegate at register time, install pokeme's, forward unrelated calls to the previous one. This survives plugin-registration order shuffles.

Also: iOS plugin uses `registrar.addApplicationDelegate(instance)` which correctly chains via Flutter's plugin mechanism. macOS should match (or use the swizzling approach from #3 below for delegate methods too).

---

### 3. `registerOnLaunch(requestPermission: true)` hangs 30s on iOS on second call within a session

When the SDK calls `UIApplication.shared.registerForRemoteNotifications()` a second time in the same app session (e.g. once on launch with `requestPermission: false`, then again from a feedback-submit path with `requestPermission: true`), iOS does **not** re-fire `application:didRegisterForRemoteNotificationsWithDeviceToken:` — but the SDK's `pendingTokenResult` is armed waiting for it. The 30s timeout fires, `getToken` throws `PushTokenException(code: 'TIMEOUT')`, and the caller's `registerAndIdentify`-style flow is dead.

We worked around by reordering to "try `identify` first; only call `registerOnLaunch` if identify throws `StateError`". But the SDK itself could:

- Cache the token from the first `didRegister` callback and return it synchronously on subsequent `getToken` calls (until next push-token-refresh notification).
- OR detect that `UIApplication.shared.isRegisteredForRemoteNotifications` is true and skip the round-trip.

### 4. `registerOnLaunch(requestPermission: false)` returns void in two indistinguishable cases

```dart
Future<void> registerOnLaunch({bool requestPermission = true}) async {
  ...
  try {
    pushToken = await _tokenService.getToken(...);
  } on PushTokenException catch (e) {
    if (!requestPermission && e.isPermissionDenied) {
      return;   // <-- silently deferred
    }
    rethrow;
  }
  ...
  // OR succeeds and POSTs to /devices
}
```

Both paths return `void` with no exception. The caller cannot distinguish "registered with server" from "permission isn't granted, did nothing". Our diagnostic logs initially showed `silent-register:ok` for the silently-deferred case, which sent us hunting in the wrong direction for an hour. Suggested fix: return a status enum (`Registered` / `PermissionDeferred` / `AlreadyCurrent`), or replace the silent return with a typed exception the caller can catch deliberately.

Same complaint about the SDK's own internal log line if it has one.

---

### 5. `identify(...)` unconditionally sends `apns_environment` on every call

The SDK passes whatever `apnsEnvironment` was set in `PokeMe.init` on every `identify` request body. Server-side this is a `coalesce($5, d.apns_environment)` — so a wrong value (see #1) overwrites the row on **every identify**, not just first-bind. This is what kept our manual server-side fix from sticking: as soon as the iPad's next feedback-submit fired identify, the env flipped back from sandbox→production and the next push cascade-revoked again.

Suggested fix: `apnsEnvironment` parameter on `identify(...)` (explicit per-call), and `PokeMe.init` no longer sends it implicitly. Or send it once on first identify only.

---

### 6. No public accessor for the cached push token or device credentials

For diagnostics we had to:
- Invoke `MethodChannel('io.pokeme.pokeme/push_token').invokeMethod('getToken')` directly to read the iOS token.
- Open the SDK's `pokeme.db` sqlite store ourselves to read `device_id` and `device_token` from `sync_state`.

Both are reasonable to keep internal, but a debug-accessible getter would have shortcut several hours of "is the SDK actually registered?". Suggested:

```dart
class PokeMe {
  /// Last APNs/FCM/Web push token returned to the SDK, if any.
  String? get currentPushToken;

  /// The server-issued `dt_…` token the SDK uses for `/devices/me/*` calls.
  Future<String?> get deviceToken;

  /// The server-assigned device UUID.
  Future<String?> get deviceId;
}
```

---

### 7. Server-side cascade-revoke is invisible to the SDK

When poke-me's fanout worker receives a "token dead" response from APNs/FCM, it runs:

```sql
update fanout.devices
   set push_token = null, push_token_updated_at = now(), mtime = now()
 where id = $1
```

…with no notification back to the device. The SDK keeps thinking it's registered when it isn't. This is half a server-side concern (perhaps a webhook / push), but the SDK could mitigate by periodically calling `GET /api/v1/devices/me` and re-registering if `push_token` is missing from the response.

---

### 8. `IdentifyRequest.app_id` body field requires UUID; URL path's `{app_id}` initially also required UUID

This is partly fixed already — kudos for landing the slug-acceptance patch on `/identify` (#6). Same patch is needed on `/api/v1/apps/{app_id}/notify`'s URL path: currently `boost::uuids::string_generator{}(...)` on the server, so passing a human-readable slug returns 400 there even though the same slug is accepted on `/devices/me/identify`. Inconsistent. Not strictly SDK, but caller-facing.
