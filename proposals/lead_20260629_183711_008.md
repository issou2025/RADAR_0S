## Questions

1. Is the intended primary dev workflow to (a) run inside `Dockerfile.dev` (the documented Nix replacement), (b) run natively on the host with locally-installed tools, or (c) both? The answer decides whether the fix is "document the container path everywhere" or "provide a host-side bootstrap script".
2. If host-side is supported: should `task setup` install missing tools itself (e.g. `pip install pre-commit`, `curl … dagger install.sh`), or should it only **check** for them and print install instructions when missing?
3. May the Dockerfile.dev image be pulled from GHCR (it is already published by `publish-dev-container.yml`) and used via `docker run` for one-off commands like `task check`, or do you want a slimmer host-only path?

## Plan

### Goal

After nix was dropped, the dev shell no longer guarantees `dagger`, `pre-commit`, `task`, `fvm`, `stalwart-mail`, etc. Developers who follow the current `DEVELOPMENT.md` / `README.md` get errors deep inside `task` runs. Make tool availability explicit, fail fast with actionable messages, and document the post-nix path.

### Concrete changes

1. **Remove the now-impossible `IN_NIX_SHELL` gate** in `Taskfile.yml` (`_nix-check`, `_preflight` lines 12–45). Replace with a `_tools-check` precondition that verifies each binary actually used downstream is on `PATH` and prints the *one* install command per missing tool:
   - `dagger` → `curl -fsSL https://dl.dagger.io/dagger/install.sh | DAGGER_VERSION=0.21.7 BIN_DIR=/usr/local/bin sh`
   - `pre-commit` → `pip install --user pre-commit==4.5.1` (version from `Dockerfile.dev`)
   - `fvm` → existing FVM install snippet
   - `task` (self-check) → taskfile.dev install one-liner
   - Bonus: keep `pre-commit install` (already there at line 45) but only after the check passes.

2. **Rewrite `hooks/pre-commit`** so the "skipping (run inside nix develop)" branch is replaced with an explicit error pointing at the new install instructions. With `.pre-commit-config.yaml` already driving everything via `pre-commit`, consider deleting `hooks/pre-commit` entirely — `task install-hooks` already wires `pre-commit install` (Taskfile.yml:91-94).

3. **Add a `task doctor` (or repurpose `setup`) target** that runs the new `_tools-check`, prints versions of `dagger`, `pre-commit`, `task`, `fvm`, `flutter`, and exits non-zero if anything is missing. This is the one command we tell new contributors to run first.

4. **Rewrite the Prerequisites section of `DEVELOPMENT.md` and `README.md` (lines 78–90 / 4–134)** to drop all Nix/direnv/`nix profile add` instructions. Replace with two clearly labelled paths:
   - **Container path (recommended):** `docker run --rm -it -v $PWD:/src ghcr.io/guettli/sharedinbox/dev:latest task check` — the image already exists (`publish-dev-container.yml`).
   - **Host path:** install `task` + `fvm` + `dagger` + `pre-commit` via the one-liners above, then `task doctor && task setup`.

5. **Clean stale Nix references** in `AGENTS.md` (lines 69–71, 94), `DAGGER.md` (line 64), `Taskfile.yml:615` (`website-dev` calls `nix develop`), and the `# Run inside nix develop` comments at the top of `scripts/pre_commit_check.sh`, `scripts/sync_reliability.sh`, `scripts/playstore_launch_check_local.sh`, `stalwart-dev/integration_android_test.sh`, `stalwart-dev/run_android.sh`, `stalwart-dev/integration_ui_test.sh`. The `deploy.sh:16` comment about nix profile PATH should be reviewed in context — the actual `PATH` munging may still be needed on the production host.

6. **Leave the systemd-unit `nix run github:dagger/nix/...` line in `DAGGER.md`** alone unless the admin host has actually been re-provisioned — that one is about the server's engine, not local dev tooling.

### Files touched

- `Taskfile.yml` — drop `_nix-check`, rewrite `_preflight`, add `_tools-check` + `doctor` target; fix `website-dev`.
- `hooks/pre-commit` — delete or rewrite.
- `DEVELOPMENT.md`, `README.md`, `AGENTS.md`, `DAGGER.md` — purge dead nix prose.
- Six `scripts/` and `stalwart-dev/` shell scripts — replace comment headers.

### Out of scope

- Re-shaping CI (`.github/workflows/*.yml`) — these run on the self-hosted runner image which already bundles `dagger`/`task`, so they are unaffected by this issue.
- Bumping the pinned `dagger` / `pre-commit` versions; we just propagate whatever `Dockerfile.dev` currently pins so there is a single source of truth.
