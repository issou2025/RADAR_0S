# Decision Integrity Review Packets

## Problem Statement

CodeTalk can recover decision notes and related history for a local change, but
its current review contract can blur materially different situations. A valid
empty result, a failed history lookup, a malformed cached record, and a
generated-only explanation can all appear too similar. Review judgments are not
bound to an immutable candidate, so a later change at the same location can
inherit a stale local judgment. The current product also has no repository-owned
way for a maintainer to turn recovered supporting material into a scoped,
lifecycle-aware confirmed decision record.

Maintainers need to know exactly which change was reviewed, which sources were
successfully consulted, what authoritative decision evidence was found, what was
only supporting material or generated interpretation, and which limitations
prevent a complete claim. They also need a privacy-preserving way to confirm
durable decisions without allowing enrichment, coding agents, or model output to
create repository authority.

## Solution

Introduce a versioned review-packet contract that binds every review to an
immutable change packet. The packet reports operation-scoped source health,
evidence availability, a bounded review result, reliability warnings, coverage,
and packet-bound decision review cards. A valid empty result remains distinct
from a degraded or unavailable source.

Add repository-owned confirmed decision records. Each record is one structured
JSON object with a maintainer-authored statement and rationale, explicit scope,
lifecycle, and minimal approved source references. The local review interface
and an explicit interactive CLI flow may create or update a draft only after
showing the final redacted payload. A record becomes authoritative only after it
is committed and appears in the reviewed change's base version.

Keep review outcomes in minimal local receipts bound to the packet fingerprint.
Raw diffs, paths, source excerpts, sessions, and identity are not persisted in
receipts or automatic analytics. Feedback and aggregate evaluation remain
user-reviewed, sanitized, explicit exports.

## Implementation Decisions

### Architecture and ownership

- Introduce two shared application-level use cases: review-packet construction
  and decision-draft confirmation. CLI, browser, and MCP adapters must not
  reimplement decision classification, source health, redaction, scope matching,
  or packet identity.
- Split packet capture, source health, decision-record storage, receipt storage,
  and adapter rendering into focused modules. Existing modules already near the
  300-line project limit must become thinner rather than absorb these features.
- Keep the deterministic core on the Python standard library. Optional web
  dependencies remain isolated from CLI and MCP imports.
- All external-data parsers remain tolerant. A malformed optional record produces
  a warning and is skipped; an unavailable required input fails the operation.

### Review contract version 2

- Structured review output declares schema version 2.
- The top-level contract contains the change packet, operation-scoped source
  health, aggregate evidence availability, reliability warnings, review cards,
  and bounded analysis metadata.
- Each card contains a packet-bound ID, the reviewed change region, association
  facts, one of the three review results, per-card evidence availability,
  separate authoritative decision evidence, supporting material, generated
  interpretation, provenance, next action, and unresolved or saved judgment.
- `kind` remains in every schema-version-2 response. It maps potential conflict to
  `potential_conflict`, decision evidence found to `decision_context`, and no
  decision evidence found to `no_evidence`. The new `result` field is
  authoritative. Removing the alias requires a future schema version and
  release notes.
- Generated-only material maps to no decision evidence found even though the
  legacy behavior previously labeled it decision context.
- Legacy local judgments without a packet fingerprint remain stored but never
  attach to version-2 cards.

### Change packet identity and coverage

- A committed comparison records verified base and candidate SHAs and the exact
  diff fingerprint.
- A worktree comparison records verified `HEAD` as base and a fingerprint of the
  exact captured worktree diff.
- An unborn repository with readable candidate content remains reviewable with
  no authoritative base decisions and a `version_unverified` warning. A missing
  `HEAD` alone is not a required-input failure when the candidate snapshot is
  valid.
- A supplied diff records its content fingerprint. Base and candidate identity
  remain unverified unless independently proven.
- Packet identity is a SHA-256 digest over a canonical representation of the
  input kind, verified version identifiers, exact captured diff bytes, and a
  sorted omission manifest. Display may shorten the digest; storage and
  comparison use the complete value.
- Candidate content is captured once. Cards, terminal output, browser output,
  JSON, and receipts must derive from that captured snapshot.
- Collection first enumerates candidate paths. It reads regular tracked and
  untracked text content only within fixed limits and never follows symlinks.
- Git path collection uses unambiguous machine-readable delimiters and treats
  unusual names as data. Renames consider both old and new paths for scope
  association. Gitlinks, symlinks, binary content, and unsupported paths are
  omitted with explicit reason codes.
- Initial fixed limits are 60 analyzed hunks, 60 untracked files, 1,000,000
  bytes per untracked file, 5,000,000 total untracked bytes, and 10,000,000
  total captured diff bytes. Limits are constants, documented, and covered by
  tests rather than exposed as new configuration.
- The collector excludes a whole file or hunk when a bound would be crossed; it
  does not create a syntactically partial diff fragment.
- The local packet may list skipped paths and reasons. The persisted receipt
  stores only counts and reason codes.
- A packet with omissions remains usable as a best-effort result but carries
  `snapshot_incomplete` or `analysis_truncated` and makes no whole-candidate
  coverage claim.

### Source health and evidence availability

- Every consulted source reports `complete`, `degraded`, or `unavailable`, plus
  whether it is required, bounded item counts, reason codes when applicable,
  and a concise next action.
- Complete means the requested read finished without known loss, including a
  valid zero-item result. Degraded means some usable material was returned with
  known loss. Unavailable means no usable material was returned.
- Sources not consulted by an operation are omitted rather than reported as
  unavailable.
- Required-input unavailability returns the existing command failure code.
  Optional-source degradation or unavailability returns a best-effort result.
- Evidence availability inventories change records, decision evidence,
  supporting material, and generated interpretation independently of source
  health. It has no sufficiency or confidence score.
- The shared source-health contract applies to every evidence-reading command
  and MCP tool. Human-readable output stays compact when all consulted sources
  are complete and emphasizes limitations otherwise.
- `doctor` reports full-environment readiness but does not stand in for
  operation-scoped health.

### Review results and conflict candidates

- The only system results are `potential_conflict`,
  `decision_evidence_found`, and `no_decision_evidence_found`.
- An active confirmed record of kind `chosen` whose explicit scope intersects a
  changed path yields decision evidence found.
- An active confirmed record of kind `rejected` or `constraint` whose explicit
  scope intersects a changed path yields a potential conflict.
- Potential conflict takes precedence when a changed region intersects multiple
  active records. Otherwise chosen evidence takes precedence over no decision
  evidence found.
- Existing authored `Vibe-Decision` and `Vibe-Rejected` notes remain decision
  evidence through their deterministic Git association.
- `Vibe-Watch` remains a risk or supporting-material signal and does not become
  an active constraint unless a maintainer confirms it as a decision record.
- A potential conflict asserts scope or provenance association only.
  `semantic_match` remains not evaluated, and no LLM or keyword heuristic
  decides whether the candidate truly violates the decision.
- Superseded and retired records remain visible history but cannot raise a
  potential conflict.
- A confirmed record that cites an existing `Vibe-*` note becomes its lifecycle
  overlay. The source key is derived from the commit SHA, note type, and exact
  note text. Valid overlays suppress duplicate standalone signals; invalid
  overlays suppress nothing and produce `decision_record_invalid`.
- Generated narratives, generic session alignment, bare PR references, inferred
  tests, commit subjects, and diffs remain interpretation or supporting material
  and never elevate the result.

### Confirmed decision records

- The canonical repository contract is one JSON file per decision at
  `.codetalk/decisions/decision-<stable-id>.json`.
- Stable IDs use a lowercase UUID-derived hexadecimal identifier and do not
  contain timestamps, text, repository names, or identity.
- Decision-record schema version 1 contains only: schema version, ID, kind,
  statement, rationale, scope, status, sources, and the conditional
  `superseded_by` or `retirement_reason`.
- Kind is exactly `chosen`, `rejected`, or `constraint`. Status is exactly
  `active`, `superseded`, or `retired`.
- Statement and rationale are required non-empty maintainer-confirmed text.
  A missing real rationale leaves the material unconfirmed.
- Scope is either explicit whole repository or a non-empty collection of
  normalized repository-relative exact files and directories. Absolute paths,
  parent traversal, line ranges, regular expressions, and arbitrary glob
  semantics are rejected.
- Directory scope matches on repository path-segment boundaries. Rename
  association checks both names; ordinary prefix matching is not sufficient.
- `superseded` requires a replacement decision ID. `retired` requires a
  maintainer-authored reason. Records are retained rather than deleted.
- A record filename must match its internal ID. Unknown fields, duplicate JSON
  keys, invalid scalar types, self-references, dangling replacements, and
  lifecycle cycles are invalid rather than silently normalized.
- Sources are a bounded non-empty list of repository commit notes, repository
  files, same-repository pull requests, local-session fingerprints, or an
  explicit maintainer statement.
- Repository-native sources store the minimum verifiable locator. Local-session
  sources store only source type, a locally keyed HMAC-SHA-256 fingerprint, and
  an optional approved redacted excerpt. The local key is not stored in the
  repository. Raw session IDs, absolute paths, full prompts, model responses,
  identity, and model metadata are rejected.
- A local-session fingerprint supports correlation only on the installation
  that created it. It is a privacy-preserving locator, not portable proof that
  another maintainer can independently resolve.
- Record files are bounded to 64 KiB, at most 100 scope entries, at most 10
  source references, 1,000 characters for statement, 2,000 for rationale, and
  2,000 for each approved excerpt.
- Record parsing uses the committed base tree rather than worktree files. A
  committed malformed or oversized record is skipped with
  `decision_record_invalid`.
- Untracked, staged, added, or modified worktree records are drafts. They are
  shown locally with `draft_decision_ignored` and cannot affect the candidate
  being reviewed.
- Existing lightweight `Vibe-*` notes remain valid and require no migration.
  Confirmed decision records are for explicit scope, lifecycle, and migrated or
  durable cross-file decisions.

### Decision confirmation and lifecycle updates

- The browser review flow and explicit interactive CLI commands share one
  confirmation service.
- The service validates input, normalizes scope, applies secret redaction,
  constructs the exact final JSON object, and returns it with redaction counts
  and a SHA-256 preview fingerprint without writing.
- A subsequent confirmation writes only when the final canonical payload still
  matches the preview fingerprint. Changed or stale previews are rejected and
  must be shown again.
- The browser requires same-origin loopback requests, the existing origin
  protections, a per-server random write token, and the matching preview
  fingerprint. Responses use no-store caching.
- The CLI shows the exact final record and requires an interactive TTY
  confirmation. Version 1 has no `--yes`, unattended, or MCP write bypass.
- Decision commands support listing and validating committed records and drafts,
  creating a draft, and preparing lifecycle or scope updates. They do not stage,
  commit, push, or delete files.
- Draft writes reject symlinked destination directories, path traversal,
  existing-ID collisions, and any resolved destination outside the repository.
- Creation is exclusive and updates require the previewed prior-record
  fingerprint. The writer revalidates containment at write time and uses an
  atomic same-directory replacement so races or interruption cannot silently
  overwrite a different draft or leave partial JSON.
- Web and CLI updates use the same preview and write protocol as creation.
- Enrichment may prefill candidate material but never invokes the writer.
- MCP exposes read-only listing and retrieval of confirmed decisions and their
  health. It exposes no create, update, confirmation, stage, or activation tool.
- The supported confirmation flow is a deliberate operator checkpoint, not
  proof of human identity. A process with repository write access can still edit
  a draft directly; committed-base authority and normal Git review controls are
  the enforceable boundary.

### Reliability warnings and next actions

- The initial warning vocabulary is limited to `source_degraded`,
  `source_unavailable`, `version_unverified`, `snapshot_incomplete`,
  `history_lookup_failed`, `decision_record_invalid`,
  `draft_decision_ignored`, and `analysis_truncated`.
- Every warning includes its code, an observed fact, affected source or local
  scope, the resulting claim limitation, and a deterministic next action.
- Warning text contains no intent, honesty, laziness, hallucination, or task
  completion judgment.
- Warnings have no confidence score, hidden severity threshold, or automatic
  blocking behavior.

### Judgments, receipts, and feedback

- Human judgments remain `confirmed_conflict`, `intentional_exception`,
  `unrelated`, or `insufficient_evidence`.
- Confirmed conflict requires `action_changed` as a boolean. Only confirmed
  conflict with changed action is a verified interception.
- Intentional exception requires a short reason and the applicable decision ID.
  It applies only to the current packet and never changes decision lifecycle.
- Unrelated reason codes are scope too broad, scope applies but change does not
  violate, record may be stale, or other.
- Insufficient-evidence reason codes are source unavailable, source ambiguous,
  missing rationale, version unverified, or other.
- Other requires bounded free text. Common structured reasons do not.
- A receipt stores the schema version, locally keyed HMAC-SHA-256 project
  identity, complete packet fingerprint, card fingerprint, referenced decision
  IDs, judgment, reason code, action-changed value, bounded exception or other
  text, elapsed review time, coverage counts and reason codes, source-health
  summary, warning codes, and local update time.
- Receipts exclude repository names, paths, raw SHAs other than verified packet
  version identifiers needed locally, diffs, decision text, evidence excerpts,
  session identifiers, and author or model identity.
- Card fingerprints include the complete packet fingerprint, change-region
  identity, and sorted associated decision IDs.
- Existing unbound judgment rows remain untouched and are ignored by version-2
  cards.
- Feedback preview and export use an allowlist. Aggregate pilot summaries may
  contain only counts, durations, evidence types, health statuses, warning
  codes, judgment reasons, action-changed counts, and verified-interception
  counts.
- No feedback or aggregate summary is uploaded automatically. Optional free text
  remains excluded until separately previewed and approved.

### Human-facing review

- The review page shows packet identity, verified or unverified version state,
  coverage declaration, source-health summary, and reliability warnings before
  cards.
- Complete source health remains compact. Degraded, unavailable, incomplete, or
  unverified states are visually prominent and include a next action.
- Each card visibly separates review result, current change, decision evidence,
  supporting material, generated interpretation, provenance, and maintainer
  judgment.
- Generated interpretation always carries a non-authoritative label and cannot
  occupy the decision-evidence region.
- Supporting material offers a confirm-as-decision action. The form requires
  kind, statement, rationale, scope, source preview, and final confirmation.
- After writing, the interface labels the record as a draft and instructs the
  maintainer to review and commit it; it never stages or commits.
- The interface supports lifecycle changes without offering deletion.
- Intentional exception and other judgments reveal only the fields required for
  that selection.
- Existing loopback security, output redaction, CSP, and no-store behavior remain
  mandatory.
- Paths, decision text, commit messages, excerpts, warnings, and all other
  repository-derived strings are rendered through text-safe escaping and never
  interpreted as HTML.
- Desktop and mobile layouts must keep text, source excerpts, warnings, radio
  controls, and action buttons inside their containers without overlap or
  horizontal page scrolling.

## Testing Decisions

- The primary read seam is the shared review-packet builder. Tests use real
  temporary Git repositories and assert the complete external packet contract,
  rather than testing private helper implementation.
- The primary write seam is the shared decision-confirmation service. Tests
  exercise preview, payload fingerprinting, stale-preview rejection, redaction,
  draft creation, update, and Git activation behavior through its public
  application interface.
- Existing review-card, browser-review, feedback, doctor, Git-history, session
  parser, CLI, and MCP tests provide prior art. Extend their external assertions
  instead of duplicating internal unit tests.
- Real-Git packet tests cover committed ranges, tracked worktree changes, staged
  changes, untracked text, unborn repositories, binary files, symlinks,
  gitlinks, unusual path characters, renames, oversized content, external diffs,
  changed worktrees after capture, and deterministic fingerprints.
- Source-health tests inject failures only at source boundaries and verify
  complete zero results, partial degradation, total unavailability, required
  failure, optional best effort, reason codes, and no false no-evidence result.
- Decision-record tests cover every kind and lifecycle, exact-file, directory,
  and whole-repository scope, path-segment matching, normalization, traversal
  rejection, malformed and oversized JSON, duplicate keys, unknown fields,
  source bounds, duplicate IDs, dangling or cyclic lifecycle references, and
  base tree authority.
- Authority tests prove that untracked, staged, and candidate-added records do
  not affect the same packet and that committed records affect the next packet.
- Migration tests prove `Vibe-*` notes remain effective, valid lifecycle overlays
  deduplicate them, and invalid overlays do not suppress evidence.
- Result tests prove chosen scope intersections yield decision evidence found,
  rejected and constraint intersections yield potential conflict, inactive
  records do not trigger, and generated-only material yields no decision
  evidence found.
- Receipt tests prove a changed packet produces different card IDs, legacy
  judgments are ignored, intentional exceptions remain packet-scoped, and
  persisted rows contain no forbidden source-bearing fields.
- Adapter tests stay thin: CLI JSON, terminal rendering, browser rendering, and
  MCP reads must expose the same packet or health contract produced by the
  shared use cases.
- Browser integration tests cover origin and write-token rejection, preview
  invalidation, concurrent-update rejection, text-safe rendering of hostile
  repository strings, visible draft state, conditional judgment fields,
  keyboard operation, and responsive layouts at representative desktop and
  mobile widths.
- Network-denial tests fail if deterministic review, record validation, draft
  confirmation, or source-health collection attempts network egress.
- Compatibility tests assert schema version 2, the authoritative `result`
  values, and the one-cycle legacy `kind` mappings.
- Redaction tests seed synthetic API keys, credentials, private-looking paths,
  session IDs, author data, and model metadata and assert that records, receipts,
  rendered HTML, JSON, and exports honor their separate allowlists.
- Resource-bound tests assert every fixed file, byte, and hunk limit at the
  boundary and one unit beyond it, with complete omission declarations.
- Completion requires the full existing test suite, focused new tests, formatting
  checks, browser QA on desktop and mobile, and local dogfood against multiple
  repositories without committing private evidence.

## Out of Scope

- Semantic conflict classification by an LLM or keyword heuristic.
- Generic AI hallucination, laziness, deception, completion, or productivity
  scoring.
- Automatic commit blocking, merge gates, alert daemons, background monitoring,
  or hidden severity thresholds.
- Owner inference, assignment queues, notifications, escalation routing, SSO,
  RBAC, or team policy administration.
- Hosted PR or CI integration, cross-repository decision aggregation, management
  dashboards, billing, or a proprietary control plane.
- Automatic promotion of enrichment, PR text, test references, session text, or
  generated interpretation into decision evidence.
- Raw packet persistence, raw source export, automatic telemetry, or automatic
  feedback upload.
- Automatic staging, committing, pushing, history rewriting, or record deletion.
- Line-number scope, regular-expression scope, arbitrary glob scope, or
  model-inferred scope.
- Removal of the legacy `kind` field from schema-version-2 output.
- Replacing `Vibe-*` commit notes or forcing existing projects to migrate.
- Making a candidate-added decision authoritative for the same candidate.

## Further Notes

- The repository domain glossary and ADRs 0001 through 0008 are normative.
- Trust-critical privacy, evidence, review, and decision-record behavior remains
  part of the open deterministic product core.
- The specification intentionally strengthens the existing review moment rather
  than introducing a new monitoring product.
- The confirmation UI is a product checkpoint, not an identity system. CodeTalk
  denies automated write capability through its own MCP and unattended
  interfaces; repository permissions and Git review govern direct edits.
- Public implementation artifacts include only the product scope needed to
  build and review the change. Private positioning, customer hypotheses, and
  unpublished evaluation data do not belong in release notes or the public
  issue.
- Formal documentation and repository record filenames remain English and do
  not use dates as document identity.
