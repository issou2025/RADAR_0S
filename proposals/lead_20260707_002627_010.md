## Title

**First-class Jetpack Compose + Single-Activity support for Android Standard SDK**

---

## Body

### Problem

The Android Standard SDK integration model is still **Activity-centric** (2023+), which conflicts with how modern Android apps are built:

- **Single-Activity** apps with Compose `NavHost`
- **Multi-module** architectures (`:app` shell + `:feature-*` modules)
- **No SDK coupling** in `MainActivity`

Today, the [official integration docs](https://razorpay.com/docs/payments/payment-gateway/android-integration/standard/integration-steps/) only document:

```kotlin
class PaymentActivity : Activity(), PaymentResultWithDataListener {
    checkout.open(this, options)
    override fun onPaymentSuccess(...) { }
    override fun onPaymentError(...) { }
}
```

This forces every host activity to implement `PaymentResultWithDataListener` / `ExternalWalletListener`, or forward callbacks manually from Activity → Fragment/Compose (see also [#258](https://github.com/razorpay/razorpay-android-sample-app/issues/258)).

Related: [#363](https://github.com/razorpay/razorpay-android-sample-app/issues/363) has been open since **July 2023** with no official response or SDK update.

---

### Why the current model is inadequate

| Modern Android practice | Razorpay Standard SDK |
|-------------------------|------------------------|
| `MainActivity` is a thin `setContent { }` shell | Activity must implement payment listener interfaces |
| Payment logic in feature modules | SDK tied to Activity class hierarchy |
| Activity Result API (`registerForActivityResult`) | Docs assume `onActivityResult` era |
| Compose callbacks / `rememberLauncher` pattern | No official Compose entry point |
| Comparable to Stripe `rememberPaymentSheet` | No equivalent |

**Stripe (reference):**

```kotlin
val paymentSheet = rememberPaymentSheet { result -> /* handle */ }
paymentSheet.presentWithPaymentIntent(clientSecret)
```

**Razorpay today:** no documented Compose API; integrators must reverse-engineer internal flows.

---

### Undocumented SDK surface (already shipped, not in docs)

Inspecting `com.razorpay:checkout:1.6.39` and the [razorpay-flutter plugin](https://github.com/razorpay/razorpay-flutter/blob/master/android/src/main/java/com/razorpay/razorpay_flutter/RazorpayDelegate.java) shows Razorpay **already supports** an alternate integration path used internally by Flutter — but it is **not documented** for native Android:

1. Launch `CheckoutActivity` directly with `OPTIONS` JSON intent extra
2. Receive result via Activity Result
3. Parse with `checkout.merchantActivityResult(...)` (instance method)
4. Fallback: deprecated static `Checkout.handleActivityResult(...)`

The Flutter plugin also uses reflection to call `merchantActivityResult` as static (which does not match the actual instance API in 1.6.39).

**This proves the SDK team has the pieces for a listener-decoupled flow — they are just not exposed or documented for native/Kotlin/Compose developers.**

---

### Pain points we hit integrating in KMP + Compose

1. **`checkout.open(activity, options)`** requires the calling Activity to implement listener interfaces — breaks single-activity shell pattern.
2. **Direct `CheckoutActivity` launch** requires `OPTIONS` JSON to include `"key"` explicitly (not obvious when migrating from `setKeyID()` + `open()`).
3. **`handleActivityResult` is deprecated**; replacement `merchantActivityResult` exists but is undocumented.
4. **No official sample** for:
   - `rememberLauncherForActivityResult`
   - Delegate-based `PaymentResultWithDataListener`
   - Multi-module / Compose Multiplatform apps

We ended up reimplementing what the Flutter plugin does, without official guidance.

---

### Requested API (minimal)

Please ship a **supported, documented** API along these lines:

```kotlin
// Option A: Compose-first (preferred)
@Composable
fun rememberRazorpayCheckout(
    onResult: (RazorpayPaymentResult) -> Unit,
): RazorpayCheckoutLauncher

// Usage
val checkout = rememberRazorpayCheckout { result -> viewModel.onPaymentResult(result) }
checkout.present(options) // or present(session)
```

```kotlin
// Option B: Activity Result contract (non-Compose)
val launcher = registerForActivityResult(RazorpayCheckoutContract()) { result -> }
launcher.launch(options)
```

**Requirements:**

- Host Activity must **not** implement `PaymentResultWithDataListener`
- Listener/callback passed explicitly to `open()` or `present()`
- Works with **Activity Result API** (not deprecated `onActivityResult`)
- Document `OPTIONS` payload schema when launching `CheckoutActivity` directly
- Document `merchantActivityResult` vs deprecated `handleActivityResult`
- Official Compose + single-activity sample in this repo

---

### Suggested implementation (SDK-side)

Razorpay already has most of this in the Flutter bridge:

| Already exists | Needed |
|----------------|--------|
| `CheckoutActivity` + `OPTIONS` extra | Document + stabilize API |
| `merchantActivityResult()` | Make public, documented, non-deprecated |
| `Checkout.RZP_REQUEST_CODE` | Expose in docs |
| Flutter `RazorpayDelegate` | Port pattern to `razorpay-android` artifact |

`Checkout.open()` could be extended without breaking changes:

```kotlin
fun open(
    activity: Activity,
    options: JSONObject,
    listener: PaymentResultWithDataListener,
    walletListener: ExternalWalletListener? = null,
)
```

Internally: register listener, launch checkout, deliver callbacks to the passed listener — **not** require `activity is PaymentResultWithDataListener`.

---

### Migration / adoption

- **Backward compatible:** keep existing `open(activity)` + Activity implements listener
- **New recommended path:** `rememberRazorpayCheckout` / explicit listener parameter
- Update docs and sample app (`PaymentActivity.kt`) to show Compose + single-activity pattern
- Deprecate docs that only show Activity-implemented listeners as the *only* way
