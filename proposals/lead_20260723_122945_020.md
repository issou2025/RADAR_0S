# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**[Proposal] Extend Kea2 to Support Flutter Apps via Dart Wrapper/Bridge**

*Project description:*
Hi Kea2 team,

I've been exploring the Kea2 codebase and I'm interested in extending it to support Flutter applications. I'd like to propose an approach and get your feedback before I start implementing.

The Problem
Kea2 currently relies on the uiautomator2 driver to fetch native Android view hierarchies and run property-based tests . However, Flutter apps render their entire UI onto a single native FlutterView surface. From the native view hierarchy perspective, the canvas is empty – uiautomator2 simply cannot inspect Flutter widgets like TextField, ElevatedButton, or any widget with a Key .

This means Kea2's precondition system (which uses self.d(resourceId="...").exists to decide when to run property tests) cannot effectively work with Flutter apps today.

Proposed Solution
I propose building a Dart wrapper or custom bridge that exposes Flutter's internal widget tree to Kea2's Python-based property assertion engine. This would allow Kea2 to query widget properties (keys, types, text, etc.) and perform actions (tap, enter text, scroll) on Flutter widgets.

This approach is already proven in the industry:

Appium Flutter Integration Driver embeds a Flutter Integration Server in the app and exposes the widget tree via the Dart VM service protocol

Patrol provides a similar bridge between Flutter widget tests and native platform interactions

Implementation Sketch
My rough plan would be:

Embed a Flutter server (similar to appium_flutter_server) in the app under test

Build an instrumented APK that includes this server

Extend Kea2's driver architecture to support Flutter queries:

Replace/extend U2StaticChecker with a FlutterStaticChecker that queries the widget tree

Replace/extend U2ScriptDriver with a FlutterScriptDriver that performs actions on Flutter widgets

Potential Challenges
Performance: Kea2's current precondition evaluation caches XML hierarchies in <1ms . Querying the Flutter widget tree over the Dart VM protocol will introduce network latency, potentially slowing down the fuzzing loop significantly. This would need careful optimization.

Integration: Kea2's driver injection mechanism expects to interact with self.d using uiautomator2's selector API . I'd need to map Flutter selectors (by key, by text, by type) to a similar API structure to maintain compatibility with existing test scripts.

Questions for the Team
Does this align with the project roadmap? I noticed the roadmap mentions future support for Appium and other UI testing tools. Would a direct Flutter bridge be preferred, or would it be better to wait for Appium support (and use Appium's Flutter driver)?

Are there any existing efforts on this? I want to make sure I'm not duplicating work that's already in progress.

Any pointers to relevant code sections? I'm particularly interested in understanding the driver architecture (U2StaticChecker, U2ScriptDriver, etc.) and where a Flutter extension would best fit.

Would you consider this for the core project, or would it be better as a plugin/extension?

Next Steps
If this sounds reasonable, I'd like to start with a minimal proof-of-concept:

Use appium_flutter_server (or similar) in a demo Flutter app

Build a simple Python wrapper that can query widget properties

Test basic precondition evaluation and action execution

Once the PoC works, I'd integrate it into Kea2's architecture.

I'm available to work on this in the coming weeks. Looking forward to your feedback!
