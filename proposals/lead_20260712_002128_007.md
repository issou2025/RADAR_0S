*Project description:*

## Problem Statement

An operator at Reebaplus can already compose and send a **Console Broadcast** (an official announcement — "Scheduled maintenance Sunday 2am", "New feature live", "Your trial ends in 3 days") to whole businesses from the web console. Today that message only lands in the in-app notification **bell**: a staff member sees it *if and only if* they happen to open the bell. If the app is backgrounded or fully closed — which, on a shared till between customers, is most of the time — nobody is alerted. So the one channel Reebaplus has to reach its users in the moment is, in practice, silent. On top of that, the operator picks a **Severity** (Info / Warning / Alert) when sending, but on mobile that choice is cosmetically dead — every announcement renders as an identical grey bell, so an urgent "Alert" looks exactly like a routine "Info".

From the shop's side: staff miss time-sensitive messages from Reebaplus, and when they do read one there's no visual cue about how important it is.

## Solution

When an operator sends a Console Broadcast, every phone signed into a targeted business gets a real **OS-level push notification** — it appears in the Android/iOS notification tray and alerts the user even when the Reebaplus app is closed or in the background. Tapping the push opens the app straight to the notification, marked read. The push is styled by the announcement's Severity (an "Alert" reads as urgent, an "Info" as routine), and the same Severity styling now shows inside the in-app feed too, with announcements marked by a distinct megaphone icon so an official Reebaplus message is unmistakable.

The user never sets anything up beyond a one-time "turn on notifications?" ask. Nothing about how they sell changes; they simply stop missing messages.

## User Stories

1. As a **cashier**, I want the phone to alert me in the notification tray when Reebaplus sends an announcement, so that I see it without having to open the app and check the bell.
2. As a **CEO**, I want to receive an operator's announcement even when the app is fully closed, so that a time-sensitive message (e.g. "your subscription lapses tomorrow") actually reaches me.
3. As a **staff member**, I want the push to appear whether the app is in the foreground, backgrounded, or killed, so that delivery does not depend on how I happened to leave the app.
4. As a **staff member**, when I tap the push, I want the app to open and show me the announcement (marked read), so that I can read it in full immediately.
5. As a **staff member** who taps a push while the app was killed, I want it to open, let me pick myself and enter my PIN as normal, and *then* show me the announcement, so that the alert never fights the sign-in flow.
6. As a **CEO setting up a new till**, I want a short, plain-language explanation of why notifications help before the system permission prompt appears, so that I can make an informed choice instead of reflexively denying.
7. As a **staff member**, I want to be asked for notification permission only once per device, so that I am not nagged on every launch.
8. As a **staff member** who declined earlier, I want a Notifications toggle in Settings that can turn them on (sending me to system settings if needed), so that changing my mind is easy.
9. As a **manager** on the same shared till as a cashier, I want the device to receive the announcement regardless of who is currently signed in, so that whoever is at the till sees it.
10. As a **CEO with two phones**, I want the announcement to reach both of my devices, so that I get it on whichever I'm holding.
11. As a **staff member**, I want an urgent "Alert" announcement to look visibly more serious than a routine "Info" one — both in the push and in the bell — so that I can triage at a glance.
12. As a **staff member**, I want official Reebaplus announcements to carry a distinct megaphone icon in my notification feed, so that I can tell them apart from ordinary app events (new order, low stock, etc.).
13. As a **staff member**, I want the announcement to still appear in my in-app bell even if the push failed to arrive, so that a flaky network never makes me miss it entirely.
14. As a **staff member**, I want tapping the push to clear its unread badge, so that my notification count reflects what I've actually read.
15. As a **cashier working offline**, I want announcements to arrive and alert me as soon as my connection returns, so that going offline only delays the alert, never drops it.
16. As a **business owner**, I want announcements meant for *my* business to never reach a phone that now belongs to a different business, so that one shop's messages can't leak to another after a device changes hands.
17. As a **staff member who logs out**, I want my device to stop receiving that business's announcements, so that a shared or resold phone doesn't keep getting messages it shouldn't.
18. As an **operator**, I want to keep composing and sending broadcasts exactly as I do now, so that adding push requires no change to my console workflow.
19. As an **operator**, I want the Severity I choose to actually drive how the message looks on the recipient's phone, so that the control I'm given is meaningful.
20. As a **staff member**, I want to dismiss or clear an announcement from the bell as I can any notification, so that read messages don't pile up.
21. As a **staff member**, I want the push title to make clear the message is from Reebaplus (not from my own business), so that I understand it's a platform announcement.
22. As a **CEO**, I want push to work on an Android device out of the box after install, so that the common case needs no extra setup on my part.
23. As the **development team**, I want the push to be an alert only — never the record of the message — so that a push bug can never corrupt or lose notification data.
24. As the **development team**, I want the client push code written cross-platform so that turning on iOS later is a configuration step, not a rewrite.
25. As a **staff member**, I want a stale announcement I never opened to still be readable in the bell days later, so that missing the momentary push doesn't lose the content.
26. As a **staff member on an Android emulator (dev/QA)**, I want push to be verifiable end-to-end without building an APK, so that the team's emulator workflow can test it.

## Implementation Decisions

**Overall shape (ADR 0016).** Firebase Cloud Messaging is the push transport (`firebase_messaging` + `flutter_local_notifications`, with `firebase_core` initialised at app bootstrap). The send pipeline mirrors the existing `send-invite-email` pattern (trigger → `pg_net` → Edge Function, migration 0126). **No web-console change is required** — the console already inserts `console_broadcast` rows; the whole pipeline hangs off that existing insert.

**Send pipeline (server).**
- A new `AFTER INSERT` trigger on `public.notifications`, filtered to `NEW.type = 'console_broadcast'`, POSTs the new row to a `send-push` Edge Function via `pg_net`. `AFTER INSERT` only — the Sync Engine re-pushes rows as upserts, which fire `UPDATE`, so a re-push can never re-send. Fire-and-forget: a push failure never aborts the insert and never blocks sync (exactly as 0126 treats email).
- Secrets in Vault (`project_url`, a push hook secret), read by the trigger; the FCM service-account credential and the hook secret live as Edge Function secrets. An unconfigured project degrades to "no push", never a broken insert.
- New `send-push` Edge Function: validates the hook secret; reads `record.business_id`; resolves **Device Tokens** via `service_role` (`SELECT fcm_token FROM devices WHERE business_id = record.business_id AND fcm_token IS NOT NULL`); builds one FCM **HTTP v1** message per token; sends via an injectable sender; prunes tokens FCM reports `UNREGISTERED`/`NOT_FOUND` (set `fcm_token = NULL`). The message-shaping is a pure `buildFcmMessages(record, tokens)` function.

**Message shape.** Hybrid **notification + data**: `notification { title, body }` + `data { notification_id, type, severity, business_id }`. Background/killed → the OS shows the tray notification automatically; foreground → the client suppresses the auto-display and renders it via `flutter_local_notifications` on a high-importance Android channel. `title` is synthesized server-side from Severity (e.g. *Reebaplus Announcement* / *Reebaplus — Important* / *Reebaplus — Alert*) so it always reads as a platform message; `body` is the operator's `message`.

**Token storage — extend `devices` (0129).** Add `fcm_token text`, `push_permission_granted boolean`, `fcm_token_updated_at timestamptz`. Written by the existing **direct-upsert** sanctioned exception (same as device presence / `sessions`), on the existing cadence (sign-in, app-open-when-online, reconnect) plus on permission-grant and FCM `onTokenRefresh`. Cloud-only: no Drift mirror, never in the outbox.

**Tenant isolation (Invariant #5) via Device Token uniqueness.** A `SECURITY DEFINER` `BEFORE INSERT OR UPDATE` trigger on `devices` nulls `fcm_token` on every *other* row bearing the same token (most-recent-login wins); the client additionally clears its token on logout / business-switch / wipe. Together, a broadcast addressed to one business can never reach a phone that now serves a different tenant.

**Client `PushService` (new, `lib/core/services/`) behind a `PushMessagingPort`.** One seam owns the whole client lifecycle: permission state, token register/refresh → `devices` upsert, the foreground render decision, and tap-intent capture + replay. The port wraps `firebase_messaging` + `flutter_local_notifications` + the permission API; a fake port drives tests (modelled on `InMemoryCloudTransport`, ADR 0001). **New invariant: Push never writes Drift** — the Notification row of record still arrives through the Sync Engine, so the FCM background isolate stays trivial and a push failure loses nothing.

**Permission UX.** Soft-ask rationale sheet on first `MainLayout` reach → OS prompt only on "Turn on" → register token only after grant. Asked once per install (device-level, not per staff); declined → no nag; re-enable via a new **Settings → Notifications** toggle that deep-links to OS settings when already denied.

**Tap routing.** `onMessageOpenedApp` (background), `getInitialMessage` (killed), and the foreground local-notification tap all open the existing **Notifications bell sheet** with the broadcast at top and mark it read. A killed-app tap stores a pending intent consumed **only after** the PIN / "Who's working?" gate and `MainLayout` are up.

**In-app rendering fix (feed).** `_NotificationCard` renders `type = 'console_broadcast'` with a **megaphone** icon coloured by Severity (`info` → primary/blue, `warning` → amber, `alert` → red) via a small shared severity→colour helper; **all other types keep their existing type-based icon/colour** (no regression). The `NotificationModel` already carries `severity`, the Drift table already has the column, and `NotificationsDao` already surfaces `recipient_user_id IS NULL` rows business-wide — so this is display-only.

**Idempotency & fan-out.** One push per broadcast row (`AFTER INSERT`); each device in the targeted business gets its own push; a shared till gets one push regardless of who is signed in (business-wide targeting).

**Optional observability.** A cloud-only `push_sent_at` on `notifications` (mirrors `invite_email_sent_at`, 0126) plus Edge Function logs; the client never reads/writes it, so a re-push upsert can't clobber it.

**Native config.** Android: google-services Gradle plugin + `google-services.json` (gitignored on the public repo with a documented local-placement step — coordinate with the secret-push guard), a `POST_NOTIFICATIONS` manifest entry + runtime request, and a default notification channel. iOS: entitlements/plist added but inert until APNs is configured.

**Docs to update at build time** (not now — these follow the implementation): `architecture.md` (add the **Push** background pathway, the *push-never-writes-Drift* invariant, and the `devices` token columns in the storage model), and `progress-tracker.md`.

## Testing Decisions

A good test asserts **external behaviour**, not wiring: given an input state, what the user/other systems observe. No Firebase platform channels in unit tests — everything testable is behind a port or a pure function.

- **`PushService` against a fake `PushMessagingPort`:** permission-grant → a `devices` token upsert happens; `onTokenRefresh` → a re-upsert with the new token; a foreground message → a local notification is shown (and the OS auto-display is suppressed); a tap intent arriving while the app is gated → it is replayed *after* `MainLayout`, not before; logout → the token is cleared. Assert the observable calls/effects, not internal state.
- **Edge Function:** the pure `buildFcmMessages(record, tokens)` — title-by-severity, body = message, the data block, one message per token, empty result when the token set is empty; token-pruning when the injectable sender simulates an `UNREGISTERED` response.
- **SQL (migration tests):** the `AFTER INSERT` trigger fires the pipeline for `console_broadcast` only, never for other types and never on `UPDATE` (re-push); the `devices` token-uniqueness trigger nulls the token on the older row when the same token is written to a new `(business_id, device_id)`.
- **Widget:** `_NotificationCard` shows the megaphone + severity colour for a `console_broadcast`, and renders an unchanged card for a non-broadcast type (guards against regressing the existing mapping).

**Prior art to mirror:** the existing `NotificationsDao` tests; `InMemoryCloudTransport` (ADR 0001) as the model for the `PushMessagingPort` fake; the `send-invite-email` Edge Function + `0126_invite_email_trigger.sql` for the server pipeline shape; `test/sync/oversell_orphan_notification_test.dart` for the notification-fire pattern.

## Out of Scope

- **iOS activation.** The client code is cross-platform, but the APNs auth key, Apple Developer push capability, and physical-device verification are a tracked follow-up prerequisite — not built or verified in this slice.
- **Pushing app-generated or targeted notifications** (`sale_rejected`, `new_order`, `low_stock`, `staff.*`, stock-approval requests). The pipeline is built general enough to add types via an allowlist later, but only `console_broadcast` opts in now. Per-user/per-device routing on a shared till is deferred with them.
- **Console-side changes:** delivery/read receipts, per-device push stats, scheduling, rich media / CTA links, or a title/image on broadcasts (the schema is `message` + `severity` only). The console keeps its current compose → `console_broadcast` flow.
- **In-app rendering changes for non-broadcast notification types.**
- **Notification preferences beyond a single on/off** — no per-category mute, no quiet hours.
- **Localization of the server-synthesized push title** (English for now; the body is the operator's own text in whatever language they wrote it).
- **Replacing the realtime Broadcast Signal or the in-app feed** — both remain; push is additive.

## Further Notes

- **The reception half already exists.** The `notifications` table already has `severity` (0066) and `recipient_user_id` (0026, NULL = business-wide), is a `SyncedTable` with realtime, and the app already has the bell, the modal, the DAO business-wide handling, and severity on the model. This PRD adds the **OS-push half** and the **severity rendering** — it does not build a new feed.
- **Emulator-friendly.** An Android emulator image with Google Play services receives FCM, so the team's Android-emulator workflow can verify end-to-end with `flutter run` — no APK build needed.
- **Secret handling on a PUBLIC repo.** `google-services.json` and the FCM service-account credential must be handled per the local secret-push guard (gitignore + documented placement / Vault + Edge Function secrets), never committed.
- **Deploy ordering** (as with 0129 / 0126): ship the `devices` ALTER + the two triggers + the `send-push` Edge Function + the Vault/secret setup *before or with* the app build that registers tokens. An unconfigured pipeline degrades to "no push", never a broken insert, login, or sync.
