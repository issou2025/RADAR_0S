*Project description:*
# [security] Daytona upload errors can echo provider bearer tokens into diagnostics

## Summary

The Daytona provider sends the configured Daytona credential as an `Authorization: Bearer ...` header when uploading the sync archive to the toolbox upload endpoint. If that upload fails with an HTTP `4xx` or `5xx`, Crabbox reads up to 4096 bytes of the upstream response body and returns it directly in the error message without applying any Daytona-specific secret redaction.

That means a Daytona toolbox endpoint, reverse proxy, or Daytona API error body that reflects the request authorization header or includes token-shaped JSON can cause the configured `DAYTONA_API_KEY`, `DAYTONA_JWT_TOKEN`, or CLI-derived Daytona credential to appear in Crabbox-generated diagnostics such as CLI stderr, automation logs, copied support output, or local failure artifacts.

This crosses Crabbox's supported credential-protection boundary. The maintainer security policy treats credential exposure beyond the current trusted OS user as in scope, and `docs/security.md` tells operators that Crabbox-generated diagnostics should redact secret-looking strings and configured provider credentials. This report does not depend on hostile tenants sharing a broker or on an operator intentionally running arbitrary commands on their own lease.

## Affected Components

- Target commit: `5001cd1179a42c174da65c8e8a87bf16218aef95`
- Daytona provider archive upload and provider diagnostics
- `internal/providers/daytona/upload.go`
  - `daytonaToolboxHeaders` builds `Authorization: Bearer ` plus `auth.token()`.
  - `uploadDaytonaFileStream` copies failed upload response bodies into returned errors without redacting the configured bearer value or token-shaped JSON fields.
- `internal/providers/daytona/client.go`
  - `daytonaError` appends summarized Daytona `GenericOpenAPIError` bodies to returned errors without a Daytona credential redaction pass.

## Attack Path

1. A trusted operator configures Crabbox with `provider=daytona` and supplies a Daytona API key, Daytona JWT token, or authenticated Daytona CLI profile.
2. Crabbox prepares a Daytona-backed run and calls the toolbox upload path to send the generated sync archive.
3. `daytonaToolboxHeaders` attaches the configured credential to the toolbox upload request as `Authorization: Bearer <token>`.
4. The toolbox upload endpoint, a proxy in front of it, or a Daytona API error path returns a `4xx` or `5xx` response body that reflects the authorization header or embeds the credential in a token-shaped field such as `token` or `access_token`.
5. `uploadDaytonaFileStream` returns `daytona upload archive: <status>: <raw body>` to its caller. The raw body can then be printed by the CLI, captured by automation, copied into support output, or included in local failure artifacts.

The attacker does not need code execution inside the Crabbox process. The boundary crossing is that lower-trust provider-controlled response text is merged into trusted Crabbox diagnostics while still containing the operator's provider credential.

## Impact

A reader of affected diagnostics can recover and reuse the Daytona credential with the same Daytona privileges Crabbox had. Depending on the token scope, that can expose or mutate Daytona sandboxes, workspace data, SSH access operations, and provider resources reachable through the configured account.

The immediate exposure path is diagnostic output rather than an unauthenticated public web response, and exploitation requires a Daytona-backed workflow plus an upstream error body that reflects or includes the credential. That keeps severity in the medium range, but the issue is still reportable because Crabbox is expected to keep provider credentials out of generated diagnostics.

## Recommended Remediation

- Add a Daytona-specific redaction helper that removes:
  - the configured Daytona API key and JWT,
  - `Authorization: Bearer ...` values,
  - JSON `token` and `access_token` fields,
  - URL userinfo and query parameters that can carry secrets.
- Apply the helper to `uploadDaytonaFileStream` before including failed upload response bodies in returned errors.
- Apply the same helper to `daytonaError` before returning summarized `GenericOpenAPIError` bodies.
- Add regression tests that make a fake Daytona toolbox/API endpoint return an error body containing the configured token, a bearer header echo, `{"token":"..."}`, and `{"access_token":"..."}`; assert that returned errors contain redaction markers and never contain the raw credential.
- Consider validating the toolbox proxy URL origin and scheme before sending the bearer header, matching the defensive destination checks used by other provider clients.
