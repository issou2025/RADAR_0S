*Project description:*
## Problem Statement

The quickstart currently stores raw NYC TLC taxi parquet files in MinIO (`s3://data/tlc-trip-data/`), and notebooks read them directly via boto3. There is no abstraction between the raw data and consumers — notebook authors must know the parquet schema, handle differences between yellow and green trip data formats (e.g., `tpep_pickup_datetime` vs `lpep_pickup_datetime`), and decode integer-coded enum columns manually. This makes the data harder to explore, query, and govern, and doesn't leverage the Zetaris data virtualization platform that is already installed on the cluster.

## Solution

Configure the Zetaris NDP Fabric Builder on lab01 to register MinIO as a data source, create a virtual data mart with a unified `taxi_trips` table, and define a semantic layer with business-friendly column names and decoded enum labels. Expose the semantic layer via REST API. Document the entire setup process so it can be reproduced on a fresh Zetaris install.

## User Stories

1. As a data scientist, I want to query taxi trip data through a single unified `taxi_trips` table, so that I don't need to handle yellow and green parquet files separately.
2. As a data scientist, I want columns named in readable snake_case (`pickup_datetime`, `dropoff_datetime`), so that I don't need to remember vendor-specific prefixes like `tpep_` or `lpep_`.
3. As a data scientist, I want a `trip_type` column that tells me whether a row is "yellow" or "green", so that I can filter or group by trip type without checking which source file a row came from.
4. As a data scientist, I want decoded enum labels alongside the original integer codes (e.g., `payment_type=1` + `payment_type_label="Credit Card"`), so that I can explore data without looking up code tables, while still having numeric values for ML feature engineering.
5. As a data scientist, I want to query the semantic layer via REST API from a Jupyter notebook, so that I can use standard Python HTTP libraries without JDBC/ODBC driver dependencies.
6. As a platform engineer, I want a step-by-step setup guide in the repo, so that I can reproduce the Zetaris data mart configuration on a new cluster without tribal knowledge.
7. As a platform engineer, I want validation queries documented, so that I can verify the semantic layer is working correctly after setup.
8. As a quickstart contributor, I want the data source registration documented, so that I understand how MinIO connects to Zetaris as an S3 file source.
9. As a downstream developer working on the Isolation Forest model (APPENG-5662), I want the semantic layer to preserve original numeric column values, so that I can use them directly as ML features without re-encoding strings.
10. As a downstream developer working on the Zetaris notebook (APPENG-5665), I want a working REST API endpoint and a sample validation query, so that I have a concrete starting point for notebook integration.

## Implementation Decisions

### Scope
- This ticket is **Zetaris platform configuration only** — done through the NDP Fabric Builder UI. No code changes to this repository.
- Notebook integration is handled separately in APPENG-5665.

### Data Source Registration
- Register MinIO as an **S3 file source** in the Fabric Builder, pointing at the MinIO instance on the cluster (endpoint TBD — see open questions).
- Bucket: `data`, prefix: `tlc-trip-data/`.

### Trip Types
- **Yellow and green only.** FHV and FHVHV trip types are deferred — they have materially different schemas and are not needed by the downstream ML pipeline (APPENG-5662).

### Virtual Data Mart — `taxi_trips` Table
- Unified table combining yellow and green parquet data.
- A `trip_type` discriminator column added with values `"yellow"` or `"green"`.

### Semantic Layer — Column Naming
- All columns standardized to `snake_case`.
- Yellow/green datetime prefixes unified: `tpep_pickup_datetime` / `lpep_pickup_datetime` → `pickup_datetime`; same for `dropoff_datetime`.

### Semantic Layer — Enum Decoding
All integer-coded enum columns keep the original integer **and** get a companion `_label` column:

| Raw Column | Kept As | Label Column | Decoded Values |
|---|---|---|---|
| `payment_type` | `payment_type` | `payment_type_label` | 1=Credit Card, 2=Cash, 3=No Charge, 4=Dispute, 5=Unknown |
| `RatecodeID` | `rate_code_id` | `rate_code_label` | 1=Standard, 2=JFK, 3=Newark, 4=Nassau/Westchester, 5=Negotiated, 6=Group |
| `VendorID` | `vendor_id` | `vendor_label` | 1=Creative Mobile, 2=Verifone |
| Green's `trip_type` | `hail_type` | `hail_type_label` | 1=Street-hail, 2=Dispatch (renamed from `trip_type` to avoid collision with the yellow/green discriminator) |

### Access Method
- Semantic layer exposed via **REST API**.

### Deliverables
1. Working semantic layer + data mart configuration on lab01.
2. Step-by-step setup guide committed to `docs/zetaris-semantic-layer-setup.md`.
3. Validation SQL query demonstrating the semantic layer works.

## Testing Decisions

Since this is a platform configuration task, testing takes the form of **validation checkpoints** documented in the setup guide:

1. **Data source registration** — Verify Zetaris can see the MinIO S3 bucket and list the parquet files under `tlc-trip-data/`.
2. **Data mart query** — `SELECT * FROM taxi_trips LIMIT 10` returns rows with both yellow and green trip data, with the `trip_type` discriminator populated correctly.
3. **Semantic layer naming** — Columns come back as `pickup_datetime`, `dropoff_datetime` (not `tpep_`/`lpep_` prefixes); all names are snake_case.
4. **Enum decoding** — `SELECT payment_type, payment_type_label FROM taxi_trips LIMIT 5` returns both the integer and the decoded string label.
5. **REST API access** — An external consumer (e.g., `curl` from the cluster) can hit the Zetaris REST endpoint and receive query results as JSON.

## Out of Scope

- **Notebook integration** — Querying the semantic layer from Jupyter notebooks is covered by APPENG-5665.
- **FHV/FHVHV trip types** — Different schemas; deferred to a future ticket.
- **Derived/calculated fields** — No `trip_duration_minutes`, `cost_per_mile`, `speed_mph`, etc. Feature engineering belongs in the ML pipeline (APPENG-5662).
- **Automation of Zetaris configuration** — No Helm hooks or scripts to auto-create the data mart. This is manual UI configuration.
- **Code changes to this repository** — Only the setup doc is committed; no chart/notebook/pipeline changes.
