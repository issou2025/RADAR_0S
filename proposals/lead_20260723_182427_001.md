# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**AI DX: make AnalyseTool fully understandable and iterable for AI agents**

*Project description:*
Today the AI loop is one-directional: a human pastes `LLM.md` into an AI, the AI writes code,
the human copies it into the extensions folder, hits Reload, and manually relays any error back.
This plan closes the loop and improves what the AI can know — using infrastructure that already
exists (script extensions, `ExtensionDiagnostics`, MCP, `GetQueueStatus`).

## Phase 1 — Close the feedback loop (biggest lever)

Goal: an agent connected via MCP can author, load, diagnose, fix and test an extension
**without a human relaying errors**.

- [ ] `GetExtensionDiagnostics` command (ReadOnly, MCP-visible): per-extension compile/load
      errors from `ExtensionDiagnostics`, plus zone/compatible/enabled state
- [ ] Make `ReloadExtensions` MCP-visible (it is the "apply" step of the loop)
- [ ] `WriteScriptExtension` command: write a `.cs` script + `plugin.json` into the DEV zone.
      **Gated behind the existing "C# code execution" setting** (off by default), `Destructive`,
      and refuses to touch the managed zone
- [ ] Agent loop works end-to-end: write script → reload → read diagnostics → fix → invoke the
      new command → verify result (manual test with Claude Desktop; document the flow in LLM.md)

## Phase 2 — MCP beyond tools

- [ ] Expose `LLM.md` (and ONBOARDING) as MCP **resources** — AI clients read the authoring guide
      through MCP, no web access or copy-paste needed
- [ ] Add an MCP **prompt** ("Write an AnalyseTool extension") that pre-loads the guide and
      conventions — one click in the client's prompt menu

## Phase 3 — Schemas and structured errors

- [ ] Audit: every non-hidden command declares `InputType` (or is genuinely argument-less) and
      every input field carries `[Description]`
- [ ] CI check enforcing the above (next to Check-Boundaries) — a command without a schema fails CI
- [ ] Structured command errors: `{ code, message, hint }` from the dispatcher, so agents branch
      on `code` instead of parsing prose (the package validator already writes actionable messages —
      generalize the idea)
- [ ] Later (ties into #70): `OutputSchema` so agents know what a command returns

## Phase 4 — Cookbook (examples beat rules)

- [ ] `samples/cookbook/` with 6–8 small, real, tested commands: read elements; write a parameter
      inside a transaction; progress via `IProgressAware`; a dockable UI page; calling another
      command; an AI-provider call
- [ ] Built in CI (like Acme.Sample) so examples can never rot
- [ ] `LLM.md` references the cookbook for implementations instead of inlining everything

## Small follow-ups

- [ ] Generated `LLM.md` header states host + SDK versions automatically (the AI knows which API
      surface it targets)
- [ ] Write-safety (`IFailuresPreprocessor`) — already a backlog blocker; note it here because a
      Revit failure dialog kills an unattended agent run
- [ ] Periodic "newcomer test": give `LLM.md` to a fresh AI, ask for extension X, log where it
      stumbles, fix the docs (this is how #38 was found)

Done criteria: an MCP-connected agent, starting from zero context, can produce a working
extension and prove it works — with no human in the error loop.
