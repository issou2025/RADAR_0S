# Project Proposal — Bim Modeling

## Project Understanding
You need assistance with: 
**Automatic idempotent regeneration of the filling/opening/voided element**

*Project description:*
Looking at the basic opening creation guide in #8411 , the regeneration Vs apply void confusion doesn't cleanly separate inherent semantic modeling from implementation quirks and convenience shortcuts.

To hide regen functionality, a dependent object (opening/wall) needs to track its driver (door/filling) whenever the driver changes. Also, multiple kinds of changes require different regeneration work.

More details:

<Details>

Let's look how other bim tools solve it.

**1. Hosted relationships are live, not button-triggered**

In Revit, a door is a "hosted" family instance. Move it, and the wall cut moves with it in the same drag — there's no separate "regen" step, because the host/hosted relationship is maintained by the parametric engine on every recompute, not by the user remembering to press a key. The equivalent here would be: dragging a door in the viewport updates the opening's placement live (or at latest, on mouse-release), the same way Blender's own modifier stack updates a mesh as you move its target empty. The two-step "move, then remember to press Shift+G" is a workaround for Blender's object model, not something the user should have to think about.

**2. One operation, not two operators the user has to choose between**

Professional tools don't ask "did you change placement or shape? Pick the right command." The system diffs old vs. new state and decides what work is needed. Concretely: a single "Update" action on a door/window could compare the cached matrix and profile hash against current values, and internally call the cheap placement-only path when only position changed, and the full regenerate-void path when shape changed. The user never needs to know Regen and Apply Void are different code paths — that's an implementation detail currently leaking into the UI as two hotkeys.

**3. Dirty-state is visible, not something you find out by pressing the wrong button**

SolidWorks and Fusion 360 show a rebuild indicator (a small icon in the feature tree, or a lightning bolt) on any node whose dependents are stale. Applied here: if a door has moved since its opening was last synced, the door (or the wall, or the outliner row) should show a small "out of sync" badge. That converts "I moved the door, pressed Regen, but the hole didn't move — why?" from a documentation problem into something the interface tells you before you even ask.

**4. Invariants are enforced by the system, not by user discipline**

"Don't press Apply Void twice" is exactly the kind of rule a robust CAD kernel wouldn't leave to the user. The fix is that Apply Void should be idempotent: detect an existing opening for that filling/wall pair and update it in place rather than create a new `IfcOpeningElement`, every time, unconditionally. The current code already has a partial `existing_opening_occurrence` reuse path — a professional-grade version makes that the only path, with no user-facing failure mode for repeated presses.

**5. Deletion is a guided decision, not a silent orphan**

Revit prompts you when deleting a hosted element's host, or cascades deletion of the hosted element automatically. Here, deleting a door should trigger something like: *"This door has an opening cut into [Wall Name]. Also remove the opening? [Remove opening] [Keep as blank opening] [Cancel]"* — rather than leaving a dangling `IfcOpeningElement` that the user discovers later via a corrupted export or a confusing "Show Voids" toggle they didn't know existed.

**6. Implementation vocabulary stays out of the primary workflow**

"Regen," "Apply Void," "Filling," "Voids Elements" are all schema/operator names, not user intent. A professional tool would surface intent-level actions — *Move*, *Flip Swing*, *Resize*, *Delete* — and handle the IFC relationship bookkeeping invisibly. The Filling/Opening/Voided-Element vocabulary belongs in an "IFC Properties" or "Relationships" inspector panel for power users, not in the primary modeling workflow, where the terms confuse.

-----

What the current door.rst is documenting is largely a set of manual workarounds for an issue of not maintaining the dependency graph automatically. Good documentation can make those workarounds obvious, but the actual UX fix is closer to "make regeneration automatic and idempotent" than "explain the two buttons better."
