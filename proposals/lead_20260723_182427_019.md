# Project Proposal — Permit Drawings

## Project Understanding
You need assistance with: 
**[Enhancement] Provide a credential-free lookup path for DigiKey and Mouser**

*Project description:*
## Summary

`--providers digikey,mouser` cannot retrieve any distributor record for a guest user. Both adapters stop before product lookup when API credentials are absent, even though useful public product data is visible without signing in on the distributors' official product pages.

This makes the advertised multi-distributor flow unavailable out of the box and is particularly confusing because the overall run is still reported as `PARTIAL` with exit code `0` while KiCad CAD is generated from EasyEDA.

## Reproduction

Tested with public beta `v1.1.0b1` (`0998d15`) on Windows 11 / Python 3.13 / KiCad 10.0.3, with no DigiKey or Mouser credentials in the process, user, or machine environment:

```powershell
python -m easyeda2kicad --full `
  --mpn OPA333AIDBVR `
  --providers lcsc,digikey,mouser `
  --output .\libs\project_parts `
  --manifest-json .\build\OPA333AIDBVR.json `
  --show-conflicts
```

Observed provider results:

| Provider | Result |
| --- | --- |
| LCSC | Live anonymous lookup succeeds; record is present |
| DigiKey | `AUTH_MISSING`, operation `oauth`; no record |
| Mouser | `AUTH_MISSING`, operation `part-search`; no record |

The run still generates the symbol, footprint, and STEP/WRL data, because all CAD comes from EasyEDA/LCSC rather than DigiKey or Mouser.

Manual guest checks for the same MPN found that the official DigiKey and Mouser product pages expose public product metadata without login. DigiKey's official ECAD page also allowed a guest KiCad/STEP download during this test. Mouser displayed an ECAD preview, while the actual CAD file download required authentication. The repository currently consumes distributor **metadata**, so the public product metadata is the relevant minimum here.

## Expected behavior

Provide a policy-compliant credential-free lookup path for the public subset of DigiKey and Mouser product metadata, with the authenticated official API remaining preferred when credentials are configured.

If distributor terms or technical restrictions make a guest implementation unsuitable for a provider, the CLI should make that limitation an explicit capability result rather than looking like a successfully queried provider.

## Suggested acceptance criteria

- A no-credential run can return an exact-MPN DigiKey/Mouser distributor record from an officially permitted public source where available.
- Authenticated official APIs remain the preferred, fully supported path.
- No prohibited scraping, authentication bypass, CAPTCHA bypass, or secret persistence is introduced.
- If guest lookup is unsupported or blocked, emit a provider-specific capability code such as `GUEST_LOOKUP_UNSUPPORTED` instead of only `AUTH_MISSING`.
- Add an opt-in strict mode (for example `--require-providers`) so a requested provider that yields no record can make the command non-zero; the current `PARTIAL`/exit-0 behavior can remain the default for compatibility.
- Add live or contract E2E coverage for anonymous mode and document which fields are available anonymously versus with API credentials.
- Never treat a guest-access failure as product `NOT_FOUND`.

## Why this matters

A tool named and documented around LCSC/DigiKey/Mouser should have a useful first-run path for all three providers, or clearly declare at execution time that a provider was not actually queried. Today a user or automation can see exit code `0` and generated KiCad files while silently receiving no DigiKey or Mouser distributor record.
