## Summary

`runPinnedWriteFallback`'s post-rename identity check (`sameFileIdentity` comparing `(dev, ino)`) always fails on rclone FUSE mounts, even with a single process and zero concurrency. The filesystem creates a fresh inode number on every path lookup, including a lookup that happens right after we renamed a file we wrote ourselves. Any overwrite-write through `writeFileInRoot`/`runPinnedWriteFallback` against a root on this kind of mount doesn't work.

I hit this through `openclaw wiki status`, which throws `FsSafeError: fallback target changed during write` on every run against a `memory-wiki` vault stored on an rclone-mounted Google Drive workspace.

## How this differs from #92134 / #85876 / #83740

Those are intermittent `path-mismatch` failures from a real concurrent rename racing a read, and they go away on retry. This doesn't match that pattern since it is one process without concurrency that fails every time. There's no race window to retry past, because the filesystem's `rename(2)` doesn't preserve inode identity the way `sameFileIdentity` expects.

## Root cause

`src/pinned-write.ts`, `runPinnedWriteFallback`:

```ts
tempStat = await handle.stat();                 // identity via held fd, before rename
...
await fs.rename(tempPath, targetPath);
targetStat = await fs.lstat(targetPath);         // fresh path lookup, after rename
if (targetStat.isSymbolicLink() || !sameFileIdentity(targetStat, expectedTempStat)) {
  throw new FsSafeError("path-mismatch", "fallback target changed during write");
}
```

`sameFileIdentity` (`src/file-identity.ts`) requires `left.ino === right.ino` plus matching `dev`. That only holds if `rename(2)` preserves the source's inode at the destination which is true on POSIX filesystems but not guaranteed elsewhere.

### Reproduction (rclone mount, `--vfs-cache-mode writes --dir-cache-time 1m`, Google Drive remote)

Plain `mv` inside the mounted directory, run 5 times in a row, nothing else touching the file:

```
run 2: pre-rename ino=16151713510612461369  post-rename ino=3761494080712081409
run 3: pre-rename ino=16151712411100833158  post-rename ino=3761492981200453198
run 4: pre-rename ino=16151715709635717791  post-rename ino=3761496279735337831
run 5: pre-rename ino=16151714610124089580  post-rename ino=3761495180223709620
```

As a control, I ran the same `mv` test on the container's `overlay` root filesystem, which isn't FUSE:

```
pre-rename ino=186377  post-rename ino=186377   (matches)
```

I also checked whether this was general cache churn unrelated to rename. Stat'ing an unrenamed, unchanged file twice, 2 seconds apart, on the same rclone mount gives the same inode both times. The instability is specific to rename, not the mount being flaky in general.

## I looked for a way to keep the same TOCTOU protection on this kind of filesystem, and couldn't find one

Before reporting this I wanted to check whether there's some inode-independent way to get the same guarantee on a mount like this. One obvious idea was to keep the file descriptor open across the rename instead of closing it first, which is what the code does today, and compare `fstat()` on that same fd before and after, but that didn't address the check's goal.

```
fd9 identity before rename (held fd):          ino=17015438226736127255
fd9 identity after rename  (same held fd):     ino=17015438226736127255   (matches)
fresh lookup of target by path after rename:   ino=7749167912470427203    (different)
```

A held fd does stay stable across the rename. But that doesn't actually buy you anything. To check whether `targetPath` still points at what you wrote, you have to look it up by name again, and on this mount every fresh lookup gets a new inode no matter what. Comparing the held fd against itself would always pass, even in the exact attacker-swap scenario the check is meant to catch. As far as I can tell, this filesystem doesn't have any way to ask whether a name currently points at the same object as a given fd.

I also checked whether rclone has a flag for stable inodes. `rclone mount --help` has nothing for this. Rclone creates inode numbers from a process-local counter (`vfs.newInode()`), not anything tied to a stable remote file ID, so I doubt rclone could fix this on its end.

## Previous context on the check

This is not an ask to weaken or drop the intent behind `sameFileIdentity`. From `docs/security-model.md` and the `66bf772` commit ("harden deepsec-reported filesystem races"), this check catches a same-UID racer swapping the rename target between `rename()` and the verifying `lstat()`.

Instead I'm suggesting using hashing to compare that the written content matches the expectation.

## Suggested next step

At minimum this seems worth a line in `docs/security-model.md` / `docs/root.md` saying the overwrite path in `runPinnedWriteFallback` needs rename to preserve inode identity, and that it won't work at all, not just be more race-prone, on FUSE mounts that don't do that. Rclone is a confirmed example. Happy to dig up more detail if it helps.
