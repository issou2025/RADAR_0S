# Project Proposal — Permit Drawings

## Project Understanding
You need assistance with: 
**Make Engraph semantic memory setup user-friendly with backend manager and Settings UI**

*Project description:*
## Background

Draft PR #1095 provides default-off, owner-safe hybrid retrieval, but it still assumes manual environment and service setup. Nontechnical staff should be able to enable and understand local semantic memory without setting environment variables, managing a service, or touching a shell.

**Explicit dependency:** merge and manually approve draft PR #1095 before implementation begins. This issue builds on its retrieval contract and should not duplicate or bypass its owner-safe fallback behavior.

## Goal

Deliver a fail-safe, device-local Engraph setup experience for Rhythm that:

- makes optional semantic memory understandable and usable by nontechnical staff;
- keeps all indexing and process state local to the signed desktop app;
- preserves the existing FTS path when Engraph is unavailable, unhealthy, disabled, or slow; and
- is self-improving and token-efficient through better retrieval, without making prompts or staff workflows fragile.

## Non-goals

- Bundling, downloading, or auto-installing Engraph in the MVP.
- Editing user environment variables, shell profiles, or global Engraph configuration.
- Touching or reusing the user's normal `~/.engraph` state.
- Indexing arbitrary folders, entire Obsidian vaults, or multiple vaults.
- Replacing FTS entirely, making semantic retrieval mandatory, or adding a pure-semantic mode in this milestone.
- Sending memory content, paths, or secrets to a remote service.

## North star

A self-improving, token-efficient memory experience that remains fail-safe and genuinely useful for nontechnical staff: when local semantic search works, use it; when it does not, Rhythm continues safely with FTS and clearly explains the next action.

# Work package 1 — Device-local Engraph manager and API

> **Implement first.** Work package 2 depends on its persisted configuration, lifecycle operations, and health API. Separate PRs are welcome if the repository workflow prefers, but this issue is the complete source of scope.

## MVP decision: installation and discovery

Do **not** bundle or auto-install Engraph.

The backend manager must:

- detect an executable from `PATH` and reasonable common Homebrew locations;
- accept a user-selected binary path from the UI and validate it before persisting it;
- expose plain-language Homebrew and official-install guidance when no valid binary is found;
- never ask users to edit environment variables;
- never mutate environment variables, shell startup files, or the user's global Engraph setup.

## Data, ownership, and path boundaries

- Create/use a **Rhythm-only** Engraph HOME and index location under the app's Application Support directory.
- Never read, write, migrate, delete, or otherwise touch normal `~/.engraph`.
- Index only Rhythm's canonical agent-memory directory; do not permit arbitrary folder selection or whole-vault indexing in this MVP.
- Resolve and validate canonical paths before use; reject paths outside the approved agent-memory root.
- Treat the executable path and all file paths as untrusted settings input until validated.
- Persist only local configuration/status metadata needed for this feature; never persist secrets or memory contents in configuration or diagnostics.

## Persisted local configuration

Persist local settings in Rhythm-owned app data (not environment variables), including at minimum:

- enabled/disabled preference;
- selected or discovered executable path and discovery source;
- approved canonical agent-memory root;
- Rhythm-only HOME/index location or derived version marker;
- last successful health-check timestamp and sanitized failure category/message;
- setup/rebuild state needed to resume safely after app restart.

Do not require users to create, edit, or export environment variables.

## Backend lifecycle and service contract

Implement a small backend manager/API responsible for:

- asynchronous setup and index preparation so app startup and request handling do not block;
- exact child-process ownership: track only the process Rhythm starts, with PID/identity validation before stop/retry/cleanup; never kill an unrelated process based only on a port or name;
- a Rhythm-owned, loopback-only, authenticated **read-only** Engraph service/API;
- startup, orderly shutdown, retry after recoverable failure, and explicit rebuild behavior;
- a one-second real-search health gate (not merely process/port existence); health is good only if an authenticated search against the approved local index succeeds within the budget;
- safe fallback to existing FTS retrieval on every unavailable, unhealthy, disabled, timeout, parse, permission, or lifecycle failure;
- sanitized diagnostics suitable for Settings, with no secrets, memory content, query text, or sensitive filesystem details exposed in logs or UI.

Use direct process APIs with fixed executable/argument construction and no shell interpretation. Do not expose a network listener beyond authenticated loopback access, and do not expose write, index-arbitrary-path, or administrative operations through the service interface.

## macOS distribution considerations

Design and validate for the signed/notarized app:

- Binary selection, spawning, Application Support writes, and local listener behavior must work from the packaged app, not only a developer shell.
- Document/handle expected macOS code-signing, quarantine/Gatekeeper, sandbox/TCC, entitlement, and filesystem-permission failure states with actionable, non-alarming UI-safe diagnostics.
- Never attempt to bypass macOS security controls. Explain the required user action when macOS blocks a selected executable or directory access.

## Likely systems/files

Exact names may differ after #1095 lands; preserve these responsibilities in the existing API-server and desktop-app architecture:

- `apps/api_server/src/services/` — Engraph manager, process/service lifecycle, health/search adapter, and safe FTS fallback integration.
- `apps/api_server/src/routes/` or existing local API registration — authenticated loopback read-only setup/status/action endpoints.
- `apps/api_server/src/config/` and existing local persistence/database layer — persisted device-local configuration and migration/schema changes if needed.
- `apps/api_server/src/services/memory_retrieval.ts` and the #1095 retrieval integration seam — consume only healthy optional semantic capability; retain owner-safe hybrid/FTS behavior.
- Existing desktop application-support/path utilities and shutdown/startup hooks — Rhythm-only HOME/index path derivation and managed cleanup.
- API-server unit/integration tests for manager, ownership, paths, and fallback behavior.

## Dependencies

- Draft PR #1095 merged and manually approved.
- Existing canonical agent-memory-directory contract and owner-safe retrieval behavior.
- Existing authenticated local API conventions and app lifecycle hooks.

## Acceptance criteria

- [ ] With no Engraph executable, Rhythm remains fully functional on FTS; setup/status reports a clear install path without requiring shell or environment-variable edits.
- [ ] Discovery checks `PATH` and supported common Homebrew locations; a binary picker can choose another executable and persists only a validated path.
- [ ] Rhythm creates and uses only its Application Support HOME/index; `~/.engraph` remains untouched in verified tests/manual inspection.
- [ ] Only the canonical agent-memory directory is indexable; arbitrary, traversal, symlink-escape, and whole-vault paths are rejected or safely normalized to the approved root.
- [ ] Setup/indexing is asynchronous and does not block app startup or normal requests.
- [ ] Managed service access is loopback-only, authenticated, and read-only; external/non-loopback and unauthenticated access fail.
- [ ] The health state requires a real authenticated search to complete within one second. Process existence or an open port alone is insufficient.
- [ ] Disabled, missing, unhealthy, timeout, malformed-result, or permission-blocked Engraph always falls back safely to FTS without breaking prompt construction.
- [ ] Start, shutdown, retry, and rebuild affect only the exact process Rhythm owns; no unrelated process can be terminated.
- [ ] Stored settings, logs, API responses, and UI-safe diagnostics contain no secrets, indexed memory content, user query text, or unnecessary sensitive path details.
- [ ] Signed-app behavior accounts for expected Gatekeeper/TCC/notarization/signing failures without bypassing macOS protections.

## Required tests/evaluation

- Unit tests: binary discovery/validation, path/root enforcement, persisted config round-trip, command construction without shell execution, ownership matching, and sanitized errors.
- Integration tests: managed loopback authenticated read-only service; real-search health success/failure/one-second timeout; start/retry/shutdown/rebuild; all fallback routes preserve FTS.
- Security tests: unauthenticated/non-loopback rejection, no write endpoint, malicious executable/path input rejection, and refusal to stop an unowned process.
- **Required live behavioral test:** configure a valid local Engraph binary against a fixture canonical agent-memory directory, start through Rhythm, verify a real semantic query succeeds through the managed service in <=1 second, then force unavailability and verify the same retrieval request safely uses FTS with no user-facing failure.

# Work package 2 — Flutter Settings setup and health UI

> **Depends on work package 1.** Implement only after the manager/API contract and health states are available.

## Plain-language user journey

1. A staff member opens **Settings → Semantic Memory** and sees that semantic memory is optional and that Rhythm will continue using its standard memory search if it is unavailable.
2. If Engraph is found and healthy, the user sees **Ready** and can enable/disable it without touching environment variables.
3. If it is not found, the user sees **Engraph isn’t installed yet** with concise Homebrew and official-install guidance plus **Choose Engraph…** to select an installed binary.
4. After selection/enabling, the UI shows a non-blocking setup/indexing state and explains that the app is preparing local memory search.
5. When a real search passes, it shows **Ready — local semantic memory is available** and the last checked time.
6. On an actionable failure, it shows a safe explanation and one relevant action (retry, choose another binary, review install guidance, or rebuild). Rhythm continues using standard search.
7. A user can disable optional semantic memory at any time; this stops managed use without deleting unrelated/global Engraph data.

## Required UI states and copy

Use clear, nontechnical language. Avoid raw commands/errors by default; offer install guidance, not a requirement to edit configuration files.

- **Off:** “Semantic memory is off. Rhythm is using standard memory search.” Action: “Turn on”.
- **Not installed/not found:** “Engraph isn’t installed yet. Install it with Homebrew or from the official Engraph instructions, then choose the app here.” Actions: “Install guidance”, “Choose Engraph…”, “Check again”.
- **Selected/validating:** “Checking Engraph…” No blocking modal.
- **Preparing/indexing:** “Preparing local semantic memory. You can keep using Rhythm while this finishes.”
- **Ready:** “Ready — local semantic memory is available.” Include sanitized last-checked metadata. Action: “Check health”.
- **Fallback/degraded:** “Semantic memory is unavailable, so Rhythm is using standard memory search.” Provide the sanitized reason and one safe recovery action.
- **Blocked by macOS:** explain that macOS prevented access to the selected app/file or memory folder and direct the user to the applicable system permission/security action; do not recommend bypasses.
- **Rebuild confirmation:** explain that rebuilding refreshes Rhythm’s local semantic index only; it does not change the user’s normal Engraph data. Require confirmation and show non-blocking progress.

## UI boundaries

- Do not show or expose indexed memory contents, search queries, auth tokens, or secrets.
- Do not present arbitrary index-folder selection, multi-vault selection, service host/port controls, environment-variable controls, or a terminal workflow.
- Reflect backend truth; the Flutter client must not infer “healthy” from a selected path alone.
- Actions must be safe to retry and disabled/guarded while an incompatible lifecycle operation is in progress.
- Follow current Flutter Settings patterns, localization/accessibility conventions, error presentation, and state-management architecture.
