# Project Proposal — Bim Modeling

## Project Understanding
You need assistance with: 
**[FamilyManager v3.0.0] Rename-invariant content-hash dedup + full hash recalculation migration**

*Project description:*
## Context

FamilyManager v3.0.0 introduces rename-invariant content-hash deduplication for loadable and system families. Currently the content hash includes `FamilyName` (actually `Document.Title`), so renamed files produce different hashes and cross-name duplicates are not detected. The goal is to:

1. Exclude the family name from the content hash (bump hash format to v2).
2. Search for duplicates across **all** catalog items by hash, regardless of name.
3. When `MakeActive` points to a version with a different name, update the catalog item name to match that version's file name.
4. Provide full backward compatibility for existing databases by recalculating all old hashes through a one-shot migration dialog with a progress bar.
5. Show a warning icon + tooltip in the batch dialog for cross-name duplicates.

This is a **breaking change** (v3.0.0). Old hashes (format v1) remain in the database but will be recalculated to format v2 during the migration.

---

## Architecture principles (Exa-confirmed)

- **Content identity ≠ name.** The hash is the stable identity; the display name is mutable metadata.
- **Version = immutable snapshot.** Each version stores its own file name and hash. Activation is a pointer switch.
- **Activation adopts the snapshot name.** When switching to a version whose file name differs, the catalog item name must follow.
- **Data repair is not schema migration.** Heavy hash recalculation must be a user-initiated, cancellable, progress-reporting operation, not part of `LocalCatalogMigrator`.
- **Batch open/close is the bottleneck.** Keep documents open in batches (SmartCon already holds 50–100 families open during batch import). Avoid repeated open/close cycles.
- **Progress UI must be STA-threaded.** Use `Dispatcher.Invoke(DispatcherPriority.Background)` updates while Revit API work stays on the Revit main thread.
- **SQLite updates should be chunked.** Use WAL mode, `busy_timeout`, and transactions per batch to avoid long locks.

### 8. FamilyManager — hash recalculation service

**New file:** `src/SmartCon.FamilyManager/Services/LocalCatalog/CatalogHashRecalculationService.cs`

**Algorithm:**
1. Query all versions with files that need recalculation:
   ```sql
   SELECT ci.id AS itemId, ci.name AS itemName, ci.family_source,
          cv.id AS versionId, cv.version_label, cv.revit_major_version,
          cv.content_hash, cv.hash_format_version,
          ff.id AS fileId, ff.relative_path, ff.file_name
   FROM catalog_items ci
   JOIN catalog_versions cv ON cv.catalog_item_id = ci.id
   JOIN family_files ff ON ff.id = cv.file_id
   WHERE cv.hash_format_version IS NULL OR cv.hash_format_version != 2
   ORDER BY ci.id, cv.version_label;
   ```
2. Process in batches of **50 files** (matching SmartCon batch import size).
3. For each batch:
   - Open all files in the batch before computing any hash (avoid repeated open/close).
   - Compute hashes via `IFamilyContentHasher`.
   - Close all documents.
   - Update SQLite in a single transaction per batch (50 versions).
   - Update `catalog_items.content_hash` when the processed version is the current active version.
   - Report progress.
   - Check cancellation before each batch.
4. On error for a single file, log `Warn` with `[Action: ...]`, skip the file, and continue.
5. Return `CatalogHashRecalculationResult`.

**SQLite settings:**
- `PRAGMA journal_mode = WAL;`
- `PRAGMA synchronous = NORMAL;`
- `PRAGMA busy_timeout = 5000;`

## Performance notes

| Operation | Cost |
|---|---|
| Hash search | O(1) via `ix_catalog_versions_content_hash`. |
| Name search | O(log n) via `ix_catalog_items_normalized_name`. |
| Hash compute | Already done during import; migration is the same work. |
| Migration batch | 50 files opened, held, computed, closed, then one SQLite transaction. |
| SQLite writes | WAL + chunked transactions minimize locking. |

## Edge cases and mitigations

| Edge case | Mitigation |
|---|---|
| File fails to open | Log `Warn [Action: ...]`, skip, continue. |
| User cancels | Roll back current batch; remaining v1 hashes recalculated on next startup. |
| Mixed Revit versions in DB | `OpenDocumentFile` may trigger upgrade UI; investigate `OpenOptions` for silent mode if possible, otherwise document it. |
| System families | Recalculate via `ExtractFromProject` using `type_unique_id` from `family_types` and `revit_category` from `catalog_items`. |
| Rename after import | `RenameFamilyFilesAsync` only renames current version; historical names remain in `family_files` per version. Activation correctly adopts the old version's file name. |
