**Security: reject insecure HTTP URLs in transaction image flows**

*Project description:*

## Problem

The current transaction-image presentation flow accepts both `http://` and `https://` remote values as valid network image sources. This conflicts with the Production HTTPS-only release rule and can allow an insecure cleartext image URL to be loaded when a legacy or malformed value reaches the client.

## Confirmed locations

- `lib/features/transactions/presentation/functions/show_transaction_image_preview.dart`
- `lib/features/transactions/presentation/widgets/table/transactions_table_row.dart`
- `lib/features/transactions/presentation/widgets/details/transaction_proof_image_preview.dart`

The Android main manifest also declares an external `http` VIEW query alongside `https`; its necessity should be reviewed as part of the focused fix:

- `android/app/src/main/AndroidManifest.xml`

## Required behavior

- Production transaction image/document remote URLs must use HTTPS.
- Supabase signed URLs must continue to work.
- Local Windows/Android file paths used before upload must continue to work.
- Cloud storage object paths such as `companyId/transactions/...` must continue to resolve through signed URLs.
- Invalid or insecure remote URLs must produce a safe localized user-facing failure rather than attempting a cleartext network request.
- Do not weaken Android network-security settings or enable global cleartext traffic.

## Architecture and localization constraints

- Keep URL validation outside Domain if it depends on platform/network representations.
- Do not introduce Supabase access into UI/Cubit layers beyond existing approved patterns.
- No user-facing hardcoded text; add English and Arabic localization entries together where needed.
- Preserve Demo Mode and Live Mode behavior.
- Do not alter the current Microsoft Store submission as part of this issue.
