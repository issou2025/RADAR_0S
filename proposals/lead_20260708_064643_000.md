### Summary

EF Core throws `DbUpdateConcurrencyException` whenever a generated `UPDATE`/`DELETE` reports 0 rows affected instead of the expected 1 â€” even with **no** optimistic concurrency token configured. The `User` entity in this project has no concurrency token (no `xmin`, `RowVersion`, `[Timestamp]`, `[ConcurrencyCheck]`, `IsConcurrencyToken`, or `UseXminAsConcurrencyToken`), so this exception is telling us the row was already gone (or never existed by that PK) at the moment `SaveChangesAsync` executed â€” not that a version stamp changed.

**Confirmed reproduction path:** the failure occurs on a single click, but a single user action results in `GET /api/auth/logout` being called more than once at the server. The first call succeeds and deletes the row; subsequent calls, arriving while the first is still in-flight or right after it completes, find 0 rows to delete and throw.

**Agreed fix approach:**
- Change `LogoutUser` from `[HttpGet]` to `[HttpPost]` and make it idempotent (treat "row already gone" as success). This eliminates the root cause â€” `POST` prevents browsers and prefetchers from issuing the request multiple times.
- Add a unique index on `Users.HashedSessionKey` in both migration chains (SQL Server and PostgreSQL). Verified to work identically on both providers â€” see Decisions section for the concrete proof.

### Key Discoveries

- The logout route is `GET /api/auth/logout` (`Controllers/AuthController.cs:90`), which is inherently unsafe for a destructive action and prone to being invoked multiple times per user click by browser prefetchers, speculative fetch, or HTTP client retries.
- **Confirmed:** the failure reproduces on a single click because `GET /api/auth/logout` is called more than once per logout action. The first invocation deletes the row; subsequent concurrent invocations find 0 rows and throw `DbUpdateConcurrencyException`.
- `LogoutUser` performs **two** `FetchByContext` reads (one inside `Authentication.IsAuthorized`, one explicit) before calling `Users.Remove(user)` + `SaveChangesAsync()` â€” a wide TOCTOU window with no transaction (`Controllers/AuthController.cs:101,103,106,107`).
- The `User` entity has **no concurrency token**. The `DbUpdateConcurrencyException` is EF Core's default reaction to `affectedRows != expectedRows`, not a version-stamp mismatch. Verified across `Models/User.cs`, `Context/RevitIfcContext.cs`, `Context/PostgreSQLRevitIfcContext.cs`, and both migration snapshots.
- The PostgreSQL migration for the `Users` table (`Migrations/PostgreSQL/20260707114647_InitialCreate.cs:64-82`) is structurally identical to the SQL Server version â€” same PK (`Id uuid`), same columns, no additional concurrency column. So the exception is **not** a schema/migration bug from the PostgreSQL port.
- The exception surfaces on PostgreSQL specifically because it is the currently-configured provider. The same bug exists on SQL Server; it just hadn't been observed there.
- Each login creates a **new** `User` row keyed by a fresh `Guid` and a fresh `HashedSessionKey` (`Controllers/OauthController.cs:55-69`). There is no upsert on `AutodeskId`. If a stale cookie is presented after the row was already deleted, the second logout attempt with the same cookie will still hash to the same key and try to delete a row that no longer exists.

### Detailed Findings

#### 1. The immediate failure point â€” `LogoutUser`

`Controllers/AuthController.cs:89-119`

```csharp
[HttpGet]
[Route("api/auth/logout")]
public async Task<ActionResult> LogoutUser()
{
    try
    {
        var url = ...;
        if (Authentication.IsAuthorized(HttpContext, RevitIfcContext))   // read 1
        {
            var user = RevitToIfcScheduler.Models.User.FetchByContext(HttpContext, RevitIfcContext);   // read 2
            if (user != null)
            {
                RevitIfcContext.Users.Remove(user);
                await RevitIfcContext.SaveChangesAsync();               // DELETE
                HttpContext.Response.Cookies.Append(AppConfig.AppId, "");
            }
        }
        return Redirect(url);
    }
    catch (Exception ex) { ... }
}
```

- `Authentication.IsAuthorized` at `Utilities/Authentication.cs:27-38` calls `User.FetchByContext` internally and only returns a bool â€” the tracked entity from that first query is discarded.
- The subsequent explicit `FetchByContext` at line 103 issues a second `Users.First(u => u.HashedSessionKey == hash)` query.
- `Users.Remove(user)` marks the entity as `Deleted` in the change tracker. `SaveChangesAsync()` then issues `DELETE FROM "Users" WHERE "Id" = @id`. EF Core verifies exactly 1 row is affected; 0 rows â†’ `DbUpdateConcurrencyException`.

#### 2. Why the row can be gone at DELETE time

- **`GET` semantics â€” confirmed as the trigger.** The failure reproduces on a single user click, but the server receives multiple `GET /api/auth/logout` requests for that one action. HTTP `GET` is defined as safe (no side effects) and idempotent â€” browsers, prefetchers, link previewers, antivirus scanners, and HTTP clients are therefore entitled to issue it multiple times without warning. ASP.NET Core has no built-in protection against this on a `GET` route. The first concurrent request to complete its `DELETE` succeeds; all subsequent ones hit 0 affected rows.
- **No transaction.** There is no `BeginTransaction`/`SaveChanges`/`Commit` scope around the read + delete. Two concurrent logouts each observe the row and both attempt to delete it.
- **No idempotency guard.** The code does not use `ExecuteDelete`/`ExecuteDeleteAsync` (EF Core 7+) with a filter that treats "already gone" as success, nor does it catch `DbUpdateConcurrencyException` to short-circuit.
- **Stale cookie replay.** After a successful logout, the cookie is cleared to `""` server-side, but the client may still submit an old cached copy in a follow-up request (back button, cached tab). The old hash still points at a now-deleted row â†’ same failure.

#### 3. Related write paths that share the same TOCTOU pattern

Any endpoint following the `IsAuthorized` â†’ `FetchByContext` â†’ `Users.Update`/`Users.Remove` â†’ `SaveChangesAsync` pattern has the same class of bug. If a logout completes between the fetch and the save, the update/remove will hit 0 rows and throw:

| Location | Operation | Trigger |
|---|---|---|
| `Controllers/AuthController.cs:61-62` | `Users.Update(user)` + `SaveChangesAsync()` | `GET /api/auth/user` |
| `Utilities/TokenManager.cs:125-126` | `Users.Update(user)` + `SaveChangesAsync()` | Token refresh path (`ThreeLeggedTokenGetter.GetToken`) |
| `Controllers/OauthController.cs:68-69` | `Users.Add(user)` + `SaveChangesAsync()` | New session â€” not exposed to this bug |

#### 4. What was ruled out

- **Optimistic concurrency mis-configuration.** The `User` entity carries no concurrency token in the C# model (`Models/User.cs:41-98`), no Fluent API concurrency call in `RevitIfcContext.OnModelCreating` (`Context/RevitIfcContext.cs:45-53`), and no such column in either provider's migration (`Migrations/PostgreSQL/20260707114647_InitialCreate.cs:64-82`, `Migrations/RevitIfcContextModelSnapshot.cs:178-214`). The exception is purely a row-count-mismatch signal.
- **Schema drift between SQL Server and PostgreSQL.** Both providers project the identical columns and PK; PostgreSQL uses `uuid`/`text`/`timestamp with time zone`, SQL Server uses `uniqueidentifier`/`nvarchar(max)`/`datetime2`. No column-name casing plugin is registered, so `"Id"`/`"HashedSessionKey"` are the on-disk names in both databases.
- **Background job interference.** No Hangfire job or `Schedule.Run()` path writes to the `Users` table. `HangfireJobs.PollConversionJob` touches only `ConversionJobs`; `Schedule.Run()` builds its own service provider but only reads/writes `Schedules`/`ConversionJobs`.
- **Session-key rotation racing the delete.** `HashedSessionKey` is written exactly once at OAuth callback (`Controllers/OauthController.cs:59`) and is never rotated. Not a factor.

#### 5. Design context worth capturing

- **Sessions live in the DB, keyed by SHA-256 of a random `Guid` cookie value** (`Models/User.cs:293-309`). Every request that needs auth performs a live `Users.First(u => u.HashedSessionKey == hash)` query. There is no index on `HashedSessionKey` in either migration â€” a separate perf concern, not causing this bug, but will be fixed in this PR.
- **Provider selection** (`Startup.cs:79-140`): reads `DatabaseProviderConfiguration`; for `PostgreSQL`, registers `PostgreSQLRevitIfcContext` and aliases `RevitIfcContext` to the same instance via `services.AddScoped<RevitIfcContext>(sp => sp.GetRequiredService<PostgreSQLRevitIfcContext>())`. Controllers inject `RevitIfcContext`, so the same instance is used throughout a single HTTP scope â€” no cross-context change-tracking issues, but also no cross-request coordination.
- **No `[Authorize]` middleware** â€” every controller action manually calls `Authentication.IsAuthorized(...)` at the top of a `try` block, and the `Hangfire` dashboard's `MyAuthorizationFilter` (`Startup.cs:220-229`) constructs a separate `ServiceProvider` for its own DB reads. Not causing this bug, but relevant to any fix that touches the auth pipeline.

### Decisions

**Logout â†’ `POST` (confirmed).** Switching `LogoutUser` from `[HttpGet]` to `[HttpPost]` is the agreed approach. `POST` eliminates the root cause â€” browsers never prefetch or speculatively retry `POST` requests, so a single user click produces exactly one server hit. The frontend logout trigger must be updated from a plain link to a form submit or `fetch({ method: 'POST' })` call. Anti-forgery (`[ValidateAntiForgeryToken]`) is good practice to add at the same time to prevent cross-origin logout attacks, though risk is low for an internal tool.

# Fix DbUpdateConcurrencyException on Logout Implementation Plan

## Overview

`GET /api/auth/logout` is called more than once per user click (browser prefetch/retry), causing the first request to delete the session row while concurrent requests find 0 rows and throw `DbUpdateConcurrencyException`. The fix changes logout to `POST` (preventing repeated browser calls), rewrites the delete as an idempotent predicate-based `ExecuteDeleteAsync`, and hardens the two other write paths (`GetUser` and token refresh) against concurrent logout. A unique index on `Users.HashedSessionKey` is added to both migration chains as a correctness guard and perf win.

## Current State Analysis

### Key Discoveries:
- `LogoutUser` is `[HttpGet]` at `Controllers/AuthController.cs:89` â€” browsers/prefetchers freely repeat `GET` requests
- Two redundant `FetchByContext` reads happen before the delete (`Controllers/AuthController.cs:101` via `IsAuthorized`, then explicit at line 103), creating a wide TOCTOU window
- `GetUser` (`Controllers/AuthController.cs:61-62`) does `Users.Update` + `SaveChangesAsync` with no guard against concurrent logout
- `TokenManager.RefreshThreeLeggedToken` (`Utilities/TokenManager.cs:125-126`) does the same `Users.Update` + `SaveChangesAsync` pattern
- Frontend logout at `ClientApp/src/Utilities/ApiCalls.ts:23-25` uses `SimpleGet`, which must change to `SimplePost`
- No unique index on `Users.HashedSessionKey` in either migration chain â€” each auth lookup does a full table scan on this column

## Desired End State

- `POST /api/auth/logout` replaces `GET /api/auth/logout`; repeated calls for the same session all succeed (idempotent) without throwing
- `GetUser` and `RefreshThreeLeggedToken` handle a concurrent logout gracefully without surfacing `DbUpdateConcurrencyException` to the caller
- Both migration chains (SQL Server and PostgreSQL) have a unique index on `Users.HashedSessionKey`
- Frontend sends a `POST` to `/api/auth/logout` instead of a `GET`

**Verification:** click logout once; no `DbUpdateConcurrencyException` appears in server logs; user is redirected to the home/login page.

## What We're NOT Doing

- Not adding ASP.NET Core `[ValidateAntiForgeryToken]` (low risk for internal tool; out of scope)
- Not refactoring `Authentication.IsAuthorized` to return the resolved `User` entity (would reduce redundant queries, but is a broader refactor out of scope for this bug fix)
- Not changing the OAuth login flow (`OauthController`) â€” `Users.Add` is not exposed to this race
- Not adding distributed locking or per-session mutexes

Change `LogoutUser` to `[HttpPost]` and replace the read-then-delete with a single `ExecuteDeleteAsync` predicate call (EF Core 8 â€” already in the project). This is atomic at the DB level and inherently treats "0 rows deleted" as success. Harden the two remaining TOCTOU write paths by catching `DbUpdateConcurrencyException`. Update the frontend to POST. Add the unique index migrations.
