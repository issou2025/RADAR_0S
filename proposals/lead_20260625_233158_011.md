# Project Proposal — Python Automation

## Project Understanding
You need assistance with: 
**pytest-twister-harness: QemuAdapter does not wait for gcov_dump to complete before ending process.**

*Project description:*
### Describe the bug

Background:
I'm turning on code coverage for a suite of pytest tests.
- qemu-x86 target
- configured for coverage via console, not semihosted

Observed behavior:
I'm getting truncated gcov_dump outputs when running console tests i.e. I can see GCOV_DUMP_START but no paired GCOV_DUMP_END message. 

Analysis: 
I'm doing a bunch of silly things in my setup that are exacerbating this situation like creating a DUT and then not actually interacting with it, but the flaw is there, I think. Unlike the native twister behavior in harness.py:process_test:181, the pytest-twister-harness QEMUAdapter stdout is monitored as a FIFO by the DeviceConnection class, which is just blindly dumping to a log file without looking for a coverage dump start in the log stream. When the test finishes, QEMU will be closed out regardless of dump state.

Workarounds:
I added an extra pytest teardown test fixture (attached) to wait for the GCOV_START and GCOV_END as part of the test suite rather than as part of the twister automation. This works well and also allows for test harness->dut dump trigger coordination.

Additional thoughts:
There's nothing I can see that would prevent this from being a problem in Semihosted mode as well.
