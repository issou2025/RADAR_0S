# Project Proposal — Unknown

## Project Understanding
You need assistance with: 
**[Bug]: stuck-session recovery aborts healthy runs while the gateway's own event loop is stalled (memory pressure/swap) — gate recovery on liveness**

*Project description:*
### Summary

Stuck-session recovery classified two healthy cron runs (and one user session) as `stalled_agent_run` and aborted them (`action=abort_embedded_run`) while the **gateway's own event loop was stalled for minutes** under host memory pressure. The runs were fine — their progress notifications simply weren't being processed by the frozen event loop. Recovery acted on telemetry that its own liveness monitor had just declared unreliable.

### Environment

- OpenClaw v2026.7.1 (gateway, systemd user unit), Node 22
- Ubuntu, 4 vCPU / 8 GiB RAM host, ~30-user Slack workspace, codex backend (embedded runs)

### What happened (2026-07-14, times UTC)

A nightly `openclaw update` + restart compressed the morning cron backlog; 5–6 embedded-run session trees ran concurrently. The service cgroup peaked at **6.57 GiB memory + 15.19 GiB swap** on an 8 GiB host → kernel swap storm (sar: major faults up to **11,168/s**, swap 74 % full).

The gateway's liveness monitor saw its own starvation clearly:

```
06:06:14 [diagnostic] liveness warning: reasons=event_loop_delay interval=30s eventLoopDelayP99Ms=1422.9 ...
06:10:23 [diagnostic] liveness warning: ... eventLoopDelayP99Ms=11416.9 eventLoopUtilization=1 cpuCoreRatio=0.337
06:14:49 [diagnostic] liveness warning: ... eventLoopDelayP99Ms=57445.2 eventLoopUtilization=1 cpuCoreRatio=0.143
06:20:12 [diagnostic] liveness warning: ... eventLoopDelayP99Ms=161463.9 eventLoopUtilization=1 cpuCoreRatio=0.121
06:37:13 [diagnostic] liveness warning: ... eventLoopDelayP99Ms=642634.5 eventLoopUtilization=1 cpuCoreRatio=0.028
```

(`eventLoopUtilization=1` at `cpuCoreRatio=0.028` = blocked on page faults, not computing.)

Minutes later, the stuck-session recovery in the same diagnostics subsystem treated the resulting absence of progress notifications as an agent-side stall and aborted the runs:

```
06:37:14 [diagnostic] stalled session: sessionId=7ecfa863-… state=processing age=2029s
         reason=active_work_without_progress classification=stalled_agent_run
         activeWorkKind=model_call lastProgress=codex_app_server:notification:item/agentMessage/delta
         lastProgressAge=741s cronJob="belegsammel-pruefen-datev-upload (täglich 08:00)" recovery=checking
06:40:11 [diagnostic] stuck session recovery: sessionId=7ecfa863-… action=abort_embedded_run
         aborted=true drained=false released=1 stopped="belegsammel-pruefen-datev-upload (täglich 08:00)"
06:40:11 [diagnostic] stuck session recovery: sessionId=e7c29011-… action=abort_embedded_run
         aborted=true drained=false released=1 stopped="daily-goc-audit-and-groom"
```

An interactive user session had already been aborted the same way at 06:16:52 (`age=376s`, `lastProgress=embedded_run:started`). User-visible result: cron failure DMs (`AbortError: agent run aborted`) and a silently dropped user turn.

### Why this is a bug

"No progress observed" is not evidence of a stalled run when the observer itself isn't scheduling. With `eventLoopDelayP99Ms` around 642,000 ms, every `lastProgressAge` below ~11 minutes is measurement noise — yet the classifier compared those ages against its normal stall thresholds and escalated straight to `abort_embedded_run`, killing healthy work and losing the runs' results.

### Proposal

1. **Gate stalled-classification (and especially `abort_embedded_run`) on the gateway's own liveness:** if recent event-loop delay (p99 or max over the classification window) exceeds a fraction of the stall threshold, treat progress ages as unreliable — skip the abort and re-check after a healthy interval, or subtract the observed loop delay from `lastProgressAge`.
2. **Make the stall/abort thresholds configurable**, in the spirit of #87441 (configurable memory-pressure thresholds).
3. This composes with the recovery-ladder idea in #86159 — a liveness gate is the "don't act on unreliable telemetry" rung before any steer/nudge/abort.
