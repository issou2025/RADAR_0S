## Problem and evidence

`test_request_headers_handler.py` is a 1,902-line umbrella module with 48 test
functions and 54 collected pytest cases. It currently owns several independent
request-hook contracts:

- request-stream installation, buffering, and probe-metadata cleanup;
- platform API upstream admission and destination binding;
- connector upstream admission, TLS evidence, and host-policy rejection; and
- stream-safe firewall auth, connector intent, fallback, and cancellation cleanup.

The module docstring describes only request-stream setup, while most of the file
protects other security-sensitive behavior. Related admission scenarios are also
spread across distant sections and repeat substantial registry and flow setup.
The file has changed in 15 commits since it was introduced on June 17, so the
mixed ownership is active maintenance debt rather than a hypothetical size
concern.

The current 54 collected cases pass. This is not a production behavior bug, and
file length alone is not the reason to change it; the problem is that unrelated
contracts share one catch-all module and helper/import surface.

## Decision

Proceed as one test-only refactor PR. Split the umbrella module by observable
request-hook responsibility so each security contract has a clear test owner.
Do not change production behavior or reduce coverage as part of the move.

## Chosen direction and constraints

- Keep these as integration tests through the real `requestheaders()` and
  `request()` hook entry points and real mitmproxy flow objects.
- Preserve existing test names and security-relevant assertions where practical.
- Preserve all 54 currently collected pytest cases.
- Keep distinct security cases explicit, including platform bypass, prior-client
  binding, unverified TLS, public-destination rejection, auth fallback, and
  cancellation cleanup.
- Reuse existing focused fixtures and helpers. Extract only small helpers with a
  stable shared meaning; do not introduce a generic requestheaders scenario
  builder or test framework.
- Parameterize only cases that have the same lifecycle and expected behavior.
  Large, bounded, and unknown-length requests must remain distinguishable when
  they exercise different header/request hook paths or cleanup contracts.
- Update the mitm-addon testing guide's test-file ownership table to describe the
  resulting modules.

## Alternatives considered

- **Keep the current file:** avoids path churn but leaves the active ownership and
  discovery problem unresolved.
- **Introduce a generic flow/registry scenario builder:** could remove more lines,
  but would hide security-relevant preconditions and create a new shared
  maintenance hotspot.
- **Split only by file size or into many small files:** reduces line counts without
  establishing durable behavioral ownership.

## Risks and mitigations

- Moving tests can expose hidden import-order or module-state dependencies. Run
  the full mitm-addon suite after focused checks.
- Test node paths will change. Preserve function names and update the documented
  file inventory so failures remain searchable.
- Combining similar-looking body-framing cases can erase phase-specific
  assertions. Consolidate only cases proven behaviorally equivalent.

## Acceptance criteria

- Replace the catch-all module with focused modules whose names and docstrings
  clearly separate request streaming, platform API admission, connector
  admission, and stream-safe auth/fallback ownership.
- Preserve all 54 currently collected pytest cases and their security-relevant
  assertions.
- Keep the distinct security cases listed above explicit and easy to locate.
- Reduce repeated setup only where a focused helper or descriptive
  parametrization improves readability without hiding lifecycle differences.
- Update `docs/testing/mitm-addon-testing.md` with the resulting test ownership.
- Pass Ruff formatting and linting, basedpyright, the focused moved tests, and the
  full mitm-addon pytest suite.
