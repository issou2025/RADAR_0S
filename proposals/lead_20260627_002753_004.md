## Project Understanding

**Fix iOS Firebase SPM Deployment Target**

*Project description:*
## User Story
As an iOS developer, I want the app's iOS deployment target to match Firebase Swift Package requirements, so that simulator and device builds do not fail with Target Integrity errors.

## Context
iOS builds can fail because Firebase Swift Package products require iOS 15.0+, while one Xcode project configuration still advertises iOS 13.0. The Podfile already sets iOS 15.0, so the fix should align the Xcode project settings and avoid unnecessary dependency churn.

## Execution Plan
Plan — Align iOS Deployment Target For Firebase SPM

Problem: iOS builds can fail with Target Integrity errors because Firebase Swift Package products require iOS 15.0+, while one Xcode project build configuration still advertises IPHONEOS_DEPLOYMENT_TARGET = 13.0.

Root Cause:
- ios/Podfile already sets platform :ios, '15.0'.
- Most Runner target configs are already 15.6.
- ios/Runner.xcodeproj/project.pbxproj still has a project-level/Profile config at 13.0.
- When Flutter/Xcode uses the Swift Package Manager path, Xcode validates package minimum platforms against the project/target deployment settings and fails if any active config resolves to iOS 13.

Approach:
- Normalize all IPHONEOS_DEPLOYMENT_TARGET values in ios/Runner.xcodeproj/project.pbxproj to 15.0.
- Prefer 15.0 as the consistent minimum unless there is a strong reason to keep 15.6; the goal is compatibility with Firebase's minimum, not unnecessarily raising it beyond that.
- Leave ios/Podfile unchanged because it is already correct.
- Preserve the existing dirty ios/Podfile.lock change unless verification proves a pod regeneration is required.
- Verify by checking all iOS deployment targets and running a focused configuration/build check if feasible.

Alternatives Considered:
- Downgrade Firebase packages: preserves iOS 13/14 support but is fragile and fights current dependency requirements.
- Disable or avoid SPM path: may work locally, but leaves the underlying target mismatch waiting to break again.
- Raise only the single 13.0 setting: smallest change, but could leave inconsistent 15.0/15.6 config drift.
- Normalize all deployment targets to 15.0: most stable and easiest to reason about.

Recommended Approach: Normalize the iOS project deployment target to 15.0 across project.pbxproj, keep Podfile at 15.0, and avoid touching the existing dirty Podfile.lock unless explicitly needed.
