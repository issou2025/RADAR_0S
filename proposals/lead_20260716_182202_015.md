# Make startup truth report unavailable data honestly and honor injected credentials

## Context

The read-only startup tools can currently turn failed checks into healthy-looking zeros:

- `scripts/check_current_state_drift.py` uses `len(prs_json or [])` and `len(issues_json or [])`, so a failed `gh` command becomes `0`.
- `scripts/morning_truth_report.py` reports `0` smoke failures when smoke JSON is unavailable.
- Both queue-count helpers require `scripts/notion-to-wp/.env` to exist before constructing `WPClient`, so Varlock/process-env/Cursor Cloud secrets cannot work without a plaintext compatibility file.

This undermines the repo's primary orientation command and contradicts `docs/current-state/VARLOCK-ROLLOUT-2026-07-16.md`.

## Acceptance Criteria

- [ ] Failed or unparsable GitHub issue/PR queries render as `unavailable` or `unknown`, never `0`.
- [ ] An unavailable public smoke result renders as unavailable/degraded, never `0 failures`.
- [ ] Drift rows distinguish `no drift` from `not evaluated`.
- [ ] `WP_USER` and `WP_APP_PASSWORD` supplied through process env work when `scripts/notion-to-wp/.env` is absent.
- [ ] The gitignored `.env` compatibility path continues to work.
- [ ] Default read-only operation still degrades gracefully, with an explicit opt-in nonzero error gate for automation.
- [ ] Regression tests cover GitHub command failure, invalid JSON, smoke failure, process-env credentials, and `.env` fallback.
- [ ] No test performs a live WordPress write or requires real credentials.
