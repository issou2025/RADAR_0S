# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**Agent-native completion summary: report the Crawl/Clone Outcome**

*Project description:*
## Problem Statement

When a coding agent (or a human driving one) runs `getdocs crawl <url>`, the command succeeds **silently** — on success it returns exit code 0 and prints nothing. The agent has no machine-readable signal telling it what just happened: how many Pages were produced, where the output landed, whether the run was truncated at the limit, or — critically — whether getdocs even crawled at all. Because of source-first (ADR-0006), the same command may instead produce a **Clone** (a cloned source repo + a generated `mkdocs.yml`, with no Pages and no Manifest). The agent currently cannot tell a Crawl from a Clone, cannot find the output without guessing, and cannot decide what to do next (grep the Pages tree? serve the Clone? raise the limit because it was truncated?).

This blocks the whole "agent-native" use case: an agent that launches getdocs — synchronously for a section, or as a background task its harness resumes on completion (ADR-0007) — needs one parseable result on completion to act on, without re-deriving state from the filesystem.

## Solution

`getdocs crawl` reports a **completion summary** describing the run's **Outcome** — exactly one of a **Crawl** or a **Clone**. The summary comes in two coordinated forms:

- An **always-on human/log line on stderr**, e.g. `getdocs: crawled 42 Pages → ./out (crawl.json)` or `getdocs: cloned acme/docs → ./out/docs (mkdocs.yml)`.
- An **opt-in structured object** via a new `--summary-json` flag, discriminated by an `outcome` field, carrying the fields an agent needs to branch and act.

With this, an agent reads one line (or one JSON object) and immediately knows: did it crawl or clone, did it succeed, how many Pages, where the output is, where the Manifest is, and whether it was truncated — then proceeds (grep the tree, serve the Clone, or re-run with a higher limit).

## User Stories

1. As a coding agent, I want a single machine-readable line when `getdocs crawl` finishes, so that I know the run completed without inspecting the filesystem.
2. As a coding agent, I want the summary to tell me whether the run produced a **Crawl** or a **Clone**, so that I branch correctly instead of looking for Pages that don't exist.
3. As a coding agent, I want the output directory in the summary, so that I can immediately grep/read the produced Pages.
4. As a coding agent, I want the **Manifest** path in the summary when a Crawl happened, so that I can read Nav Order and Reading Order without searching for `crawl.json`.
5. As a coding agent, I want a `truncated` signal, so that I know the Crawl hit its limit and the Pages are incomplete.
6. As a coding agent, when truncated, I want enough context (page count vs limit) to decide whether to re-run with a higher `--limit`.
7. As a coding agent, when the run produced a **Clone**, I want the cloned repo location and the generated serve config path, so that I can offer to serve the docs locally instead of grepping a Pages tree.
8. As a coding agent, I want the source repo identity (e.g. `acme/docs`) in a Clone summary, so that I can tell the user which repository was cloned.
9. As a coding agent, I want a structured (`--summary-json`) form, so that I can parse the outcome reliably instead of scraping a prose line.
10. As a coding agent, I want the structured summary to use a stable `outcome` discriminator, so that I can write one branch for `crawled` and one for `cloned`.
11. As a coding agent running on Claude Code, I want to launch a long Crawl as a background task and, when my harness resumes me on its completion, immediately read the summary, so that I can continue the user's task with the docs in hand.
12. As a coding agent on a non-Claude-Code harness, I want the same summary after a synchronous (blocking) run, so that the behavior is portable across agents.
13. As a coding agent that polls, I want the summary printed deterministically at completion, so that I can detect "done" by watching for it.
14. As a human driving an agent, I want a readable one-line confirmation on stderr, so that I can see what happened at a glance in my terminal.
15. As a human, I want the summary to never pollute stdout in files mode unless I asked for it, so that piping behavior is predictable.
16. As a scripting/CI user, I want `--summary-json` to emit a stable object, so that I can gate a pipeline on the run's outcome.
17. As a scripting user, I want the exit code to keep its current meaning (non-zero when no Pages were produced), so that existing automation keeps working.
18. As a coding agent, I want an empty/failed Crawl to be reported distinctly (status reflects "empty"), so that I don't treat a zero-Page run as success.
19. As an agent, I want the summary in both files and jsonl formats, so that whichever output mode I chose, I still get a completion signal.
20. As an agent consuming jsonl, I want the structured outcome to be available without a second mechanism, so that I rely on the existing final Manifest record rather than a duplicated summary.
21. As a user reading the README, I want a "Use with your agent" recipe, so that I know how to wire getdocs into my coding agent (synchronous for a section, background for a whole-site mirror).
22. As a user of the recipe, I want it to show how to read the summary and branch on Crawl vs Clone, so that I can copy a working pattern.
23. As a maintainer, I want the summary built from the run's actual Outcome (not re-derived guesses), so that it never disagrees with the produced Pages/Manifest.
24. As a maintainer, I want the human line and the JSON to carry the same facts, so that the two forms can't drift apart.
25. As an agent, I want `--no-clone-source` runs to always report a `crawled` outcome, so that disabling source-first gives me predictable output.
26. As an agent, I want a resumed Crawl (`--resume`) to also produce a completion summary, so that resumption is observable like a fresh run.

## Implementation Decisions

- **Two coordinated surfaces, one source of truth.** The CLI entry flow emits (a) an always-on stderr line and (b) an opt-in structured object behind a new `--summary-json` flag. Both are rendered from the same in-memory **Outcome** value so they cannot disagree.
- **Outcome discriminator.** The structured summary is a tagged object keyed by `outcome`:
  - `crawled` → `{ outcome, status: "ok" | "truncated" | "empty", pages, output_dir, manifest, truncated }`
  - `cloned` → `{ outcome: "cloned", status: "ok", repo, output_dir, mkdocs_config }`
  A **Clone** carries no `pages`/`manifest` (it produces neither), consistent with the glossary (a Clone is a sibling Outcome of a Crawl, not a kind of Crawl).
- **Engine run entry returns a structured run result.** The engine's crawl-run entry point returns an Outcome value (status, page count, output directory, Manifest location, truncation) instead of a bare page count, so the CLI can report uniformly across files and jsonl modes. The exit-code-1-on-zero-Pages behavior is preserved (derived from the result).
- **Source-first orchestrator returns Clone details.** The source-first orchestrator already returns the cloned repo directory; it is extended to surface enough about the **Clone** (source repo identity, output directory, generated serve-config path) for the CLI to report it. The clone path still short-circuits before the crawl (ADR-0006), and now emits the `cloned` summary there.
- **stdout discipline (ADR-0002).** stdout remains the jsonl Page stream in jsonl mode. Therefore:
  - In **files** mode, `--summary-json` writes the structured object to **stdout** (otherwise unused).
  - In **jsonl** mode, the final Manifest record already carries the structured Outcome, so `--summary-json` does not emit a second object; the structured outcome lives in that record.
  - The **stderr** human line is emitted in both modes and both Outcomes.
- **No new agent surface.** This is delivered entirely through the existing CLI (ADR-0007): synchronous runs work on every agent; background-with-resume is the Claude Code bonus. No MCP server, no async job protocol, no daemon.
- **`--summary-json` is additive and off by default.** Default behavior is unchanged except for the new always-on stderr line (which lands where existing notes like "starting over" already go), so stdout-piping consumers are unaffected.
- **README "Use with your agent" recipe.** Documentation deliverable: a section showing the synchronous one-liner for a section, the background pattern for a whole-site mirror, and how to read the summary / branch on `crawled` vs `cloned`.

## Testing Decisions

- **Test external behavior, not internals.** Tests assert what an agent observes — the stderr line, the `--summary-json` stdout object, exit codes, and which Outcome was reported — not how the summary is constructed internally. The structured object's shape (keys, `outcome` value) is the contract and is asserted directly.
- **Crawled Outcome — existing subprocess CLI e2e seam (highest).** Drive `python -m getdocs crawl` against the fixture site and assert: the stderr summary line is present; with `--summary-json`, stdout parses to `outcome: "crawled"` with correct `pages`, `output_dir`, `manifest`, and `truncated`. Cover a truncated run (low `--limit`) and a normal run. Prior art: `test_crawl_e2e.py`, `test_resume_e2e.py` (already asserts on stderr text), `test_jsonl_e2e.py` (asserts the final Manifest record on stdout). Also assert a `--resume` run reports a summary.
- **Cloned Outcome — existing in-process `cli.main()` seam.** A subprocess e2e cannot drive the Clone path (source detection requires a real git-host link), so verify one seam lower: call `main([...])` in-process with the source-first orchestrator monkeypatched to return a fake repo dir, capture stdout/stderr, and assert the `outcome: "cloned"` summary (repo, output_dir, mkdocs_config) on both the stderr line and `--summary-json`. Prior art: `test_source.py` already monkeypatches the orchestrator and fabricates a repo dir; `test_cli.py` drives the CLI layer.
- **jsonl-mode discipline.** Assert that in jsonl mode stdout still contains only Page records + the final Manifest record (no extra summary object), while the stderr line is still emitted. Prior art: `test_jsonl_e2e.py`.
- **Empty/failed run.** Assert an unreachable seed yields the existing non-zero exit code and a summary/stderr reflecting an empty Crawl. Prior art: `test_limits_errors_e2e.py`.

## Out of Scope

- **The Claude Code plugin and its distribution (branch 2).** The `/getdocs` plugin (bootstrap via `uvx`, sync-vs-background mode selection, Outcome-driven post-run guidance), the repo-as-plugin-marketplace, and community-list submissions are a **named follow-on** that depends on this summary landing first. Track separately.
- **An MCP server or any agent-facing job/daemon protocol.** Explicitly rejected in ADR-0007.
- **Changing crawl/clone behavior itself.** No changes to Scope, extraction, source-first detection, rendering, or the Manifest contents beyond surfacing existing facts in the summary.
- **A `--quiet` flag / suppressing the stderr line.** Not in this PRD; the line is always emitted (it lands where existing stderr notes do).
- **Progress reporting during a long run.** Only the completion summary is in scope, not mid-run progress.
- **Promotion tactics** (demo GIF, Show HN, awesome-list PRs) — go-to-market, not this code feature.

## Further Notes

- Domain vocabulary: this feature is built around **Outcome** (a run yields exactly one of a **Crawl** or a **Clone**), **Pages**, and the **Manifest**, as defined in `CONTEXT.md`. A **Clone** has no Pages or Manifest by definition.
- Relevant ADRs: **ADR-0007** (agent integration is the CLI, not an MCP/async-job surface — this summary is the one code change that ADR calls for), **ADR-0006** (source-first clone-over-crawl — the reason two Outcomes exist), **ADR-0002** (subprocess-per-Crawl and the stdout-is-the-jsonl-stream rule that governs where `--summary-json` may write).
- The completion summary is the contract the future plugin (branch 2) reads to choose its next action, so its `outcome` discriminator and field names should be treated as a stable interface.
- Rationale captured during a grilling session: the agent-native value is "a path the agent can grep," not inline markdown; and the "wake me when the crawl finishes" UX is a Claude-Code harness feature, not an MCP capability — so the summary, not a server, is what agents actually need.
