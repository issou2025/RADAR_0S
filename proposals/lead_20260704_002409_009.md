# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**PRD: Business-Scoped Stream primitive — kill the build-time-poison provider bug by construction**

*Project description:*
## Problem Statement

Right after a user creates a business, store pickers (Receive Stock, Transfer/Request Stock, POS scope) render **empty**, and only a cold app restart fixes it. From the developer's side, this is a recurring, easy-to-reintroduce trap: a business-scoped live-query provider bakes the current businessId into its query at first build, and if that first build happens in the brief window before a business is bound, it either **throws and sticks errored for the whole session** or **silently sticks empty**. The only defense today is remembering to hand-write a businessId guard on every new provider — and it gets forgotten.

## Solution

Every business-scoped stream provider is declared through a single guarded **factory**, so the guard is automatic and the empty-screen bug cannot be written. A developer writes `businessScopedStream(whenAbsent: …, (db, businessId) => …)` instead of a raw `StreamProvider`. The factory watches the current businessId, emits the declared empty value **only** while no business is bound, swaps in real data the instant a business binds, and re-runs on a business switch. A CI ban test refuses any new raw business-scoped stream provider, so the class of bug stays dead.

## User Stories

1. As a user who just created a business, I want the Receive Stock store picker to populate as soon as my business is ready, so that I can receive stock without restarting the app.
2. As a user who just created a business, I want the Transfer / Request Stock store pickers to populate, so that I can move stock between stores immediately.
3. As a user who just created a business, I want the POS store scope to populate, so that I can start selling immediately.
4. As a user, I want the app-wide currency to show a sensible default before my business binds, so that money is never rendered wrong during the handoff.
5. As a user who switches between businesses on one device, I want every live list to re-query for the business I switched to, so that I never see the previous business's data.
6. As a developer, I want a single factory to declare business-scoped stream providers, so that I cannot forget the businessId guard.
7. As a developer, I want the factory to hand my closure a guaranteed non-null businessId, so that my query code never has to null-check or risk a throw.
8. As a developer, I want to state the null-window value explicitly per provider, so that a list shows `[]`, a count shows `0`, and the currency shows its default — each correct for its type.
9. As a developer, I want keyed (family) business-scoped streams to have their own guarded factory, so that argument-parameterised providers are equally safe.
10. As a developer, I want providers that run custom SQL against `business_id` to use the same factory, so that the silent-empty variant of the bug is eliminated too.
11. As a developer, I want one named `currentBusinessIdProvider` seam, so that nothing re-derives "who is the current tenant" and I can override it in tests.
12. As a developer, I want a CI test that fails when a business-scoped stream provider is declared without the factory, so that the invariant is enforced, not just conventional.
13. As a developer, I want genuinely global streams (the permissions catalogue, unscoped roles) to remain declarable outside the factory via a small, ratcheting allowlist, so that non-tenant data isn't forced through a tenant guard.
14. As a developer, I want the factory unit-testable by flipping the businessId seam, so that I can prove the guard without spinning up a database.
15. As a reviewer, I want the migration to preserve each provider's existing null-window behaviour 1:1, so that I can review it as behaviour-neutral.
16. As a maintainer, I want the design recorded as an ADR and the vocabulary in the glossary, so that the next person understands why the factory exists rather than "fixing" it away.

## Implementation Decisions

- **A guarded provider factory owns the declaration.** `businessScopedStream<T>` replaces the raw `StreamProvider<T>` declaration at each business-scoped site; the businessId watch and null-guard live inside it and cannot be omitted.
- **A second factory covers the family variant.** `businessScopedStreamFamily<T, Arg>` handles keyed providers, because Riverpod cannot emit both plain and family providers from one function. No hand-written family carve-out.
- **The closure receives `(db, businessId)` with a guaranteed non-null id.** DAO-watch closures ignore the id; custom-SQL closures use it directly. This makes the closure total — `requireBusinessId()` can never throw from a factory-built provider — and lets the factory subsume both the throw-poison and the silent-empty families.
- **`whenAbsent: T` is required.** The null-window value varies by type (`[]`, `{}`, `0`, `null`, currency default) and is not inferred; requiring it keeps the migration a behaviour-preserving 1:1 lift.
- **A new `currentBusinessIdProvider` is the single watchable businessId seam**, derived from the auth state (`authProvider.select(currentUser?.businessId)`). The factory watches it; nothing else re-derives it; non-reactive `db.currentBusinessId` reads are retired where they back a stream.
- **Rebuild-on-switch is a property of watching the seam** — free, and it pre-solves a future switcher's staleness without touching the deferred eviction decision.
- **Enforcement is a source-scan ban test** modeled on the existing gate ban, with a small non-empty allowlist naming the genuinely-global streams and a shrink-only ratchet, plus a companion strictness test proving a planted violation is caught and the migrated form is not flagged.
- **Global/unscoped streams stay off the factory** (permissions catalogue, unscoped roles) — they are not tenant-scoped.
- Recorded in **ADR 0003 (`docs/adr/0003-business-scoped-stream.md`)**; glossary terms *Business-Scoped Stream* and *Current Business Id* added to `CONTEXT.md`.

## Testing Decisions

- **Test external behaviour, not internals.** Assert what the provider *emits* as the businessId transitions, not how the factory is wired.
- **Primary module under test: the factory itself**, via the `currentBusinessIdProvider` seam. Drive `null → bound → switched` and assert `whenAbsent → data → re-queried data`; assert it never throws in the null window. No database required — the closure returns a test `StreamController`.
- **Prior art:** `test/permissions/guarded_test.dart` (override a context provider with a flippable `StateProvider` source and assert the consequence); `AppDatabase.forTesting()` + `databaseProvider.overrideWithValue()` (`test/receiving/receive_flow_mode_test.dart`) for any end-to-end query check.
- **The ban test** carries its own companion test (planted violation caught / migrated form not flagged), exactly as `test/permissions/gate_static_ban_test.dart` does.

## Out of Scope

- The `AsyncLoading`-while-unbound behaviour (deferred opt-in via omitting `whenAbsent`); the default stays "emit `whenAbsent`".
- Eviction-on-business-switch (a separate, deliberately-deferred data-loss decision).
- Forcing global/unscoped streams through the factory.
- Any change to provider **consumers** — `ref.watch(xProvider)` call sites are untouched.
- Non-stream providers (`FutureProvider`, plain `Provider`) — this primitive is stream-only.

## Further Notes

- The originating item's "257 providers" was a count of consumers; the actual migration surface is **~41 declarations across two provider files** (`lib/core/providers/stream_providers.dart`, `lib/core/providers/app_providers.dart`), consumers untouched.
- Delivered as **two independently-grabbable issues**: (1) the factories + `currentBusinessIdProvider` + tests + a ban test whose allowlist temporarily names all current raw providers (CI green); (2) the mechanical migration of the ~41 sites, shrinking the allowlist to just the globals, plus the strictness test.
