## iOS full validation sweep — 76 scenarios (performance + rendering + relay-diagnostics + offline)

This is the thorough polish + performance + correctness pass for the Chirp iOS app on the **iOS Simulator**. It is executed by tester agents who capture screenshots + performance metrics per scenario, and reviewed by an Opus reviewer who reads **every** screenshot before any scenario is marked done.

> **REFRESHED 2026-07-05 for the post-UX-overhaul build.** A large IA/UX overhaul landed on `master` (PRs #100 Liquid Glass tab bar + unified design system, #103/#79 relay redesign, #104/#96 nav/IA restructure, #106/#95 diagnostics dev-console framing, #107/#61 dead-zap removal; #108/#93 NIP-29 multi-relay discovery is landing). Every scenario below is grounded in the **current** `apps/ios/Chirp/` source. The execution plan (which prior attempts are still valid vs must be re-validated, and the batch breakdown) is in a pinned comment titled **"Execution plan (post-UX-overhaul refresh)"**.

### Current information architecture (READ FIRST — the old nav is gone)

- **Tab bar (native Liquid Glass, #100):** **Home · Chats · Discover · Wallet · Settings**. The old **Groups** tab is retired.
- **Chats** (`ChatsView`) is now the **single conversation surface**: NIP-17 DMs, MLS/Marmot groups, and the NIP-29 demo room are **ONE recency-sorted list**. A shared `ConversationRow` renders every entry; a leading glyph badge distinguishes them (**🔒** encrypted group, **#** public group, avatar for a DM). There is no separate DM-vs-group list.
- **Discover** (`DiscoverView`) occupies the old Groups slot and is for finding things you are **not** already in: a **"Find people"** row (hands off to the omnibox `SearchView`) and **NIP-29 public-group browse**. With #108 landing this is a **curated cross-relay browse** (`CuratedGroupRelays.defaults` = Chirp's demo relay + `wss://groups.0xchat.com` + `wss://groups.fiatjaf.com`, all live-aggregated at once) with a **name filter** (`.searchable`) and an **"Add a relay"** affordance. Until #108 merges, Discover is the single-relay `join-group-relay-input` form.
- **Creating a group** ("New Group", private/MLS or public/NIP-29) now lives in the **Chats compose menu** (`chats-compose-menu`), not a Groups toolbar.
- **Search is back** (`SearchView`, `ChirpRoute.search`): reachable from the **Home top-bar magnifier** (`home-search-button`) and Discover's "Find people". Real NIP-50 free-text + entity (npub/nprofile/nevent/nip05) resolution. The prior "search deleted, verify absence" note is void.
- **Home top-bar** (`HomeFeedView`): leading = profile avatar; trailing = **magnifier** (search) + **pencil** (`square.and.pencil`, new note). The old paperplane is gone — a **transient "N pending publish" pill** (`home-outbox-status-pill`) floats in via `safeAreaInset(.top)` **only while the publish outbox is non-empty**, and tapping it opens the outbox (`NotificationsView`).
- **Zap UI is removed** (#61/#107). Every former zap scenario now **verifies absence** — no bolt button on note rows, no `ZapAmountSheet`, no zap affordance in Wallet. (`ChirpColor.zap` survives only as a color token for the wallet "warning" tone.)
- **Relays** (`RelaySettingsView`, #103/#79): Settings → Relays; each relay is a `NavigationLink` → `RelayConfigDetailView` carrying the **same live per-relay status** Diagnostics shows (connection dot, role); add via `+`; remove via swipe **or** `EditButton` delete (#80); the "Publish as DM inboxes" row is an honest kind:10050 state machine.
- **Diagnostics** (`DiagnosticsView`, #106/#95): Settings → **Developer** → Diagnostics (DEBUG builds). Now explicitly framed as a **developer console** with a footer — *"Developer diagnostics — internal counters and raw relay traffic. Not a user-facing surface."* Content (kernel/perf/relays/publish-queue/active-account) is unchanged and remains the canonical in-app metrics source.

### Rules (read before testing)

1. **Screenshots are mandatory.** Post at least one screenshot per scenario as a comment on this issue, tagged with the scenario ID (e.g. `S07`). Capture the state that proves the acceptance list — not a generic screen. For multi-step scenarios, post the before/after (or the full sequence).
2. **Performance metrics are mandatory per scenario.** Every scenario names exactly which metrics to capture and how. The app's **Diagnostics screen** (Settings → Developer → Diagnostics) is the canonical in-app metrics source and truthfully reports: Kernel Status/Rev/Snapshots/Update-Seq, First Event ms, Max Apply µs, Decode µs, cb→screen µs, Events Rx, Bytes Rx, Stored, Note Events, Queue Depth, per-relay connection/auth/role/subs/events/reconnect/notices/errors, Logical Interests, Publish Queue. Screenshot the relevant Diagnostics rows as the metric evidence. Supplement with wall-clock (screenshot/video frame timestamps) and OS-level memory/CPU (see Setup).
3. **Relay provenance must be shown.** For any scenario that loads network data, the tester must show **which relays** the data came from and their live state — via the Diagnostics → Relays section (connection dot, role, sub count, events count) and/or per-relay `RelayConfigDetailView`/`RelayDetailView`. "Content loaded" is not acceptance on its own; "content loaded, and Diagnostics shows N relays Connected serving M events" is. For Discover, the per-row **host-relay chip** is the provenance surface.
4. **Offline = SIMULATOR only, never the whole computer.** Put the *simulator* offline (see Setup). The canonical in-app proof of offline is the Diagnostics → Relays section showing every relay `Disconnected` with reconnect counts climbing. Re-enable connectivity immediately after each offline scenario.
5. **NO HACKS in later remediation.** This issue only authors the plan and records findings. Any fix that follows must be a real fix (no stubs, no fake data, no disabling the assertion) — see repo AGENTS.md zero-tolerance rule.
6. **Objective pass/fail.** Each scenario's acceptance list is written so a reviewer judging only the screenshots + metrics can rule PASS or FAIL without re-running the app. If a scenario cannot be reached (feature absent/gated), mark it **N/A with a screenshot proving absence** rather than skipping.

## Setup the testers need

**App / build.** iOS Simulator (iPhone 16 or 15, iOS 17+). Project at `apps/ios/Chirp.xcodeproj`, scheme `Chirp`. Build+install+launch via the XcodeBuildMCP `xcode` tools (`build_run_sim`) or `just rust-ios-sim` then install. Confirm `session_show_defaults` first. Build from `master` (post-overhaul); for the Discover multi-relay scenarios (S71–S74), build from `master` once #108 has merged, otherwise use the `chirp-nip29-discovery` branch or fall back to the single-relay form.

**Test identities (need at least two).**
- **Identity A — "primed" account:** an nsec with an existing follow graph and real published content, so the timeline, profile, and social counts populate. Used for the bulk of read/render scenarios.
- **Identity B — "counterparty" account:** a second nsec on the same simulator (multi-account) or a second simulator, used for two-party DM, group, reply/reaction round-trips.
- A **fresh/empty** identity (create-new-identity path) for empty-state and onboarding scenarios.

**Relays.** Two options, pick per batch:
- **Real wss relays** (e.g. `wss://relay.damus.io`, `wss://nos.lol`, `wss://relay.primal.net`) for provenance realism — Diagnostics will show which served what. **Required** for Discover (S71–S74 curated relays are real NIP-29 wss hosts) and for NIP-17 DM inbox routing (kind:10050).
- **Local seeded relay** (`nak serve` on `ws://127.0.0.1:...`, seeded with deterministic events) for reproducible rendering/offline-cache scenarios. Preferred for content-rendering correctness (S18–S27) so screenshots are stable across runs. Seed **≥50 historical note events** for the pagination scenario (S15).

**Putting the simulator offline (SIM only).** In priority order:
1. **Network Link Conditioner "100% Loss"** profile — the standard iOS offline test. If host-wide, enable only for the offline scenario and disable immediately after.
2. **Status-bar override:** `xcrun simctl status_bar <udid> override --dataNetwork none --wifiMode failed --cellularMode notSupported` (sets the visible indicator; pair with #1 or #3 to actually cut traffic).
3. **Point the app at an unreachable relay set** (a dead `ws://` host) as a sim-scoped way to force "no relay reachable".
- **Regardless of method, the acceptance proof is in-app:** Diagnostics → Relays shows every relay `Disconnected`, reconnect counters climbing, and Bytes Rx flat. Re-enable after.

**Capturing performance metrics.**
- **In-app (primary):** screenshot Diagnostics rows named per scenario (`metric-events-value`, `metric-first-ms-value`, `metric-apply-us-value`, `metric-rx-value`, `metric-visible-value`). This is the truth source for kernel/render/network-settle timings.
- **Cold-start / render wall-clock:** `xcrun simctl launch --console-pty <udid> <bundleid>` for launch console + timestamped screenshots, or `record_sim_video` and count frames tap→first-content.
- **Memory + CPU:** Instruments (`xctrace record --template 'Allocations'` / `'Time Profiler'` / `'Core Animation FPS'`), **or** `ps -o rss,pcpu -p <pid>`. Report peak RSS (MB) and peak CPU%.
- **Scroll / frame perf:** `record_sim_video` during the scroll, inspect for dropped frames / stutter. Report observed smoothness + any hitch.
- **Network settle time:** wall-clock from screen-open to Diagnostics Events Rx / Bytes Rx going flat (steady state).
