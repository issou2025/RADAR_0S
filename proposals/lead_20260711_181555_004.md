# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**IfcConvert hangs forever (infinite loop in OCCT corner trimming) on IfcSweptDiskSolid with directrix segments much shorter than the disk radius**

*Project description:*
Converting an IFC exported by Revit MEP (flexible electrical conduit) makes IfcConvert hang forever with one core at 100%. This is not a slow conversion, it is a true infinite loop; we observed a production worker stuck for 3.5 hours on a 15 MB file until an external watchdog killed it.

Verified on the official builds:

- `IfcConvert 0.8.3-0aae576 (OCC 7.8.1)` linux64: hangs
- `IfcConvert 0.8.4-158fe92 (OCC 7.8.1)` linux64 (latest build available on ifcopenshell-builds, 2025-11-19): hangs
- Our v0.8.0-based fork against OCCT 7.6.3 and OCCT 7.8.1: hangs

Current master should be affected as well: the relevant code now lives in `src/ifcgeom/kernels/opencascade/sweep_utils.cpp` (line 169 still selects `BRepBuilderAPI_RightCorner` for non-C1 directrices) and there is no directrix sanitization anywhere in that path. We could not test a post-OCCT-8.0 build (none published yet), but the loop is inside OCCT's `ProjLib` walking code, which behaves identically in 7.6.3 and 7.8.1.

## Root cause

- The file contains `IfcSweptDiskSolid` entities whose composite-curve directrix is a discretized spline containing chords **much shorter than the disk radius**: segments down to 238 um against an 11.2 mm radius (about 1/50).
- The sweep code selects `BRepBuilderAPI_RightCorner` for non-C1 directrices. The call chain then is:
  `BRepOffsetAPI_MakePipeShell::Build()` -> `BRepFill_Sweep::PerformCorner` -> `BRepFill_TrimShellCorner::Perform` -> `BOPAlgo_PaveFiller::MakePCurves` -> `BOPTools_AlgoTools2D::MakePCurveOnFace` -> `ProjLib_CompProjectedCurve::Init()` -> `math_NewtonFunctionSetRoot` fails to converge while projecting an edge onto two nearly-coincident pipe surfaces (the two sides of a corner whose adjacent segment is shorter than the trim region), and the walking loop inside `Init()` never advances past the `x = 2*pi` parameter boundary.
- **The hang cannot be interrupted externally:** no `Message_ProgressRange` / `UserBreak` reaches this path. `BRepFill_PipeShell::Build()` (OCCT) does not propagate any progress range into `BRepFill_Sweep`/`BRepFill_TrimShellCorner` (grep of OCCT 7.8.1: zero occurrences in those files).
- The cgal/cgal-simple kernels are unaffected (they reject `sweep_along_curve` items), but the default and hybrid kernels hang.

Two gdb stack samples captured seconds apart, both inside `ProjLib_CompProjectedCurve::Init` (attached below).

Possibly related (same area, different symptom): #5474.

## Reproduce

`minimal_el67.ifc` (21 KB, single `IfcCableCarrierSegment`, inlined below):

```
IfcConvert minimal_el67.ifc out.obj    # hangs forever at 'Creating geometry'
```

The element's directrix is an `IfcCompositeCurve` of 65 two-point polylines: 16 chords are shorter than 1/10 of the disk radius (min 238 um vs radius 11.2 mm), corner angles between 0.15 and 11.25 degrees. Collapsing only the 3 shortest chords (under 300 um) already makes the file convert, so the trigger is specifically the sub-radius-scale segments, not the shallow corner angles (a sibling element in the same file with min chord 1.3 mm and the same angle profile converts fine).

## Fix we applied (for reference)

Before feeding the wire to `BRepOffsetAPI_MakePipeShell`, we collapse straight directrix segments shorter than `max(radius / 10, 10 * Precision)` to their midpoint (directrix endpoints pinned; wires with non-linear segments or nearly-closed wires untouched). With this the full 15 MB file converts in about 1 minute (304 elements, 118 of them sanitized), and models without sub-scale segments are byte-identical.

The patch below is against v0.8.0's `sweep_along_curve.cpp`; the same code lives in master's `sweep_utils.cpp` today, so it should apply with minimal adjustment. Happy to turn this into a PR if the approach sounds right. An alternative or complementary fix would be guarding `PerformCorner` against segments shorter than the corner trim region. Arguably `ProjLib_CompProjectedCurve::Init()`'s unbounded walking loop is also an OCCT bug worth reporting upstream.
