## Problem Statement

HELM is a working product Daniel uses daily, but it is also a public portfolio piece on his GitHub and resume (SWE job search is his top priority). A visitor — recruiter, hiring manager, or engineer — currently gets a strong README and nothing else: no way to *see* HELM without cloning it, no way to *run* it without a vault, the Claude CLI, and a Python voice stack, no visible quality signal (no CI), and the best artifact in the repo (the interactive architecture explainer) is invisible behind a clone. The engineering that would land in an interview — the security perimeter, the three-tier router, files-as-message-bus, the contract tests guarding the load-bearing couplings — is scattered across code and CLAUDE.md where no skimmer will find it.

## Solution

Make the repo legible to a visitor who will spend between 60 seconds and 10 minutes on it, in escalating depth:

- **A — Demo recording**: a short screen recording (GIF inline, optionally linked video) at the top of the README showing the real loop: hold Space, speak, watch the router tier fire (instant vault answer vs. queued skill), hear HELM speak back.
- **B — Demo vault**: a committed `demo-vault/` of realistic sample files so `VAULT_ROOT=./demo-vault npx next dev -p 3107` renders a fully populated HUD — every tab, tile, and feed panel showing data — with zero credentials, no Claude CLI, no Ollama, no voice server. A contract test keeps the demo data honest against the real vault parsers.
- **C — Hosted architecture explainer**: serve the existing architecture one-pager via GitHub Pages so the README can link straight to it; screenshot it in the README.
- **D — CI**: a GitHub Actions workflow running the existing quality gates (`npm test` + `npx tsc --noEmit`) on push/PR, with a status badge in the README.
- **E — Design decisions doc**: a short "interesting decisions" document (fail-closed auth, method-based CHAT_ONLY gate, files as the message bus, three router tiers with offline fallback, contract tests guarding skill/tab/runner couplings, deterministic feeds vs. LLM skills) — five-to-eight entries, each a few sentences of *why*, linking into the code. Doubles as interview talking points.

## Implementation Decisions

- **Single tested seam**: everything rides the existing `VAULT_ROOT` environment boundary. The demo vault is pure data exercised through the real vault-reading module; no demo flags, no mock layer, no code branches for demo mode. If a panel can't render from files alone, it degrades exactly as production does (that graceful degradation already exists and is part of what's being demonstrated).
- **Demo vault content**: fictional but realistic — a daily note for a plausible day, a handful of job-application records, metrics CSV rows spanning enough days for the weekly delta to compute, one or two morning reports for the AI Wire, a few completed run records, a fresh-looking runner heartbeat is explicitly NOT faked (runner shows OFFLINE in demo mode — honest, and it demonstrates the degradation design). Dates in demo files should be either recent-relative (generated) or the README should note that stale dates are expected; decision: keep static files and let the daily-note ≤today fallback and empty-state paths do their job, since regenerating dates adds machinery for marginal polish.
- **Contract test**: a new test in the existing test-suite style (temp-free variant: point `VAULT_ROOT` at the committed demo vault before importing the vault module, assert each parser returns non-empty, well-formed results — applications parse, metrics rows parse and produce a weekly delta, morning report yields a headline, queue/runs scan cleanly). Wired into the `npm test` chain like every other suite.
- **CI**: one GitHub Actions workflow, ubuntu runner, Node setup + `npm ci` + `npm test` + `npx tsc --noEmit`. All python suites are stdlib-only (verified), so `python3` on the runner suffices — no pip install step. No matrix, no caching cleverness beyond the standard npm cache, no separate lint job. Badge in the README header.
- **GitHub Pages**: deploy-from-branch serving the docs folder; the explainer becomes reachable at the Pages URL. Add a minimal index redirect (or link directly to the page) — whichever requires less; no site generator, no theme. The explainer already works offline/self-contained so it ships as-is.
- **README changes**: GIF at top (committed to the repo or a stable asset URL), CI badge, demo-mode quickstart promoted above the full quickstart, architecture screenshot + Pages link, link to the design-decisions doc. Keep the existing voice/runner setup sections as the "full install" tier.
- **Design-decisions doc**: one Markdown file in docs/, 5–8 entries, each: decision, why, what it cost, link to code. Content sourced from the security-perimeter issue, the load-bearing-couplings section of the project instructions, and the deterministic-feed decision. Written for an external engineer, not for Claude.
- **Recording (A)**: 30–45s, captures: hold-Space push-to-talk, a tier-2 instant answer ("what's in the queue" style — safe, queues nothing), a tier-1 skill dispatch with the deck/tab reaction, and TTS playback. GIF for autoplay in the README; source video kept locally. This is a manual deliverable with a checklist, not automation.
- **Colorblind rule applies** to anything visual added (badge choice, screenshots, explainer tweaks): never red-vs-green alone.
- **Security perimeter unchanged**: Pages hosts static docs only; demo vault contains no secrets; CI needs no repository secrets (no API spend in the gates — that property is load-bearing and must stay).

## Testing Decisions

- Tests assert external behavior only: "the committed demo vault parses through the real vault readers into non-empty, well-formed view data" — not how parsers work internally (that's already covered by the existing vault sweep).
- Modules under test: the vault data layer, exercised via the `VAULT_ROOT` boundary against `demo-vault/`. Nothing else gets new tests; CI is the existing gates re-hosted, and A/C/E are verified manually (GIF renders on GitHub, Pages URL serves the explainer, doc reads well).
- Prior art: the existing vault sweep test, which already sets `VAULT_ROOT` to a throwaway directory before dynamic-importing the config module — the demo-vault test copies that pattern with the committed directory instead of a temp one.
- The new test joins the `npm test` chain, so CI (D) automatically guards the demo vault (B) — the deliverables reinforce each other with zero extra wiring.

## Out of Scope

- A hosted live HUD demo. The security perimeter exists precisely so this isn't internet-facing; a public interactive instance fights the design.
- Demo/mock modes in application code — no flags, no branches, no fake runner heartbeat. Data-only.
- Voice in demo mode (needs the local Python stack; the GIF is how visitors experience voice).
- CONTRIBUTING.md, issue templates, code-of-conduct — nobody is contributing; it reads as cosplay on a personal project.
- A full user manual (single-user product; README quickstart suffices).
- Running the voice-server model downloads or any GPU/onnxruntime work in CI.
- Date-regeneration machinery for demo files (static files + existing fallback behavior instead).
- Any change to auth, binds, CHAT_ONLY, or .stignore.

## Further Notes

- Suggested implementation order: B → D → C → E → A. The demo vault makes the recording session trivial (record against demo data — also avoids leaking real vault contents in the GIF), and CI should exist before the new test lands so it's born guarded. A is last and manual; Daniel records it (or Claude drives it via browser automation and captures a GIF) once B is merged.
- The GIF doubles as the asset for the resume line and any application portfolio links — worth the extra take to get a clean one.
- Recording checklist (for re-records after UI changes): demo vault active, one browser tab only (double-audio gotcha), click once in the tab before recording (autoplay), show hold-Space, one tier-2 question, one tier-1 dispatch, end on the spoken answer.
