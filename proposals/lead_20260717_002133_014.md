# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**Expand fsspec-cli beyond plain ls with truthful POSIX filesystem primitives**

*Project description:*
## Problem Statement

The plain **ls** tracer proved that an embedded, async, POSIX-shaped command library can be useful over explicitly configured fsspec sources without claiming that every fsspec backend is a POSIX filesystem. The project now has a stable host seam, invocation-owned async source lifecycle, explicit mapped operands, strict source-free option rejection, deterministic output, isolated-wheel evidence, and a command-scoped tested matrix.

The next work must not revive the historical draft's broad assumptions. Familiar method names and inherited fsspec composites do not prove POSIX semantics or complete mutation. Backends vary in paths, types, overwrite, metadata, and failures, while remote stores often lack current-directory state, allocated blocks, capacity, modes, ownership, links, special files, and atomic rename. Transfer and removal can leave completed earlier operands or destination residue; rebuilding fsspec traversal would add risk without capability.

CANFAR data commands, vostools-compatible tools, and later hosts need one migration target with exact streams, exits, and unsupported boundaries. Native vosfs remains the priority, but support stays scoped per command, option, source form, version, and evidence. Generic interoperability is proved, never presumed.

This specification therefore covers only POSIX Issue 8 filesystem-related primitives and their direct lexical helpers. It does not attempt every POSIX utility. It also does not add command wrappers for composite traversal and content-processing behavior such as find, walk, glob, checksums, comparisons, line slicing, counting, archives, or conversion.

## Solution

Expand **fsspec-cli** beyond plain **ls** as a sequence of small, independently tested command profiles. The semantic baseline is the [Open Group Base Specifications Issue 8 / IEEE Std 1003.1-2024 utilities volume](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/idx/utilities.html). The package describes itself as providing an **Issue 8-aligned supported subset**. It does not describe itself, a configured source, or fsspec generally as POSIX-conforming.

The sole stable host and behavioral seam remains **App(sources).typer_app**. The source mapping, async ownership, mapped operand grammar, preflight model, cleanup rules, release boundary, and matrix vocabulary already approved for plain **ls** remain locked.

### Filesystem-utility catalog

The catalog separates commands that can be implemented truthfully now from commands whose POSIX meaning depends on capabilities absent from the generic fsspec contract.

| Utility | Decision | Initial or eventual profile |
| --- | --- | --- |
| **ls** | Already implemented; unchanged by this issue | Plain listing and **-A** remain supported; **-a**, **-l**, and every other unapproved option remain strictly unsupported |
| [basename](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/basename.html) | Implement as a source-free lexical primitive | **basename string [suffix]**, using Issue 8 string rules and no filesystem acquisition |
| [dirname](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/dirname.html) | Implement as a source-free lexical primitive | **dirname string**, using Issue 8 string rules and no filesystem acquisition |
| [cat](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/cat.html) | Implement incrementally | First mapped source-reported file entries with exact byte output; later stdin and **-** sequencing; keep **-u** unsupported until its forwarding contract is proven |
| [mkdir](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/mkdir.html) | Implement incrementally | Base creation without parents, then **-p** through the fsspec-owned parent-creation operation; keep **-m** unsupported |
| [rmdir](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/rmdir.html) | Implement base form | Empty-directory removal; defer **-p** until its parent-boundary profile is independently approved |
| [unlink](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/unlink.html) | Implement as an XSI-optional thin primitive | Exactly one mapped file operand and no options |
| [rm](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/rm.html) | Implement incrementally | Source-reported file entries first, then **-f** and **-d**; **-R/-r** and **-v** require stronger operation results; keep **-i** unsupported |
| [cp](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/cp.html) | Implement incrementally as a declared reduced profile | Two source-reported file operands first, same-source and cross-source as separate evidence, then multiple sources; recursive and capability-dependent options stay unsupported until their complete contracts exist |
| [mv](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/mv.html) | Implement only positively evidenced same-source reduced profiles | Cross-source move remains unsupported until metadata preservation and source-version safety can be proven; directory and multiple-source forms are separate profiles |
| [cd](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/cd.html) and [pwd](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/pwd.html) | Unsupported by this architecture | They mutate or report persistent shell/process state, while every command invocation requires an explicit source and path |
| [umask](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/umask.html) | Unsupported by this architecture | It mutates persistent process creation policy that the invocation-owned source model neither owns nor carries between commands |
| [df](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/df.html) | Unsupported without a future capacity profile | fsspec has no portable total, used, available, capacity, filesystem identity, or mounted-root contract |
| [du](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/du.html) | Unsupported as POSIX **du** | fsspec **du** normally aggregates logical byte sizes; Issue 8 requires allocated space in specified block units |
| [chmod](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/chmod.html), [chgrp](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/chgrp.html), and [chown](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/chown.html) | Capability-gated and initially unsupported | Require authoritative modes, ownership namespaces, mutation, recursion, and link handling |
| [ln](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/ln.html) and XSI [link](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/link.html) | Capability-gated and initially unsupported | A copied object or redirect record is not a hard or symbolic link |
| [readlink](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/readlink.html) and [realpath](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/realpath.html) | Capability-gated and initially unsupported | Require native link text, component resolution, loop detection, and exact existence rules |
| [touch](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/touch.html) | Capability-gated and initially unsupported | Generic fsspec touch behavior may create or truncate; it cannot stand in for timestamp-only mutation |
| [mkfifo](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/mkfifo.html) | Unsupported generically | A zero-byte object is not a FIFO and no generic fsspec special-file contract exists |
| XSI [fuser](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/fuser.html) | Unsupported generically | Requires host process tables, open-file state, user identities, signals, and mounted-filesystem identity |
| [pathchk](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/pathchk.html) | Deferred | Default behavior needs backend pathname and component limits; lexical **-p/-P** alone does not justify admitting the base command |
| [file](https://pubs.opengroup.org/onlinepubs/9799919799.2024edition/utilities/file.html) | Outside the primitive command set | It is content classification driven by magic rules, locale, and host data rather than a filesystem namespace primitive |
| **stat** | Explicitly not a POSIX Issue 8 utility | A future fsspec-cli extension may expose authoritative **info** fields, but must never be presented as POSIX **stat** or used to smuggle incomplete long-listing metadata into the package |
| **mount**, **sync**, **install**, **truncate**, and **mktemp** | Not Issue 8 filesystem utilities in this catalog | No command is added under this specification |

Commands such as **find**, **walk**, and **glob** are deliberately absent. fsspec already owns these composite operations. The command library does not rebuild their traversal algorithms or add a POSIX expression engine. Likewise, **cksum**, **cmp**, **dd**, **head**, **od**, **pax**, **split**, **strings**, **tail**, **tee**, **test**, and **wc** are content, transformation, archive, expression, or stream-composition utilities rather than the filesystem primitive surface approved here.

### POSIX-shaped reduced-profile ledger

“Supported” always means the exact fsspec-cli command compatibility profile, not complete conformance of the corresponding Issue 8 utility. Omitting an option does not erase mandatory base-utility semantics. Every admitted reduction is therefore explicit:

- **basename** and **dirname** follow the Issue 8 lexical algorithms over the host-decoded argument string. The operating system cannot carry NUL in argv; newline is processed as data rather than rejected.
- The first **cat** profile requires mapped file operands instead of implementing the Issue 8 stdin cases. Stdin and **-** are a later profile.
- Base **mkdir** cannot apply the Issue 8 creation mode derived from requested or default mode and process umask on a mode-less source. It creates with the source's documented default and is a reduced profile for that source form.
- Base **cp** proves target resolution, replacement, bytes, diagnostics, and partial state, but a mode-less source cannot prove all Issue 8 destination creation mode, ownership, link, and characteristic requirements. Such a row passes only the named reduced profile.
- Base **mv** is admitted only as a same-source, source-form-specific reduced profile. It does not imply atomic rename, POSIX identity preservation, or generic metadata preservation unless those facts are separately evidenced.
- Base **rm** cannot perform the Issue 8 implicit non-writable-file prompt when a source exposes no authoritative mode and access model. Explicit interactive behavior remains unsupported.
- A source-reported **type=file** value is only the common fsspec type shape. It does not by itself prove a POSIX regular file or prove absence of a symbolic link. Link behavior stays outside each first profile unless the source supplies authoritative evidence.

Documentation and matrix rows must repeat the relevant divergence. No reduced-profile pass may be summarized as “POSIX compliant.”

### Incremental delivery order

Each numbered item is a separate delivery slice and normally a separate pull request. Implementation slices require a command compatibility profile; research slices either approve one or preserve strict rejection. A later item cannot silently broaden an earlier profile.

1. Generalize the existing private command lifecycle and diagnostic machinery so it accepts the concrete command name instead of hard-coding **ls**, without adding a public runner or changing **ls**.
2. Add source-free **basename** base behavior.
3. Add the optional **basename** suffix operand.
4. Add source-free **dirname** base behavior.
5. Add mapped-file **cat** with no options and exact binary output.
6. Add POSIX stdin behavior for operand-free **cat** and **-** operands only after binary stdin and mixed-input order are proven.
7. Add base **mkdir** with parent creation disabled.
8. Add **mkdir -p** by delegating to the fsspec parent-creation composite.
9. Add base **rmdir** for empty non-root directories.
10. Add XSI **unlink** for exactly one source-reported file operand.
11. Add base **rm** for one or more source-reported file operands.
12. Add **rm -f**, including its zero-operand and missing-operand behavior.
13. Add **rm -d** as an extension of base **rm** that additionally admits empty directories.
14. Research **rm -R/-r** through the source recursive-removal composite. Keep the flags unsupported unless that operation exposes a directly verifiable complete-result and containment contract; do not implement a second traversal engine.
15. Add **rm -v** only where each reported removal can be confirmed.
16. Add two-operand, same-source source-reported-file **cp**.
17. Add two-operand, cross-source source-reported-file **cp** through bounded local staging.
18. Add multiple source-reported file sources into an existing destination directory.
19. Research same-source **cp -R** through the fsspec recursive-copy composite. Keep it unsupported unless the operation exposes a directly verifiable complete tree result; cross-source recursive copy also remains unsupported.
20. Add two-operand, same-source source-reported-file **mv** only for source forms whose complete move outcome and reduced-profile divergences are positively evidenced.
21. Keep cross-source **mv** unsupported until a profile can preserve the Issue 8 characteristics, detect source changes, and condition source deletion on the exact copied generation.
22. Add same-source directory and multiple-source **mv** forms only when the relevant operation has its own complete profile.

The first implementation of a command accepts only the listed base form. Every unlisted option, grouped option containing an unsupported letter, long option, operand count, source type, or destination shape is rejected or reported exactly as that command profile specifies. Unsupported behavior does not fall back to a weaker command.

### Common command contract

- Filesystem operands use only **name:/path**. Both operands of copy and move are mapped operands. There is no bare-local shortcut, default source, current source, current directory, tilde expansion, dot-segment resolution, separator normalization, protocol URL, or mount-point interpretation.
- Lexical **basename** and **dirname** operands are strings, not mapped filesystem operands. They do not validate a source name, test existence, or acquire a source. The complete input token is processed by the Issue 8 lexical algorithm, so a source-looking prefix is ordinary string content.
- Every filesystem command performs complete argument, option, mapped-operand, known-source, and statically detectable safety preflight before calling a source factory.
- All distinct referenced sources are acquired once, sequentially, in first operand appearance order, before filesystem I/O, mutation, or stdout. Repeated references reuse the invocation-owned filesystem. All acquired sources are cleaned up in reverse order on the same loop.
- The CLI uses the documented, version-tested async fsspec operation that directly represents the requested primitive. When fsspec already owns a required composite, such as parent creation, recursive removal, or same-source recursive copy, the CLI delegates and validates; it does not rebuild that algorithm from listings.
- Method presence is not capability evidence. A base-class default, inherited composite, successful construction, or **NotImplementedError** does not automatically classify a source. Compatibility comes from executing the exact command profile and validating the result and final state.
- There are no backend-type branches, protocol-name branches, hidden capability probes, runtime matrix registry, retries, alternate-operation fallbacks, or fabricated metadata.
- CLI-originated operations are sequential in v1. A delegated fsspec-owned composite may schedule its internal work differently; that behavior belongs to its source-form profile. Operand order controls CLI work order. A later failure does not roll back already completed earlier operands, and a nonzero result must not imply rollback.
- Only source-reported **file** and **directory** types are admitted where a profile names them. These are fsspec type values, not proof of a POSIX regular file or non-link. Links and other or missing types are incompatible until an explicit profile says otherwise.
- Utility output is stable between terminal and redirected execution. Data and successful command output go to stdout; diagnostics go to stderr. Names are not colored, quoted, escaped, decorated, or arranged in columns unless a future command profile explicitly requires an Issue 8 format.
- Ordinary command success is status 0, runtime/source/result/output/cleanup failure is status 1, and source-free usage or preflight rejection is status 2. The command library does not appropriate status 1 for a future utility whose standard gives it a normal negative-result meaning.
- Broken-pipe and other output failures stop further output, preserve already accepted bytes, run source cleanup, and return failure under the command profile. Control-flow exceptions are cleaned up and propagated unchanged.
- Diagnostics identify the concrete command and mapped operand, use the existing safe control-character escaping, never include credentials or file bytes, and never expose tracebacks for ordinary failures.
- A command with multiple independent operands continues after an ordinary per-operand failure when Issue 8 permits useful later work. A command stops immediately when output is unusable, source lifecycle failed, or continuing could delete data after an unverified transfer.

### Command-specific behavior

#### basename and dirname

- Both commands are pure, source-free lexical utilities. They do not require the source mapping to contain a name that happens to appear before a colon.
- **basename** accepts one required string and one optional suffix. It removes leading directory components and applies the Issue 8 suffix rule. It writes exactly one result followed by one newline.
- **dirname** accepts exactly one string and applies the Issue 8 lexical algorithm to the complete token. It writes exactly one result followed by one newline.
- Neither command normalizes a mapped path for later filesystem use or promises that its output is a valid mapped operand. For example, the qualifier-like text is part of the input string, not a filesystem selection.
- Extra operands, unsupported options, and NUL are source-free usage failures. Newline is processed verbatim by the lexical algorithm and the command still appends its required terminating newline. Exact **--help** remains framework-owned.

#### cat

- The first profile accepts one or more mapped source-reported file operands and no options. Directories and all other reported types fail per operand. The profile makes no independent POSIX regular-file or link-identity claim.
- Files are emitted in operand order as exact bytes, with no headers, separators, decoding, newline insertion, newline removal, or text normalization.
- Each remote object is transferred through the source file-download primitive into a secure invocation-owned local temporary file, then forwarded to binary stdout in bounded chunks. At most one operand is staged at a time. This avoids relying on a whole-object in-memory **cat** implementation and keeps memory independent of object size.
- A source read or local staging failure for one operand produces a diagnostic and allows later operands to run. Bytes already emitted for earlier operands remain emitted. An output failure stops the invocation.
- Temporary files are always closed and removed after success, failure, cancellation, or output break. Temporary paths and file content never appear in evidence or diagnostics.
- The first profile rejects no operands, bare **-**, and **-u** during source-free preflight. A later stdin profile may make no operands mean stdin and may treat **-** as stdin at its exact position. That profile must preserve mixed mapped-file/stdin order and reject attempts to consume stdin more than the approved contract allows.
- **-u** stays unsupported until the implementation can prove without-delay, unbuffered forwarding and flushing rather than merely accepting a no-op flag.

#### mkdir

- Base **mkdir** accepts one or more mapped directory operands, calls the source directory-creation primitive with parent creation disabled, and processes operands in order.
- A missing parent, existing file, existing directory, unsupported operation, or incompatible result is a failure for that operand. Earlier successful directories remain.
- The command verifies that a successful result names a directory at the requested path. It emits no stdout.
- **mkdir -p** delegates to the source parent-creation composite with existing directories allowed, then verifies the requested directory. It does not hand-build parent traversal.
- **-m** is strictly unsupported because generic fsspec and native vosfs do not provide an authoritative POSIX mode contract. Backend defaults are not described as applying a requested mode.

#### rmdir and unlink

- Base **rmdir** accepts one or more mapped operands, requires each to identify a directory, and delegates empty-directory removal to the source. It emits no stdout and continues across ordinary per-operand failures.
- No generic async **rmdir** capability is assumed. Each exact source form and fsspec version must expose and prove the required async empty-directory operation; absent evidence leaves the source row **unverified**, while a reached profile contradiction records **fail**.
- **rmdir** rejects a configured source root and final dot or dot-dot component before source acquisition. It never treats recursive removal as empty-directory removal.
- **rmdir -p** is not part of the first profile. If admitted later, it must remove successive parents without crossing the configured source root and must define partial completion when an ancestor cannot be removed.
- XSI **unlink** accepts exactly one mapped source-reported file operand, delegates to the source single-file removal primitive, verifies absence without hiding access or service errors, and emits no stdout. It makes no independent non-link claim.
- A directory, source root, final dot or dot-dot component, missing operand, extra operand, or any option is rejected or reported according to the command profile. **unlink** is not implemented as an alias that accidentally enables recursive **rm** behavior.

#### rm

- Base **rm** accepts one or more mapped operands and removes only exact file-type entries through the single-file removal primitive. A mapped path that does not exist is a per-operand runtime failure; zero operands is a source-free usage failure except under **rm -f**. Directories are rejected for lack of **-d** or **-R/-r**, and ordinary failures do not prevent later operands.
- All **rm** profiles reject configured source roots and final dot or dot-dot components before source acquisition. Recursive profiles additionally reject dot-segment path components that could make lexical containment ambiguous.
- **rm -f** makes zero operands successful, makes missing entries silent and non-failing, and never suppresses permission, service, incompatible-result, containment, or uncertain-mutation failures. Option order with future **-i** is not accepted until **-i** exists.
- **rm -d** retains base file-removal behavior and additionally removes empty directories through the empty-directory primitive. It does not treat a directory marker as an ordinary file.
- **rm -R** and **rm -r** are equivalent candidate profiles that would call the source recursive-removal composite. The CLI does not enumerate and delete a tree itself. The flags retain a command-preflight **unsupported** profile unless the delegated operation exposes a directly verifiable complete-result contract covering containment, non-following of links, partial failure, cancellation, and root protection. Happy-path tests or an inherited successful return are not enough. If later admitted, native vosfs source evidence begins **unverified** and becomes **fail** only when a qualifying reached-command test contradicts the profile.
- **rm -v** writes a mapped pathname only after that removal is confirmed. If the delegated recursive operation cannot report individual confirmed entries, verbose recursive removal stays unsupported rather than inventing a list.
- **rm -i** and implicit permission-based prompting remain unsupported. Generic fsspec does not expose portable writability/mode facts, and this release does not own a locale-aware prompt and affirmative-response contract.
- Removal success is verified by an absence check that distinguishes not-found from permission, timeout, service, and parse failures. An uncertain post-mutation check returns failure and invalidates no truth by claiming the entry is gone.

#### cp

- The first **cp** profile accepts exactly two mapped operands and no options. The source must be a source-reported file. If the destination identifies an existing directory, the source basename is appended; otherwise the destination is the exact target. Its parent must already be a directory. An eligible existing destination is replaced under base **cp**; replacement is not introduced by **-f**.
- An exact same-source, same-path copy is rejected without mutation. Source and destination source names, rather than Python object identity or protocol strings, determine whether an operation is same-source.
- Same-source source-reported-file copy delegates to the source copy primitive. Cross-source source-reported-file copy downloads to a secure local temporary file and uploads to the destination source. No source contents pass through a text interface.
- A command reports success only after the destination is a source-reported file with the expected byte count and content. Portable verification uses bounded, disk-backed byte comparison when authoritative matching digests are unavailable. Any digest optimization needs its own tested profile.
- A failed or unverifiable copy may leave a partial or complete destination. The command reports that truthfully, does not claim rollback, and never deletes the source.
- Multiple-source **cp** requires the final operand to be an existing destination directory. Sources are processed in original order, each target uses the source basename, eligible name collisions use the base replacement behavior, and earlier successes remain after later failure.
- Same-source **cp -R** remains unsupported unless the source recursive-copy composite exposes a directly verifiable complete tree result. The CLI does not invoke a CLI-owned find/walk implementation to manufacture that proof. Links, special types, destination-inside-source ambiguity, dot-segment containment, partial completion, and cancellation must all be represented by the operation contract; happy-path tests and a success return are insufficient.
- Cross-source **cp -R** remains unsupported until a separate profile demonstrates that the operation can be bounded and verified without adding a second traversal implementation.
- **-p** is unsupported without complete mode, owner, group, time, and related metadata preservation. **-H**, **-L**, and **-P** are unsupported without native symbolic-link identity and traversal semantics. **-i** is unsupported without the prompt contract. **-f** is not ordinary replacement: it requires unlink-and-retry when opening or truncating an existing destination fails. It stays unsupported until that exact failure classification and retry behavior are profiled.
- In **cp**, bare **-** is not stdin or stdout. Because all filesystem operands are mapped, a remote or local file literally named hyphen is addressed through its mapped path.

#### mv

- The first **mv** profile accepts exactly two mapped source-reported file operands and no options, with the same target-directory resolution, eligible existing-destination replacement, and parent requirements as **cp**.
- Same-source **mv** delegates to the source move operation only when the exact source form has positive evidence for source retention or removal, destination completeness, replacement, failure, and cancellation. An inherited default is not automatically trusted.
- After both operands exist and destination-directory resolution is complete, an exact same source name and backend path is a successful no-op: no move operation, stdout, or diagnostic, and status 0. Different spellings are not assumed to identify the same entry without authoritative source identity.
- Cross-source **mv** is unsupported in this specification. Issue 8 cross-filesystem behavior includes duplicating required access time, modification time, owner, group, and mode characteristics before source removal. The fail-closed project contract also requires proof that the source did not change between transfer and deletion, using immutable generation identity and conditional deletion or an equivalent source-owned guarantee. Byte verification alone satisfies neither requirement.
- A future cross-source profile must still specify the case where destination verification succeeds but source removal fails or becomes uncertain: the verified destination remains, the source may remain, the command reports failure, and it does not delete the destination to manufacture rollback.
- Existing-destination replacement, same-path detection, source changes during transfer, destination changes during verification, and cancellation each have explicit failure tests. No atomic rename claim is made across configured sources.
- Moving a directory does not require **-R** under Issue 8, but it is a later profile. Same-source directory movement delegates to the source move composite. Cross-source directory movement stays unsupported until it can satisfy the same verification and source-retention rules without recreating traversal.
- Multiple-source **mv** requires an existing destination directory and processes sources in order. A later failure leaves completed earlier moves completed.
- **-i** is unsupported without the prompt contract. **-f** suppresses overwrite prompting; it does not create base replacement semantics. Because prompting and authoritative writability are not yet supported, **-f** also remains unsupported. Option precedence is not approximated before both options exist.

### Command admission and completion

A command or option is complete only when all of the following exist:

1. A command compatibility profile defines accepted argv, preflight, source acquisition, exact fsspec operation, accepted types and result shapes, output, diagnostics, continuation, partial completion, cleanup precedence, status, and unsupported behavior.
2. The implementation is reachable only through **App(sources).typer_app** and preserves the single synchronous Typer adapter plus one awaited command coroutine.
3. Hermetic external-behavior tests pass through the App seam.
4. The isolated built wheel passes the same profile without repository imports or undeclared dependency leakage.
5. The tested command matrix records exact source form, dependency versions, platform, status, gates, and immutable evidence. Missing or stale evidence remains **unverified**.
6. Native vosfs evidence exists where a positive native claim is made. An admitted native source row may remain **unverified**; a deliberately rejected command or option is represented separately by its source-free command-preflight **unsupported** profile.
7. User documentation and the independent fsspec-cli changelog describe only the supported forms and link unsupported forms to their profile.

## User Stories

1. As a host author, I want one embedded command library for filesystem primitives, so that CANFAR, vostools-compatible tools, and later products do not reimplement command parsing and failure behavior.
2. As a host author, I want the only stable seam to remain **App(sources).typer_app**, so that adding commands does not create a second invocation API.
3. As a host author, I want to provide multiple preconfigured async filesystem sources with independent names, so that two endpoints using the same fsspec protocol can coexist.
4. As a security owner, I want authentication, endpoint discovery, and filesystem construction to remain in the host, so that the command library never owns credentials.
5. As a user, I want every filesystem path to name its source explicitly, so that a command never guesses local versus remote state.
6. As a user, I want both sides of copy and move to use mapped operands, so that upload, download, and remote-to-remote transfer share one grammar.
7. As a user, I want POSIX Issue 8 conventions plus an explicit divergence ledger for each reduced profile, so that familiar command knowledge transfers without implying full utility conformance.
8. As a user, I want unsupported commands and options rejected plainly, so that a familiar spelling never hides weaker semantics.
9. As a maintainer, I want compatibility scoped per command and option, so that successful **ls** evidence cannot imply **cp**, **mv**, **rm**, or metadata support.
10. As a maintainer, I want no CLI-owned traversal implementation, so that fsspec remains responsible for find, walk, glob, and its existing recursive composites.
11. As a maintainer, I want each base command to land before its flags, so that failures are attributable to one small semantic surface.
12. As a maintainer, I want every flag to have its own profile and evidence row, so that grouped options cannot accidentally admit unsupported behavior.
13. As a script author, I want **basename** to apply lexical Issue 8 rules without touching a source, so that it is deterministic and side-effect free.
14. As a script author, I want the optional **basename** suffix behavior to be independently tested, so that edge cases do not depend on a backend.
15. As a script author, I want **dirname** to process the complete input string lexically, so that it does not require the path to exist.
16. As a host author, I want lexical commands to acquire no sources, so that they work without network, credentials, or backend lifecycle.
17. As a user, I want **cat** to emit exact file bytes in operand order, so that binary data is never decoded or rewritten.
18. As a user, I want **cat** to add no headers or separators, so that concatenated output is byte-compatible with the requested files.
19. As a user, I want large **cat** inputs staged and forwarded with bounded memory, so that object size does not become process memory size.
20. As a user, I want a failed **cat** operand diagnosed while later operands remain usable, so that one missing file does not conceal independent requested data.
21. As a pipeline author, I want broken stdout to stop further reads and still clean up sources and temporary files, so that closed pipelines do not waste remote work.
22. As a user, I want stdin and **-** behavior added only when mixed binary ordering is proven, so that a nominal POSIX feature is not a text-mode approximation.

## Implementation Decisions

### Locked architecture

- Preserve the independently versioned, separately installable fsspec-cli distribution in the shared uv workspace and shared lock.
- Preserve a library-only package with no console script, module executable, public main helper, public run helper, or public async invocation seam.
- Preserve **App(sources).typer_app** as the sole stable composition and external behavioral seam.
- Preserve public exports as **App** and the typing contract for an async filesystem source unless a later public-API issue explicitly changes them.
- Preserve a non-empty **Mapping[str, AsyncFilesystemSource]**. Each source callable returns a fresh async context manager yielding one invocation-owned async filesystem for each referenced name.
- Preserve explicit host adaptation of synchronous backends. The command library never auto-wraps a raw synchronous filesystem.
- Preserve one zero-command-logic synchronous Typer adapter per concrete command. Everything beneath it executes in one awaited coroutine and one event loop.
- Preserve independent fsspec-cli versioning, changelog, Release Please lane, tag prefix, GitHub Releases-only publication, and no PyPI publication.

### Primitive boundary

- Admit only the filesystem primitives and lexical helpers listed as implementable in the catalog. Do not expand this issue into a general shell, a complete POSIX utilities package, or a mirror of every fsspec convenience method.
- Delegate directly to the relevant async fsspec primitive. If fsspec already supplies a necessary parent, recursive removal, recursive copy, or move composite, use it only when its directly verifiable result contract is sufficient; never reconstruct it from lower-level listing merely to manufacture proof.
- Do not add **find**, walk, glob, traversal expressions, content transforms, checksums, comparison, line selection, counting, archive, or stream fan-out commands in this work.
- Cross-source source-reported-file **cp** is the sole approved command-level composition because no one configured source owns both endpoints. Keep it bounded, sequential, disk-staged, and independently evidenced. Cross-source **mv** is not approved by this specification.
- Method existence and inheritance do not constitute support. Every source form must prove the complete observable command profile.

### Parsing and source lifecycle

- Reuse the exact mapped filesystem operand grammar approved for **ls**. Do not normalize the backend path or accept protocol URLs and bare local paths.
- Parse options strictly per command. Preserve **--** as the end of option processing and framework-owned exact **--help**. Reject the first unsupported token in argv order before acquiring sources.
- Treat lexical helper operands as strings rather than filesystem operands. They run without a source factory or filesystem operation.
- Complete command preflight before source acquisition. Acquire all distinct referenced sources in first appearance order before any filesystem operation, mutation, or output.
- Validate every yielded filesystem as a native or explicitly adapted asynchronous fsspec filesystem in asynchronous mode.
- Reuse each acquired source for repeated operands and close all entered sources once in reverse order on the same event loop.
- Generalize command labels inside the existing private lifecycle and diagnostic flow. Do not create a second orchestration stack or public runner.

### Operations, validation, and failure

- Keep CLI-originated v1 operations sequential. Do not add CLI concurrency, automatic retries, fallback operations, background loops, threads, nested-loop workarounds, or cancellation shielding. A delegated source composite may retain its own documented scheduling.
- Use source-reported file and directory type values as the common floor, without calling them POSIX regular-file or non-link proof. Treat links, special entries, missing type, malformed metadata, and unexpected result shapes as incompatible unless a command profile admits them.
- For mutation, validate the final observable state where that validation is safe and meaningful. Distinguish confirmed success, confirmed failure, and uncertain outcome.
- Never claim transactionality or rollback. Earlier operand success and destination residue may survive a later failure.
- For cross-source copy, stage one source-reported file at a time and verify destination bytes before success. Do not implement cross-source move under this specification.
- Never emit file contents, credentials, source configuration, temporary paths, or unsanitized remote responses in diagnostics or matrix evidence.
- Protect configured source roots and final dot or dot-dot components for destructive commands. Add stricter dot-segment rejection wherever a recursive operation would otherwise have ambiguous lexical containment.
- Do not infer absence from a generic false result that can hide permission or service errors. Verify absence through an operation whose not-found outcome is distinguishable.
- On output failure, stop producing output and further mutation where safe, then run normal source and temporary-resource cleanup.
- Ordinary cleanup exceptions append diagnostics and force status 1 without replacing an existing ordinary failure. An original escaping **BaseException** wins and propagates unchanged after every exit attempt. If no original control-flow exception exists, the first cleanup or diagnostic-rendering **BaseException** propagates after all exits even when ordinary command failure already occurred; previously accepted output and diagnostics remain.

### Compatibility, documentation, and release

- Create one command compatibility profile for every base form and every later flag or operand-shape expansion.
- Extend the tested command matrix with exact command profile, source or command-preflight scope, source form, dependency and package versions, platform, status, required gates, and immutable evidence.
- Preserve the four matrix states: **pass**, **fail**, **unsupported**, and **unverified**. Missing, stale, blocked, or unavailable evidence is **unverified**.
- Record **unsupported** only when the command profile deliberately rejects the surface and negative tests prove source-free rejection. Do not derive it from **NotImplementedError** alone.
- Keep native vosfs first among real-backend evidence targets. Generic Local and Memory evidence is useful but does not override the native vosfs row.
- Keep hermetic, isolated-wheel, and trusted live evidence as distinct gates. Live infrastructure or credential failure yields **unverified**; a reached command contradiction yields **fail**.
- Keep fsspec-cli command work and vosfs backend hardening in separate pull requests and issues. This issue consumes corrected backend behavior but does not implement it.
- Do not modify or delay the approved fsspec-cli 0.1.0 release automation and first-release issues. New commands belong to a later independent release.
- Update user documentation and changelog with the exact supported forms, strict rejections, partial-state rules, and matrix links. Never publish a blanket “POSIX-compatible” or “works with all fsspec backends” claim.

## Testing Decisions

- The highest and default test seam is a full invocation through **App(configured_sources).typer_app**. Assert argv, stdout bytes or text, stderr, exit status, awaited source behavior, cleanup, and final filesystem state.
- Do not test private command handlers as the primary contract. Focused pure tests are acceptable only for the Issue 8 lexical basename/dirname algorithm, control-character rendering, or a similarly isolated protocol boundary; every result still needs App-seam coverage.
- Extend the existing recording async source and filesystem doubles with the actual fsspec operations used by each command. Record factory, entry, operation, result, exit, event loop, and failure order without mocking the command layer.
- Run every filesystem profile against explicitly adapted Local and Memory source forms where the operation is meaningful. Keep adaptation in the host fixture, never production CLI code.
- Run every claimed native vosfs profile against the hermetic transport simulator. Use the same public App seam and validate remote request sequence and resulting namespace rather than internal helper calls.
- Block unexpected network in hermetic tests. Expected routes must be both exhaustive and exclusive.
- Preserve dedicated tests for source-free preflight: unsupported options, grouped options, **--**, missing and extra operands, malformed mapped operands, unknown source names, roots, dot entries, source order, and the guarantee of zero factory calls.
- Preserve lifecycle fault matrices for first and later source acquisition failure, wrong-mode yields, operation failure, reverse cleanup, multiple cleanup failures, output failure, cancellation, and control-flow propagation.
- Add lexical golden tests for Issue 8 **basename** and **dirname** edge cases: empty-looking results, root slashes, repeated slashes, trailing slashes, suffix equal to the complete basename, suffix absence, source-looking prefixes, option-looking operands after **--**, Unicode, NUL rejection, and verbatim embedded-newline processing plus the required terminating newline.
- Add binary **cat** tests with empty files, every byte value, invalid UTF-8, embedded carriage returns and NUL bytes, large content, multiple sources, mixed successes and failures, short local writes, temporary-disk failure, cleanup failure, broken pipe, and cancellation.
- Use a real subprocess for byte-level stdin/stdout and pipe behavior. Typer's text capture alone is insufficient proof for binary commands.
- Add **mkdir** matrices for missing parent, existing directory, existing file, parent file, root, multiple operands, partial success, **-p** idempotence, backend over-eager parent creation, post-success type verification, and the exact source-default mode divergence where process umask cannot be represented.
- Add **rmdir** and **unlink** matrices for source forms with and without a version-tested async empty-directory operation, empty and non-empty directories, files, missing paths, roots, dot and dot-dot, access failures, ambiguous post-state, multiple operands where allowed, and absence verification.
- Add base **rm** matrices for missing files, files, directories, other types, multiple sources, continuation, root guards, and partial mutation.
- Add **rm -f** matrices for no operands, all missing, mixed missing and existing, permission failure, service failure, and confirmation that non-not-found failures remain visible.
- Add recursive **rm** negative and admission tests for delegated-operation invocation, absence or presence of a directly verifiable result contract, source-root rejection, dot-segment rejection, non-following of links, malformed descendants, cancellation, partial failure, and uncertain final state. Happy-path behavior alone cannot admit the flag. Do not make a native vosfs pass claim until the backend operation satisfies the runtime contract.
- Add **cp** target-resolution matrices for absent target, existing file, existing directory, missing parent, same path, same source name, distinct source names yielding similar backend classes, base replacement, multiple sources, duplicate basenames, unsupported directory sources, and the declared creation-mode, ownership, and link divergences.
- Add cross-source copy fault injection at download construction, download streaming, local close, upload construction, upload streaming, destination verification, temporary cleanup, source cleanup, and output diagnostics. Assert source retention and truthful destination residue.
- Verify copied content byte-for-byte, including empty, binary, large, and changing-source scenarios. Test authoritative digest optimization separately if ever admitted.
- Add **cp -R** negative and admission tests only through a source's real recursive-copy operation. Assert that the CLI does not enumerate through its own find/walk path and that a void or unverifiable success cannot admit the flag. Cover links, special types, destination-inside-source, root, partial result, and cancellation.
- Add same-source **mv** matrices for destination failure, source disappearance, destination replacement, same-path input, source and destination characteristics, cancellation, and cleanup failure.
- Keep cross-source **mv** as source-free unsupported behavior. A future admission suite must add source-generation changes during transfer, conditional-delete failure, all required characteristic-copy outcomes, destination verification, source deletion failure, and cancellation before and after the deletion guard.
- For same-source move, test the exact backend-owned operation and final namespace. Do not substitute cross-source orchestration in a test that claims native move compatibility.
- Add matrix-negative tests for every deliberately unsupported option and operand shape of an admitted command. The tests must prove empty stdout, one stable diagnostic, status 2, no source factory call, and no filesystem mutation. Unadmitted catalog utilities receive a documented disposition; they do not require Typer stubs or source-form matrix rows.
- Build the fsspec-cli wheel, install it in a clean environment with only declared runtime dependencies, prove imports resolve from site-packages rather than the repository, and rerun the command profiles. Install vosfs only in the separate integration environment.
- Test supported Python versions on the established Linux matrix plus the established macOS and Windows representatives. Keep path expectations source-profile-specific without weakening mapped operand grammar.
- For trusted live OpenCADC tests, allow read-only commands in an explicitly configured test namespace. Mutation commands require a generated run-owned namespace, collision-resistant naming, exact allowed prefix validation, an unconditional reverse cleanup attempt, recorded cleanup outcome, and sanitized evidence. Incomplete cleanup or unavailable safe setup leaves the live row **unverified** or **fail** according to whether the command contract was reached.
- Keep live destructive testing separate from the vosfs package hardening gate. A CLI test must not compensate for an unsafe backend operation.
- Add release acceptance that the matrix, documentation, changelog, built wheel, and GitHub Release all describe the same command profiles. New command release tags retain the **fsspec-cli-v** prefix.
- Prior art to reuse includes the current plain-**ls** App-seam tests, source lifecycle recorder, adapted Local and Memory fixtures, hermetic native-vosfs transport fixture, isolated-wheel workflow, trusted live evidence workflow, command compatibility profile format, and tested command matrix.

## Out of Scope

- Reimplementing **find**, walk, glob, recursive traversal, expression evaluation, or any other composite already owned by fsspec.
- Adding checksum, comparison, slicing, counting, conversion, archive, pager, terminal, or stream-fan-out utilities such as **cksum**, **cmp**, **dd**, **head**, **more**, **od**, **pax**, **split**, **strings**, **tail**, **tee**, **test**, or **wc**.
- Claiming complete POSIX Issue 8 conformance, a POSIX environment, GNU compatibility, BSD compatibility, macOS command parity, or generic support across all fsspec backends.
- Implementing **cd**, **pwd**, or **umask** through hidden process state, a default source, or a virtual current directory.
- Approximating **df** from object totals or **du** from logical byte sizes.
- Inventing modes, owners, groups, link counts, allocated blocks, device identities, filesystem capacity, access checks, timestamps, hard links, symbolic links, FIFOs, sockets, or atomic rename.
- Implementing **chmod**, **chgrp**, **chown**, **ln**, **link**, **readlink**, **realpath**, **touch**, **mkfifo**, or default **pathchk** before an exact capability profile and evidence exist.
- Treating **stat**, **mount**, **sync**, **install**, **truncate**, or **mktemp** as POSIX Issue 8 utilities under this work.
- Changing existing plain **ls**, **ls -A**, strict **ls -a/-l** rejection, mapped operand grammar, source lifecycle, public exports, or the sole App seam.
- Adding bare local paths, protocol URLs, implicit uploads/downloads, automatic synchronous wrapping, backend construction, authentication, endpoint discovery, config files, a console script, module execution, or a public async runner.
- Adding runtime capability negotiation, a backend registry, protocol branches, retries, fallback operations, concurrency, progress bars, color, columns, TTY-dependent output, or Rich UI.
- Solving vosfs transport, path, cache, staging, link, mutation, containment, resource, or lifecycle defects. Those remain owned by their vosfs issues.
- Altering fsspec-cli 0.1.0 release automation or first-release acceptance, publishing to PyPI, extracting the package to another repository, or coupling its release schedule to vosfs.
- Migrating CANFAR, vostools, or other downstream host commands in the same work.
