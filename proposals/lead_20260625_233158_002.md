## Questions to Resolve

### 1. Plugin vs Companion App Boundary
- Does VolteX require a Revit plugin (.NET, loaded inside Revit), or can it run as an external companion process?
- - Revit API is only accessible from within the Revit process — a plugin is required for live model access.
- - - Decision needed: what lives inside Revit vs what runs outside?
### 2. Live Data Model
- What model data does Phase 1 expose? (elements, parameters, views, sheets, warnings?)
- - **Phase 1 is read-only. No write-back to Revit until the read-only bridge is stable.**
- - - What is the minimum useful payload for a first working demo?
### 3. Snapshot vs Event Stream
- Does the bridge push data on change (event-driven) or respond to poll requests (snapshot)?
- - Revit API events: `DocumentChanged`, `ViewActivated`, `SelectionChanged` — available but constrained.
- - - Decision: event-driven or polling? Hybrid?
### 4. Element Identity Strategy
- Revit element IDs are session-local and can change across sync. UniqueId is stable.
- - What is the canonical key for a VolteX element? `UniqueId`? Custom parameter? Both?
### 5. IPC Transport Options
- Options for plugin to external process communication:
-   - Named pipes (Windows, low latency)
-   -   - Local HTTP / WebSocket (portable, debuggable)
-   -   -   - Shared memory (fast, complex)
-   -   -   -   - File-based (simple, slow)
-   -   -   -   - - Decision: which transport for Phase 1?
### 6. Revit API Constraints
- Revit API calls must happen on the main thread (external event or idling event).
- - No async/await inside Revit event handlers without `IExternalEventHandler`.
- - - `SelectionChanged` is useful for live awareness but must be treated as read-only — do not modify the model or open write transactions inside restricted event handlers.
- - - - The External Events / Idling pattern needs to be evaluated for safe execution before use.
- - - - - Decision: what is the threading model for Phase 1?
### 7. Security / Trust Boundary
- The plugin runs inside Revit (elevated, trusted).
- - The external receiver runs as a normal user process.
- - - What data leaves the machine? (Phase 1: nothing — local only)
- - - - Authentication: Phase 1 is localhost only. No auth required yet.
### 8. Phase 1 Acceptance Test

> A CLI command or minimal browser page displays the active Revit view name, ElementId, and UniqueId, updated live or near-live, after the plugin is loaded, with no model write-back.

### 9. Out of Scope for Phase 1
- Multi-user collaboration
- - Cloud sync or remote access
- - - Write-back to Revit model
- - - - UI / frontend beyond a minimal CLI or raw JSON endpoint
- - - - - Discord integration
- - - - - - Authentication / user management
- - - - - - - Signed installer
### 10. Revit Version / .NET Target
- Which Revit version is Phase 1 targeting first?
- - Which .NET target is required for that Revit version? (Revit 2024+ requires .NET 8; earlier versions use .NET Framework 4.8)
- - - Are we building for one Revit version first, or planning multi-version support later?
### 11. Deployment / Loading Model
- How will the Phase 1 plugin be loaded into Revit?
- - `.addin` manifest for local development?
- - - Manual copy to Revit Addins folder for Phase 1?
- - - - Signed installer is out of scope for Phase 1.
