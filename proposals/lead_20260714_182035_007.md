# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**Join links use framed:// scheme: not tappable when shared in chat apps**

*Project description:*
Join links use the custom `framed://join?v=1&t={join_token}&k={base64url game key}` scheme (`lib/core/crypto/qr_payload.dart`, `QrPayload.encode()`/`.parse()`). `app_router.dart`'s `_redirectJoinLink` rewrites the incoming `framed://join` URI to the internal `/join` route, and the Android intent-filter in `android/app/src/main/AndroidManifest.xml` registers `<data android:scheme="framed" android:host="join"/>` with `flutter_deeplinking_enabled`.

Custom schemes aren't registered as tappable links in most contexts outside the app. When a join link is shared via #48 (OS share sheet) into WhatsApp, Signal, iMessage, or Slack, the recipient sees plain text, not a link — they have to copy/type the URI by hand. QR scanning still works, but that only covers players who are physically together; anyone joining remotely hits this dead end.

## Fix: Android App Links / iOS Universal Links

Switch to an `https://` URL, e.g. `https://getframed.fun/join?v=1&t=...&k=...`, that opens Framed directly (no chooser dialog) if installed, and falls back to a web page if not.

### Checklist

- [x] Static landing page for the fallback + verification files, hosted on GitHub Pages for this repo (`docs/` folder on `main`).
- [x] `/.well-known/assetlinks.json` served from `getframed.fun`, for Android App Links verification.
- [x] `/.well-known/apple-app-site-association` served from `getframed.fun`, for iOS Universal Links verification.
- [x] `android/app/src/main/AndroidManifest.xml`: `<intent-filter android:autoVerify="true">` with an `https` scheme and `android:host="getframed.fun"` `<data>` block, replacing the old `framed://join` one. `app_router.dart`'s `_redirectJoinLink` and `QrPayload` now speak the new `https` shape only — no real users exist yet, so there was no reason to keep both.
- [x] iOS: Associated Domains entitlement (`applinks:getframed.fun`, `ios/Runner/Runner.entitlements`, wired into all three build configs in `project.pbxproj`).

The key moved from the query string into the URL fragment while this was being built (not in the original checklist, but the same principle as the rest of this app's crypto): a fragment never leaves the client, so it can't leak into `getframed.fun`'s own request logs or a chat app's link-preview bot fetching the shared URL server-side. `v`/`t` stay in the query — only the key needed that treatment.

App Link/Universal Link verification silently fails until those last two are real values — that's a Google/Apple-side check against what's actually served at `getframed.fun`, nothing client-side can route around it.

### App id: staying as `me.roboroads.framed`

Considered renaming to `fun.getframed.framed` to match the domain, decided against: `getframed.fun` is just this app's domain, not a company — Roboroads is, and `me.roboroads.framed` is Robbin's own personal app id. Not required for App Links either way (`assetlinks.json` just lists whatever the real package name is, already handled above).
