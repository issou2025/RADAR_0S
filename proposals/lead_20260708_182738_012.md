# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**BE-905 · Realtime: messaging & reservation status (SSE/WebSocket)**

*Project description:*
> **Decision note (2026-07-08):** Migrated from GraphQL/Ferry to REST. This issue was rewritten to describe the REST equivalent; acceptance-criteria intent is unchanged.

**Story:** As the developer, I want realtime endpoints for in-app messaging and reservation status changes so that the Flutter app receives realtime updates through the REST layer with no direct Supabase realtime dependency.

## Context
Replaces BE-504 and the realtime aspect of BE-1601. The server bridges Supabase realtime channels (Postgres changes) into a streaming HTTP endpoint. Prefer **Server-Sent Events (SSE)** for one-way push; fall back to WebSocket only if bidirectional messaging is needed.

## Acceptance criteria
- [ ] `GET /subscriptions/messages?conversationId=` (SSE) — bridges Supabase realtime `messages` table changes; streams `Message` events to the subscriber
- [ ] `GET /subscriptions/reservations?reservationId=` (SSE) — bridges reservation status updates; streams `Reservation` events
- [ ] Transport: SSE (`text/event-stream`) unless a bidirectional need forces WebSocket
- [ ] Auth: the connection must carry a valid JWT (`Authorization` header) or be rejected with `401`
- [ ] Server-side fan-out: one Supabase realtime channel per resource (not one per subscriber)
- [ ] Stream is cleaned up when the client disconnects
- [ ] Messaging schema from BE-1601 feeds into `MessageService`; no duplication of schema logic
- [ ] Integration test: a mocked `MessageService.subscribeToMessages` emits events, the subscriber receives them

## Definition of Done
- [ ] `melos run test` green (stream handler unit tested)
- [ ] `melos run analyze` green
- [ ] Manual smoke test: two sessions in the app exchange a message via the realtime endpoint
- [ ] No secret committed
