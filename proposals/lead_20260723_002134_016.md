## Proposed direction (from @tyeth)

Dependabot-style CI in this repo: on merge to `master`/`api-v2`, rebuild wrappers with that branch's own `bin/build-*` and open **rolling, force-updated PRs** downstream (single stable branch per target, auto-closed when the diff disappears):

| Merge to | Downstream PRs |
|---|---|
| `master` (V1) | Arduino ŌåÆ `main` (`src/wippersnapper/`); protomq ŌåÆ `main` (`protobufs-v1/` only) |
| `api-v2` (V2) | Arduino ŌåÆ `migrate-api-v2` (`src/protos/`); Python ŌåÆ `main` (via the repo's own `update_protobuf.py`); protomq ŌåÆ `main` (V1 bundle only ŌĆö V2 regenerated as a compile check but never committed) |

Requirements / notes:
- The workflow file must exist identically on **both** trigger branches (push workflows run the pushed branch's copy).
- Needs a PAT secret (`WS_SYNC_PAT`: contents + PR write on Arduino, Python (private), protomq). Existing `IO_BOT_PAT` (2021) is unverified against the private Python repo.
- Python sync must fail loudly when upstream adds a module missing from `update_protobuf.py`'s list (never silently drop a component); removals already fail inside that script.
- protomq: never un-ignore `protobufs/` ŌĆö V2 flows via submodule/local import by design.
- Question of sequencing raised in review: fix + reinstate the legacy workflow's still-wanted jobs first, then supplant? io-node is with Loren; docs deploy (dead since 2023) ŌĆö worth reviving? Both could land as extra jobs in the new workflow, after which `protoc-wrapper-generation.yml` gets deleted rather than left dormant.
- Possible extension: the Python-repo PR could also bump the `tools/Wippersnapper_Protobuf` (and optionally `tools/protomq`) submodule pointers in the same rolling PR.
- Forward-looking: if generation moves to `buf generate` (`buf.gen.yaml` driving nanopb/python/js plugins, possibly BSR), only the build step changes ŌĆö the sync/PR mechanics are agnostic. Separately, `buf-checks.yml` deserves modernization and V1 could re-enable breaking-change detection.

Input wanted from @lorennorman on: io-node coordination, any buf plans that should shape the build step, and protomq V2 submodule intentions. Once direction is agreed the closed PRs can be reopened/rebased in a day.
