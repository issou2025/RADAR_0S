*Project description:*
## Problem

CodeDecay now has a persistent JS/TS symbol graph (#636), framework-aware route evidence (#35/#144), and a language capability boundary (#565). The remaining gap is real downstream behavior mapping across frameworks, languages, and non-import relationships.

Current limits include:

- full semantic support is JS/TS-only
- Python is limited and other languages are unsupported
- route detection is primarily Next.js and Node-style patterns
- symbol propagation is bounded and static
- database consumers, queues, jobs, events, environment variables, contracts, and frontend/backend relationships are not represented in one evidence graph
- dynamic dependency injection and framework registries can be missed

## Goal

Define and ship the next adapter-backed impact-graph slice without rebuilding parsers or framework analyzers that maintained OSS projects already provide.

## Acceptance criteria

- Define an adapter contract that normalizes framework/language graph nodes, edges, confidence, source tool, and limitations.
- Model at least route/API, symbol/import, package, persistence/schema, job/event, config/env, test, and product-flow node/edge kinds.
- Evaluate maintained OSS parser/graph options before adding custom parsing; document the selection.
- Add one non-JS language adapter using a language-native or Tree-sitter-class parser.
- Add one high-value JS/TS framework adapter beyond existing generic/Next.js behavior.
- Map adapter evidence into impacted routes/areas, redteam reports, agent bundles, and MCP without treating uncertain edges as proof.
- Preserve raw tool identity and confidence in evidence.
- Degrade cleanly when optional tools are unavailable; no hidden installs or network calls.
- Add representative multi-package fixtures where a changed shared module reaches an API/UI/job consumer.
- Add false-positive fixtures for dynamic-looking but unrelated files.
- Document supported and unsupported capabilities per adapter.
