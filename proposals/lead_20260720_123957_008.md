# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**ci: add PR size guard and OSV vulnerability scanning across Rust/JS/Flutter lockfiles**

*Project description:*
This repo currently has **zero** GitHub Actions workflows, including no PR hygiene or security scanning. This issue covers both guard-rail workflows; see related issues for the actual Rust/JS/Flutter build pipelines.

### Part 1 — PR size guard
The sibling repo `Echo-Mirror-Butler/Echo-Mirror-Butler-` already has a working `pr_guard.yml` (job `size-check`) that:
- Reads `pr.additions`/`pr.deletions` via `actions/github-script@v7`
- Warns (doesn't fail) over 600 changed lines
- Fails over 3000 changed lines, unless the PR has a `large-pr-approved` label
Port this workflow to `echomirror-sdk` as-is (it's pure GitHub API logic with no repo-specific assumptions) — add the `large-pr-approved` label to this repo if it doesn't already exist.

### Part 2 — Security/vulnerability scanning
The sibling repo's `security.yml` runs Google's OSV scanner (`google/osv-scanner-action`) against `pubspec.lock` on PR + a weekly Monday cron. This repo needs the same scanner but pointed at **three** lockfiles instead of one, since this is a multi-language monorepo:
- `Cargo.lock` (Rust workspace — depends on the Rust CI issue generating and committing this)
- `package-lock.json` (npm workspace — depends on the JS scaffolding issue generating and committing this)
- `packages/flutter/pubspec.lock` (Flutter package)

Check whether those lockfiles exist yet before starting — if the dependency issues haven't landed, either wait or coordinate with whoever's picking those up, since this workflow needs all three lockfiles committed to actually scan anything.

Scope:
- `.github/workflows/pr_guard.yml` — ported size-check job, triggered on PR to `main`.
- `.github/workflows/security.yml` — OSV scan job using `google/osv-scanner-action/.github/workflows/osv-scanner-reusable.yml@v2.3.3` (pin the same version the sibling repo uses), with `scan-args` listing all three lockfile paths, triggered on PR to `main` plus a weekly cron (match the sibling's `0 3 * * 1`).
- Correct `permissions` block (`contents: read`, `security-events: write`, `actions: read`) matching the sibling repo's.

Acceptance criteria:
- A PR that changes 3000+ lines without the `large-pr-approved` label fails the size-check job
- The OSV scan job runs successfully against all three lockfiles and would flag a known-vulnerable dependency if one were introduced
- The weekly scheduled scan is configured (verify via the Actions tab that it's registered, even though you can't wait a week to see it fire)
