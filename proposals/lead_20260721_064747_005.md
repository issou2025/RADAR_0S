# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**[gda-balancing] PRD: Standard Schema 2.0 language, runtime, and evidence architecture**

*Project description:*
## Problem Statement

The accepted Standard Schema v1 design established useful foundations—canonical JSON, layered validation, typed refusals, version bundles, semantic catalogs, and normative vectors—but its authored data model is too narrow for production RPG and Roguelike balancing. A single root document, float-only parameters, attribute/tier addressing, direct AST interpretation, and a monolithic effect shape cannot represent the full lifecycle of model composition, deterministic execution, experimentation, calibration, approval, and migration without accumulating special cases.

The attempted RPG template implementation exposed that this is a Standard Schema boundary, not a template-content problem. Adding more template fields would preserve the underlying coupling and make every future genre extension harder to validate, compile, execute, and migrate.

Standard Schema 2.0 therefore needs a specification-first redesign. The design must define an extensible language, a deterministic runtime contract, and an evidence pipeline while preserving the repository's existing operational strengths. It must also close the gaps identified during review: one explicit Schema-major Kernel Specification → Language Definition Bundle authority chain, formal semantics, scoped authored authority domains, executable package compatibility, complete RPG/Roguelike coverage, deterministic scheduling, statistically meaningful calibration, CLI failure contracts, and an explicit clean-break/deprecation boundary for unpublished v1 work.

This issue is the design/specification work item. It is ready for an agent to produce the domain decisions and normative specification, but it does **not** authorize implementation of the 2.0 compiler or runtime until the human decision gates and acceptance criteria below are complete.

## Solution

Define Standard Schema 2.0 as a restricted, typed, non-Turing-complete modeling language plus compiler, runtime, and evidence architecture. The end-to-end conceptual pipeline is:

`Wire representation → Authoring AST → Typed HIR → Resolved Model (RIR) → Execution IR (EIR) → Runtime → Metrics/Evidence`

The specification will distinguish authored authorities from generated artifacts, define the non-self-hosted Schema-major Kernel Specification and the Language Definition Bundle interpreted beneath it, formalize name resolution and typing, specify lowering and runtime behavior, and provide package contracts for reusable game-domain concepts.

The first disposable tracer has executed one local RPG path through the proposed layers, but it is connectivity evidence rather than Standard Schema conformance. Remaining proof proceeds in order: a Semantic-authority conformance prototype, an Orthogonality/extensibility prototype, then a production RPG tracer and Roguelike cross-genre tracer. Roguelike coverage must extend the same abstractions with procedural generation, run/meta progression, and reset semantics rather than creating a parallel language.

The design will preserve and generalize the proven v1 infrastructure: preflight resource caps, structural/semantic validation separation, typed refusals, numeric contracts, canonical emission, version bundles, semantic catalogs, normative vectors, self-describing command surfaces, and explicit random seeds.

## Dogfooding Evidence and Remaining Validation

The disposable RPG tracer at [commit `77f8ffb`](https://github.com/aigengame/godot-agent/commit/77f8ffbf6867409600fdba33f5001aace6896d1e) demonstrated one local exact-integer path across Language Definition Bundle admission, Model Source Package, Authoring AST, Typed HIR, canonical RIR, cross-process artifact identity, Event transactions, Metrics, Evaluation run, and prototype-local Evidence. The detailed findings remain in the issue's dogfooding comment and prototype `DOGFOODING.md`.

This result is partial layer-integration evidence only. It closes no Genre coverage row and does not establish independent bundle interpretation, semantic authority, cross-evaluator conformance, complete package resolution, transport-independent publication, or normative Evidence issuance.

The feedback has changed the design authority in these areas:

- a non-self-hosted Schema-major Kernel Specification now owns bundle interpretation and irreducible Numeric/RNG/transition laws, while the Language Definition Bundle owns language content under that kernel;
- RIR and Package Lock contain semantic identity only; compiler and resolver implementation provenance live in Build and Resolution receipts;
- Runtime profile definitions are Language Definition Bundle content, while generated Resolved Runtime profiles bind evaluator/platform and execution facts without an identity cycle;
- gameplay alternatives are closed discriminated outcomes rather than Schema refusals;
- Event atomicity and invocation artifact publication are separate, with terminal-audit sets, commit-before-envelope ordering, and caller-known Invocation keys for recovery;
- Replay comparison and validated prerequisites are required before issuing `reproducible` Evidence;
- the Command schema profile owns cross-command schema rules while each Command descriptor remains the sole per-command surface authority; and
- the Genre contract now distinguishes outcome/refusal vectors and includes RPG economy flows, exchange/pricing, and party/enemy encounter composition.

The next validation work proceeds in order:

1. Semantic-authority conformance prototype.
2. Orthogonality/extensibility prototype.
3. Production RPG tracer and then Roguelike cross-genre tracer only after both prototype gates pass.

## User Stories

1. As a balancing designer, I want to author a model as a composable source package so that a large game does not depend on one monolithic JSON document.

2. As a balancing designer, I want values to have explicit types, quantities, units, supports, and numeric policies so that health, rates, costs, probabilities, durations, and capacities cannot be mixed accidentally.

3. As a balancing designer, I want constants, parameters, inputs, state, derived values, outputs, and random variables to be distinct symbols so that mutability and tuning intent are inspectable.

4. As a balancing designer, I want tunable parameters to declare discrete or continuous variability and legal domains so that optimization does not generate invalid candidates.

5. As a game designer, I want entities and components to compose orthogonally so that new attributes or mechanics can be added without editing unrelated core schemas.

6. As an RPG designer, I want target selection, resources, actions, damage stages, mitigation, immunity, interruption, effects, builds, progression, economy, loot, and encounters to have explicit semantics so that production combat systems are representable without genre-specific escape hatches.

7. As a Roguelike designer, I want constrained procedural generation, effect pools, build conflicts, run state, meta progression, and reset rules to reuse the same core model so that run-based games do not require a forked schema.

8. As an extension author, I want versioned packages and operations with declared capabilities and compatibility rules so that I can add domain concepts safely.

9. As an extension author, I want unknown types, attributes, operations, and capabilities to fail according to an explicit compatibility policy so that extensions do not degrade silently.

10. As a compiler implementer, I want a versioned Schema-major Kernel Specification plus one exact Language Definition Bundle, with derived or conformance-checked projections, so that bundle interpretation, schemas, registries, documentation, evaluators, and diagnostics cannot drift.

11. As a compiler implementer, I want formal name-resolution, typing, error, and lowering rules so that multiple implementations accept and reject the same programs.

12. As a runtime implementer, I want resolved models and execution plans to be immutable, hashable artifacts so that each evaluation can identify exactly what ran.

13. As a runtime implementer, I want effects separated into application requirements, value capture, contributions, transitions, scheduling, stacking, reapplication, and removal so that effect behavior is composable rather than monolithic.

14. As a runtime implementer, I want signals distinguished from persistent state so that events and durable values have different lifecycle and replay semantics.

15. As a runtime implementer, I want event ordering, snapshots, read/write conflicts, cancellation, rollback, external inputs, fairness, and RNG stream derivation specified so that replay is deterministic within a declared numeric/platform profile.

16. As a CLI user, I want parse, validation, compilation, runtime, capability, and evaluation failures mapped to stable envelopes, channels, diagnostic codes, and exit statuses so that automation can distinguish them.

17. As a tool integrator, I want one Command schema profile to close cross-command schema rules and each Command descriptor to remain the sole per-command authority so that CLI evolution and conformance tests cannot drift.

18. As an experiment author, I want scenarios, experiment plans, metrics, targets, and calibration policies to reference an exact model artifact without becoming hidden model authority.

19. As an analyst, I want simulated and future observed playtest metrics to share one round-trippable Metrics schema so that comparison and calibration use the same contract.

20. As an analyst, I want observation models, replication units, correlation assumptions, identifiability checks, model-discrepancy treatment, uncertainty intervals, holdouts, and multi-objective acceptance rules specified so that a `calibrated` result has one meaning.

21. As a reviewer, I want evidence and approval records to bind exact model, experiment, Resolved Runtime profile, data, seed, and result hashes so that an approval is auditable.

22. As a maintainer, I want Schema versioning independent from the toolkit package version so that adopting Schema 2.0 does not incorrectly force a `2.0.0` product release.

23. As a maintainer, I want accepted v1 balancing decisions explicitly superseded or retained by new decisions so that two contradictory standards cannot both appear authoritative.

24. As a pre-release v1 author, I want a deterministic source converter to migrate only semantics-preserving constructs and report every deprecated construct so that a valid-looking 2.x model never hides a lossy conversion.

25. As a maintainer with no published 1.x artifacts, I want Schema 2.0 to be the sole forward standard without save, replay, old-ruleset, gray-rollout, or reverse-migration machinery so that hypothetical compatibility does not constrain the new architecture.

26. As a contributor, I want a requirements-to-operations-to-golden-scenario coverage matrix for RPG and Roguelike mechanics so that claims of genre completeness are falsifiable.

27. As a contributor, I want normative positive, negative, limit, migration, replay, and cross-implementation vectors so that the specification is executable.

28. As a project maintainer, I want implementation to proceed in vertical slices so that source-to-evidence behavior is validated before broad package expansion.

## Implementation Decisions

- Treat Standard Schema 2.0 as a language, compiler/runtime contract, and evidence system rather than a larger template document.
- Keep Standard Schema versioning independent from the `gda-balancing` product release version; product version changes continue to follow the repository release policy.
- Use separate authored authority domains for Model Source Package, Experiment Specification, and Approval Record, linked by exact content identities without copying each other's facts.
- Establish the Schema-major Kernel Specification as the non-self-hosted authority for bundle interpretation, irreducible kernel primitives, Numeric/RNG laws, and atomic transition primitives. Establish the exact Language Definition Bundle as the sole language-content authority under that kernel for grammar, structured rules, types/effects, reducible operations, diagnostics, packages, compatibility, Runtime/Numeric profile definitions, lowering rules, and normative vectors. Language projections are generated or guarded by reverse conformance; Command descriptors remain the per-command surface authority.
- Model the pipeline with distinct authoring, typed, resolved, and execution representations. Resolved and execution representations are immutable generated artifacts, not alternative authored sources.
- Use a closed core expression language with literals, reads, calls, conditionals, local bindings, aggregates, lookup, and sampling. Core expressions are pure, recursion is forbidden, iteration is statically bounded, mutation occurs only through declared transitions, unit conversion is explicit, and randomness uses named streams.
- Define values through orthogonal concerns: representation, semantic kind, unit, support/domain, and numeric policy. `current`, `capacity`, `cost`, and `rate` are roles rather than primitive numeric types.
- Define versioned, namespaced operations and packages with explicit required/optional dependencies, capability negotiation, conversion legality, and compatibility behavior.
- Separate effect application, capture, contributions, transitions, scheduling, stacking, reapplication, and removal.
- Split the Language Definition Bundle-owned Runtime profile definition from a generated Resolved Runtime profile that binds the evaluator/platform and concrete budgets. Specify deterministic execution relative to the Resolved Runtime profile, event ordering, and exact RNG law. Do not claim unconditional cross-platform bit identity.
- Preserve one Metrics schema for simulated samples and future observed playtest samples.
- Generalize the existing version bundle, semantic catalog, typed refusal, numeric-contract, canonical-emission, normative-vector, preflight-cap, seed, and command-descriptor machinery instead of replacing those contracts implicitly.
- Supersede affected accepted balancing ADRs explicitly. New decisions must state which v1 decisions remain, which are replaced, and which CLI/conformance contracts are updated.
- Treat 2.0 as a clean forward baseline. Permit a best-effort, semantics-preserving 1.x source conversion, but deprecate/refuse unsupported constructs and provide no compatibility runtime for unpublished saves, active effects, replays, or old rulesets.
- Treat disposable prototypes as design-validation probes, never production or 1.x implementation. Run the Semantic-authority conformance prototype first, then the Orthogonality/extensibility prototype, then a production RPG tracer spanning authoring, compilation, runtime, metrics, evidence, and CLI behavior. Expand Roguelike coverage through the same abstractions only after those gates pass.
- Treat Modelica, FMI, SBML, MLIR, UCUM, and ONNX as design inputs only where the specification provides an explicit mapping, verifier, conversion, or lifecycle contract; naming a borrowed concept is not sufficient.
- Re-triage the existing v1 template, growth/economy, customization, simulation, and tuning backlog after the 2.0 decisions land; do not retrofit those issues silently.

## Testing Decisions

- **Compiler seam:** Given a Model Source Package, Schema-major Kernel Specification, and Language Definition Bundle, assert canonical diagnostics or a canonical Resolved Model plus separate Build/Resolution receipts, including stable spans and diagnostic codes.
- **Formal-semantics seam:** Maintain declarative rules for name resolution, typing, evaluation, errors, and lowering; test progress/preservation where applicable and document the boundary where equivalence is established by conformance rather than proof.
- **Runtime seam:** Given an exact Resolved Model, Experiment Specification, external-input sequence, Resolved Runtime profile, and seed bundle, assert the canonical trace, snapshots, Metrics, terminal status, and terminal-audit behavior.
- **CLI seam:** Assert Command schema profile and descriptor projections, structured binding, Invocation-key conflict/retry behavior, artifact commit-before-envelope ordering, stdout/stderr channels, diagnostics, exits, and publication faults for every compiler/runtime/evaluation state.
- **Metrics/evidence seam:** Round-trip simulated and observed samples through one Metrics schema and bind every evaluation, calibration, and approval artifact to exact input and implementation identities.
- **Genre seam:** Maintain a requirements → operations → Golden scenarios matrix. The production RPG tracer covers target selection, resource cost, staged damage, mitigation/immunity, interruption, stacking/reapplication, and Metric emission; the complete RPG claim additionally requires economy flow/exchange/pricing and party/enemy encounter-composition rows. Roguelike additions cover constrained generation, build conflicts, run reset, and meta progression.
- **Compatibility seam:** Test dependency conflicts, required and optional packages, unknown capabilities, operation upgrades, legal and illegal conversions, cross-package effects, and downgrade/refusal behavior.
- **Determinism seam:** Test priority direction, simultaneous events, snapshot visibility, read/write conflicts, rollback, cancellation, external input ordering, fairness, named RNG streams, and declared floating-point tolerances.
- **Calibration seam:** Test observation-model choice, identifiability analysis and verdict/refusal boundaries, replication units, correlated observations, discrepancy handling, holdout separation, multi-objective acceptance, and data-version drift.
- **Migration seam:** Test deterministic semantics-preserving source mappings and explicit deprecated-construct refusals. A refused conversion emits no partial 2.x authority. Runtime-state, replay, old-ruleset, staged-rollout, and reverse-migration compatibility tests are deliberately absent because no 1.x artifact was published.
- **Cross-implementation seam:** Require two independently implemented bundle-bootstrap interpreters and two evaluators that share no host domain primitive implementation. They must consume each other's artifacts and agree on positive, outcome/refusal, limit, semantic-mutation, lowering, RNG, runtime, and Replay comparison vectors before either implementation can claim conformance.
- Keep resource-limit tests adversarial and deterministic; exhaustion must produce typed refusal rather than an uncontrolled failure.

## Acceptance Criteria

- [ ] The Schema-major Kernel Specification closes bundle admission, fact/term schemas, premise operators, binding/substitution, rule selection, irreducible primitives, diagnostics, and resource laws.
- [ ] Two independently implemented bootstrap interpreters agree on admission, judgments, diagnostics, mutation behavior, and canonical Language Definition Bundle identity.
- [ ] Two evaluators sharing no host RPG primitive implementation consume each other's RIR and agree on operation, Numeric, RNG, scheduler, effect, trace, Metric, refusal, and Replay comparison vectors.
- [ ] RIR is a semantic normal form independent of source/debug/compiler provenance; Build and Debug Map receipts cannot change Resolved Model identity.
- [ ] Package Lock closes the transitive dependency/capability/type/conversion/operation graph; resolver implementation provenance lives only in a Resolution receipt.
- [ ] Runtime profile definitions and Resolved Runtime profiles are separate, executable contracts with no identity cycle.
- [ ] Gameplay alternatives are exhaustive discriminated outcomes and are never mislabeled as Schema refusals.
- [ ] Event transaction atomicity and invocation artifact-publication atomicity have separate contracts, terminal-audit behavior, fault vectors, commit-before-envelope ordering, and Invocation-key recovery.
- [ ] Experiment Specification, Metric, Metric dataset, Evaluation run, evaluator/tool, Evidence, Replay comparison, and assertion-prerequisite schemas are closed and validated before issuance.
- [ ] Command schema profile and per-command descriptor authority close dialect, keywords/formats, inputs, defaults, decoding, conflicts, stages, channels, exits, artifacts, and conformance fixtures without duplicated authority.
- [ ] Target and Signal vectors cover empty/cardinality/tie/capability, payload/effect/cycle/order/snapshot, and rollback boundaries.
- [ ] The Orthogonality/extensibility prototype adds the selected attribute/mechanics through one normative package/LDB edit per feature, without RPG-specific core compiler/runtime branches.
- [ ] RPG economy source/sink/transfer/exchange/pricing and party/enemy encounter composition rows have executable scenarios and outcome/refusal/boundary vectors.
- [ ] Every required Genre row resolves to admitted operations and executable public-artifact observations; all rows remain open until then.
- [ ] No acceptance or conformance claim relies only on disposable prototype code.

## Out of Scope

- Implementing the complete Schema 2.0 compiler, runtime, optimizer, or calibration engine in this issue.
- Shipping every RPG or Roguelike mechanic before the tracer-bullet architecture is proven.
- General-purpose scripting, unbounded loops, recursion, arbitrary host-language callbacks, or unrestricted side effects in the core language.
- Claiming universal coverage for every game genre; other genres must be addable through the extension contracts without changing core semantics.
- Implementing the complete 2.x artifact-oriented CLI surface in this specification-only issue; its taxonomy, outcomes, descriptors, structured input, and conformance behavior are decided by the new balancing ADRs.
- Coupling Schema 2.0 to a `gda-balancing` 2.0.0 release.
- Treating framework terminology from Modelica, FMI, SBML, MLIR, UCUM, or ONNX as normative without an explicit adopted contract.
- Compatibility adapters for unpublished 1.x saves, active effects, scheduled events, replays, rulesets, evidence, gray rollout, or reverse migration.
- Automatically closing or rewriting the existing product PRD and implementation issues; they require explicit re-triage after this specification is accepted.
