# Project Proposal — Flutter App

## Project Understanding
You need assistance with: 
**SSE upload progress never reports incremental bytes — jumps 0%→90%→100%, contradicts own design doc**

*Project description:*
## Problem
The SSE upload-progress feature never reports real transfer progress. It can only ever announce two step-changes — 0% (request start) then 90% (request body fully received) then 100% (processing done) — instead of incrementally tracking bytes as they arrive. This directly contradicts the implementation the project's own design doc specifies, committed in the very same commit range as the code: the doc explicitly calls for a `req.on('data', ...)` byte-accumulation hook that the shipped code never added. The demo UI consumes the SSE `progress` value directly with no client-side smoothing, so end users see the progress bar jump 0%→90%→100% instead of animating during the actual upload.

## Evidence
- `src/upload/progressAwareUploader.js:50-90`: `activeProgressManager.startUpload(...)` fires once before the body is parsed (progress 0). The only other update before completion is `activeProgressManager.markProcessing(uploadId, { fileSize: contentLength, receivedBytes: contentLength })` at line 79-82, which fires **after** `upload[isSingle ? 'single' : 'array'](...)` (multer) has already consumed the entire request body (line 57). There is no `ctx.req.on('data', ...)` or any chunk-level hook anywhere in the file.
- `flutter_upload_progress_implementation_plan.md:65-69` (added in the same commit range) explicitly documents the intended mechanism:
  ```js
  ctx.req.on('data', (chunk) => {
    receivedBytes += chunk.length;
    progressManager.updateProgress(uploadId, { receivedBytes, ... });
  });
  ```
  and its own feature comparison table (lines 103-108) advertises "服务器端进度监控" / "实时进度推送" / "进度精确性: 网络+服务器处理" as the differentiator of this new implementation vs. the old one.
- `src/fileServer/demo.html:313-327` (`listenToServerProgress`) applies `data.progress` from each SSE message directly via `onProgress(data.progress, data.status)` with no interpolation/smoothing — so the sparse 0/0.9/1 sequence renders as visible jumps in the demo's progress bar, not the smooth "real-time" progress the plan doc and UI copy advertise.

## Proposed Solution
- Implement the byte-accumulation hook already specified in `flutter_upload_progress_implementation_plan.md` (`ctx.req.on('data', ...)` in `src/upload/progressAwareUploader.js`, calling `progressManager.updateProgress` with running `receivedBytes` before multer finishes), so SSE progress reflects actual transfer progress.
- Alternatively, if true streaming progress is out of scope for now, update `flutter_upload_progress_implementation_plan.md` and any user-facing copy (e.g. `enhanced-demo-features.md`) to describe the two-phase (0%→90%→100%) behavior actually shipped, instead of documenting an unimplemented per-chunk design as if it were live.

## Acceptance Criteria
- Either: an upload of a reasonably large file shows SSE `progress` values incrementing smoothly between 0 and 0.9 as bytes are received (verifiable via a test asserting multiple intermediate `updateProgress` calls during body upload), or: the design doc and any UI copy claiming real-time/precise server-side progress are corrected to match the actual two-phase behavior.
