## Description

`pr-unreplied-comments.sh --coverage-only` reported `patch_pct: 0` on stillwater PR #2574 when
the real Codecov patch coverage was **77.78%**. A parse artifact, not a coverage failure.

This is the same shape as #315 and it is the reason both are worth fixing rather than
remembering: **a green-looking (or red-looking) signal that reads that way because a check DID
NOT PARSE, not because it measured anything.** A `patch_pct: 0` sends a lead hunting a
catastrophically untested PR that does not exist - or, worse in the other direction, invites
someone to "fix" the threshold.

## Steps to Reproduce

Reported MEASURED on stillwater PR #2574 (merged e3cbea14), 2026-07-16: the
`--coverage-only` extraction returned `patch_pct: 0` while the `codecov/patch` commit status on
the same head reported 77.78%. Not yet re-derived on this repo (cc-orchestrator has no
`codecov.yml`, so the coverage path self-skips here - reproduction needs a repo with an active
coverage service).

## Expected Behavior

The extractor returns the real percentage, or an explicit "cannot determine" - never a
plausible-looking `0` it did not measure.

## Actual Behavior

`patch_pct: 0`, indistinguishable from a genuine 0% patch coverage.
