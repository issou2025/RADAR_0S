## User Story
As a field worker on a company-issued or personal Android device, I want a native EquipmentTracker app from the Google Play Store, so that I can check out, return, and track equipment reliably in zero-connectivity environments regardless of Android OS memory pressure, Doze mode, or browser restrictions.

---

## Problem Statement
If PWA hardening (Phase 1) does not resolve offline checkout failures to below a 5% rate in real field conditions, the underlying issue is that browser-imposed constraints on Android (storage eviction, service worker termination, Doze mode) cannot be overcome at the application layer. A native Android app eliminates all browser constraints definitively and is required for:

1. **Enterprise procurement qualification:** Enterprise construction RFPs explicitly require "native Android app on Google Play," "MDM/Android Enterprise compatible," and "reliable full offline." A PWA-only product is disqualified from these deals.
2. **Hardware integration:** QR scanner (hardware camera API for speed), NFC tag reading for equipment identification, Bluetooth printer support for labels — all require native APIs unavailable to mobile browsers.
3. **Market competitiveness:** All 6 major competitors have native Android apps. Category average Play Store rating is 3.4★. No competitor currently combines >4.0★ satisfaction with reliable offline-first checkout. This is the ownable quality gap.
4. **BYOD deployment:** A Play Store listing removes the friction of PWA "Add to Home Screen" instruction flows that BYOD workers frequently skip.

## Acceptance Criteria
- [ ] **Play Store Publication:** Native Android app is published to Google Play Store, targeting Android 9+ (API 28+) as minimum supported version
- [ ] **Offline-First Checkout:** Equipment checkout (item selection, confirm, submit) completes in zero-connectivity mode; the transaction is persisted locally and syncs automatically when connectivity is restored
- [ ] **Offline Return + Condition Notes:** Equipment return with written condition notes and photo capture completes offline; all data persists and syncs on reconnect
- [ ] **Offline Sync Queue:** All offline-created transactions are stored in a local SQLite-backed sync queue; on reconnect, queue flushes within 30 seconds with per-item success/failure status visible to the user
- [ ] **QR Code Scanner:** Hardware-integrated QR code scanner (native camera API) identifies equipment items; scanning to checkout initiation completes in ≤1.5 seconds under normal lighting conditions
- [ ] **NFC Tag Reading:** NFC tag tap initiates equipment checkout or return flow; works on NFC-capable Android devices without additional user setup
- [ ] **≤3-Tap Checkout Flow:** From app home screen to checkout confirmation is achievable in ≤3 taps under normal conditions (no connectivity required)
- [ ] **App Launch to Checkout in ≤15 Seconds:** Cold app launch to the checkout initiation screen takes ≤15 seconds on Samsung Galaxy A34 (BYOD reference device)
- [ ] **BYOD No-MDM Requirement:** App installs and runs fully from the Play Store as a standard user app; no MDM profile, device enrollment, or enterprise permissions are required for BYOD install and normal use
- [ ] **MDM/Android Enterprise Support:** IT administrators can deploy and configure the app via Managed Google Play (silent install); a managed configuration schema supports at minimum: server URL, auth mode, and certificate pinning toggle — without requiring user interaction
- [ ] **Reference Device Compatibility:** All above criteria pass on Samsung Galaxy A34 (BYOD reference, Android 12), Samsung Galaxy S22 (mid-tier, Android 13), and Zebra TC53 (enterprise reference, Android 11)
- [ ] **Android OS Fragmentation:** App is tested and free of critical crashes on Android 9, 11, 12, 13, and 14; no crashes on devices included in Android vitals baseline
- [ ] **Sync Conflict Resolution:** When the sync queue flush generates a conflict (same equipment checked out by two offline users), the conflict is routed to the Operations Coordinator dashboard with item identity, both transactions, timestamps, and a resolution action (accept/override)
- [ ] **Background Process Survival:** Unsent sync queue entries survive app backgrounding and device reboot; entries resume sync automatically on next app open with network available
- [ ] **Play Store Rating Gate:** App achieves ≥4.0★ average rating on Google Play within 90 days of launch (measured via Google Play Console); 4.5★ target at 6 months
- [ ] **Crash-Free Session Rate:** ≥99.5% crash-free session rate in Firebase Crashlytics within 30 days of launch

## Notes
- **Target launch segment:** AV/Events field workers as early adopter cohort — higher install willingness (~5.5/10 vs. 4.5/10 BYOD), faster iteration cycles, urgency-driven (equipment leaves venue if not tracked). Validate with this segment before broad construction rollout.
- **Android-first, not React Native:** Research #135 recommends Kotlin-native (not React Native/Flutter) for hardware integration quality (QR, NFC, Bluetooth). Do not compromise on native hardware APIs.
- **iOS gap strategy:** Android ships first; PWA continues to serve iOS users during gap; iOS native roadmap planned within 2 quarters of Android launch. PO to communicate iOS roadmap externally to prevent churn.
- **BYOD privacy messaging:** Play Store listing and onboarding must lead with worker-benefit framing, not employer surveillance. No MDM profile on BYOD is a non-negotiable product decision.
- **BA clarification needed:** Managed configuration schema keys (minimum viable set for enterprise IT admin) need BA to confirm with enterprise GC IT admin interviews before build.
- **Phase trigger:** If Phase 1 (#139) field test reports ≥5% failure rate, move this issue to "Ready for Development" and begin Design sprint.
