# Use Case Description

I'm building a **telephony agent using Vobiz + Pipecat**.

Currently I'm using **Pipecat Cloud** in production, but I'm planning to move to my own infrastructure because I need to run around **10 instances of the same agent continuously** for better cost efficiency.

The main issue I'm trying to solve is **startup latency**.

## Current Flow

1. Vobiz sends a request to my `/answer` endpoint.
2. The endpoint returns XML containing my WebSocket URL.

```python
ws_url_base = f"wss://{myServerUrl}/voice/ws"
```

3. When the **call is answered**, Vobiz connects to `/voice/ws`.
4. Inside the WebSocket handler I start the Pipecat agent:

```python
await bot(
    runner_args,
    call_id=call_uuid,
    stream_id=stream_id,
    gemini_api_key=gemini_api_key,
)
```

From the moment **the call is answered** until the agent speaks its **first opening sentence**, I'm seeing approximately **4–5 seconds of latency** during local testing.

I haven't deployed this architecture on my own cloud infrastructure yet, so I don't know whether the latency will improve or remain similar.

## What I'm Looking For

I'd like to reduce the initial response time to around **1–1.5 seconds**.

I'm mainly looking for guidance on:

- Best practices for reducing startup latency in self-hosted Pipecat deployments.
- Whether there is a recommended **warm pool / pre-warmed worker** pattern.
- Examples of running multiple long-lived Pipecat workers instead of creating a new pipeline for every call.
- Whether agents, models, or transports can be initialized ahead of time and reused.
- Any production architectures others are using for telephony workloads.

I believe this is essentially a **cold-start problem**, but I couldn't find much documentation or examples around warm-starting Pipecat agents.
