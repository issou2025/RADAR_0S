### Paywall requirements for App Review

The current paywall would be rejected. Apple checks subscription paywalls on specifics, and these are the most common causes:

- [x] **Price and duration shown explicitly** (#161) — `"Subscribe for {price}/month"` / `"Assinar por {price}/mês"`. The price stays the store's own localized `priceString`; only the period is ours. Hardcoding it is correct while there is exactly one monthly product — **an annual package would have to read the period off the RevenueCat `Package`**
- [x] **Restore Purchases** action — required, and the only way a reinstalling user recovers their entitlement
- [x] Links to **Terms of Use (EULA)** and **Privacy Policy** (both tracked in #157)
- [x] Auto-renewal disclosure: that it renews until cancelled, and where to cancel

## Phase 4 — verify in sandbox

TestFlight builds use the sandbox IAP environment automatically, so all of this works **before** submitting anything.

Sandbox runs on an accelerated clock: a 1-month subscription renews every **~5 minutes** and auto-renews **6 times before stopping**. So the cancel and expiry checks below resolve in minutes, not a month — plan them as one sitting.

- [ ] Buy from a TestFlight build (real Apple ID, sandbox environment, no charge); confirm the webhook writes `status = 'active'` and the scans unlock
- [ ] Cancel; confirm the row becomes `cancelled` and access **persists** until `expires_at`
- [ ] Request a refund; confirm the row becomes `expired` and access is revoked **immediately, while `expires_at` is still in the future** — this is the case the mapping exists for (#156). **Sandbox has no self-service refund flow** — drive it with a RevenueCat test webhook carrying `cancel_reason: CUSTOMER_SUPPORT` instead, which is what the mapping actually needs exercised
- [ ] Reinstall; confirm Restore recovers the entitlement
