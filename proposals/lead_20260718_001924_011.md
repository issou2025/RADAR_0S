class TestDocumentChunker:
    """Test suite for the semantic structural chunking logic."""

    def test_chunk_pages_creates_valid_chunks(self):
        # 1. Setup: Initialize chunker with artificially small limits to force a split
        chunker = DocumentChunker(chunk_size=50, chunk_overlap=10)
        
        # Create mock input mirroring the output of ingest.py
        raw_pages = [
            {
                "text": "This is a very long sentence that needs to be split into smaller pieces for testing.",
                "metadata": {"source": "textbook.pdf", "page": 1}
            }
        ]
        
        # 2. Execute: Run the chunker
        chunks = chunker.chunk_pages(raw_pages)
        
        # 3. Assert: Validate the output
        assert len(chunks) > 1, "The text should have been split into multiple chunks."
        assert isinstance(chunks, list), "Output must be a list."
        
        for chunk in chunks:
            # Check structure
            assert "text" in chunk, "Each chunk must have a 'text' key."
            assert "metadata" in chunk, "Each chunk must have a 'metadata' key."
            
            # Check metadata preservation
            assert chunk["metadata"]["source"] == "textbook.pdf", "Source metadata must be preserved."
            assert chunk["metadata"]["page"] == 1, "Page metadata must be preserved."
            
            # Check length constraint
            assert len(chunk["text"]) <= 50, f"Chunk size exceeded limit: {len(chunk['text'])} > 50"

    def test_empty_pages_handled_gracefully(self):
        # 1. Setup
        chunker = DocumentChunker()
        raw_pages = []
        
        # 2. Execute
        chunks = chunker.chunk_pages(raw_pages)
        
        # 3. Assert
        assert chunks == [], "Empty input should return an empty list."

## 🏗️ Technical Architecture Decision Records (ADR Set)

### ADR-002: Deploying to Managed Free-Tier Container Spaces (Hugging Face) vs. Traditional Virtual Private Servers (VPS)

* **Context:** The underlying pipeline utilizes local deep-learning embeddings (`BAAI/bge-small-en-v1.5`) alongside structural parsing frameworks (PyMuPDF). Containerizing and deploying this stack on conventional, lower-tier commercial VPS options often results in immediate server crashes due to strict RAM constraints during model initialization.


* **Decision:** We chose to target **Hugging Face Spaces** (leveraging the persistent, free-tier hardware container allocation that offers up to 16GB of dedicated CPU RAM) as the primary cloud hosting environment.
* **Consequences:**
* *Positive:* The 16GB RAM buffer easily accommodates local PyTorch model loading, text vectorization, and multi-tenant persistence layers without running out of memory. Deploys seamlessly via automated Git push mirrors.
* *Negative:* The persistent file container storage layer is ephemeral; any application state or uploaded text assets are lost whenever the Space transitions into a low-power sleep state. This requires the pipeline to dynamically build vector embeddings in memory or initialize local Chroma instances on demand per execution workspace session.





### ADR-003: Implementing Hybrid Ensemble Retrievers vs. Standalone Sparse/Dense Geometric Vector Search

* **Context:** Relying strictly on dense semantic vector embeddings often misses critical technical matches when students look up specific keywords (e.g., precise code names, system error definitions, or specific metadata attributes). Conversely, using simple keyword-matching approaches (BM25) fails to capture semantic themes or context across long paragraphs.


* **Decision:** We deployed an **EnsembleRetriever combining a dense semantic vector space engine (Chroma) and a traditional sparse BM25 keyword index**, utilizing a balanced 0.5/0.5 reciprocal weight distribution function.


* **Consequences:**
* *Positive:* Combines the strengths of both systems. The pipeline can accurately look up exact keywords (like error logs) while still capturing the broader conceptual context of user queries, significantly reducing hallucinations.
* *Negative:* Increases data preprocessing overhead. The local ingestion pipeline must now systematically build and keep both an active sparse document term matrix and a structural dense vector lookup system in memory.

## 🧠 What Disappointed Me / What I'd Do Differently

I underestimated how messy real-world text splitting can be. While working with simple text paragraphs went smoothly, I noticed that documents containing complex tables or structured bullet points would often break across arbitrary boundaries when handled strictly by character length constraints. If I were starting the data modeling phase over, I wouldn't jump straight to default character splitting variables. Instead, I would build an explicit structural layout analyzer to isolate tables before chunking, ensuring that critical data blocks stay grouped together.
