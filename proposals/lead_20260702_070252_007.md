# Autopublishing Design for GenUI (Melos Edition)

## Goal
Transition from manual, local package publishing to an automated, Continuous Deployment (CD) workflow using GitHub Actions. We will adopt **Melos**, the industry-standard tool for Dart/Flutter monorepos, to replace custom release scripts.

## Why Melos?
-   **Standardization**: widely used in the Flutter community (e.g., Firebase, Flame).
-   **Features**: Built-in dependency graph analysis, versioning, changelog generation, and batch publishing.
-   **OIDC Support**: `melos publish` works seamlessly with Dart's OIDC authentication.
-   **Simplification**: Replaces the custom `tool/release` maintenance burden with a maintained package.

## Proposed Design

### 1. The Workflow
We will use a "Tag-then-Publish" workflow to comply with `pub.dev` OIDC requirements and GitHub "Squash & Merge" behaviors.

#### Step 1: Developer Bumps Version (Locally)
-   Developer runs `melos version` (or a simplified command).
-   Melos updates `pubspec.yaml`, `CHANGELOG.md`, and handles inter-package dependency versioning automatically.
-   Developer commits and pushes to `main` (via PR).

#### Step 2: Auto-Tagging (CI on `main`)
-   Triggers on merge to `main`.
-   Action detects if a package version in `pubspec.yaml` is newer than the latest git tag.
-   Action pushes the missing tag (e.g., `genui-v0.9.0`).
-   *Why?* Tags created locally by developers are lost/orphaned during "Squash & Merge". CI must recreate them on the merge commit.

#### Step 3: Publishing (CI on `tag`)
-   Triggers when step 2 pushes a tag.
-   Action runs `melos publish --no-dry-run`.
-   Melos automatically filters to the package matching the current tag (or we can be explicit).

### 2. Migration Plan

#### A. Install & Configure Melos
1.  Add `melos` as a dev dependency in the root (already a workspace).
2.  Create `melos.yaml`:
    ```yaml
    name: genui_monorepo
    packages:
      - packages/*
      - examples/*
    command:
      version:
        # Configures how versioning/changelogs work
        updateGitTagRefs: true
    ```

#### B. Replace `tool/release`
-   **Delete**: `tool/release/bin/release.dart`, `bump.dart`, `publish.dart`.
-   **Keep**: We might keep a small script for the "Step 2: Auto-Tagging" logic, as Melos doesn't strictly have a "tag-if-missing" CI command (it assumes you are running `melos version` in CI or manually).
    -   *Optimization*: We can write a tiny Dart script `tool/ensure_tags.dart` that uses `melos list` to get versions and checks git tags.

### 3. GitHub Actions

#### Workflow 1: `tag_releaser.yml`
-   **Triggers**: `push` to `main`.
-   **Run**: `dart run tool/ensure_tags.dart --push`
    -   This script will:
        1.  Parse all package versions.
        2.  Check if `package-vVersion` tag exists.
        3.  If missing, `git tag package-vVersion && git push`.

#### Workflow 2: `publisher.yml`
-   **Triggers**: `push` to `tags` (`*-v*`).
-   **Permissions**: `id-token: write`.
-   **Run**:
    -   `dart pub global activate melos`
    -   `melos publish --no-dry-run --yes`
    -   Melos is smart enough to check if a package is already published. We can also use `melos publish --missing` logic if needed.

### 4. OIDC Setup (Same as before)
1.  **Enable on pub.dev** for ALL packages.
2.  **Tag Pattern**: `{{package}}-v{{version}}` (Matches Melos default).

## Comparison

| Feature | Old Custom Script | New Melos Workflow |
| :--- | :--- | :--- |
| **Bumping** | Manual `bump` command | `melos version` (Automatic changelogs!) |
| **Publishing** | Manual `publish` loop | `melos publish` (Parallel/Batch) |
| **Maintenance** | High (Custom logic) | Low (Community tool) |
| **Tagging** | Manual/Custom CI | Custom CI Script (Small) |
