# Project Proposal — Python Automation

## Project Understanding
You need assistance with: 
**Add basectl logs last for the latest failed run tail**

*Project description:*
## Goal

Add a `basectl logs last` surface that prints the most relevant tail of the latest failed Base command log, with optional JSON output for automation.

## Background

The v1.6.1 product evaluation calls out a UX gap: Base records history JSONL and per-run log file paths, but users still have to manually discover and open a path from `basectl logs`. `docs/observability.md` also describes a future rule-based `basectl explain last-error`; `logs last` can provide the concrete local evidence surface without any AI provider dependency.

## Scope

- Add `basectl logs last` and consider `basectl logs last --format json` if it matches existing log command patterns.
- Locate the latest failed history record and its linked log path when available.
- Print command metadata, exit status, time, project if known, and a bounded redacted log tail.
- Handle missing or cleaned log files gracefully.
- Reuse existing history/log redaction helpers rather than exposing secrets from command arguments or URLs.
- Document how `logs last` relates to the broader future `explain last-error` concept.

## Acceptance Criteria

- `basectl logs last` finds the latest failed run and prints a useful bounded tail without requiring users to open the log path manually.
- The command reports a clear no-failure state when history has no failures.
- Missing log files produce an actionable message that still includes available metadata.
- JSON output, if included, has stable keys and focused tests.
- Output redacts secret-looking arguments and URL credentials at least as defensively as existing history/report surfaces.

## Validation

- Focused Python tests for last-failure selection, missing log handling, tail truncation, and redaction.
- Bash dispatch/help/completion tests if a new subcommand or option is added.
- `git diff --check`.
