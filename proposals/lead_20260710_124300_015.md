# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**[FEAT] - Telegram Markdown Rendering Design**

*Project description:*
## Executive Summary

Today, **every Telegram message sent by a ds-team agent carries no formatting**. Asterisks, underscores, backticks and links reach Telegram users as literal characters because `eve`'s built-in `message.completed` handler issues plain-text `sendMessage` calls. This document describes the chosen fix.

**Approach:** convert the agent's Markdown output to `MessageEntity[]` server-side, then call `sendMessage` with `{ text, entities }` and no `parse_mode`. All formatting is supplied by structured entities that Telegram renders natively; nothing on the wire is parsed server-side, so nothing can fail with a 400 on a stray character.

**Library:** [`@vcc-community/telegramify-markdown`](https://github.com/Vibe-Coding-Community/telegramify-markdown) `@1.0.2` — TypeScript, MIT, 15.3 kB unpacked, uses `marked` under the hood.

**Where:** one override of `message.completed` in `packages/agent-core/src/channels/telegram.ts`, plus one call inserted before `send(...)` in the voice-fork path. Both routed through a single `formatOutbound(text)` helper. All ds-team Telegram agents pick the change up through the shared channel.

**Why not `parse_mode`: MarkdownV2` or `HTML`** is summarised briefly in §3; the short version is that both are server-side-parsed and all-or-nothing (one bad character → 400 + the entire message is rejected). Entities avoid that failure mode by construction.

**Why not Rich Messages** (Bot API 10.1, 11 Jun 2026): Telegram Web still cannot render them ([hermes-agent#45785](https://github.com/NousResearch/hermes-agent/issues/45785), open as of 2026-07-10); only Desktop and mobile render, so any rich reply would break for Web users. Tabular/nested-list benefits also don't apply to current ds-team outputs (paragraph, list, code, link).

## 1. Problem statement

### 1.1 Observed behavior

A user receives from `fitness-coach` (or any other ds-team Telegram agent) responses like:

> For 3x5 at RPE 7, brace *before* the bar leaves the floor.

instead of italic for "before". Code samples show as backticks, tables as ASCII pipes, links as raw `[label](url)`. The LLM emits well-formed Markdown; the channel throws the structure away.

### 1.2 Root cause (verified against source)

```
eve/dist/src/public/channels/telegram/telegramChannel.d.ts:52-60
export interface TelegramMessageBody {
    readonly disable_notification?: boolean;
    readonly link_preview_options?: Readonly<Record<string, unknown>>;
    readonly message_thread_id?: number;
    readonly protect_content?: boolean;
    readonly reply_markup?: Readonly<Record<string, unknown>>;
    readonly reply_parameters?: Readonly<Record<string, unknown>>;
    readonly text: string;
}
```

- `parse_mode`, `entities`, and `link_preview_options` (the latter restricted to a plain `Record`) are not present on the type passed to `sendMessage`.
- The default `message.completed` handler (`docs/channels/telegram.mdx:46`) sends plain text — *"passes no `parse_mode`, so any Markdown shows up literally."*
- The `parseTelegramMessage` API also never sets `entities` on inbound messages, so even if a user sends bold text, the model sees raw markdown.

### 1.3 Impact

Two deployed Telegram bots today (`fitness-coach`, `home-automation-assistant`) emit outputs unfit for the channel. Adding new Telegram-fronted agents under this posture is not viable.

## 2. Why entities, not `parse_mode`

Both `MarkdownV2` and `HTML` cause the entire message to be rejected by Telegram if any character is mis-escaped. The `markdownToTelegramEntities` (entities) path sends a plain `text` plus a structured `entities[]` array; the Bot API stores the entities without parsing `text`, so it cannot 400 on a stray character.

| Mode | Server-side parses `text`? | Failure mode of a malformed character | Universal client support (Jul 2026) |
|---|---|---|---|
| `parse_mode: MarkdownV2` | Yes | **All-or-nothing.** 18 characters must be escaped (`_ * [ ] ( ) ~ \` > # + - = | { } . ! \`); one mishandled `.` → HTTP 400 `Bad Request: can't parse entities` and the whole message is rejected. | Yes (iOS, Android, Desktop, Web). |
| `parse_mode: HTML` | Yes | Same all-or-nothing. Smaller escape surface (`<`/`>`/`&`), but a stray `<` from an LLM shell fragment → 400. | Yes. |
| **`entities[]` (chosen)** | **No.** | Plain text degrades silently if the parser fails. No server-side validation can 400 the message. | Yes (universal). |
| `Rich Messages` (Bot API 10.1) | No, but requires `sendRichMessage` | Graceful on client (no message at all) but **`Web does not render today** ([hermes-agent#45785](https://github.com/NousResearch/hermes-agent/issues/45785), open as of 2026-07-10). Tabular/nested benefits unused in current ds-team outputs. | No (Desktop + mobile only). |

MarkdownV2 escape set cross-checked against [the Rust `telegram-escape` source](https://docs.rs/telegram-escape/latest/src/telegram_escape/lib.rs.html) and [core.telegram.org/bots/api#markdownv2-style](https://core.telegram.org/bots/api). Rich Messages status cross-checked against [the official changelog](https://core.telegram.org/bots/api-changelog) (10.1, 11 Jun 2026) and the open Web rendering bug.

## 3. Library: `@vcc-community/telegramify-markdown`

Chosen dep, pinned at `1.0.2`.

- **License:** MIT.
- **Size:** 15.3 kB unpacked, 2 deps.
- **Authoring:** TypeScript; uses [`marked`](https://marked.js.org/) under the hood.
- **Input / output:** `markdownToTelegramEntities(text) → { text, entities }`. Offsets/lengths in UTF-16 code units (`str.length`) — exactly what [`api/entities`](https://core.telegram.org/api/entities) requires.
- **Tokens covered:** `paragraph, strong, em, del, codespan, code (pre), link (text_url), blockquote, list, heading, hr, br, text`.
- **Tokens NOT covered (deliberately):** `underline, spoiler, expandable_blockquote, custom_emoji, mention, hashtag` — current ds-team outputs don't use them.
- **Maintenance status:** 0 stars, single contributor, published 2025-05-28. Source-read on 2026-07-10; algorithm is small and direct. **Fallback plan** if it disappears: vendor the package inline (~100 LOC under `packages/agent-core/src/vendor/telegramify-markdown.ts`) and pin that path; alternative, replace with a `marked` AST walker implementing the same offset logic (algorithm reference: [`sudoskys/telegramify-markdown`](https://github.com/sudoskys/telegramify-markdown) Python source under `convert()`, MIT).

## 4. Design

### 4.1 Where the change lives

`packages/agent-core/src/channels/telegram.ts`, in `makeTelegramChannel()`. Single file, single symbol (the existing `inner` channel). The override is a `message.completed` handler attached via `events: { "message.completed": … }` at the `telegramChannel({...})` constructor.

**Source-verified contract** (read 2026-07-10 against `eve@0.20.0`):

- `telegramChannel.js` constructs `events = { ...defaultEvents, ...e.events }` (one-liner, spread). The user's `events` **wins** for any key it defines, but **silently drops** every other default — so we must spread `defaultEvents` ourselves if we only want to override `message.completed`.
- `defaultEvents["message.completed"]` (in `defaults.js`) is `async (e,t,n) => { … !e.message || await t.telegram.post(e.message) }`. If we replace it, we lose the tool-call gating (`e.finishReason === "tool-calls"`) and we must re-implement the equivalent ourselves. *This revision adds back the tool-call gate in the proposed override below.*
- `defineChannel.js` wires `events[name]` into the channel runtime as `(event, ctx) => handler(event, ctx, buildCallbackContext())`. So our handler signature is `(event, ctx) => { … }`.

Revised handler shape (incorporates the source-level findings, addresses §9.5):

```ts
import { defaultEvents } from "eve/channels/telegram";

telegramChannel({
  botUsername,
  credentials: { botToken, webhookSecretToken },
  events: {
    ...defaultEvents,                          // ← keep turn.failed, input.requested, etc.
    "message.completed": async (event, ctx) => {
      if (event.finishReason === "tool-calls") return;     // reapply gate we just overwrote
      const payload = event.message;
      if (!payload) return;
      const rawText = typeof payload === "string" ? payload : payload.text;
      if (!rawText) return;
      try {
        const { text, entities } = markdownToTelegramEntities(rawText);
        await ctx.telegram.request("sendMessage", {
          chat_id: ctx.state.chatId,
          text,
          entities,                              // bypass TelegramMessageBody type
        });
      } catch {
        // Last-resort: plain text, never block delivery.
        await ctx.telegram.sendMessage(rawText);
      }
    },
  },
  // …
});
```

> **Note** on `ctx.telegram.request("sendMessage", …)`: this *bypasses* `postTelegramMessage`, which means **we lose the 4096-char auto-split**. See §9.3 for the trade-off and the proposed split path.

### 4.2 Helper: `formatOutbound`

A single module-scoped helper called from both paths:

```ts
function formatOutbound(rawText: string): { text: string; entities: MessageEntity[] } {
  const parsed = markdownToTelegramEntities(rawText);
  // ...apply 4096 chunking per §9.3 (returns a single chunk today; multi-chunk later)
  return parsed;
}
```

Throws only on programmer error — never on malformed markdown (the lib degrades gracefully). Callers wrap in try/catch and fall back to plain text.

### 4.3 Why a shared fallback

Per the repo-wide [[feedback-mcp-permissive-surfaces]] pattern: behavior is shaped by the **channel**, not by individual agents. All six current Telegram agents (and any future ones re-exporting from `@ds-team/agent-core/channel/telegram`) get formatted output for free, with one place to maintain.

### 4.4 Voice transcription interaction

**Confirmed by source reading (2026-07-10).** The voice fork in `agent-core/src/channels/telegram.ts` (around line 296) calls `send({ inputResponses, message: turnMessage, context: [contextBlock] }, …)` directly, bypassing the turn lifecycle. The `message.completed` event is **not** emitted on that path. Consequence: if we override `message.completed` alone, **voice replies stay plain text**.

Two approaches; the second is preferred because it keeps formatting in one place:

- **A. Pre-transform the transcript** before `send(...)`. Read as one extra line near line 290 — `turnMessage` (a `TelegramMessageBody`-shaped object) gets its `.text` rewritten via `markdownToTelegramEntities(...)` if it parses cleanly, plus `entities` attached. **Strays from the principle** (formatting logic in two places: voice + chat).
- **B. Wrap `send` once at the channel level** so both `message.completed` and the voice fork pass through a single `formatOutbound(text)` helper. Single source of truth. **Preferred.** Implementation surface: introduce a `formatOutbound` function inside `makeTelegramChannel()` and call it from both (a) the override of `message.completed` and (b) just before `send(...)` in the voice fork handler. Trivial refactor.

Open question on B (deferred to implementation): if the agent emits *attachments* alongside text in a voice reply (rare, but `TelegramMessageBody` permits it), the formatter only handles `.text`. Keep behaviour consistent with `e.message` from §5.2 — string → use as-is, object → extract `.text`.

### 4.5 Out of scope

- `parse_mode: MarkdownV2` — 18-char escape set, all-or-nothing failure mode.
- `parse_mode: HTML` — same failure mode; sanitize-on-send is more code than the entities approach with no upside.
- Rich Messages (`sendRichMessage`) — Telegram Web doesn't render ([hermes-agent#45785](https://github.com/NousResearch/hermes-agent/issues/45785)); tabular benefits don't apply to current outputs.
- Asking agents to emit escape-correct MarkdownV2 in their instructions — shifts an 18-char escape set onto every model we run.

### 4.6 Future hooks (not implemented now)

- **Rich Messages opt-in**: when Telegram Web renders Rich Messages, add a `process.env.TELEGRAM_RICH_MESSAGES=1` hook that switches to `sendRichMessage`. Tracked separately.
- **Streaming**: `sendMessageDraft` / `sendRichMessageDraft` for live-typing UX. Separate design doc if/when needed.

## 5. Implementation surface (when approved)

### 5.1 Code change set

1. `packages/agent-core/package.json` — add `@vcc-community/telegramify-markdown` (pinned to exact `1.0.2` for the first cut; see §8 mitigation).
2. `packages/agent-core/src/channels/telegram.ts` — `makeTelegramChannel()`:
   - Import `defaultEvents` from `eve/channels/telegram`.
   - Override `message.completed` per §4.1 shape.
   - Move `defaultEvents` spread into the new `events` object so we keep the other handlers.
   - Add a `formatOutbound(text: string): { text: string; entities?: MessageEntity[] }` helper at module scope, called from (a) the `message.completed` override and (b) the voice fork's pre-`send` step (per §4.4 option B).
3. Voice fork handler (same file, near line 290): call `formatOutbound(transcript)` and rewrite `turnMessage.text` + add `turnMessage.entities` before the existing `send(...)`.
4. Fallback wrapper around the formatted send: on `markdownToTelegramEntities` throw OR `sendMessage` 4xx, fall back to plain text (`ctx.telegram.sendMessage(rawText)`).

### 5.2 Validation (gate before merging)

- Unit test in `packages/agent-core`: feed `markdownToTelegramEntities` fixed inputs covering all 9 token types the lib claims to handle (`paragraph, strong, em, del, codespan, code, link, blockquote, list, heading, hr, br, text`). Confirm `text` and entity offsets/lengths.
- UTF-16 unit test: feed input containing a non-BMP code point (e.g. `😅` in the middle of `**bold 😅 fat**`) and assert that the entity length covers both surrogate pairs (i.e. `length === 4` for the bold span, not 3).
- Manual smoke on at least one agent (proposed: `fitness-coach`):
  - iOS, Android, Telegram Desktop, **and Web** (the Web bug §2.4 doesn't apply to entities, so it should render). Same message, four clients, confirm rendering parity.
  - Send a transcript through voice channel (regression test for §4.4): confirm formatting survives the transcription fork.
- Manual regression: each MarkdownV2 foot-gun character (`.`, `!`, `:`, `[`, `]`, …) inside the model output — must NOT 400 anywhere on the path. (With entities, there is no server-side parse, so a 400 cannot occur from formatting. Confirm by deliberate injection in a smoke message.)

### 5.3 Risks and mitigations

| Risk | Mitigation |
|---|---|
| `@vcc-community/telegramify-markdown` is unmaintained | Surface is tiny (~100 LOC); vendor it under `packages/agent-core/src/vendor/telegramify-markdown.ts` if the upstream disappears. Alternative: replace with a `marked` AST walker (algorithm reference: [`sudosks/telegramify-markdown`](https://github.com/sudoskys/telegramify-markdown) Python `convert()`, MIT). |
| Parser drops entities on edge cases | Final fallback to plain text `sendMessage` covers all failures silently. |
| LLM emits odd structures (nested bold inside code-block) | lib handles nested; if it misaligns, unrendered chars fall back to plain text. |
| Web renderer drift on entities | Universal today; if Telegram splits an entity type to Web-only, our output still degrades to plain text without breaking the channel. |

## 6. Open questions

1. Should we add a feature flag `TELEGRAM_RENDER_MODE` (`entities` default, `markdown` opt-in, `html` opt-in) per agent, or keep the channel opinionated? Default position: opinionated channel; per-agent override only when an agent demonstrably needs a different mode.
2. Should the parser output also fix the inbound side (entities → Markdown for `parseTelegramMessage`)? Affects what the LLM sees in user messages. Lower priority — agents handle plain markdown inputs today.
3. Should we vendor a Rust-backed parser (`pyromark`, `pulldown-cmark`) for perf at scale? Current throughput is 1-message-per-turn; not justified yet.

## 9. Source-level uncertainties (added 2026-07-10)

These items came up while reading the actual `eve@0.20.0` source on 2026-07-10 and the published `@vcc-community/telegramify-markdown@1.0.2` source. They do **not** change the chosen path, but they narrow the scope of what we can claim is "ready to code" vs "needs runtime check".

### 9.1 Imperfect markdown from the LLM

`@vcc-community/telegramify-markdown` uses `marked` internally and walks the token list with simple offsets. **It is not runtime-tested against actual LLM output yet.** LLMs regularly produce:
- Overlapping delimiters (`**bold *italic***`) — `marked` will pick one path, the other entity may be missing or misaligned.
- Unclosed fences — `marked` treats as inline code, offsets may leave dangling entities.
- LLM-emitted `\n\n` then lib-emitted `\n` between list items — visual double-newlines.

**Mitigation:** fallback to plain text on throw. **Trade-off:** we lose formatting on these messages, but never deliver a broken message. Per §5.2, must smoke-test with at least 5 real `fitness-coach` outputs.

### 9.2 Nested formatting edge cases

Looking at the lib's switch — `paragraph → processTokens(token.tokens)` is called *after* the entity is recorded with `startOffset = getUtf16Length(plainText)`. For `text` tokens that themselves have sub-tokens (`**bold *italic* bold**`), the recursion appends text before the entity is recorded. Offsets should still be correct *as long as* the lib never produces an empty span; the open question is what happens at token boundaries that mix inline + block levels.

**Mitigation:** unit-test the nested cases in §5.2 explicitly. If specific patterns misbehave, document and decide if acceptable.

### 9.3 Loss of 4096-char auto-split when bypassing `postTelegramMessage`

The proposed handler uses `ctx.telegram.request("sendMessage", { chat_id, text, entities })`. `request` calls `callTelegramApi` directly — **bypassing** `postTelegramMessage` and its `splitTelegramMessageText`. For a long reply, Telegram will return HTTP 400 with `message is too long`, and our fallback would re-send as plain text *above* the 4096 limit, causing the *same* error in the fallback. Dead end.

**Options (verify in implementation):**
1. **Pre-split the parsed text ourselves** at 4096 UTF-16 units, distributing any open entities to subsequent chunks (rare, complex).
2. **Loop our own split** — there is no native `sendMessage` continuation; we send each chunk with its own `entities`.
3. **Don't bypass.** Cast `as any` and call the typed `sendTelegramMessage({body: {text, entities, chat_id}})` (which we can't reach from `ctx.telegram` directly). Or re-import `sendTelegramMessage` privately.

The cleanest is **(2)** — write a small chunker in our own `formatOutbound` helper. Should be ~30 LOC. Recommended during implementation.

### 9.4 UTF-16 offsets and astral characters

`marked` returns strings. The lib computes `offset = getUtf16Length(plainText)` where `getUtf16Length(str) => str.length`. In JavaScript, `str.length` counts UTF-16 code units — *which is exactly what Telegram's [api/entities](https://core.telegram.org/api/entities) requires*. **The offset mechanism is correct in principle.**

However: if we do any custom text manipulation between `markdownToTelegramEntities(...)` and `sendMessage(...)` (e.g. trim, collapse whitespace, prepend a footer), and that manipulation introduces a BMP character while leaving entity offsets as-is, we will get **silent offset drift**. The proposed handler does no such manipulation *between* parse and send — it sends `text` and `entities` as returned by the lib. So drift should be zero in the canonical path.

**Watch out for:** §9.3 split logic (sub-chunking must recompute offsets — strings get sliced and the entity lists must be re-offset). The §9.3 implementation should derive sub-entity offsets from the per-chunk substring only.

### 9.5 Voice fork does not emit `message.completed`

Source-confirmed at `agent-core/src/channels/telegram.ts` around line 296: the voice fork calls `send({ inputResponses, message: turnMessage, context: [contextBlock] }, …)` directly. It does not pass through the `message.completed` event lifecycle. If the formatter only listens to `message.completed`, **voice replies go out plain text**.

**Mitigation:** §4.4 option B — a single `formatOutbound` helper called from both `message.completed` and the voice fork. Adds ~3 lines to the voice fork and removes the risk of drift between the two paths.
