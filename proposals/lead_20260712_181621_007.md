# Project Proposal — Flutter App

## Project Understanding

**Phase 4: PowerSync integration (optional, self-hosted sync)**

*Project description:*
Phase 4 of the sync program (PRD #189). Integrate **PowerSync** as an **optional, self-hosted** sync layer, targeting the developer self-hosting on a **tailnet** for personal trial use. Phases 1–3 (export/import, UUIDv7 PKs + timestamps + soft-delete, DB-backed timer) are all merged; the schema (v15) is sync-ready.

## Guiding constraint — sync is ADDITIVE and OPTIONAL

The app MUST keep working exactly as today with no sync present and with the sync toggle off (plain drift, single local file). Sync is an opt-in layer, off by default. This is groundwork the developer self-hosts and trials locally — not yet a shipped end-user feature. The same code path later becomes the paid, managed offering (Phase 5).

## Architecture (from research, verified against live PowerSync docs 2026)

PowerSync **owns its own SQLite file** — content tables become views over its internal `ps_data` store; the schema is declared in a PowerSync `Schema`, not by drift migrations. There is **no in-place** "convert my existing local DB" path. But drift sits on top either way, so the opt-in is a runtime **connection swap**:

- sync **off** → today's `drift_flutter` (native) / WASM (web) connection.
- sync **on** → `PowerSyncDatabase` (own file + `Schema`), wrapped in `SqliteAsyncDriftConnection`, handed to the unchanged `AppDatabase`.

Same table names both ways → **every DAO is untouched**. Enabling sync **seeds the PowerSync DB from the Phase-1 backup snapshot** (`exportBackupBytes` → `restoreBackup`), since there is no in-place conversion.

- Backend: PowerSync ships **no** CRUD/write endpoint — host one (adapt `powersync-ja/self-host-demo` Node backend). It writes uploaded ops to Postgres.
- Auth: **HS256 shared-secret dev token** — sufficient behind the tailnet perimeter; real JWKS deferred to Phase 5.
- **active_timer + app_settings are NOT synced** — device-local by design.
- **Web is deferred** — needs a second WASM/worker set on top of drift's and is SDK-beta. Native (Linux/Android) only for the trial.

## Slices

- **4a — Self-host stack** (infra): docker-compose (Postgres `wal_level=logical` + `journeyapps/powersync-service`), server Postgres schema mirroring the 6 content tables + publication, `service.yaml`, single global-bucket `sync_rules.yaml`. Runs on the tailnet host. Scaffolding drafted in scratchpad `phase4-powersync/`.
- **4b — Backend upload/auth endpoint**: adapt the self-host-demo Node backend to timedart's Postgres + a dev-token endpoint; add as a compose service.
- **4c — App connection factory + connector** (native, flag off by default): connection-swap seam in `database.dart`, PowerSync `Schema` mirroring the 6 tables, `PowerSyncBackendConnector` (`fetchCredentials`/`uploadData`). Sync-off path byte-for-byte unchanged.
- **4d — Enable-sync flow**: Settings toggle ("Enable sync (experimental)") behind a flag + seed-from-snapshot on enable (reuse Phase 1 backup/restore). End-to-end desktop⇄phone round-trip over tailnet.
- **4e — Web support** (deferred / Draft): second WASM/worker set; revisit once native trial is proven.

## Definition of done (trial)

Developer runs the stack on the tailnet host, enables sync on desktop + phone, and a client/project/task/entry created on one device appears on the other; turning the toggle off returns the app to pure-local behaviour with no data loss. No public exposure; no real auth.

## Non-goals (this phase)

Multi-tenant auth, per-account isolation, managed ops, real JWKS, web, syncing the live timer. All Phase 5 / later.

Research summary + rationale in memory (sync-program). Scaffolding: scratchpad `phase4-powersync/` (docker-compose, init SQL, service.yaml, sync_rules.yaml, README, .env.example).
