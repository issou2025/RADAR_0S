# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**Built-in Split View (Dual-Pane) Support for Large Screens and Foldable Devices**

*Project description:*
### Use case

When running a Flutter app on large-screen devices (tablets, foldables, desktops), only one page is displayed at a time, leaving most of the screen unused. This is a significant gap compared to native platforms:

Android provides SlidingPaneLayout for master-detail two-pane UIs
iOS provides UISplitViewController
Web has CSS multi-column / grid layouts
Common apps like email clients, settings, file managers, and note-taking apps universally use a master-detail split layout on large screens. Flutter developers currently have no built-in way to achieve this that transparently works with existing Navigator/Router APIs.
To achieve a split view today, developers must write a significant amount of application code: manually detecting screen size with MediaQuery, conditionally composing Row/Column layouts, duplicating navigation logic for each screen, and managing state synchronization between panes. This adaptation must be repeated for every page in the app, and it breaks standard Navigator/Router semantics (back button, route transitions, deep linking). The result is high development cost, fragile code, and inconsistent behavior across platforms.

### Proposal

### Existing Alternatives and Their Limitations

| Solution | Type | App Adaptation Effort | Supports Router Routing? |
|----------|------|----------------------|--------------------------|
| `adaptive_scaffold` | Official package | High — requires redesigning layout tree and slot structure | N/A — does not involve navigation stack |
| `flutter_split_view` (3rd-party) | Package | Medium — requires replacing routing API | No — does not support RouterDelegate/RouterConfig/go_router |
| `flow_nav` (3rd-party) | Package | Medium — requires replacing routing API and custom route function mapping | Partial — large-screen implementation uses page replacement, no back navigation support |
| Our approach (dual overlay) | Framework-level change | Low — just enable the toggle and specify split mode; app code essentially unchanged | Yes — supports both Navigator and Router routing |

### Expected Effect

An app that currently looks like this:

```
┌─────────────────────────┐
│                         │
│      Single Page        │
│                         │
└─────────────────────────┘
```

Should automatically become this on wide screens, **without modifying the app's navigation code**:

```
┌───────────┬─────────────┐
│           │             │
│  Home     │  Detail     │
│           │             │
│           │             │
└───────────┴─────────────┘
```

## Proof of Concept

We have completed a working proof of concept using the **dual-overlay approach**. The implementation supports all common Flutter routing patterns:

### Supported Routing Patterns

| # | Pattern | Entry Point |
|---|---------|-------------|
| 1 | `initialRoute` + `onGenerateRoute` | `MaterialApp` |
| 2 | `initialRoute` + `routes` | `MaterialApp` |
| 3 | `home` + `onGenerateRoute` | `MaterialApp` |
| 4 | `home` + `routes` | `MaterialApp` |
| 5 | `CupertinoApp` | `CupertinoApp` |
| 6 | `RouterDelegate` + `pages` | `MaterialApp.router` |
| 7 | `RouterConfig` | `MaterialApp.router` |
| 8 | `go_router` | `MaterialApp.router` |

### How It Works (Overview)

1. **Dual overlay**: Replaces the `Navigator`'s single `Overlay` with two side-by-side `Overlay`s.
2. **Route allocation**: Assigns route overlay entries to the appropriate overlay, enabling two pages to be visible simultaneously.
3. **Navigation interception**: Intercepts push and pop operations to maintain split-view invariants (each overlay displays the expected page).
4. **Popup handling**: `PopupRoute`s are assigned to the correct overlay, with a mirror barrier placed on the opposite side to block interaction.

### Configuration

Our local implementation currently reads configuration items such as split mode from a specific JSON configuration file. This is merely an implementation choice for the proof-of-concept stage. The configuration approach (file format, programmatic API, or framework integration) can be entirely adjusted according to upstream recommendations.

What remains unchanged: no need to modify the app's `Navigator.push()`, `Navigator.pop()`, `context.go()`, or any other navigation code.

## Comparison with Existing Solutions

- **vs. `adaptive_scaffold`**: Zero intrusion, no need to redesign the layout tree. The two have different positioning: `adaptive_scaffold` is a layout slot system, while our approach is split view at the navigation stack level.
- **vs. `flutter_split_view`**: No need to replace routing API, supports declarative routing; but requires modifying Navigator internals.
- **vs. `flow_nav`**: The right pane maintains a real navigation stack (supports back navigation), no need to configure callbacks for each routing pattern; but requires modifying Navigator internals.

## Implementation Approach

The proof of concept adds `SplitViewNavigatorPolicy` in `navigator.dart`, responsible for:

- **Overlay entries reallocation**: In `NavigatorState.build`, route overlay entries are distributed to left and right overlays, enabling two pages to be visible simultaneously.
- **Navigation operation interception**: Intercepts key push/pop operations to maintain split-view invariants, avoiding routing anomalies in multi-page scenarios.
- **Popup and barrier handling**: Assigns `PopupRoute` to the correct overlay and places a mirror barrier on the opposite side to block interaction.
- **Dynamic split view visibility**: Controls the visibility of left and right overlays to dynamically adjust whether the current page displays split view on screen orientation or size changes (landscape/portrait switch, fold/unfold), enabling seamless transitions.
- **Forced split/no-split**: Supports forcing specific pages into split view or fullscreen mode (e.g., video player forced fullscreen).

We are also open to other approaches preferred by the official team.
