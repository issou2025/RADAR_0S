# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**Fix CI pipeline: restore green builds with critical test suite**

*Project description:*

## Problem Statement

The CI pipeline fails on every push to `main` and `develop`. The immediate cause is a deprecated Supabase SDK API (`anonKey` → `publishableKey`) that `flutter analyze` treats as an error. Beyond that, the test suite is in a degraded state: 190 of 1050 tests fail due to drifted mocks, missing integration infrastructure, and outdated assertions. The `test/` directory is excluded from `flutter analyze` to hide ~500 analyzer errors. The three build jobs (Android, iOS, Web) are blocked because they depend on the `test` job succeeding. The `build_runner` step takes ~47 seconds per job with no caching, making the full pipeline take ~3 minutes even when nothing has changed.

## Solution

Restore a meaningful green CI pipeline by fixing the immediate analyzer error, un-excluding `test/` from analysis for a curated set of critical test files, and adding build caching. Integration tests remain out of scope until proper infrastructure is available.

## Implementation Decisions

- **Supabase API migration**: Replace `anonKey` with `publishableKey` in `Supabase.initialize()` in `lib/main.dart`. The environment variable name (`SUPABASE_ANON_KEY`) remains unchanged — only the Dart SDK parameter name changes.
- **CI test include list**: The `test` job in `.github/workflows/ci.yml` will run an explicit set of test files rather than the full suite: `test/widget_test.dart` (existing smoke test) plus the 4 critical test files. This is an explicit include, not an exclude — new passing tests are not automatically included.
- **Critical test files**: `auth_service_test.dart`, `profile_repository_test.dart`, `hike_details_view_model_test.dart`, `checkout_view_model_test.dart`. These cover authentication, user data, the core hike booking feature, and payment flow logic.
- **Analyzer scope**: Remove `test/**` from the `analyzer.exclude` list in `analysis_options.yaml`. Only the 4 critical test files and their dependencies will be fixed to pass analysis — the remaining failing test files will be re-excluded via a targeted exclude pattern to keep CI green.
- **Mock infrastructure**: The existing `mockito`-based mock setup in `test/mocks/mock_repositories.dart` and per-file `@GenerateMocks` annotations will be used. Drifted mocks in the 4 critical files will be updated to match current production interfaces.
- **Build caching**: CI will cache `~/.pub-cache` (Dart package cache) and `.dart_tool/build` (build_runner artifacts) using `actions/cache`. Cache keys will be derived from `pubspec.lock` to invalidate on dependency changes.
- **Integration tests**: All files under `test/integration/` are excluded from the CI test run. These require Supabase database access, payment provider credentials, and potentially device emulators — infrastructure that will be addressed in a separate effort.
- **build_runner deprecation**: The `--delete-conflicting-outputs` flag is deprecated in current `build_runner`. Replace with the default behavior (it now deletes conflicting outputs by default) or the replacement flag if one exists.

## Testing Decisions

- **What makes a good test here**: Tests should verify external behavior — e.g., "given a valid email and password, `signInWithEmailPassword` returns a successful `AuthResponse`" — not internal implementation details like "the auth client was called with these exact arguments." However, the existing test suite uses mockito verification extensively; the fix phase should preserve the existing test style while ensuring tests pass.
- **Modules tested**: `AuthService` (auth flow), `ProfileRepository` (user CRUD), `HikeDetailsViewModel` (hike detail state), `CheckoutViewModel` (payment orchestration), plus all domain model tests that currently pass.
- **Prior art**: The domain model tests in `test/domain/models/` are the gold standard — pure Dart, no mocks, test constructor defaults, `copyWith`, JSON round-trips, equality, and edge cases. The `widget_test.dart` smoke test follows the same pattern. The `checkout_view_model_test.dart` uses TDD-style naming ("Red Phase", "Green Phase") and mockito mocks.

## Out of Scope

- Integration tests (`test/integration/`) — require database and payment infrastructure setup
- UI widget tests for pages and components — widget test infrastructure needs pump setup work
- Provider tests (dashboard, route management) — depend on complex state provider mocks
- Remaining 190 failing tests beyond the 4 critical files — will be addressed incrementally in follow-up work
- `build_runner` performance optimization beyond caching — switching to watch mode or incremental strategies is a separate concern
- Security scan false-positive tuning in Trivy
- Node 20 deprecation warnings in GitHub Actions runners

## Further Notes

- The current test suite runs in ~50 seconds locally. With caching, CI should drop from ~3 minutes to ~1-2 minutes.
- The `test/` exclusion in `analysis_options.yaml` references ticket "WHI-11" — this work partially addresses that backlog item.
- After this PRD is implemented, the path to adding more tests is: fix the test file, add it to the CI include list, verify green, merge.
