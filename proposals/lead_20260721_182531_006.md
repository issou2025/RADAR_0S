# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**[BUG][auto] arch-config-ancestor-leakage-unbounded-ascent: architecture config search crosses repo boundaries with no warning**

*Project description:*
## Description

`FindConfigFileFromPath` in `internal/config/toml_loader.go` walks upward from the analyzed target path looking for `.pyscn.toml` (or `pyproject.toml` with `[tool.pyscn]`), with **no boundary check at the target's own repository root**. It only stops at the filesystem root (`parent == current`). If the analyzed directory happens to be nested inside a parent directory tree that contains an unrelated `.pyscn.toml` — e.g. a checked-out target repo living under `<pyscn-checkout>/.pyscn/audit/repos/<slug>/`, or any monorepo/nested-checkout layout — pyscn silently applies that ancestor's config to the target, with no warning that the resolved config lives outside the analyzed tree.

## Repro

This was found auditing `geldata/gel-python` (commit `8a956d6`) cloned to `.pyscn/audit/repos/geldata-gel-python` inside this pyscn checkout, which has its own dogfooding `.pyscn.toml` with:
```toml
[architecture]
enabled = true
validate_layers = true
strict_mode = true

[[architecture.layers]]
name = "presentation"
packages = ["router", "routers", ..., "controller", ...]
# ... application / domain / infrastructure layers, all generic web-app package names
```

None of gel-python's packages (`gel`, `edgedb`, `gel.cli`, `gel._internal.*`) match any of these generic layer package-name patterns, so every module falls into `"unknown"` layer. With the leaked `strict_mode = true`, every cross-module edge becomes a `strict_mode` violation — including ordinary same-package imports like `gel.__main__ -> gel.cli` and the project's own auto-generated backwards-compat shim (`edgedb/*.py`, which literally re-exports `gel/*.py`).

Result: 334 of 411 (81%) reported architecture violations, and an Architecture score of 28/100, that have nothing to do with gel-python's actual structure — they are an artifact of the pyscn repo's own unrelated config leaking across a repository boundary.

Verified independently:
- Built the current binary and ran it against two synthetic zero-config repos (no ancestor `.pyscn.toml` anywhere in the path) — both scored Architecture 100/100 with zero `strict_mode` violations, confirming `autoDetectArchitecture` correctly hardcodes `StrictMode: false` when no config is found.
- Read `internal/config/toml_loader.go:428-465` (`FindConfigFileFromPath`): the loop `for { ... parent := filepath.Dir(current); if parent == current { break }; current = parent }` has no check against the target path's own VCS root or a `.git` boundary — it will ascend arbitrarily far.
- Confirmed `edgedb/__init__.py` and `edgedb/abstract.py` in gel-python are auto-generated re-export shims (`# Auto-generated shim`), so the "violations" attributed to them are especially misleading.

## Expected

Config discovery should stop ascending at the analyzed target's own repository boundary (e.g. its nearest `.git` directory, or the originally-specified target path itself), and/or emit a visible warning when the config file resolved is outside the analyzed directory tree, so users aren't silently scored against an unrelated project's architectural conventions.

## Actual

Config search walks all the way up the filesystem tree with no boundary check, silently picking up and applying an ancestor project's `.pyscn.toml` — including `strict_mode` and layer definitions that have no relationship to the analyzed code — whenever the target happens to be nested under such a directory (monorepos, nested checkouts, or tooling that clones targets under its own working tree, as this project's own audit-automation skills currently do).

## Priority: P1

Substantially misleading output a user would act on: an entire top-level score dimension (Architecture, 28/100) and the majority of reported violations (81%) do not reflect any real property of the analyzed code, with no indication to the user that a foreign config file was used. This can silently and drastically understate code quality for any nested-checkout or monorepo-subdirectory analysis.
