# Project Proposal — Bim Modeling

## Project Understanding
You need assistance with: 
**Benchmark: speed and cost of IFC export — desktop Revit vs this app vs Model Derivative scheduler**

*Project description:*
## Goal

Measure and communicate the **speed** and **cost** of exporting IFC from Revit across three routes, so we can make an informed recommendation (and defend the cloud approach, or not, with numbers):

| # | Route | Engine |
|---|-------|--------|
| A | Revit on desktop (manual export) | Local Revit + IFC addin |
| B | **This app** | Design Automation for Revit + [RevitIfcExporter appbundle](https://github.com/ADN-DevTech/aps-revit-ifc-exporter-appbundle) |
| C | [aps-revit.ifc.scheduler](https://github.com/libnypacheco/aps-revit.ifc.scheduler) (the original this repo was forked from) | Model Derivative API |

## Phase 1 — Define tests and metrics

### Test matrix (proposal, to be refined)

- **Models:** at least one small (`rac_basic_sample_project.rvt`), one medium (`rac_advanced_sample_project.rvt`), and one real production model. Note model size (MB) and element count for each.
- **Scenarios per model:**
  - Full model export
  - View-filtered export (3D view UniqueId + only-visible-elements — route C cannot do this; record it as a capability gap, see the README section *Exporting from a specific 3D view*)
  - At least two setups, e.g. IFC2x3 CV2.0 and IFC4x3 Reference View, with base quantities on
- **Repetitions:** ≥ 3 runs per cell, ideally spread across times of day — cloud queue time varies and a single run is not representative.

### Metrics

**Speed**
- End-to-end wall clock: user action → IFC available next to the RVT in ACC.
- Breakdown for route B from the Design Automation workitem `stats` (the workitem status JSON carries queue/download/instructions/upload timestamps) — this separates *our* overhead and queue time from pure Revit processing time.
- Breakdown for route C from the Model Derivative manifest timestamps.
- Route A: stopwatch (or journal-file timestamps) for the export itself, recorded separately from the human overhead (open model, dialog, upload result to ACC).

**Cost**
- Route B: Design Automation bills per processing hour (Flex/cloud credits); route C bills per conversion. Confirm current rates on the [APS pricing page](https://aps.autodesk.com/pricing) and capture actual token consumption per test batch from the APS usage dashboard — don't rely on published rates alone.
- Route A: person-minutes per export (from the speed measurements) + Revit license amortization; state assumptions explicitly.
- Express results as **cost per export** and **cost per 100 exports/month** so scheduled bulk conversion (this app's main use case) is visible.

**Fairness / output parity** (needed for the comparison to mean anything)
- Same setup JSON across routes wherever possible.
- Record: output IFC size, entity counts (quick pass with IfcOpenShell or the buildingSMART validation service), success rate and any retries/timeouts.

## Phase 3 — Communicate results

Open question — options considered:

1. **Markdown report in this repo** (`docs/benchmarks.md`): cheapest, versioned, but oddly placed since the benchmark spans two other routes.
2. **Separate small repo** (e.g. `ifc-export-benchmarks`): harness scripts + raw CSVs + a README.md report. Neutral ground for a three-way comparison, keeps test data out of the app repos.
3. **Static webpage**: GitHub Pages on top of option 2 with a simple chart page generated from the CSVs — worth it only if the results need to circulate beyond the team.

**Recommendation:** start with option 2 (markdown-first, raw data committed), and promote to GitHub Pages later if stakeholders need charts. Decide before Phase 2 so the harness writes results to their final home.
