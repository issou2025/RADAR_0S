## Research Findings

### Summary
Both source repos (`adamenagy/aps-design-automation-nodejs` and `adamenagy/aps-design-automation-dotnet`) underwent large restructuring on their `standardize` branches. Current tutorial docs in `docs/03-tutorials/04-design-automation/` still reflect the older layout. Three dimensions of drift:

1. **Realtime → polling** — both repos removed their realtime layer (Node.js dropped `socket.io`, .NET dropped `SignalR`). The new UI polls `GET /api/workitems/{id}` every 2s via `setInterval`.
2. **Folder + file layout changes** — both repos moved static assets from `public/` (or `designAutomationSample/wwwroot/`) to root `wwwroot/`. Server code split into `services/` (Node.js) or `Models/` (.NET). Controllers renamed. Router mount paths changed (`/api/aps/designautomation/*` → `/api/setup/*` + `/api/workitems/*`).
3. **SDK/package upgrades** — Node.js migrated from `forge-apis` to `@aps_sdk/*` packages. .NET migrated from `Autodesk.Forge` to `Autodesk.Authentication` + `Autodesk.Oss` and bumped target from `net6.0` to `net8.0`.

### Key Discoveries
- Node.js `standardize` no longer depends on `socket.io`, `express-formidable`, `cookie-session`, `body-parser`, `form-data`. Client uses `fetch` polling.
- .NET `standardize` no longer references `Microsoft.AspNetCore.SignalR.*`, and the CDN import `microsoft-signalr/3.1.7/signalr.min.js` in `wwwroot/index.html` is gone.
- .NET solution renamed: project `designAutomationSample` → `aps-design-automation-dotnet`; solution filename `designautomation.sln` → `aps-design-automation-dotnet.sln`; project targets `net8.0`.
- Bundle folder path in .NET repo changed: `wwwroot/bundles/*.zip` → `Bundles/*.zip` (capital `B`, at repo root).
- All plugin source folders (`UpdateDWGParam/`, `UpdateIPTParam/`, `UpdateMAXParam/`, `updateRVTParam/`) removed from .NET repo; only prebuilt `.zip` bundles remain under `Bundles/`. Docs still describe building these plugin projects.
- Node.js `package.json` intro in `setup/project.mdx` is inconsistent with `setup/config.mdx` (only lists three packages while `config.mdx` lists ten including `socket.io`) — regardless, both are out of date.
- `docs/03-tutorials/04-design-automation/index.mdx:25-26` links to `autodesk-platform-services/aps-design-automation-nodejs` / `-dotnet` (org repos), while the issue references the `adamenagy/*` forks' standardize branches. Ownership/upstream question — flagged as open below.
- Duplicate NuGet line in `dotnet-vscode/setup/project.mdx` (`Microsoft.AspNetCore.Mvc.NewtonsoftJson --version 6.0.11` appears twice).

### Detailed Findings

#### Node.js source — `standardize` branch layout
Files exposed on standardize:
```
config.js, server.js, package.json
routes/{setup.js, workitems.js}
services/{aps.auth.js, aps.da.js, aps.oss.js}
wwwroot/{index.html, main.js, main.css}
bundles/{UpdateDWGParam.zip, UpdateIPTParam.zip, UpdateMAXParam.zip, UpdateRVTParam.zip}
sample files/{...dwg, .ipt, .max, .rvt, .f3d}
```

`package.json` deps (new): `@aps_sdk/authentication@1.0.0`, `@aps_sdk/autodesk-sdkmanager@1.0.0`, `@aps_sdk/oss@1.1.1`, `autodesk.forge.designautomation@^3.0.5`, `dotenv`, `express`, `multer`.

`server.js` mounts `routes/setup.js` and `routes/workitems.js`; no `app.io`; serves `wwwroot` statically. Endpoints:
- `GET /api/setup/engines`
- `GET /api/setup/appbundles`
- `GET /api/setup/activities`
- `POST /api/setup` (body `{zipFileName, engine}`)
- `DELETE /api/setup/account`
- `POST /api/workitems` (multipart: `inputFile` + JSON `data`)
- `GET /api/workitems/:id`
- `GET /api/workitems/files/:name/url`

`services/aps.auth.js` uses `@aps_sdk/authentication` `AuthenticationClient.getTwoLeggedToken` (not `forge-apis`).
`services/aps.oss.js` uses `@aps_sdk/oss` `OssClient` (`ensureBucketExists`, `uploadObject`, `createSignedResource` with `useCdn: true`).
`services/aps.da.js` uses `autodesk.forge.designautomation` (`AutodeskForgeDesignAutomationClient`) with a fetch/refresh token hook.

`wwwroot/main.js` uses HTTP polling: `setInterval(check, 2000)` calling `GET /api/workitems/:id` until `status === "success"` or `.includes("failed")`.

#### .NET source — `standardize` branch layout
```
Program.cs, Startup.cs, appsettings.json, appsettings.Development.json
Properties/launchSettings.json
Controllers/{SetupController.cs, WorkitemsController.cs}
Models/{APS.cs, APS.Auth.cs, APS.Da.cs, APS.Oss.cs}
wwwroot/{index.html, main.js, main.css}
Bundles/{UpdateDWGParam.zip, UpdateIPTParam.zip, UpdateMAXParam.zip, UpdateRVTParam.zip}
aps-design-automation-dotnet.{csproj,sln}
```

`.csproj` targets `net8.0`. NuGet packages: `Autodesk.Authentication 2.0.0`, `Autodesk.Forge.DesignAutomation 6.0.2`, `Autodesk.Oss 2.1.0`, `Microsoft.AspNetCore.Mvc.NewtonsoftJson 8.0.15`.

`Startup.cs` reads env `APS_CLIENT_ID`, `APS_CLIENT_SECRET`, `APS_NICKNAME`, `APS_BUCKET`; registers `APS` singleton; pipeline: DeveloperExceptionPage → DefaultFiles → StaticFiles → Routing → MapControllers. **No `AddSignalR()`**, no `MapHub`.

`Program.cs` uses classic `Host.CreateDefaultBuilder(...).ConfigureWebHostDefaults(...).UseStartup<Startup>()` (still `Startup.cs`-based, not minimal hosting).

Endpoints mirror the Node.js repo (same paths).

`Models/APS.Auth.cs` uses `Autodesk.Authentication.AuthenticationClient.GetTwoLeggedTokenAsync` (scopes: `CodeAll, BucketCreate, BucketRead, DataRead, DataWrite, DataCreate`).
`Models/APS.Oss.cs` uses `Autodesk.Oss.OssClient` (`UploadObjectAsync`, `CreateSignedResourceAsync` with `Access.ReadWrite`).
`Models/APS.Da.cs` — `EngineAttributes` for 3dsMax/AutoCAD/Inventor/Revit; `Setup`, `CreateAppBundle`, `CreateActivity`, `GetAvailableEngines`, `GetDefinedActivities`, `StartWorkitem`, `GetWorkitem`, `ClearAccount`; uploads bundle ZIP via `HttpClient` multipart to `UploadParameters.EndpointURL`.

`wwwroot/index.html` no longer loads `microsoft-signalr` — loads `/main.js` as `type=module`.
`wwwroot/main.js` uses `fetch` + `setInterval` 2s polling (same behavior as Node.js version).
