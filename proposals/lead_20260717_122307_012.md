# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**[plan] Permissions as tool contracts — compositor-managed, fresh-box self-contained (epic)**

*Project description:*

## Summary

Every permission a bot holds should be emitted by the composer from a contract declared on an equipped source (skill, expertise, integration, guardrail) — so a bot ships **self-contained on a fresh box** with an empty `~/.claude`, and self-containment is enforced by CI. The compositor already does this for expertise, MCP fragments, integrations (`tool_grants`, #627 P5), and channels; this initiative closes the two remaining gap sources (skills, guardrails), widens the grant grammar beyond `mcp__` to cover `Bash(...)` and bare tools, retires the 223 hand-accumulated global allows that currently mask whether composition is complete, and adds a fresh-box CI gate that proves a bot boots and exercises its tools with only composed perms.

## Evidence

- **MCP grant union (Layer 3)** — legacy `_resolve_mcp_permissions` reads a fragment's `_permissions_contract.tools` (`composer.py:210-246`); new `_resolve_integration_grants` reads integration-file `tool_grants` frontmatter (`composer.py:249-292`); unioned and hard-gated by `_assert_grant_superset` (`composer.py:1209-1225`, called at `:1286`), which fails `generate` unless the new set ⊇ the legacy set. 10 integration files declare `tool_grants` today (github, notion, docker, gws, gmail, google-calendar, slack, spotify, granola, homeassistant).
- **Channel (Layer 4)** — `_resolve_channel_permissions` (`composer.py:1112-1120`): a Telegram handle grants the 4 `_TELEGRAM_PLUGIN_TOOLS` (`composer.py:1104-1109`). Only Telegram is wired — hardcoded, no per-channel loop.

## Architecture

Target state — one rule: **a bot's every tool permission is emitted by the composer from a contract on an equipped source; the shared `~/.claude` holds no tool allows.**

```
equipped sources ── declare ──▶ contracts ── composer unions ──▶ per-bot .claude/settings.local.json
  skills        }  additive grants (tool_grants: mcp__* | Bash(..) | bare)   allow: [complete, self-contained]
  integrations  }                                                            deny:  [expertise/guardrail + sibling isolation]
  expertise     }  deny-capable (permissions: {allow, deny, bash_allow})     enabledMcpjsonServers: [trusted set]
  guardrails    }
  (baseline)    ─  composer-emitted floor (F3)

shared ~/.claude/settings.json ── tool allows: EMPTY ── only ratified host-runtime behavior (F5)

CI gate: compose bot → temp $HOME with empty ~/.claude → static superset assert + real `claude -p` tool-exercise → no prompt, no missing-perm  (F4)
```

Design decisions: additive sources (skills, integrations) share one widened `tool_grants` grammar; deny-capable sources (expertise, guardrails) keep the `permissions:{allow, deny, bash_allow}` shape (they need deny-wins). One MCP-grant path (legacy cut, per P8). Self-containment is a test, not a claim.

## Implementation Plan

### Dependencies
- None external. Builds directly on the #627 P5 `tool_grants` machinery (`composer.py:249-292`) and the `_assert_grant_superset` gate (`composer.py:1209-1225`), both already on `main`.

### Blocks
- Unblocks the deferred **P8** legacy-MCP-contract cut (folded into P2 here).
- Unblocks any future move to per-bot `CLAUDE_CONFIG_DIR` credential isolation (F1) — but does not require it.

### Steps

Each step is one phase = one implement-time issue (broken out post-`/ironclad`). File anchors are the insertion sites, not literal diffs.

**P1 — Unified grant schema + widened grammar + validator (M).**
- Extend the loader to parse a permissions contract from the two gap sources: skills (`library/skills/<name>/SKILL.md` frontmatter) and guardrails (`library/guardrails/<name>.md` frontmatter), reusing `parse_frontmatter` (`loader.py:55`) and the `ExpertisePermissions` precedent (`loader.py:40-44`).
- Widen `_TOOL_GRANT_RE` (`validator.py:65`) per the F2/F3 ratification so a grant list accepts `mcp__*`, `Bash(<cmd> *)`, and bare tool names, validated per-kind.
- Add validator warnings: malformed grant; and a best-effort heuristic that a source whose body invokes a CLI (`gh`, `railway`, `vercel`, `tmux`, …) declares a matching `Bash(...)` grant. Extends the existing migration-gap warning (`validator.py:157-166`).

**P2 — Generalized union resolver + execute the deferred P8 cut (L).**
- Add `_resolve_skill_grants` and `_resolve_guardrail_permissions` beside the existing resolvers; wire them into `compose_settings_local`'s layers (`composer.py:1256-1321`). Result: every equipped source's declared grants union into the complete per-bot allow/deny.
- Per no-shims + consolidate-don't-fork: **cut** legacy `_resolve_mcp_permissions` + `_permissions_contract` JSON reading (the deferred P8) now that integration `tool_grants` proves superset fleet-wide via `_assert_grant_superset`. One MCP-grant path, not two parallel ones.

**P3 — Author contracts for every source that needs one; migrate the 223 (XL).**
- Walk the categorized 223 (167 mcp / 28 skill / 20 bash / 8 bare) → owning source → write its contract:
  - mcp (fragment-backed): already covered by integration `tool_grants`; verify each equipped bot resolves it.
  - CLI-backed skills (`prs`, `deploy-status`, `dispatch` → `gh`/`railway`/`vercel`/`tmux`): add `tool_grants` to their SKILL.md.
  - guardrail denies (`snowflake-read-only` → deny non-SELECT patterns): add `permissions.deny`.
  - baseline tools (8 bare + generic shell utils): route to the F3-ratified owner.
  - plugin-owned skills (`claudna:audit`, `claudna:session`, …): resolve their perm home (fleet `tools.allow` vs a plugin-integration contract); drop the 24 dead pre-consolidation `Skill` allows.
  - collapse the parallel Google paths (connectors vs `gws` instances) to one.
- Recompose; diff each bot's composed allow union **before vs after** to prove coverage of what the bot actually uses (not of the sloppy 223). Drop redundant sub-globs.

**P4 — Fresh-box CI gate: static superset + real-boot smoke (L).**
- Static gate: assert every bot's composed allow ⊇ the union of its equipped sources' declared contracts, and that no perm depends on the global file. Extends `_assert_grant_superset` philosophy and the `tests/test_grant_superset.py::TestGrantSupersetSweep` per-bot sweep.
- Real-boot smoke (per F4): compose a **scoped** bot (e.g. navi — no `allow_all`) into a temp `$HOME` with empty `~/.claude`, run `claude -p "<prompt exercising its tools>" --output-format stream-json`, assert no permission prompt and no missing-perm failure. This resolves the replace-vs-union open question (R1) and is the enforced form of "ships self-contained on a fresh box." Build on `lib/validate-bot-change.sh`.

**P5 — Retire the global list (S, high-risk).**
- Empty `permissions.allow` in the shared `~/.claude/settings.json` (or reduce to the F5-ratified host-behavior-minimum). Route behavior flags to their composed owners where possible (`permission_mode` is already composed).
- Gate the merge on P4 green (real-boot smoke passes with empty global allow). Canary on one bot (throwaway supervised-bot method) before fleet-wide. Irreversible moment-of-truth.

## What NOT To Do
- **Do not build a central `library/permissions/` registry** — it's Telegram access policy; follow the inline per-source frontmatter precedent.
- **Do not keep the legacy `_resolve_mcp_permissions` path alongside the new resolver** — no shims; cut it (P8) once the superset gate is green.
- **Do not prove self-containment with an `allow_all` bot** (e.g. software-engineering expertise) — it passes trivially and masks incomplete skill/integration contracts. The gate must exercise a scoped bot.
- **Do not bundle the `CLAUDE_CONFIG_DIR` per-bot migration (F1) into the allow-retire** — keep it separable.
- **Do not retire the host behavior flags in the same step as the tool allows** — `skipAutoPermissionPrompt` etc. gate the #638 permission-prompt-hang class; treat as a separate ratified sub-decision (F5).
- **Do not comment perms with the dated event** ("added for Chris's 2026-07-17 ask") — document the durable convention near the schema (convention-over-event).

## Decision Forks

### Fork F1: Config-dir scope — shared `~/.claude` vs per-bot `CLAUDE_CONFIG_DIR`
- **Context:** Self-containment can be proven with the composed per-bot `settings.local.json` alone, or pushed further to full per-bot user-tier isolation. The mechanism already exists (`composer.py:336-373`).
- **Options:** **(a)** Keep the shared `~/.claude` — project-tier composition (settings.local) is the unit of self-containment. **(b)** Scope `CLAUDE_CONFIG_DIR` per bot — full user-tier isolation, empty per-bot global by construction.
- **Lean:** **(a).** The composed settings.local already scopes perms per bot; per-account isolation adds ~10× credential/login management (a separate Claude auth per bot) for an identity concern orthogonal to self-containment, and the CI gate uses a temp `$HOME` regardless. **Ratifier:** Chris. **Status:** open.
- **Evidence:** `composer.py:336-373`; `config.py:449`; eng fleet all-default account.

### Fork F2: Contract schema — one unified shape vs semantic split (additive vs deny-capable)
- **Context:** Additive sources (skills, integrations) only add allows; deny-capable sources (expertise, guardrails) need deny-wins. Today they differ: `tool_grants:` (flat list) vs `permissions:{allow,deny,bash_allow}`.
- **Options:** **(a)** One unified schema across all four source types that supports allow + deny. **(b)** Two schemas by semantic — additive `tool_grants:` (skills, integrations) and deny-capable `permissions:` (expertise, guardrails) — each already used by its category.
- **Lean:** **(b).** Matches the live precedent, keeps deny-capability where it's needed without forcing additive sources to carry deny machinery, and consolidates *within* each semantic (one grammar per kind). Includes widening the `tool_grants` grammar to `mcp__* | Bash(..) | bare` (F3). **Ratifier:** ari. **Status:** open.
- **Evidence:** `loader.py:40-44`; `composer.py:249-292`; `library/permissions/README.md` (no central registry).

### Fork F3: Grant grammar — one heterogeneous list vs typed sub-lists
- **Context:** To own the 20 `Bash(...)` + 8 bare allows, the grant list must express more than `mcp__`. `_TOOL_GRANT_RE` is `mcp__`-only (`validator.py:65`).
- **Options:** **(a)** One heterogeneous list; kind inferred by pattern (`mcp__*` / `Bash(...)` / bare). **(b)** Separate typed keys (`mcp_grants`, `bash_grants`, `tool_names`).
- **Lean:** **(a).** The final composed allow list is already a flat heterogeneous union; mirroring that shape in the contract is the lowest-overhead, and one regex-per-kind validator keeps it safe. **Ratifier:** ari. **Status:** open.
- **Evidence:** `validator.py:65`; `composer.py:1274-1311` (flat union today).

### Fork F4: Fresh-box CI gate — static superset vs real-boot vs both
- **Context:** Chris's ask is to make fresh-box self-containment an enforced test. There's a fast static form and a high-fidelity behavioral form.
- **Options:** **(a)** Static superset only (composed allow ⊇ declared contracts; global contributes nothing). **(b)** Real-boot only (compose into empty `$HOME`, `claude -p` exercises tools). **(c)** Both.
- **Lean:** **(c).** Static gate is the fast per-PR check; real-boot is the empirical gate our MANDATORY validation rule requires and the only thing that resolves replace-vs-union (R1) and proves actual boot. Run real-boot as a gated job (not every PR) but required before P5. **Ratifier:** ari. **Status:** open.
- **Evidence:** `_assert_grant_superset` (`composer.py:1209`); `tests/test_grant_superset.py`; CLAUDE.md empirical-validation gate.

### Fork F5: Retire scope — empty allows only, and what happens to the host behavior flags
- **Context:** The 223 allows are fully coverable by contracts, so `permissions.allow` → empty. But global also carries behavior flags (`skipAutoPermissionPrompt`, `skipDangerousModePermissionPrompt`, `permissions.defaultMode:auto`, `sandbox.*`).
- **Options:** **(a)** Empty the tool allows; leave behavior flags host-global as-is. **(b)** Empty the tool allows AND move each behavior flag to its composed owner where one exists (e.g. `permission_mode`), keeping only irreducible runtime behavior as a *composed* host file — never hand-maintained.
- **Lean:** **(b)** directionally, but stage it: retire the allows first (this initiative), then treat behavior-flag migration as its own ratified step, because those flags gate the #638 prompt-hang class. **Ratifier:** Chris (fleet-wide runtime behavior + irreversible). **Status:** open.
- **Evidence:** global flags (`~/.claude/settings.json`); `permission_mode` already composed (`composer.py`, `TestPermissionMode`); #638 seeding.
