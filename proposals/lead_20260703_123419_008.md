# Project Proposal — Structural Drawings

## Project Understanding
You need assistance with: 
**PRD: Phase 2 — Stripe Webhook + License Email Delivery + Landing Page**

*Project description:*
## Problem Statement

I have a Flutter credential manager (Cred Manager Pro) with a working subscription licensing system on the client side, but no way for a customer to actually buy a license. Today there is no purchase path: a visitor lands on the repo with no way to pay, no way to receive a signed license key, and no way to unlock the paid tier. I need an end-to-end commerce pipeline — from landing page through Stripe payment to license-key delivery — so I can sell the app direct at \$29.95/year per the locked-in licensing model.

## Solution

A single-product Next.js 15 (App Router, TypeScript) web app deployed on Dokploy (primary) or Vercel (fallback). The app contains:

1. **Landing page** (`/`): hero, value prop, free-vs-paid feature comparison, $29.95/year price, 30-day money-back guarantee, "Buy" button → Stripe Checkout (Stripe-hosted). Footer with Terms + Privacy.
2. **Stripe Checkout integration**: server-side Checkout Session creation in a Next.js route handler; Stripe hosts the payment page so the app stays out of PCI scope.
3. **Webhook handler** (`POST /api/webhooks/stripe`): verifies Stripe signature, handles `checkout.session.completed`, signs an Ed25519 license payload using the same wire format the Flutter app already validates (`cmp1.<base64url(payloadJson)>.<base64url(signature)>`), persists to Postgres (idempotency + audit), and emails the buyer their key.
4. **Postgres** with one table — `licenses` — for idempotency (Stripe retries webhooks) and audit / lost-key lookup. No customer accounts, no dashboard, no revocation list — Stripe remains source of truth for "did this user pay."
5. **Self-hosted SMTP** via Nodemailer, abstracted behind a `LicenseEmailService` interface so the implementation can be swapped to Resend later if deliverability becomes a problem.
6. **Success page** (`/success`): "Check your email" + explicit spam-folder reminder (because self-hosted SMTP deliverability is a known risk).

The license signing logic gets ported from `tools/license_gen/bin/license_gen.dart` (~30 lines of Ed25519 signing) to TypeScript using `@noble/ed25519`. The output MUST be byte-identical in format to what the Dart tool produces, validated by cross-signing the same payload and comparing the wire-format strings.

- Next.js 15 App Router, TypeScript strict mode
- shadcn/ui + Tailwind CSS for landing page UI
- Postgres with Drizzle ORM (lighter than Prisma, edge-friendly)

**Schema** (`licenses` table):
- `id` uuid primary key (default gen_random_uuid())
- `stripe_event_id` text not null, **unique index** (idempotency)
- `stripe_session_id` text not null, **unique index** (lookup by session)
- `email` text not null
- `license_key` text not null (the full wire-format string)
- `product` text not null default 'cred-manager-pro'
- `tier` text not null default 'paid'
- `features` jsonb not null default '[]'
- `issued_at` timestamptz not null default now()
- `expires_at` timestamptz nullable (null = perpetual)

**Wire format** (unchanged from Flutter app):
```
cmp1.<base64url(payloadJson)>.<base64url(ed25519_signature)>
```
Payload JSON shape, identical to `tools/license_gen/bin/license_gen.dart`:
```json
{
  "product": "cred-manager-pro",
  "tier": "paid",
  "email": "buyer@example.com",
  "issued": "2026-07-03T...",
  "expires": null,
  "features": ["projects","credcli","aiServices","sync","breachAlerts","mfa"]
}
```

**Signature is over the exact bytes of `JSON.stringify(payload)` in stable key order.** The TS implementation must call `JSON.stringify` with the keys in the exact order above and verify byte-equality against Dart's `jsonEncode` output before shipping. (Cross-validation test #37 covers this.)

**Environment variables** (Zod-validated at boot):
- `DATABASE_URL` — Postgres pooled connection string
- `STRIPE_SECRET_KEY` — `sk_test_...` or `sk_live_...`
- `STRIPE_WEBHOOK_SECRET` — `whsec_...`
- `LICENSE_PRIVATE_SEED` — 64 hex chars (the Ed25519 seed, matches embedded public key)
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`
- `APP_BASE_URL` — `https://credmanagerpro.com` (used for success_url + email links)
- `STRIPE_PRICE_ID` — `price_...` for the $29.95/year product

**Webhook flow** (in `/api/webhooks/stripe/route.ts`):
1. Read raw body + `stripe-signature` header
2. `stripe.webhooks.constructEvent(rawBody, sig, STRIPE_WEBHOOK_SECRET)` — throws on bad signature → return 400
3. If event type ≠ `checkout.session.completed` → log + return 200
4. Extract: `event.id` (idempotency key), `session.id`, `session.customer_details.email`
5. `INSERT INTO licenses` with `stripe_event_id = event.id` — on unique constraint violation, return 200 (already processed)
6. Build payload, sign with `LICENSE_PRIVATE_SEED`, store `license_key` in the row
7. Send email via `LicenseEmailService.sendLicenseEmail(email, licenseKey)` — failures are logged but do NOT fail the webhook (the license is already persisted; support can re-send)
8. Return 200

**Checkout flow** (in `/api/checkout/route.ts`):
1. POST with no body
2. `stripe.checkout.sessions.create` with `mode: 'payment'`, `line_items: [{price: STRIPE_PRICE_ID, quantity: 1}]`, `customer_email: undefined` (Stripe asks), `success_url: ${APP_BASE_URL}/success?session_id={CHECKOUT_SESSION_ID}`, `cancel_url: ${APP_BASE_URL}/`
3. Return `{ url: session.url }` for the client to `window.location` to

**Landing page** (`/`):
- Server-rendered (default App Router) for SEO
- Hero: app name, one-line value prop, primary CTA ("Buy — \$29.95/year") that POSTs to `/api/checkout` and redirects
- Features section: free vs paid comparison table
- Trust signals: 30-day money-back guarantee, "data stays on your device" (links to local-only architecture story), "no auto-renew"
- Screenshots: 1-3 screenshots of the app
- Footer: Privacy, Terms, support email

**Success page** (`/success`):
- "Check your email" headline
- Explicit spam-folder reminder (because self-hosted SMTP)
- "Lost your key? Email support@credmanagerpro.com"

**Idempotency guarantee:**
The unique constraint on `stripe_event_id` is the source of truth. Even under concurrent retries (Stripe's at-least-once delivery), the DB rejects the duplicate INSERT. The handler treats unique-constraint violation as success.

## Testing Decisions

**What makes a good test here:** only test external behavior — HTTP responses, DB rows, email content (after mocking). Never assert on internal call sequence or private methods. The webhook is a state machine; tests should drive it with real-looking inputs and assert outcomes.

**Test seams (prefer existing over new):**
1. **`LicenseSigner` (pure function)** — `signLicense(payload, seedHex) → wireFormatString`. Unit tests cover: valid payload, perpetual vs annual expiry, feature subset, malformed seed. Cross-validated against `tools/license_gen` Dart output by signing the same payload in both and asserting string equality.
2. **Stripe webhook verification** — tested via `stripe.webhooks.constructEvent` using test fixtures generated by `stripe trigger checkout.session.completed` (Stripe CLI). Tests cover: valid signature, tampered body, missing header, stale timestamp.
3. **`LicenseEmailService` interface** — pure interface, mocked in tests. The Nodemailer implementation is exercised in a single integration test that connects to a test SMTP server (MailHog or similar in CI).
4. **DB layer** — Drizzle with a test Postgres (testcontainers-node). Tests cover: INSERT happy path, unique constraint violation on duplicate `stripe_event_id`, lookup by `stripe_session_id`.
5. **Webhook route handler** — Next.js route handler test that drives `POST /api/webhooks/stripe` with a real Stripe test event, mocking `LicenseEmailService` and using a real test DB. Asserts: 200 response, row in DB with correct email + license_key format, mock email called once. Drives the same event twice and asserts second call is a no-op (idempotency).

**Prior art:**
- `frontend/test/license_service_test.dart` — establishes the wire format contract. The TS signer's output must validate against the Flutter app's parser.
- `tools/license_gen/bin/license_gen.dart` — reference signing implementation. The TS signer's output must byte-match this for the same payload + seed.

**Cross-validation test (critical):**
A shared fixture (`payload.json` + `seed.hex`) committed to the repo. Both the Dart tool and the TS service sign it. The test asserts the two wire-format strings are byte-identical. This is the only way to catch format drift (e.g. JSON key reordering, base64url padding differences).

**E2E smoke (manual, documented in README):**
```bash
stripe listen --forward-to localhost:3000/api/webhooks/stripe
stripe trigger checkout.session.completed
# assert: row appears in licenses table, email arrives in MailHog
```

## Further Notes

**Self-hosted email deliverability is a known accepted risk.** The landing page, success page, and support-reply templates will all explicitly instruct buyers to check their spam folder. The `LicenseEmailService` interface exists specifically so swapping to Resend (or SES, Postmark, etc.) is a one-line implementation change if deliverability becomes a recurring support burden.

**Architecture is namespaced by product from day one** even though only one product exists:
- Stripe product ID: `cred_manager_pro_yearly`
- DB `product` column: `'cred-manager-pro'`
- URL paths: `/api/webhooks/stripe` (not product-scoped, since one Stripe account handles all products)
- This means a future second product requires zero schema migration — just a new row value in `product`.

**Drizzle over Prisma** because Drizzle is lighter, has better edge runtime support, and a single-table app doesn't need Prisma's heavier client. If the data model grows significantly (customer accounts, multi-tenant), revisit.

**`@noble/ed25519` over Node `crypto`** because the signing needs to work in any JS runtime. If the deploy target moves from Dokploy (Node) to Vercel Edge or Cloudflare Workers, the code doesn't change.

**Boot-time env validation with Zod** because Next.js will silently serve a broken route if env vars are missing at request time. Better to crash the boot. The `/api/checkout` and `/api/webhooks/stripe` route handlers each call `parseEnv()` at module load — missing vars throw immediately.

**Private seed handling:** The `LICENSE_PRIVATE_SEED` env var contains the same 64-hex-char seed used by `tools/license_gen`. Anyone with this seed can issue paid licenses. It must never be logged, never returned in any response, never written to disk outside the deploy environment. The handler reads it once at boot and caches it in memory.

**Cross-validation is the load-bearing test.** If the TS signer drifts from the Dart tool (e.g. JSON key order, base64 padding, encoding differences), every license issued is broken and customers can't unlock the app. The cross-validation test is the only way to catch this before deploy.
